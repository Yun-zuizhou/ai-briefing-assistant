## 2. 评估结论先行

本轮底层策略评估后的结论是：

1. **产品方向稳定结论**
   - 项目本质上是“简报助手”，不是“多垂类资讯产品”
2. **数据模型稳定结论**
   - 产品主模型应围绕“用户关注 -> 简报 -> 行动/记录/成长”组织，而不是围绕“频道 / 模板 / profile 产品化”组织
3. **代码框架稳定结论**
   - 当前 `Workers + D1` 作为正式在线主线、`app/` 作为离线执行与参考链，这个框架方向是稳定且符合项目目标的
4. **阶段十六实现结论**
   - `ai-daily` 可以继续保留，但必须明确为内部执行模板，不得上升为产品一级概念
5. **本轮硬约束**
   - 后续实现不得再把阶段十六演进成“多个垂类简报产品”，也不得让调试接口反客为主变成产品主页面

一句话总结：

`当前正确的稳定方向，不是做一个“AI 资讯系统”，而是在简报助手内部补齐“关注驱动的信息收集与摘要执行能力”。`

## 3. 产品方向评估

## 3.1 项目真正的产品核心

从产品设计文档反复出现的关键词看，当前产品核心是一致的：

1. 每日简报
2. 信息追踪
3. 待办提醒
4. 个人叙事
5. 对话式交互
6. 关注管理

也就是说，产品不是围绕“资讯类别”组织，而是围绕“用户每天收到的一份个人简报”组织。

因此产品层的正确结构是：

1. 用户表达关注
2. 系统围绕关注收集信息
3. 简报把这些内容整合为：
   - 信息追踪
   - 待办提醒
   - 个人叙事

## 3.2 当前最容易偏航的方向

本轮已经确认，当前最容易偏航的方向有 3 个：

1. 把阶段十六做成“AI 重点资讯产品”
2. 把内部执行模板做成用户可见频道体系
3. 把阶段十六的 `daily-digest` 调试页误当成未来产品首页

这 3 个方向都不符合你项目原始定位。

## 3.3 产品方向稳定判断

当前产品方向可判断为：

### 稳定且应坚持

1. 单一产品：`简报助手`
2. 单一主载体：`每日简报`
3. 单一交互入口：`关注 + 对话 + 简报互动`

### 不应引入

1. 用户侧多频道订阅产品模型
2. 用户侧多垂类产品导航结构
3. 将“模板”直接暴露为用户概念

## 4. 数据模型评估

## 4.1 产品主模型应围绕什么组织

当前最符合项目方向的主模型应该是：

1. `user_interests`
2. `briefings`
3. `todos`
4. `notes`
5. `history_entries`
6. `user_profiles`
7. `reports`

这几类对象共同构成：

`关注 -> 信息 -> 行动 -> 记录 -> 成长`

这是你的产品主轴。

## 4.2 阶段十六新增对象属于哪一层

阶段十六新增或使用的对象：

1. `ingestion_runs`
2. `ai_processing_runs`
3. `summary_generation_tasks`
4. `summary_generation_results`

它们不属于产品主模型，而属于：

`支撑执行层 / 运行事实层`

这层对象的定位必须固定：

1. 它们是生成简报素材和咨询上下文的中间层
2. 它们服务于产品主模型
3. 它们不能替代产品主模型

也就是说：

- `summary_generation_results` 不是最终产品页对象
- 它是“可回流到简报的信息素材对象”

## 4.3 数据模型稳定判断

当前最稳定的数据模型方向应判断为：

### 产品主模型

围绕：

1. `关注`
2. `简报`
3. `待办`
4. `记录`
5. `成长`

### 执行支撑模型

围绕：

1. `模板`
2. `候选`
3. `任务`
4. `结果`
5. `运行日志`

### 硬约束

1. 产品主模型与执行支撑模型必须分层
2. 不允许把执行支撑模型直接抬成产品一级对象
3. 不允许为了阶段十六方便，把“模板”直接写进用户心智模型里

## 5. 代码框架评估

## 5.1 当前代码框架是否符合方向

当前框架：

1. `apps/edge-worker`
   - 正式在线 API
2. `infra/cloudflare/d1`
   - 正式数据迁移与结构
3. `app/`
   - 离线/参考/执行链
4. `packages/contracts`
   - 前后端契约

这个框架当前是合理的。

原因：

1. 正式读写和用户面已收口在 Workers
2. 阶段十六首轮执行链仍需要较灵活的 Python 采集与模型调用能力
3. 目前没有必要为了“理论统一”强行把执行器一次性迁入 Workers

## 5.2 当前代码框架最危险的漂移点

当前代码框架的风险不在“技术选型不对”，而在“职责边界可能漂移”：

### 风险 1

把 `app/` 又重新抬回正式在线主线。

### 风险 2

把 `apps/edge-worker/src/routes/content.ts` 里的调试接口持续扩写，最后变成第二个产品首页。

### 风险 3

把 `summary_generation_results` 当成最终产品的内容事实总表。

### 风险 4

让前端调试页 `/ai-digest-lab` 长期承担产品主功能展示职责。

## 5.3 代码框架稳定判断

当前应固定的框架原则：

1. 在线主线继续在 `apps/edge-worker`
2. 执行链首版继续允许在 `app/` 与 `tools/scripts`
3. 前端调试入口允许存在，但必须明确标记为调试页
4. 阶段十六所有新增接口默认先归入“支撑与读侧验证”，不直接升级为产品一级页面

## 6. 当前应该继续坚持的底层方向

本轮评估后，以下方向可以视为“稳定方向”，后续应继续坚持：

1. **简报驱动**
   - 一切能力最终回到简报，而不是散成多个独立功能产品
2. **关注驱动**
   - 用户表达关注，系统围绕关注组织内容
3. **分层驱动**
   - 产品主模型和执行支撑模型严格分层
4. **Workers 正式主线**
   - 正式用户面继续以 `apps/edge-worker` 为唯一在线后端主线
5. **模板内部化**
   - 内部模板保留，但不升级为用户产品概念

---

## 7. 当前必须禁止的方向漂移

后续实现必须明确禁止以下 7 类漂移：

1. 把 `ai-daily` 写成产品长期一级功能
2. 把“模板列表”暴露给用户做产品导航
3. 把 `daily-digest` 调试页演进成正式首页替代
4. 为了阶段十六方便，绕开 `user_interests` 另起一套用户关注真源
5. 把 `summary_generation_results` 当成统一内容事实层总表
6. 把阶段十六继续扩成“全站资讯系统”或“舆情系统”
7. 在没有明确产品收益证明前，重构当前 `Workers + D1 + Python 执行链` 主框架

---

## 8. 稳定性判断：本轮后续开发边界

为了避免“底层方向反复更正”，本轮之后的开发应只允许在以下边界内推进：

### 允许继续推进

1. 跑通 `ai-daily` 这个内部样例模板
2. 补 provider 配置、重跑失败任务
3. 补“关注 -> 模板”的最小映射规则
4. 让阶段十六结果回接到简报信息追踪区
5. 继续强化执行链可观测性

### 暂停推进

1. 多模板用户产品化
2. 大规模模板配置平台化
3. 阶段十六调试页产品化
4. 新一轮底层架构迁移

## 9. 最终结论

本轮可以给出一个明确的稳定性判断：

1. **产品方向稳定**
   - 是，且已经足够清晰
2. **数据模型方向稳定**
   - 是，应坚持“产品主模型 / 执行支撑模型”分层
3. **代码框架方向稳定**
   - 是，当前 `Workers + D1 + Python 执行链` 符合项目阶段目标
4. **当前是否还需要继续改底层逻辑**
   - 不需要再大改
5. **当前更合理的动作**
   - 在已确定边界内继续收实现，而不是再改方向

一句话收口：

`当前最重要的不是再发明一套新底层，而是用已经纠偏后的稳定方向，把阶段十六跑成真正服务“简报助手”主闭环的能力。`
