## 2. 参考案例 A：Obsidian + Claude / RSS 插件

## 2.1 值得参考的形态

当前能代表“Obsidian + AI 内容整理”工作流的，不是单一产品，而是 3 种组合模式：

1. `RSS 插件 -> 落笔记 -> 手动挑选 -> AI 总结`
2. `AI 助手 -> 直接在 vault 内读写与搜索`
3. `本地知识库 -> AI 边读边整理 -> 人工保留高价值结论`

### 参考 1：`obsidian-feed`

参考链接：

1. [GitHub - fjdu/obsidian-feed](https://github.com/fjdu/obsidian-feed)

当前可借的点：

1. 把 RSS 项一条条落为可操作对象，而不是一次性合成大列表
2. 用户可以对单条内容做：
   - 保存
   - 摘录
   - 手动标记已读
   - 触发 GPT 摘要
3. AI 摘要不是自动全量跑，而是“对值得保留的条目再跑”

当前不建议照搬的点：

1. 过于依赖用户在笔记里手工挑选
2. 插件内部直接把“阅读器 + 摘要器 + 笔记器”揉在一起
3. 适合个人知识管理，不适合你当前项目这种“正式 API + D1 + 可观测执行链”

### 参考 2：`obsidian-claude-code`

参考链接：

1. [GitHub - Roasbeef/obsidian-claude-code](https://github.com/Roasbeef/obsidian-claude-code)

当前可借的点：

1. Claude 不是只当聊天框，而是当“能读写当前知识工作区的 agent”
2. conversation/history 被显式持久化，而不是一次性临时上下文
3. 工具能力和上下文边界被明确暴露出来

当前不建议照搬的点：

1. 直接给模型过强的本地文件写权限
2. 把“整理结果”完全交给交互式 agent，而不保留独立的任务表、结果表、运行表

### 参考 3：Obsidian 官方对社区插件的安全提醒

参考链接：

1. [Community plugins - Obsidian Help](https://obsidian.md/help/community-plugins)
2. [Plugin security - Obsidian Help](https://help.obsidian.md/plugin-security)

当前必须吸收的结论：

1. 社区插件本质上是第三方代码
2. 它们可能访问本地文件、联网、安装附加程序
3. 因此“插件里一把梭”不适合作为你当前项目的正式实现口径

### 对当前项目的借鉴结论

Obsidian 这条线真正值得借的，不是“做插件”，而是下面两个产品原则：

1. `每条内容先成为独立对象，再决定要不要总结`
2. `AI 不直接替代结构化存储，而是围绕结构化对象工作`

## 3. 参考案例 B：Notion AI 定向整理工作流

## 3.1 值得参考的形态

Notion 这条线更接近你当前项目，因为它是：

1. database-first
2. automation-first
3. AI property / AI agent 围绕结构化表工作

### 参考 1：Web Clipper

参考链接：

1. [Notion Web Clipper](https://www.notion.com/en-gb/help/web-clipper)
2. [Notion Web Clipper Product Page](https://www.notion.com/web-clipper)

当前可借的点：

1. 先把网页收进结构化数据库
2. 自动保留原始 URL
3. 裁剪时就可以补 tag、备注、目标库

对当前项目的启发：

1. 你的系统也应该先把原始来源当成正式对象
2. URL / source / collected_at / source_type 必须是第一层真源字段
3. 摘要结果不能替代原始来源记录

### 参考 2：AI Autofill

参考链接：

1. [Notion AI for databases / Autofill](https://www.notion.com/en-gb/help/autofill)
2. [5 AI prompts to surface fresh insights from your databases](https://www.notion.com/zh-cn/help/guides/5-ai-prompts-to-surface-fresh-insights-from-your-databases)

当前可借的点：

1. AI 不是先聊天，而是先挂在数据库属性上
2. 支持：
   - Summary
   - Key info
   - Custom autofill
3. 能对每一条 page/item 生成统一结构的摘要字段

当前必须注意的官方限制：

1. Notion AI Autofill 不直接读取互联网网页
2. 它基于页面内容工作，因此“采集”和“AI 整理”是两层系统

这对当前项目的直接启发是：

`你的阶段十六也必须把“抓取原文”和“AI 总结”拆成两层，而不是让模型直接面对外部 URL 做黑箱总结。`

### 参考 3：Custom Agents

参考链接：

1. [Custom Agents – Notion Help Center](https://www.notion.com/help/custom-agent)
2. [Best practices for creating and optimizing a Custom Agent](https://www.notion.com/help/best-practices-for-creating-and-optimizing-a-custom-agent)
3. [Custom Agent pricing – Notion Help Center](https://www.notion.com/help/custom-agent-pricing)

截至 2026-04-21，可以确认的官方口径：

1. Custom Agents 当前仍处于 beta
2. 它强调的是 recurring workflow
3. 它要求把 context 收紧到具体页面/数据库
4. 从 2026-05-04 开始，Custom Agents 使用 workspace-level credits 计费

当前最值得借的 3 个原则：

1. `只把 agent 放在重复流程上`
2. `只给 agent 必要上下文`
3. `先定义 done，再定义 prompt`

### 对当前项目的借鉴结论

Notion 这条线最适合当前项目借的是：

1. 结构化数据库优先
2. AI 作为字段加工器，而不是系统真源
3. 自动执行必须有 trigger、上下文边界和 activity log

## 5. 获取方式设计

## 5.1 获取方式分层

阶段十六建议严格按 4 层获取方式来接：

### L1：直接 RSS / Atom

适用：

1. OpenAI Blog / Newsroom RSS
2. Google AI Blog RSS
3. arXiv API / Atom
4. InfoQ AI

优点：

1. 成本低
2. 结构稳定
3. 去重简单

### L2：官方新闻页 HTML 轮询

适用：

1. Anthropic Newsroom
2. 没有明显 RSS 的官方页面

要求：

1. 只取标题、链接、日期、分类、摘要
2. 不做深度页面抓取，除非进入 top N

### L3：广义热点 API / 页面抓取

适用：

1. 微博
2. 知乎
3. 头条

要求：

1. 先抓候选项
2. 再做关键词和白名单过滤
3. 只留下 AI 相关项

### L4：全文拉取

只对进入摘要候选 top N 的内容执行。

原因：

1. 节省成本
2. 降低噪音
3. 避免全量抓取导致的稳定性问题

## 5.2 当前项目最适合的获取顺序

建议顺序：

1. 先 RSS / Atom
2. 再 HTML 轮询
3. 最后才补广义热点抓取

不建议顺序：

1. 一开始就做搜索引擎聚合
2. 一开始就做全网页全文抓取
3. 一开始就做跨平台舆情监控

## 6. 首批内部样例的整理与排序规则

## 6.1 候选项最低结构

每条候选项至少要有：

1. `source_name`
2. `source_type`
3. `title`
4. `summary`
5. `url`
6. `published_at`
7. `tags`
8. `raw_signal`

## 6.2 候选排序建议

首批建议分数：

1. `source_trust_score`：40%
2. `keyword_match_score`：20%
3. `freshness_score`：20%
4. `user_interest_score`：20%

补充规则：

1. 官方源默认更高信任权重
2. 学术源默认更低大众优先级，但更高专业价值
3. 广义热点源必须经过关键词过滤后才能参与排序
4. 同一事件多源重复时，优先保留官方源，其他源作为补充引用

## 6.3 去重建议

至少做 3 级去重：

1. URL 去重
2. 标题规范化去重
3. 同实体 + 同发布日期近似去重

## 7. 提示词设计

阶段十六建议把提示词分成 4 套，而不是一套万能 prompt。

补充说明：

1. 当前这些提示词围绕 `ai-daily` 首批样例编写
2. 未来若扩到其他关注主题，应在保留主结构的前提下替换：
   - 重点关注维度
   - 来源说明
   - 结果输出偏好

## 7.1 摘要提示词

用途：

1. 生成“今日重点信息卡”

建议系统提示词：

```text
你是“每日人工智能重点信息”整理助手。

你的任务不是泛泛改写原文，而是基于给定材料，输出一条对用户有决策价值的信息卡。

强约束：
1. 只能依据提供材料作答，不得补充未出现的事实。
2. 如果材料不足，必须明确写“信息不足”，不要编造。
3. 输出必须是结构化 JSON，不要输出解释性前言。
4. 重点关注：模型发布、产品更新、价格/API变化、Agent 能力、工程工具链、安全与政策、开源生态、研究突破。
5. 语气要求：简洁、准确、可执行，不要营销腔。
```

建议用户提示词模板：

```text
请基于以下材料，生成一条信息卡。

【基础信息】
- 来源：{{source_name}}
- 标题：{{title}}
- 发布时间：{{published_at}}
- 链接：{{url}}

【原始摘要】
{{summary}}

【正文摘录】
{{content_excerpt}}

【输出格式】
{
  "summary_title": "不超过18字的摘要标题",
  "summary_text": "80-140字的中文摘要，必须说明发生了什么、为什么值得关注",
  "key_points": ["3-5条要点"],
  "impact_level": "high | medium | low",
  "impact_reason": "为什么是这个等级",
  "tags": ["标签1", "标签2"],
  "risk_flags": ["如果没有则返回空数组"],
  "citations": [{"title": "引用标题", "url": "引用链接"}]
}
```

## 7.2 重点筛选提示词

用途：

1. 对候选项做“是否值得进入今日重点”判断

建议系统提示词：

```text
你是人工智能信息筛选器。

你的任务是判断一条候选信息是否值得进入“今日 AI 重点信息”。

保留标准：
1. 对模型能力、产品能力、API、Agent、生态、价格、安全、政策、研究突破存在真实增量
2. 能影响开发者、产品经理、研究者或 AI 使用者的判断

剔除标准：
1. 纯营销
2. 与 AI 主线弱相关
3. 重复报道
4. 只有情绪热度，没有事实增量
```

建议输出：

```json
{
  "keep": true,
  "score": 86,
  "reason": "有明确产品更新与开发者影响",
  "topic_type": "product_update"
}
```

## 7.3 咨询提示词

用途：

1. 围绕单条摘要结果回答用户追问

建议系统提示词：

```text
你是“AI 重点信息咨询助手”。

你只允许依据给定的摘要结果、原始材料和引用信息回答问题。

规则：
1. 不要把没有证据的内容说成事实。
2. 如果问题超出材料范围，明确说明“当前材料不足以判断”。
3. 回答时优先给出简明结论，再给出依据。
4. 如果问题涉及“对我有什么影响”，优先用产品/工程/研究三个角度回答。
```

建议用户提示词模板：

```text
【摘要结果】
{{summary_result_json}}

【补充材料】
{{supporting_context}}

【用户问题】
{{question}}

请输出：
{
  "answer": "简明回答",
  "evidence": ["证据1", "证据2"],
  "uncertainties": ["若无则返回空数组"],
  "suggested_next_actions": ["若无则返回空数组"]
}
```

## 7.4 失败保护提示词

用途：

1. 避免模型把营销稿、转载稿、标题党误判成重点信息

建议规则：

```text
如果材料只包含宣传性表述、缺少明确事实、缺少新增信息、或无法判断其实际影响，请降低分数并输出 keep=false。
```

## 8. 适合当前项目的最终借鉴结论

对当前项目而言：

### 8.1 应该借 Notion 的地方

1. 数据库先行
2. AI 挂在对象结果上
3. 自动化执行有 schedule / trigger / logs

### 8.2 应该借 Obsidian 的地方

1. 单条对象可单独保留、阅读、追问
2. AI 不只做一句话摘要，还要能继续围绕对象咨询
3. 人工可以介入高价值条目的保留与修正

### 8.3 不该照搬两者的地方

1. 不做重 UI 工具平台
2. 不做插件生态
3. 不把全部能力塞进一个交互面板

### 8.4 适合当前项目的组合解法

你当前项目最适合的是：

1. `Notion 式数据库工作流`
2. `Obsidian 式单条对象深读与追问`
3. `Python 执行器 + D1 结果层 + Workers 正式 API`

一句话落地形态：

`采集和执行像自动化后台，结果和咨询像结构化内容对象，而不是像聊天机器人杂糅一切。`
