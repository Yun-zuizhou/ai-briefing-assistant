## 2. 设计前提

根据当前产品设计文档，必须先固定 3 个前提：

1. 产品始终只有一个“简报助手”，不是多个垂类简报产品
2. 用户表达的是“关注什么”，不是“订阅哪个内部模板”
3. 内部执行模板只是为了把用户关注映射成可执行的采集与摘要链

也就是说：

- 用户侧概念：`关注`
- 系统侧概念：`模板`
- 产品侧载体：`简报`

## 3. 产品层数据模型

## 3.1 用户关注项

当前产品设计里，用户真正关心的是：

1. 想长期关注什么
2. 最近重点追什么
3. 哪些内容与自己当前行动目标有关

因此“关注项”应是产品层第一核心对象。

### 当前已存在的用户层真源

1. `user_interests`

当前语义：

1. 用户当前主动表达的关注主题
2. 会影响 Today 的推荐与内容组织

### 当前建议扩展，但不要求阶段十六立刻物理落库的字段

建议未来统一语义为：

```ts
type UserInterest = {
  id: number
  userId: number
  interestName: string
  interestType: 'topic' | 'goal' | 'opportunity' | 'learning'
  keywords: string[]
  priority: 'high' | 'medium' | 'low'
  status: 'active' | 'paused'
  sourcePreferences?: string[]
  createdAt: string
  updatedAt: string
}
```

说明：

1. `interestName`
   - 用户可感知名称，如“AI 前沿”“远程工作”“产品设计”
2. `interestType`
   - 帮助系统理解这类关注更像：
     - 主题
     - 目标
     - 机会
     - 学习方向
3. `keywords`
   - 用于匹配内容和内部模板
4. `priority`
   - 决定简报里显示优先级
5. `sourcePreferences`
   - 允许未来支持“更偏官方 / 更偏机会 / 更偏学术”

## 3.2 用户简报偏好

这部分产品层当前已有：

1. `user_settings`
2. `briefing_schedules`

阶段十六不需要重做它们，但要明确：

1. 它们决定的是：
   - 什么时候发
   - 发什么类型
2. 不决定“用户关注什么”

也就是说：

- `user_interests` 决定“简报内容从哪里找”
- `user_settings / briefing_schedules` 决定“简报什么时候到”

## 4. 执行层数据模型

## 4.1 内部执行模板

当前代码里出现的 `ai-daily`，在系统层应被定义为：

`内部执行模板`

它的职责是：

1. 定义一类关注的默认信息源
2. 定义这类关注的关键词筛选方式
3. 定义候选排序规则
4. 定义摘要提示词版本

它不应被定义为：

1. 用户订阅的频道
2. 产品导航页
3. 产品一级功能模块

### 建议的模板语义

```ts
type CollectionTemplate = {
  templateId: string
  name: string
  description: string
  enabledSources: string[]
  whitelistKeywords: string[]
  blacklistKeywords: string[]
  promptVersion: string
  topNCandidates: number
  topNSummary: number
}
```

当前阶段十六只需要：

1. `ai-daily` 一个模板跑通

后续可扩展：

1. `remote-opportunity-watch`
2. `writing-opportunity-watch`
3. `design-trend-watch`
4. `academic-paper-watch`

但这些仍然属于内部模板，不是用户侧概念。

## 4.2 模板与来源

当前模板与来源的关系是：

1. 一个模板绑定多来源
2. 一个来源可以被多个模板复用

所以当前最适合的是：

```ts
type TemplateSourceBinding = {
  templateId: string
  sourceId: string
  trustWeight: number
  enabled: boolean
}
```

阶段十六首批已落在：

1. `tools/config/collection_profiles/ai-daily.json`

后续若需要平台化，再迁表。

## 4.3 候选项

候选项不是最终产品对象，而是执行层中间态。

建议语义：

```ts
type CollectedCandidate = {
  candidateId: string
  templateId: string
  sourceId: string
  title: string
  summary: string
  url: string
  publishedAt?: string
  rawTags: string[]
  scoreBreakdown: {
    trustScore: number
    keywordScore: number
    freshnessScore: number
    signalBonus: number
  }
  totalScore: number
}
```

阶段十六当前实现选择：

1. 候选项先写 manifest 文件
2. 不立刻新增候选表

这是合理的，因为当前重点是：

1. 先把链跑通
2. 不是先做全量回溯分析后台

## 5. 关注项与模板的映射

这是当前最关键的一层。

## 5.1 正确映射关系

正确关系不是：

`用户 -> 直接订阅 ai-daily`

而是：

`用户关注 -> 系统判断适用哪个内部模板 -> 模板驱动收集与摘要`

例如：

| 用户表达 | 产品层对象 | 系统层映射 |
|---|---|---|
| 我想关注 AI 前沿 | `user_interest = AI前沿` | `ai-daily` |
| 我想关注远程工作机会 | `user_interest = 远程工作机会` | `remote-opportunity-watch` |
| 我想关注交互设计趋势 | `user_interest = 交互设计趋势` | `design-trend-watch` |

所以，模板是解释器，不是产品分类。

## 5.2 当前阶段十六推荐做法

阶段十六不要先做复杂规则引擎。

推荐先做一层最小映射函数：

```ts
matchInterestToTemplate(userInterest) => templateId[]
```

初期可以基于：

1. 关键词
2. `interestType`
3. 来源偏好

最小规则示例：

1. 命中 `AI / GPT / Claude / Agent / 模型 / 推理`
   - 映射到 `ai-daily`
2. 命中 `远程工作 / 求职 / 兼职 / 招聘`
   - 映射到 `remote-opportunity-watch`
3. 命中 `写作 / 投稿 / 征稿 / 专栏`
   - 映射到 `writing-opportunity-watch`
4. 命中 `设计 / Figma / 交互 / 设计系统`
   - 映射到 `design-trend-watch`

当前阶段不需要把这个映射表做成复杂管理后台。

### 5.3 当前已形成的真实证明

截至当前本轮，下面这层最小映射已形成真实证明：

1. 用户关注 `AI`
   - 已真实命中 `ai-daily`
2. 用户关注 `写作`
   - 已真实命中 `writing-opportunity-watch`
3. 用户关注 `远程`
   - 已真实命中 `remote-opportunity-watch`
4. 当前未命中的关注
   - 会被显式保留在 `unmatched_interests`

这说明当前“关注 -> 内部模板”的桥梁层已经不是纯设计，而是已进入真实可执行状态。

## 6. 摘要与咨询结果模型

## 6.1 当前已落地结果层

当前阶段十六已经有：

1. `summary_generation_tasks`
2. `summary_generation_results`

它们的职责分离是对的：

### `summary_generation_tasks`

负责：

1. 状态机
2. 执行过程
3. 重试入口

### `summary_generation_results`

负责：

1. 摘要标题
2. 摘要正文
3. 要点
4. 引用
5. 咨询上下文

这层应继续保持。

## 6.2 当前结果不应直接等于产品首页

这点很重要。

阶段十六生成的摘要结果：

1. 不是产品首页最终结构
2. 不是用户感知的最终信息组织层
3. 它更像“简报素材池”

后续应该回流到：

1. `Today / 简报首页` 的信息追踪区
2. 用户关注领域的聚合卡
3. 需要时再联动待办与叙事

---

## 7. 最终回流到简报的结构

你的产品设计里，简报最终仍然是：

1. 信息追踪
2. 待办提醒
3. 个人叙事

所以阶段十六结果不应停在“列表页”。

应回流成：

### 7.1 信息追踪

来自：

1. 用户关注主题映射后的摘要结果
2. 热点背景信号

作用：

1. 告诉用户“今天与你相关的世界发生了什么”

### 7.2 待办提醒

来自：

1. 摘要结果中可行动的信息
2. 用户在对话中确认后的待办转化

作用：

1. 把“知道”变成“行动”

### 7.3 个人叙事

来自：

1. 用户看了什么
2. 用户收藏了什么
3. 用户追问了什么
4. 用户记录了什么

作用：

1. 把“时代”回流到“我”

## 8. 当前阶段建议的数据模型结论

阶段十六当前最合适的数据模型结论是：

1. 产品层继续以 `user_interests` 为中心，不引入新的用户可见“频道订阅”模型
2. 系统层允许存在多个内部模板，但它们只是执行模板
3. 阶段十六结果层继续用：
   - `summary_generation_tasks`
   - `summary_generation_results`
4. 候选项当前先保留为文件 manifest，不急着入库
5. 最终结果必须回流到“简报”的信息追踪，而不是独立演化成新的产品面

一句话总结：

`用户表达的是关注，系统运行的是模板，产品交付的是简报。`
