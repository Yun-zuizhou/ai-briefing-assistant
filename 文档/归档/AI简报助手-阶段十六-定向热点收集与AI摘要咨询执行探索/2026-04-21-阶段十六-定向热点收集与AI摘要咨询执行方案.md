## 1. 为什么现在切到阶段十六

当前项目的 `Workers + Pages + D1` 主链已经稳定到“可以继续开发新能力”的程度，但还不适合再把时间继续大量消耗在页面尾差、历史资产清理和大范围阶段收尾上。

当前最值得尽快进入实践调试的新主线，已经变成：

1. 从“泛内容展示”进入“关注驱动的信息收集”
2. 从“静态 summary 字段消费”进入“真实大模型摘要执行”
3. 从“内容详情阅读”进入“围绕单条热点的摘要咨询”
4. 从“支撑链事实表已经落位”进入“至少一条真实执行链跑通”

一句话判断：

`阶段十六不是继续收页面，而是把“采集 -> 筛选 -> 摘要 -> 咨询”这条后端闭环第一次跑成真实执行链。`

### 1.1 本轮目标口径补充

阶段十六当前必须明确补上一条口径：

`当前追求的不是“向老师展示的最小演示单元”，而是“可答辩展示的最小完整闭环”。`

这里的差异非常关键：

1. 最小演示单元强调“能演一下”
2. 最小完整闭环强调“基础功能真实可执行，演示时只把外部波动降到最低”
3. 对当前项目来说，老师看到时允许存在的是：
   - 设计表达仍可继续优化
   - 页面层次、文案、视觉还可继续打磨
4. 不允许存在的是：
   - 登录后主链打不开
   - 页面读取依赖临时假数据才能成立
   - AI provider 一失效就把主路径拖死
   - 咨询或摘要缺配置时只能报错、没有正式空状态或兜底状态

因此阶段十六后续一切判断都应服从下面这条硬约束：

`答辩展示时，项目剩余问题可以是设计问题，但不应再是执行问题。`

## 3. 阶段十六的严格定义

阶段十六只承认一个主闭环：

`用户关注 / 对话表达 -> 内部执行模板选择 -> 候选内容入库 -> 规则筛选与排序 -> 生成摘要任务 -> 调用模型生成结构化摘要 -> 围绕单条结果进行咨询问答 -> 回流到每日简报的信息追踪区 -> 可观测、可复跑、可人工验证`

这里的“定向”强调的是：

1. 不是继续抓一切热点
2. 不是继续做通用内容列表
3. 而是围绕“用户关注的主题”建立可控的采集与整理链
4. 当前 `ai-daily` 只是其中一个内部样例，不是最终用户侧产品分类

这里的“咨询”强调的是：

1. 不是全库开放问答
2. 不是多轮 agent 工作流
3. 而是围绕单条热点/文章摘要结果做 grounded 追问

## 4. 当前阶段目标

### 4.1 P0 目标

阶段十六必须完成的最小能力只有 7 项：

1. 定义一套“关注驱动”的首批内部执行模板
2. 能手动触发一次定向采集执行
3. 能把采集运行写入 `ingestion_runs`
4. 能创建并消费 `summary_generation_tasks`
5. 能把摘要执行写入 `ai_processing_runs`
6. 能把摘要结果以正式结构落到 D1
7. 能基于单条摘要结果发起一次咨询并返回 grounded 回复

### 4.1.1 P0 目标的完成口径

这 7 项不是“做出一版能演示的样子”就算完成，而应按“最小完整闭环”判断：

1. 非 AI 基础链路必须独立稳定：
   - 登录 / session
   - `daily-digest` 结果读取
   - 页面跳转
   - 待办、历史、成长入口回看
2. AI 增值链路必须至少有一条真实成功链路：
   - 真实采集成功
   - 真实任务创建成功
   - 至少一条真实摘要结果成功落库
   - 至少一条围绕摘要结果的咨询成功返回
3. 当 provider 不可用时，系统必须返回：
   - `pending_provider`
   - `not_configured`
   - `failed`
   - 或明确标记的 `debug-fallback`
4. 不允许再用伪摘要、伪咨询、伪趋势把失败包装成成功展示

简化成一句话：

`阶段十六的 P0，不是“老师面前能点几下”，而是“这条闭环本身已经具备最基本的正式执行可信度”。`

当前首批内部执行模板取值：

1. `ai-daily`

它当前代表的是：

1. 阶段十六首轮技术验证样例
2. 一个“关注主题 -> 收集 -> 摘要 -> 咨询”的最小闭环场景

它当前不代表的是：

1. 产品长期唯一主题
2. 用户可见的最终信息架构
3. 多个垂类简报产品中的一个

### 4.2 P1 目标

可以进入阶段十六，但不作为首批硬门槛的：

1. 自动定时调度
2. 后台批量重试
3. Web 管理页
4. 多 provider 动态切换 UI
5. 摘要分享与推送

### 4.3 当前明确不进入本轮的

1. 全库 RAG
2. 向量数据库
3. 多轮 agent 编排
4. 完整前端产品页
5. “AI 替用户选题并自动生成周报/月报”的扩面链路

## 5.3 信息源接入策略

阶段十六当前执行层只承认 4 类来源：

1. 官方源
   - 官方新闻页
   - 官方博客
   - 官方 RSS
2. 行业技术媒体
   - InfoQ AI
   - 量子位
3. 学术源
   - arXiv `cs.AI / cs.LG / cs.CL`
4. 广义热点源
   - 微博/知乎/头条，但必须经过 AI 关键词过滤

当前不建议首批引入：

1. Twitter/X 全量抓取
2. Reddit 全量抓取
3. 任意站点搜索引擎抓取

原因：

1. 噪音高
2. 去重难
3. 时效虽强，但很容易把阶段十六拖成舆情系统

补充口径：

1. 当前这些来源是“内部模板源池”
2. 它们服务的是当前样例模板的执行，不等于产品未来只围绕这些源构建
3. 未来若用户关注“远程工作 / 产品设计 / 学术申请”等其他主题，应继续沿“关注驱动 -> 模板映射 -> 简报生成”扩展，而不是把产品拆成多个垂类入口

## 5.5 执行器方案

建议新增两个脚本入口：

1. `python tools/scripts/run_directed_collection.py --profile ai-daily`
2. `python tools/scripts/run_summary_executor.py --profile ai-daily`

执行顺序：

1. 读取 profile
2. 跑采集
3. 写 `ingestion_runs`
4. 归一化候选项
5. 排序并选出 top N
6. 对 top N 创建 `summary_generation_tasks`
7. 消费任务并调用模型
8. 写 `ai_processing_runs`
9. 写 `summary_generation_results`

## 5.5.1 当前必须避免的误判

阶段十六当前容易出现的误判是：

1. 误把 `ai-daily` 当成最终产品内容结构
2. 误把阶段十六做成“AI 资讯产品”
3. 误把“摘要结果列表”当成未来对用户直接暴露的最终首页

当前正确口径应固定为：

1. `ai-daily` 是内部执行模板
2. `daily-digest` 是阶段十六调试与执行结果读取口
3. 最终回到产品时，这些结果应进入：
   - 简报首页的信息追踪区
   - 用户关注领域对应的摘要卡
   - 需要时再衔接待办提醒与个人叙事

## 5.6 摘要与咨询接口方案

建议首批新增 3 个正式 API：

1. `GET /api/v1/system/summary-tasks/:id/result`
   - 读取单个摘要结果
2. `GET /api/v1/content/daily-digest?profile_id=ai-daily`
   - 返回当日 top N 摘要结果列表
3. `POST /api/v1/content/consult`
   - 输入 `result_ref` + `question`
   - 返回基于当前摘要结果与引用内容的咨询回答

其中咨询接口首批只允许：

1. 围绕单条结果问答
2. 最多带 3 条相关上下文
3. 必须返回引用依据

首批禁止：

1. 全库自由问答
2. 无引用回答
3. 为了追求“像聊天”而牺牲可追溯性

## 5.7 观测与调试方案

阶段十六必须把“可调试”当成 P0，而不是 P1。

必须能看见：

1. 本次采集跑了哪些源
2. 每个源抓了多少条
3. 过滤掉多少条
4. 哪些条进入摘要
5. 每条摘要用了哪个 provider / model / prompt_version
6. 哪条失败了，为什么失败

建议首批调试入口：

1. CLI 日志
2. D1 运行表
3. `GET /api/v1/system/summary-tasks`
4. `GET /api/v1/system/chain-health`

## 5.8 三档运行口径

为了同时兼顾开发效率、真实验证和答辩稳定性，阶段十六后续固定只承认下面三档运行口径：

### A. 正式联调档

用途：

1. 验证真实 provider 是否可用
2. 验证真实摘要质量与 grounded 咨询质量
3. 作为“阶段十六已具备真实执行能力”的事实依据

要求：

1. 真采集
2. 真任务
3. 真 provider
4. 真结果落库

### B. Debug Fallback 联调档

用途：

1. 本地无 key 时继续联调页面和读侧
2. 在不阻断开发的前提下验证完整调用链
3. 为失败场景提供可观测、可标记的最小兜底

要求：

1. 必须明确标记为 `debug-fallback`
2. 只能作为联调与本地保底
3. 不能作为“真实 AI 能力已完成”的验收证据

### C. 答辩保底档

用途：

1. 答辩演示
2. 截图录屏
3. 展示账号稳定回放

要求：

1. 默认读取预生成且已持久化的正式结果
2. 页面主路径不现场依赖 provider 即时成功
3. 可允许 `consult` 在本地或非生产环境走明确标记的 fallback
4. 不允许用伪数据临时拼装“看起来完整”的结果

这里必须特别强调：

`答辩保底档是展示保底，不是正式验收替代。`

8. 至少完成一次围绕单条摘要结果的咨询调用

### 7.2 结果质量验收

首批结果不要求“完美摘要”，但必须满足：

1. 摘要能说明：
   - 发生了什么
   - 为什么值得关注
   - 对当前用户或当前主题有什么影响
2. 咨询回答不能脱离原文与摘要结果胡编
3. 每条结果至少附一个原始链接
4. 明显无关或营销噪音内容，不得进入 top N

### 7.2.1 展示稳定性验收

除结果质量外，阶段十六还要新增一条展示稳定性验收：

1. `daily-digest` 主列表必须可稳定读取
2. 点进单条摘要结果后必须可稳定查看正文、要点与引用
3. `consult` 在 provider 不可用时，必须是：
   - 明确配置错误
   - 明确待生成状态
   - 或明确标记的 debug fallback
4. 页面不能因为 AI 即时执行失败而整体空白或链路中断
5. 答辩展示时，剩余可见问题应主要属于设计层，而不是执行层

### 7.3 人工验收

人工验收至少补 4 类证据：

1. 一次定向采集命令执行截图或日志
