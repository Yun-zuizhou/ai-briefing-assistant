# ChatPage 正式会话读侧消费判断

- 文档状态：进行中
- 建立时间：2026-04-05
- 所属阶段：阶段十一 - 二级页面正式读侧整合与会话读侧消费收口
- 文档职责：承接 `ChatPage` 从本地会话态转向正式 `chat/sessions` 读取链的判断、实现与验收

---

## 1. 为什么当前切到这条主线

当前项目已经具备以下现实前提：

1. 主闭环三页已基本站到正式读侧上
2. 后端 `GET /api/v1/chat/sessions`
3. 后端 `GET /api/v1/chat/sessions/{session_id}/messages`
4. `chat_session / chat_message` 持久化链路已成立并进入主线测试

但当前前端 `ChatPage` 的真实消费面仍主要依赖：

- `localStorage` 本地会话态

这意味着：

- 正式会话读取链已经成立
- 但用户正在使用的 `ChatPage` 还没有真正消费它

因此阶段十一第一优先不应继续扩对象，而应先把 `ChatPage` 收回到正式会话读侧上。

---

## 2. 当前真实现状

### 2.1 后端已经成立的能力

当前已成立：

1. `chat/execute`
2. `chat/reclassify`
3. `chat/sessions`
4. `chat/sessions/{session_id}/messages`

并且当前会话消息读取已能返回：

1. `messageState`
2. `candidateIntents`
3. `confirmedType`
4. `deepLink`
5. `nextPageLabel`
6. `changeLog`

这意味着当前后端读侧已经足够支撑 `ChatPage` 的主要消息展示语义。

### 2.2 前端当前问题

当前 `ChatPage / useChatLogic` 仍主要依赖：

1. 本地 `localStorage`
2. 前端即时拼接的会话消息
3. 刷新后优先恢复本地缓存，而不是正式会话读取链

当前最大问题不是“后端没有接口”，而是：

- `ChatPage` 还没有真正把正式会话读取链吃进去

### 2.3 当前这一轮实现结果

当前这一轮已经完成：

1. Chat 会话公共类型已从 `api.ts` 内联结构提升为可复用类型
2. `useChatLogic` 已补上：
   - `getChatSessions()`
   - `getChatSessionMessages()`
3. `ChatPage` 进入页面后，当前会优先读取最近一个正式 session
4. 刷新后当前会优先从正式会话读取链恢复消息，而不是继续把本地 `localStorage` 当作主会话事实源
5. `localStorage` 当前仅作为：
   - 正式读取失败时的缓存兜底
   - 页面临时过渡态缓存
6. `ChatPage` 已补上正式读取状态提示：
   - 正式会话
   - 缓存会话
   - 无正式会话
   - 正式读取失败提示

同时，这轮第二优先也已做了最小推进：

1. `JournalPage` 当前已从：
   - `getNotes() + getHistory()`
   调整为：
   - `getNotes() + getGrowthOverview()`
2. `HistoryLogsPage` 当前保持 `getHistory()` 不变
3. 这意味着阶段十一第二批页面里：
   - `JournalPage` 已开始向正式读侧整合
   - `HistoryLogsPage` 当前只做边界确认，不强行扩新接口

---

## 3. 当前边界

这条主线当前只做：

1. 正式会话读取链消费
2. 会话消息字段到 UI 消息的稳定映射
3. 刷新后从正式会话恢复最近一次会话内容

当前明确不做：

1. 完整多会话管理 UI
2. 新的 Chat 对象层设计
3. 会话归档、会话搜索、会话删除
4. 继续扩 Chat 子线覆盖面

---

## 4. 当前阶段性结论

当前这条主线应先收口为：

1. `ChatPage` 值得进入正式会话读侧消费收口
2. 默认只消费最近一个正式 session
3. 本地 `localStorage` 应降级为临时缓存或异常兜底
4. 当前不引入完整多会话产品设计
5. 这条主线的第一轮最小实现已经成立
6. 第二批页面里，`JournalPage` 已开始往正式读侧收口，`HistoryLogsPage` 当前继续后置扩展

---

## 5. 当前下一步入口

下一步不再是继续补这一轮基础实现，而是：

1. 回到阶段十一总主线，判断：
   - `JournalPage / HistoryLogsPage` 是否还值得继续做更高层正式读侧整合
2. 如果当前没有新的高收益实现点，则应把这条 Chat 子主线作为阶段十一已完成事实继续上推
