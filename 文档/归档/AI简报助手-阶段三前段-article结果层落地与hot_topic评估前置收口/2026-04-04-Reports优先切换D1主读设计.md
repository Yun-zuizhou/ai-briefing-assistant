# Reports优先切换D1主读设计

- 文档状态：进行中
- 建立时间：2026-04-04
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：在接口依赖分层已经明确后，固定“为何下一步优先推进 `reports/*` 而不是 `Today / content/by-ref`”的设计判断，作为后续实现前的最后一层方案说明

---

## 1. 本轮结论

在当前阶段二与方案 C 的前提下，下一步优先推进：

`reports/* -> Cloudflare D1 正式主读切换设计与实现`

而不是先推进：

`dashboard/today + content/by-ref（hot_topic / opportunity 子集）`

---

## 2. 为什么当前优先 `reports/*`

## 2.1 `reports/*` 属于行为聚合接口

根据 [D1接口依赖分层与切换顺序设计.md](/E:/python/杂谈-想法记录与实践/文档/进行中/2026-04-04-D1接口依赖分层与切换顺序设计.md)，`reports/*` 当前属于：

- 行为聚合接口

它主要依赖：

1. `notes`
2. `favorites`
3. `todos`
4. `history_entries`
5. `user_interests`

这些都属于：

- 正式主库层

因此，优先推进 `reports/*`，本质上是在继续扩大“正式主库层已经切到 D1”的稳定成果。

## 2.2 `reports/*` 不直接受 `rss_articles` 约束

当前最重内容包是：

- `seed.content.rss_articles.sql`

但 `reports/*` 当前并不直接以 `rss_articles` 为主依赖。

这意味着：

1. 如果先切 `reports/*`，不会立即把系统再次拖回 `rss_articles` 分片执行问题
2. 这一步能让数据库切换主线继续往前推进，同时保持执行面稳定

## 2.3 `reports/*` 更符合方案 C 的顺序

方案 C 当前强调的是：

1. 正式主库层优先
2. 内容层延后
3. 重内容详情接口最后处理

从这个顺序看：

1. 行为事实接口已经成立
2. 下一层最自然的就是行为聚合接口
3. `reports/*` 正好位于这里

所以它是当前最符合方案 C 的下一步。

---

## 3. 为什么当前不优先 `Today / content/by-ref`

## 3.1 它们已进入内容层

即便 `dashboard/today` 和 `content/by-ref` 的 `hot_topic / opportunity` 子集还没有直接撞上 `rss_articles`，它们也已经属于：

- 轻内容聚合 / 轻内容详情接口

一旦优先推进它们，当前主线就会提前转向内容层正式化。

## 3.2 会更快回到 `article` 问题

如果先推进：

1. `dashboard/today`
2. `content/by-ref` 的 `hot_topic / opportunity` 子集

那么下一步自然会被追问：

- 那 `article` 呢？

这会更快把项目重新带回：

- `rss_articles` 分片执行与内容详情正式切换

当前并不适合这样提前拉高复杂度。

## 3.3 它会把“主库层收稳”和“内容层正式化”混在一起

当前更合理的顺序应是：

1. 先把正式主库层与行为聚合层收稳
2. 再推进轻内容层
3. 最后再进入重内容详情层

如果现在直接去切 `Today / content/by-ref`，会让这三步重新缠绕。

---

## 4. 当前对 `reports/*` 的设计判断

## 4.1 当前切换性质

`reports/*` 当前不应被理解为：

- 统一内容层正式化的一部分

而应被理解为：

- 行为事实正式主库层向上提供的聚合读侧

## 4.2 当前切换前提

当前推进 `reports/*` 进入 D1 主读的前提已经具备：

1. `user_interests` 已在 D1 主读成立
2. `notes / favorites / todos / history_entries` 已在 D1 主读成立
3. 云端主线专项验证已通过

## 4.3 当前切换边界

当前推进 `reports/*` 时，应保持以下边界：

1. 只切 `reports/*` 本身的主读来源
2. 不顺手一起推进 `Today`
3. 不顺手一起推进 `content/by-ref`
4. 不把 `reports/*` 的切换解释成“内容层已经正式化”

---

## 5. 当前实现前需要先明确的设计点

虽然已决定优先 `reports/*`，但实现前仍应先做一轮小设计，至少明确：

1. `reports/*` 是否新增统一 D1 聚合读取层，还是直接在接口中拼装读取
2. 周报 / 月报 / 年报 / 报告可用性接口是否共用同一 D1 聚合入口
3. `content_ref` 在报告中的生成逻辑是否保持完全不变
4. 当前 D1 下的 `reports/*` 是否仍保留“过渡态生成口径”，不提前引入统一内容实体化

也就是说：

当前已经确定“先做 `reports/*`”，但实现前仍需要一轮针对 `reports/*` 本身的小设计。

---

## 6. 当前正确口径

当前正确口径应更新为：

1. 行为事实接口已经完成 D1 正式主读验证
2. 当前下一步优先推进的是 `reports/*`
3. 原因不是它“更显眼”，而是它位于正式主库层之后、内容层之前
4. `Today / content/by-ref` 当前不是不做，而是顺序上排在 `reports/*` 之后
