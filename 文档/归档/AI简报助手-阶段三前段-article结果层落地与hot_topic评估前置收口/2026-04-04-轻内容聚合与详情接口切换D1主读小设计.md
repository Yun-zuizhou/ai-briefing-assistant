# 轻内容聚合与详情接口切换D1主读小设计

- 文档状态：进行中
- 建立时间：2026-04-04
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：在已明确接口依赖分层之后，先固定轻内容聚合 / 轻内容详情接口切到 D1 主读的最小正确边界，再进入实现

---

## 1. 当前纳入范围

本轮只纳入：

1. `dashboard/today`
2. `content/by-ref` 的 `hot_topic` 分支
3. `content/by-ref` 的 `opportunity` 分支

本轮明确不纳入：

1. `content/by-ref` 的 `article` 分支
2. `rss_articles` 分片执行
3. 统一 `contents` 实体化

---

## 2. 设计判断

### 2.1 是否新增 D1 内容读取层

当前判断：

- 要新增最小 D1 内容读取层

它只负责：

1. 读取 `hot_topics`
2. 读取 `opportunities`
3. 按当前接口所需字段输出最小可消费对象

它不负责：

1. 读取 `rss_articles`
2. 改造统一内容层
3. 重写推荐算法

### 2.2 `dashboard/today` 的切换方式

当前判断：

- 在 `D1_USE_CLOUD_AS_SOURCE=true` 时，Today 改为：
  - 兴趣继续走 D1 行为事实层
  - 内容供给改走 D1 内容读取层
  - 继续复用当前推荐 / worth_knowing / worth_acting 的加工函数

也就是说：

- 只切主读来源，不改 Today 当前过渡态生成口径

### 2.3 `content/by-ref` 的切换方式

当前判断：

- 在 `D1_USE_CLOUD_AS_SOURCE=true` 时：
  - `hot_topic` 分支走 D1 内容读取层
  - `opportunity` 分支走 D1 内容读取层
  - `article` 分支保持本地现状或明确不支持进入 D1 分支

当前正确边界是：

- `content/by-ref` 本轮不是整体切到 D1，而是只切其中两条轻分支

### 2.4 `related_items` 的口径

当前判断：

- 对 `hot_topic / opportunity` 分支，D1 下的 `related_items` 只使用：
  - `hot_topics`
  - `opportunities`

当前不把 `rss_articles` 混进来。

原因：

1. 避免轻内容详情实现时提前重新依赖 `rss_articles`
2. 保持本轮切换边界清晰

### 2.5 `content_ref` 契约

当前判断：

- 完全保持不变

也就是：

1. `hot_topic:{id}`
2. `opportunity:{id}`

当前不改 `content_ref / by-ref` 的读契约。

---

## 3. 当前实现边界

本轮实现只应触及：

1. 最小 D1 内容读取层
2. `dashboard.py` 的 D1 主读分支
3. `content.py` 的 D1 主读分支（仅 `hot_topic / opportunity`）
4. 对应验证与文档同步

本轮不应顺手触及：

1. `rss_articles`
2. `article` 分支
3. `Today` 的统一内容实体化
4. `ArticlePage` 的长期正式化重构

---

## 4. 当前结论

当前轻内容聚合 / 轻内容详情接口切 D1 的最小正确方案已经明确：

1. 要加最小 D1 内容读取层
2. Today 只切内容来源，不改当前过渡态加工逻辑
3. `content/by-ref` 只切 `hot_topic / opportunity` 子分支
4. `article` 分支继续留在后续阶段处理
5. `related_items` 当前不重新引入 `rss_articles`
