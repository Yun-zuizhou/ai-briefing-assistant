# Article内容流水线分层设计

- 文档状态：进行中
- 建立时间：2026-04-04
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：在 `content/by-ref` 的 `article` 分支尚未进入 D1 正式切换前，先把这层的真实复杂度收束为“每日持续生产型内容系统”的分层设计，明确 `rss_articles` 在其中的定位，以及 D1 当前应先承接到哪一层

---

## 1. 当前为什么必须单独设计 `article`

当前 `article` 这层之所以复杂，不只是因为：

1. `rss_articles` 体量大
2. `seed.content.rss_articles.sql` 当前仍是重包

更重要的是，它本质上还是一条：

`每日抓取 -> 每日写入 -> 每日加工 -> 每日详情供给`

的持续生产型内容流水线。

这意味着当前难点不是：

- 单纯把 `article` 详情接口改成读 D1

而是：

- 让一条会持续增长、持续加工、持续供给的内容系统，在 D1 上先建立最小稳定结构

---

## 2. 当前 `article` 这层的真实特征

根据当前代码与数据结构：

### 2.1 它是持续增量型数据

`RSSAggregator` 会持续：

1. 抓取多个 RSS 源
2. 对同一来源进行增量同步
3. 依据 `guid` 或 `source_url` 做幂等更新

这意味着：

- `rss_articles` 不是一次性导入型数据，而是每日持续增长的数据

### 2.2 它不是单纯详情表

当前 `rss_articles` 同时承载：

1. 标题
2. 摘要
3. 正文
4. 来源信息
5. 标签 / 分类
6. 抓取时间
7. 质量分
8. 原始数据

这说明它不是“只为页面展示准备的详情表”，而是：

- 原始文章事实与部分加工结果混合承载层

### 2.3 它是可重建内容层

与 `todos / notes / favorites / history` 不同：

1. `article` 很多时候可以重新抓取
2. 部分摘要、分类、标签理论上可以重新计算
3. Today / 详情页 / 报告后续读取的，未必都要直接绑定 `rss_articles` 原始形态

这意味着：

- `article` 当前不应被视为“唯一不可替代事实层”
- 它更接近“可重建内容层”

---

## 3. 当前最合理的三层结构

当前应把 `article` 这条链路拆成三层来理解，而不是继续视为一张表。

## 3.1 原始事实层

定义：

- 系统从 RSS 源抓回来的原始文章事实

当前最接近这层的是：

- `rss_articles`

但当前 `rss_articles` 还不是纯粹的原始层，因为它已经混入了：

1. `summary`
2. `quality_score`
3. `tags / category`

这些半加工字段。

当前仍可把它近似视为：

- 原始事实层 + 轻加工字段的混合过渡表

这层的目标是：

1. 保证幂等写入
2. 保证来源可追溯
3. 保证正文与原始元信息可保存

## 3.2 加工结果层

定义：

- 由原始文章事实进一步生成的内容加工结果

典型应包括：

1. 结构化摘要
2. 主题标签
3. 分类纠正
4. 质量分
5. 相关推荐信号
6. 是否进入 Today 候选池

当前现实是：

这层还没有被单独拉出来，而是部分字段直接混在 `rss_articles` 或聚合逻辑里。

因此当前问题不是“这层不存在”，而是：

- 这层还没有独立成正式层

## 3.3 读侧投影层

定义：

- 真正给页面和接口消费的稳定对象层

对 `article` 来说，至少应包括：

1. `content_ref`
2. 标题
3. 摘要
4. 正文
5. 来源
6. 标签 / 分类
7. 可用于相关推荐的稳定读对象

当前现实是：

- `content/by-ref` 的 `article` 分支就是这层的当前过渡读侧

但它仍直接贴近 `rss_articles`，还不是正式独立投影层。

---

## 4. D1 当前最适合先承接哪一层

当前判断：

- D1 当前最适合先承接的是：
  - `article` 的原始事实层

更准确地说，是：

- `rss_articles` 作为“原始事实层 + 轻加工字段混合过渡表”的这一层

原因：

1. 当前已有真实抓取写入逻辑
2. 当前 `content/by-ref` 的 `article` 分支就是从这里读
3. 当前还没有独立的正式加工结果层
4. 当前也还没有独立的正式读侧投影层

所以如果现在就强行把 D1 承接到“加工结果层”或“正式读侧投影层”，会跳过当前阶段边界。

---

## 5. 当前不该做的事

基于以上分层，当前不应直接做：

1. 直接把 `rss_articles` 说成“正式内容层已完成”
2. 在没有单独设计的情况下，把摘要、分类、相关推荐全部正式化
3. 借 `article` 切换顺手做统一 `contents` 实体化
4. 把 `article` 分支的数据库切换和内容流水线重构混成同一轮任务

---

## 6. 当前最小正确前置顺序

如果后续要推进 `content/by-ref` 的 `article` 分支进入 D1 正式切换，当前最小正确顺序应为：

1. 先完成 `rss_articles` 分片执行规则设计
2. 再明确 `rss_articles` 在 D1 中只先承接“原始事实层 + 轻加工字段混合过渡表”
3. 再让 `article` 分支先从这层稳定读
4. 之后若需要，再单独设计：
   - 加工结果层
   - 正式读侧投影层

也就是说：

`article` 当前不应一步到位做成最终内容架构，而应先让原始事实层在 D1 稳定成立。`

---

## 7. 当前对 `rss_articles` 的定位更新

当前应更新为：

1. `rss_articles` 不是普通详情表
2. `rss_articles` 也不只是 seed 重包
3. `rss_articles` 当前是“每日持续生产型内容系统”的原始事实承接层
4. 当前它之所以复杂，核心不是接口不会写，而是：
   - 它背后是持续抓取、持续加工、持续供给的系统性问题

---

## 8. 当前结论

当前 `article` 这层的最小正确判断已经明确：

1. 它是持续生产型内容系统，不是普通详情表
2. 它至少应拆成：
   - 原始事实层
   - 加工结果层
   - 读侧投影层
3. D1 当前最适合先承接的是：
   - 原始事实层
4. 当前 `rss_articles` 应被视为：
   - 原始事实层 + 轻加工字段混合过渡表
5. 后续 `article` 分支若要进入 D1 正式切换，应先按这个判断推进，而不是先追求最终统一内容层
