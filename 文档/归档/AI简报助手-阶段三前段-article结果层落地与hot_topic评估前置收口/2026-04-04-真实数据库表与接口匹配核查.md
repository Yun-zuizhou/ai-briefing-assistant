## 2. 当前真实表和设计锚点的关键差异

当前最重要的差异有三类：

### 2.1 真实库里没有这些设计锚点表

当前真实库里不存在：

1. `contents`
2. `briefings`
3. `reports`
4. `opportunity_follows`
5. `user_profiles`

这意味着：

1. 当前统一内容层仍然是“读侧过渡态”，不是正式实体表
2. 报告和画像当前主要还是接口聚合结果，不是独立落表

### 2.2 已存在表和 SQL 设计稿命名不完全一致

例如：

1. `users` 真实字段是 `hashed_password`，不是设计稿里的 `password_hash`
2. `favorites` 真实字段是 `item_title / item_summary / item_source / item_url`
3. `todos` 真实字段是 `content`，不是设计稿里的 `title`
4. `opportunities` 真实字段比设计稿更丰富

这意味着：

- 当前如果要做正式数据库迁移，不能直接拿 SQL 设计稿当真实库结构替身

### 2.3 当前真实库偏“分表事实层”

内容相关真实事实目前分散在：

1. `hot_topics`
2. `rss_articles`
3. `opportunities`

而不是统一进：

- `contents`

当前最关键的不匹配不是“接口跑不起来”，而是：

### 4.1 统一内容层没有正式实体表

影响接口：

1. `/api/v1/content/by-ref`
2. `/api/v1/dashboard/today`
3. `/api/v1/favorites`
4. `/api/v1/history`
5. `/api/v1/reports/*`

表现：

1. 这些接口都能跑
2. 但底层还不是统一事实层

### 4.2 收藏和历史的统一引用还只是兼容层

当前：

1. `favorites` 没有 `content_ref` 字段
2. `history_entries` 没有 `content_ref` 字段

所以：

1. 接口返回能给你 `contentRef`
2. 但底层表并没有真的切过去

### 4.3 报告、画像、简报没有独立结果表

所以：

1. 接口能给数据
2. 但不是正式持久化结果层

## 5. 现在再回答你：哪些接口还在读本地 SQLite，哪些已经可以切 D1

### 5.1 当前仍在读本地 SQLite 的接口

按当前后端实现，几乎所有真实业务接口都还在读本地 SQLite：

1. `/api/v1/dashboard/today`
2. `/api/v1/content/by-ref`
3. `/api/v1/preferences/*`
4. `/api/v1/reports/*`
5. `/api/v1/chat/*`
6. `/api/v1/favorites`
7. `/api/v1/history`
8. `/api/v1/notes`
9. `/api/v1/todos`
10. `/api/v1/hot-topics`
11. `/api/v1/opportunities`
12. `/api/v1/rss/*`

原因很简单：

- `app/database.py` 当前主连接仍是 `DATABASE_URL`
- `.env.example` 当前默认仍是 `sqlite:///./info_collector.db`
- `D1_USE_CLOUD_AS_SOURCE=false`

### 5.2 当前“已经可以切 D1”的准确说法

严格说：

- 现在还没有哪条后端接口是真正“已经可以直接切 D1”

更准确的说法是：

1. D1 目标已经明确
2. `wrangler.toml` 已有正式 D1 配置
3. 但后端数据库层还没有 D1 主读写实现

所以当前不是“挑几条接口先切 D1”，而是：

`先决定数据库访问层怎么支持 D1，再谈哪些接口切。`

---

## 6. 当前最应该先做的数据库方向任务

基于上面的事实，当前最合理的顺序是：

1. 先确认第一批要进入正式数据库主读的接口
   - 我建议优先：`preferences / chat / favorites / history / todos / notes`
2. 再设计这些接口的数据库访问层如何从 SQLite 切到 D1
3. 最后才谈 Today / content / reports 这类聚合接口怎么切

原因：

1. 前一批是“单表或少表 + 用户行为事实”
2. 后一批是“聚合读侧 + 统一内容层过渡态”
3. 如果先切 Today / content，复杂度会高很多

---

## 7. 当前一句话结论

`你现在已经有真实业务表，绝大多数当前接口也都能和这些表对上；但当前整个后端真实主读写仍然是本地 SQLite，不存在“已经可以直接切 D1 的接口”，真正缺的是数据库访问层的正式切换设计，而不是再猜有没有表。`
