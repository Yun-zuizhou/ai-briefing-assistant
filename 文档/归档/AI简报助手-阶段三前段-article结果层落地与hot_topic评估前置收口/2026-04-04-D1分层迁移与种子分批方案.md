# D1分层迁移与种子分批方案

- 文档状态：进行中
- 建立时间：2026-04-04
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：将当前 Cloudflare D1 切换主线收束为“正式主库层优先、内容层延后、规则先行”的方案文档，作为后续种子拆包与聚合读侧切换的统一依据

---

## 1. 本轮方案结论

当前不采用“只做局部过渡”的纯方案 A，也不采用“立刻全量长期化”的纯方案 B。

当前采用：

`方案 C：正式主库优先、内容层延后、规则先行的可迁移方案`

它的核心目标是：

1. 让 Cloudflare D1 先在第一批行为事实接口上成为正式数据库
2. 不把 `Today / content / reports` 这类聚合读侧和重内容灌装提前做重
3. 让后续无论继续停留在 D1，还是迁移到更重的数据库，都仍然可控

---

## 2. 对 D1 的当前判断

### 2.1 D1 当前不再被视为临时库

当前项目中，D1 的定位不是 mock、演示库或一次性过渡库。

在当前架构下，D1 可以被视为：

- 云端正式数据库候选

### 2.2 D1 当前更适合先承接行为事实层

优先适合进入 D1 正式主库的，是：

1. 用户偏好
2. 待办
3. 记录
4. 收藏
5. 历史
6. Chat 执行与纠偏联动写入

原因是：

1. 这类数据是明确的用户行为事实
2. 表边界清楚
3. 写入价值高于灌装规模
4. 更适合作为 APP / 前后端统一依赖的真实云端主库

### 2.3 D1 当前不应被误判为“所有层都已正式化”

即便 D1 可以成为正式数据库，也不意味着：

1. `rss_articles / hot_topics / opportunities` 必须现在就全部完成全量云端灌装
2. `Today / content / reports` 当前就应该一起切成正式云端主读
3. 当前 seed 结构应当直接等同于长期最终数据库形态

---

## 3. 分层规则

## 3.1 正式主库层

当前定义为：

1. `users`
2. `user_interests`
3. `user_settings`
4. `todos`
5. `notes`
6. `favorites`
7. `history_entries`

这层的规则是：

1. 当前优先保证真实写入、真实读取和真实联动成立
2. 这层应优先被视为 D1 正式数据库主线
3. 未来如果继续停留在 D1，这层直接保留
4. 未来如果迁移到更正式的数据库，这层也应作为最高优先迁移层

## 3.2 可重建内容层

当前定义为：

1. `rss_sources`
2. `rss_articles`
3. `hot_topics`
4. `opportunities`
5. `briefings`
6. `user_profiles`

这层的规则是：

1. 当前允许存在于 D1，但不要求本轮完成全量正式灌装
2. 这层本质上更接近“可重新抓取、可重新生成、可重新回填”的内容与派生层
3. 后续进入 D1 时，优先走分批灌装、按需回填，而不是把它们视为当前阶段必须一次性全量同步的正式主库
4. 即便未来迁移数据库，这层的迁移优先级也低于正式主库层

## 3.3 当前不切范围

当前明确不直接切成正式云端主读的范围：

1. `dashboard/today`
2. `content/by-ref`
3. `reports/*`
4. 与上述聚合读侧直接相关的“全量内容灌装”

原因是：

1. 它们是聚合读侧，不是单表事实层
2. 它们依赖统一内容引用、兼容 `content_ref`、多表聚合和规则加工
3. 如果现在一起切，会把“数据库切换”和“内容层正式化”混成一件事

---

## 4. 迁移规则

### 4.1 当前迁移优先级

固定顺序为：

1. 正式主库层先成立
2. 可重建内容层再分批进入
3. 最后再推进 `Today / content / reports` 这类聚合读侧正式数据库切换

### 4.2 当前 seed 规则

后续 seed 不再默认只保留一份“整包远端文件”作为唯一执行入口。

应改成至少两层：

1. 正式主库 seed
2. 可重建内容 seed

必要时，可重建内容层继续拆为：

1. `rss_sources`
2. `rss_articles`
3. `hot_topics`
4. `opportunities`
5. `briefings / user_profiles`

补充规则：

`rss_articles` 当前必须被视为“可重建内容层中的重包”，不得再与其他内容表捆绑成一个默认执行单元。

### 4.3 当前 apply 规则

后续远端 apply 默认遵循：

1. migration 独立执行
2. 正式主库 seed 独立执行
3. 可重建内容 seed 独立执行
4. 禁止再把“migration + 全量内容 seed”视为一条不可拆的大动作

补充规则：

`rss_articles` 后续进入远端 apply 时，应默认遵循“分片执行”，而不是再以单文件整包执行作为默认路径。

### 4.5 `rss_articles` 分片规则（设计已确定，暂不执行）

当前结论：

1. 需要先设计 `rss_articles` 的分片规则
2. 不要求当前立刻生成 `part` 文件
3. 这项设计应被视为 `article` 详情链路切换前的前置项

#### 4.5.1 分片目标

1. 避免 `seed.content.rss_articles.sql` 再次以 1000+ statements 的单包方式远端 apply
2. 保持 `rss_articles` 表结构不变，只拆执行批次，不改数据口径
3. 让后续即便继续停留在 D1，也能拥有稳定、可重复、可回放的内容灌装规则

#### 4.5.2 分片边界

1. 只按同一张表 `rss_articles` 分片
2. 不与 `rss_sources / hot_topics / opportunities` 混片
3. 不把 `rss_articles` 分片逻辑扩展成其他内容表必须同步遵循的默认规则

#### 4.5.3 分片方式

1. 按固定批次数量分片
2. 不按人工临时估算文件大小拆分
3. 每个分片文件内保持 SQL 结构一致，只是行数不同

#### 4.5.4 文件命名规则

后续如执行分片，文件命名应固定为：

1. `seed.content.rss_articles.part01.sql`
2. `seed.content.rss_articles.part02.sql`
3. `seed.content.rss_articles.part03.sql`
4. 依此类推

#### 4.5.5 执行规则

后续如执行分片，默认执行顺序应为：

1. 按 `part01 -> part02 -> part03 ...` 顺序执行
2. 允许单独执行某一片
3. 允许从某一片重新继续
4. 不再把“全部 part 一次性串成单条大命令”视为默认执行方式

#### 4.5.6 迁移规则

后续如执行分片，仍需保持：

1. `rss_articles` 仍属于可重建内容层
2. 分片只是执行策略，不改变其在架构中的层级归属
3. 即便未来迁移到更正式的数据库，这套“同表分片、顺序执行、可单片重放”的规则仍应可复用

### 4.4 当前迁移可逆规则

无论后续 D1 是否继续作为长期正式数据库，当前都必须保留这些可逆条件：

1. 接口契约不直接暴露 D1 特性
2. 行为事实访问继续走统一存储层
3. `content_ref / by-ref` 读契约保持稳定
4. migration 与 seed/backfill 分开维护
5. 可重建内容层默认允许重新导出、重新生成、重新灌装

---

## 5. 当前执行解释

采用方案 C 后，当前正确解释是：

1. 第一批行为事实接口已经可以正式切入 D1
2. 当前不要求内容层也同时完成正式数据库切换
3. `seed.remote.sql` 的整包超时，不应再被理解为“整个 D1 切换方案失败”
4. 当前真正该做的是把整包 seed 拆成分层 seed，并优先保证正式主库层 seed 可稳定执行

---

## 6. 方案 C 相比方案 A / B 的位置

### 6.1 不是纯方案 A

因为它不把当前工作只视为临时过渡，而是承认 D1 具备成为正式数据库的资格。

### 6.2 也不是纯方案 B

因为它不要求当前就把内容层和全量灌装一起长期化。

### 6.3 方案 C 的实际优势

1. 保留方案 A 的轻量、低风险、贴合当前主线
2. 保留方案 B 的长期数据库建设方向
3. 让后续最好的情况变成：
   - D1 直接停留为正式数据库
4. 同时也保留后续迁移空间：
   - 即便未来要迁，也不会因为当前把内容层做死而付出过重代价

---

## 7. 当前执行边界

基于本方案，当前允许立即执行的事项是：

1. 拆分 seed
2. 调整 apply 入口
3. 优先保证正式主库 seed 可稳定 apply
4. 为 `Today / content / reports` 的后续切换建立前置边界

当前不应直接执行的事项是：

1. 把聚合读侧直接宣布为已正式切到云端
2. 直接重做统一 `contents` 实体化
3. 在没有单独设计的情况下，把可重建内容层一次性做成长期全量强依赖

---

## 8. 本轮后续执行顺序

1. 先把 seed 按“正式主库层 / 可重建内容层”拆开
2. 让正式主库 seed 可单独生成、单独 apply、单独验证
3. 再决定可重建内容层是否继续细分
4. 最后再进入 `Today / content / reports` 的正式数据库切换前置收束
