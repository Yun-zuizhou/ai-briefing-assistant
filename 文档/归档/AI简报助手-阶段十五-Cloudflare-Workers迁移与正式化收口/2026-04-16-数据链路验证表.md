## 2. 当前产品主链路

项目总纲中的主线是：

`今日信息 -> 行动转化 -> 自我记录 -> 周期回顾`

这条主线在当前仓库中已经能拆成 4 段正式链路。

## 2.1 主链路 A：兴趣驱动的 Today 内容聚合

### 链路定义

`用户兴趣 -> 内容源匹配 -> Today 聚合结果 -> 简报缓存/展示`

### 当前数据链

1. 用户兴趣来源正式从 `user_interests` 读取
2. 若没有正式兴趣，则进入空兴趣正式态
3. `users.interests` 仅保留影子同步职责，不再参与正式主读
4. 从 `hot_topics`、`opportunities` 读取候选内容
5. 使用内容加工结果表 `hot_topic_processing_results` 辅助标准化
6. 生成 `recommended_for_you / worth_knowing / worth_acting`
7. 将结果以 `briefings.payload` 的形式缓存
8. 前端 Today 页按 `content_ref` 进入内容详情

### 涉及表

- `user_interests`
- `hot_topics`
- `hot_topic_processing_results`
- `opportunities`
- `briefings`

### 当前验证状态

| 检查项 | 状态 | 依据 |
| --- | --- | --- |
| Today 已从兴趣事实层读取兴趣 | 已完成 | `app/api/v1/dashboard.py`、`tests/test_dashboard_today.py` |
| Today 已输出 `content_ref` | 已完成 | `content_ref` 测试与前端跳转契约 |
| Today 已具备最小排序和加工字段 | 已完成 | `ranking_score / processing_stage / quality_score` 已进入响应 |
| Today 已真正影响推荐结果 | 已完成 | `tests/test_api_mainline.py::test_chat_execute_interest_flow_updates_dashboard_recommendations` |
| Today 是否已有正式统一 `contents` 表 | 未完成 | 当前仍依赖多源拼装 |

### 这条链路要成立的必要条件

| 条件 | 当前情况 | 备注 |
| --- | --- | --- |
| 兴趣必须有正式存储位 | 已有 | `user_interests` |
| 内容源必须可读 | 已有 | `hot_topics` / `opportunities` |
| 内容必须能统一引用 | 已有 | `content_ref` |
| 内容加工结果必须可复用 | 已有 | 热点/文章结果表 |
| 推荐权重体系要足够稳定 | 部分完成 | 当前仍是规则排序，不是正式推荐系统 |

## 2.2 主链路 B：对话输入 -> 意图识别 -> 兴趣/待办/记录写入

### 链路定义

`Chat 输入 -> recognize -> execute -> 写入结构化对象 -> 写入 history/chat_messages -> 影响 Today/日志/待办`

### 示例链路

`对话输入“今天特别关注了AI，以后希望得到更多关于AI的咨询”`

-> 系统识别 `add_interest`

-> 从文本中抽取 `AI`

-> 写入 `user_interests`

-> 同步更新旧字段 `users.interests`（当前仅保留影子同步职责）

-> 写入 `history_entries(event_type=interest_added)`

-> 写入 `chat_messages`

-> Today 再读取兴趣时，把 `AI` 作为正式关注项纳入推荐

### 当前可支撑的主要意图

| 意图 | 写入对象 | 影响页面/链路 |
| --- | --- | --- |
| `add_interest` | `user_interests`，并同步影子字段 `users.interests` | Today、Reports、Profile |
| `remove_interest` | `user_interests.status=inactive` | Today、Reports、Profile |
| `create_todo` | `todos` | 待办页、行动页、报告 |
| `record_thought` | `notes` | 日志页、成长页、报告 |
| `fragmented_thought` | `notes(tags)` | 日志页、成长页、报告 |
| `chat_only` | 仅 `chat_messages` | 不写结构化业务对象 |
| `reclassify` | `todo/note/history` 间重新解释 | Chat 纠偏链路 |

### 涉及表

- `chat_sessions`
- `chat_messages`
- `user_interests`
- `users`
- `todos`
- `notes`
- `history_entries`

### 当前验证状态

## 2.3 主链路 C：内容详情 -> 行动转化

### 链路定义

`Today/热点/收藏/报告热点 -> content/by-ref -> 详情页 -> 转成待办 / 跳 Chat 动作上下文`

### 当前数据链

1. 页面持有 `content_ref`
2. 统一走 `/api/v1/content/by-ref`
3. 后端根据 `hot_topic/article/opportunity` 分流读取源表
4. 对热点/文章优先读取加工结果表
5. 返回正文、标签、相关推荐、详情状态
6. 若是 `opportunity`，前端详情页可带 `presetInput + sourceContentRef + sourceTitle` 跳到 Chat
7. Chat 再将机会文本转成 `todos`
8. 后续可以进入 `opportunity_follows` 或 `history_entries`

### 涉及表

- `hot_topics`
- `hot_topic_processing_results`
- `rss_articles`
- `article_processing_results`
- `opportunities`
- `todos`
- `history_entries`
- `chat_sessions`
- `chat_messages`

### 当前验证状态

| 检查项 | 状态 | 依据 |
| --- | --- | --- |
| 统一内容详情 API 已建立 | 已完成 | `/api/v1/content/by-ref` |
| 支持热点 / 文章 / 机会三类对象 | 已完成 | 代码与测试均覆盖 |
| 支持相关推荐 | 已完成 | 内容详情测试与契约文档存在 |
| 机会详情页可进入行动闭环 | 已完成最小版 | 通过跳转 Chat 完成 |
| 是否已直接在详情页内写 `opportunity_follows` | 未明确完成 | 当前主闭环更像“转待办”，不是直接 follow |

### 这条链路要成立的必要条件

| 条件 | 当前情况 | 备注 |
| --- | --- | --- |
| 统一引用键 | 已有 | `content_ref` |
| 统一详情读取接口 | 已有 | `content/by-ref` |
| 详情必须能返回正文和相关推荐 | 已有 | 过渡态但可用 |
| 至少一种行动入口 | 已有 | 转 Chat，再转待办 |
| 行动结果需要回流到行为层 | 已有 | `todos/history/chat_messages` |

| 简报历史回看 | Today/briefing 结果 -> `briefings` -> 历史简报 | `briefings` | 已成立最小版 | 真正晨报/晚报生产链不完整 |

## 5. 当前最关键的实现差距

这些不是“没有任何东西”，而是“已经有基础，但要说功能真正成立还差最后一段”。

## 5.1 缺口一：统一内容事实层仍未正式实体化

当前情况：

- 已有统一引用键 `content_ref`
- 已有统一详情接口 `content/by-ref`
- 但底层仍是 `hot_topics + rss_articles + opportunities` 分源读取

影响：

- 内容去重、统一排序、统一推荐、统一收藏解释成本较高
- 报告和画像很难直接基于“内容事实层”做解释

建议结论：

- 目前可以说“统一读侧已成立”
- 还不能说“统一内容事实层已成立”

## 5.2 缺口二：推送配置已存在，但正式调度与投递链未形成闭环

当前情况：

- `user_settings` 已有晨间/晚间时间
- 产品文档明确要求晨报/晚报/即时简报
- 但未看到正式的调度实例表、消息投递表、执行日志表

影响：

- “设置了 8 点推送”这件事，在数据上有配置，但在执行上缺少可验证链路

建议结论：

- 可以说“推送偏好设置已成立”
- 不应说“定时推送链路已完整成立”

## 5.3 缺口三：推荐和画像仍以规则型聚合为主

当前情况：

- Today 推荐已能用兴趣命中 + 质量分 + 热度做规则排序
- 画像已能基于记录/收藏/兴趣生成摘要

影响：

- 已可用，但解释性与稳定性高度依赖规则
- 还不是正式个性化推荐系统

建议结论：

- 这是“可用的 MVP 级主链”
- 不是“成熟的智能推荐系统”

## 5.4 缺口四：机会链路还缺最终结果沉淀

当前情况：

- 机会可进入详情
- 可转成待办
- 也有 `opportunity_follows`

但还缺：

- 申请结果 / 反馈结果 / 结果状态表
- 从机会到待办到完成结果的强绑定

建议结论：

- 现在成立的是“机会 -> 行动准备”
- 还不是“机会 -> 执行 -> 结果回收”的完整闭环

## 7. 总结

## 7.1 已经成立的核心链路

1. `兴趣配置/对话新增兴趣 -> Today 推荐变化`
2. `对话创建待办 -> 待办页/行动页/报告`
3. `对话记录想法 -> 日志/成长页/报告`
4. `统一内容引用 -> 内容详情 -> 相关推荐`
5. `行为沉淀 -> 周报/月报/年报 -> 历史报告回看`

## 7.2 已成立但仍属过渡态的链路

1. Today 推荐排序
2. 用户画像生成
3. 机会详情转行动
4. 简报结果缓存

## 7.3 还不能算完全成立的链路

1. 正式推送调度与投递链
2. 统一 `contents` 事实表
3. 机会从发现到最终结果的完整闭环
4. 更成熟的个性化推荐系统

## 7.4 最适合继续补的优先顺序

1. 把“推送设置 -> 调度执行 -> 简报生成记录”补成可验证链路
2. 把“机会 -> 待办 -> 跟进 -> 结果”补成强绑定链路
3. 决定是否正式建设统一 `contents` 表
4. 在现有规则排序之上补更正式的推荐权重与解释层
