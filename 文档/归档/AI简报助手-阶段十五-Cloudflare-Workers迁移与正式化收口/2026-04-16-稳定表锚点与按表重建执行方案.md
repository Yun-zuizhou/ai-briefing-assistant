## 2. 本轮真正按表重建的范围

当前只把 `reports` 定义为“本轮按表重建对象”。

原因：

1. `reports` 承担正式周期结果快照职责
2. 当前表缺少逻辑周期唯一性约束
3. 现有写入逻辑是“先查再更新”，如果远端已有重复数据，后续只会持续叠加
4. `reports` 没有被其他表外键引用，适合单表重建

## 3. 本轮只做 in-place 硬化、不做重建的表

### `briefings`

当前问题不是主键结构错误，而是缺少稳定元字段。

因此本轮采取：

1. 保留原表
2. 用 migration 补 `status`
3. 用 migration 补 `generated_at`
4. 不做整表重建

### `user_profiles`

当前 D1 正式列名已经是：

1. `summary`
2. `version`
3. `profile_data`
4. `generated_at`

漂移主要在 Python 本地 ORM，不在正式 D1 表。

因此本轮：

1. 不重建 `user_profiles`
2. 把它视为稳定结果快照表
3. 后续若要统一命名，先统一代码层，再决定是否重建

## 4. 执行原则

### 4.1 schema 真源原则

1. `infra/cloudflare/d1/migrations` 是正式 schema 真源
2. 远端库是否健康，不以“apply 成功”判断，而以“stable anchor audit 通过”判断

### 4.2 重建原则

1. 能用 additive hardening 解决的，不做整表重建
2. 需要 dedupe + 唯一性约束才能真正稳定的，才做整表重建
3. 有外键依赖的表，优先避免直接重建
