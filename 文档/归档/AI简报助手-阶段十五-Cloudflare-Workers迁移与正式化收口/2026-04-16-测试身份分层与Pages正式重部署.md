# 2026-04-16-测试身份分层与Pages正式重部署

- 文档状态：进行中
- 文档定位：承接“测试邮箱分层、用户身份数据重排、Pages 最新前端重部署”这一轮结果级变化
- 更新时间：2026-04-16

## 1. 为什么切到这条子线

1. 生产前端此前虽然已部署到 Pages，但线上 bundle 不是当前 `apps/web` 的最新构建。
2. 云端 D1 先前只有 `test@example.com` 一套用户事实数据，不满足“切换不同测试邮箱看到不同数据内容”的验证需求。
3. 当前登录链路以邮箱透传身份为主，如果不先把测试身份和事实数据分层，后续人工验收无法稳定复现“空白用户 / 主测试用户 / 展示用户”三种状态。

## 2. 本轮真实执行事实

### 2.1 前端登录与测试账号切换

1. 已在 `apps/web/src/pages/LoginPage.tsx` 增加测试邮箱快速切换入口。
2. 当前登录页内置 3 类测试身份：
   - `新用户新邮箱`：每次生成唯一邮箱，首次登录自动创建空白用户。
   - `test@example.com`：主测试账号，承接完整主链数据。
   - `show@example.com`：展示账号，承接偏前端 / 设计 / 远程机会的数据口径。
3. 当前登录成功后：
   - 登录现有邮箱：直接进入 `/today`
   - 新注册：仍进入 `/interest-config`
4. `InterestConfigPage` 完成后跳转已统一改为 `/today`，不再落到旧的 `/home`。

### 2.2 用户身份数据重排

1. 已新增远端同步脚本：`tools/scripts/sync_test_identities.py`
2. 脚本职责：
   - 确保 `test@example.com` 维持主测试账号身份
   - 为 `show@example.com` 创建并重置独立展示数据
   - 保持“新用户新邮箱”策略为首次登录自动建空白用户，而不是提前 seed 固定邮箱
3. 本轮执行后，云端 D1 当前测试身份分层如下：
   - `test@example.com`
     - `interests=3`
     - `favorites=5`

     - `todos=17`
     - `history=60`
     - `reports=3`
     - `briefings=1`
     - `follows=1`
     - `sessions=1`
   - `show@example.com`
     - `interests=3`
     - `favorites=3`
     - `notes=3`
     - `todos=3`
     - `history=4`
     - `reports=3`
     - `briefings=1`
     - `follows=1`
     - `sessions=1`
4. `show@example.com` 当前内容口径刻意偏向：
   - 前端
   - 设计
   - 远程机会
5. `test@example.com` 继续保留为：
   - 主链完整验证账号
   - Chat / Interests / Today / Reports 联调账号

## 3. 本轮边界

1. 这轮不改变正式鉴权方案，当前仍是“按邮箱透传并在 Workers 侧按邮箱落用户”的最小身份口径。
2. 这轮不新增第四个固定测试邮箱，只固定维护：
   - `test@example.com`
   - `show@example.com`
   - 动态新邮箱策略
3. 这轮不把“新用户新邮箱”预先 seed 到 D1，而是保留首次登录自动建空白用户的真实首登体验。
4. 这轮主要解决：
   - 测试身份分层
   - 数据事实分层
   - 生产前端 bundle 与当前代码一致

## 4. 当前阶段性结论

1. 生产 Pages 已不再停留在旧 bundle，当前线上前端已经与本地 `apps/web` 最新代码口径对齐。
2. 当前项目已具备 3 类可稳定复现的测试身份状态：
   - 空白新用户态
   - 完整主测态

3. 后续人工验收、截图级验证、答辩演示、账号切换测试，已经有稳定的事实基础，不再需要依赖单一 `test@example.com`。
