# 2026-04-16 数据库接口主线与云端前端补齐决策

- 记录日期：2026-04-16
- 记录类型：进行中主线决策文档
- 目标：为当前仓库确定一套可执行的“数据库 -> Workers -> 前端页面”正式链路标准，并补齐仍停留在本地的正式前端能力。

## 1. 当前事实

当前仓库已经明确：

1. 正式在线后端是 `apps/edge-worker`
2. 正式前端是 `apps/web`
3. 正式数据库真源是 `infra/cloudflare/d1/migrations`
4. Python `app/` 继续保留为离线/参考/验证端，而不是正式在线主后端

因此，这一轮不再讨论“谁是主后端”，而是讨论：

1. D1 表结构如何成为唯一事实真源
2. Workers 如何把事实层投影成稳定接口
3. 前端正式页面哪些已经上云，哪些仍是假云端/本地页

## 2. 正式标准

### 2.1 Schema 标准

1. D1 migration 是正式 schema 单一真源
2. Workers 路由和服务层只允许查询 migration 中真实存在的列
3. Python ORM 若继续保留，只能跟随 D1 对齐，不允许再自行演化另一套正式字段语义

### 2.2 分层标准

当前正式层次收口为三层：

1. 事实层
   - `users`
   - `user_settings`
   - `user_interests`
   - `rss_sources`
   - `rss_articles`
   - `hot_topics`
   - `opportunities`
   - `todos`
   - `favorites`
   - `notes`
   - `history_entries`
   - `opportunity_follows`
   - `chat_sessions`
   - `chat_messages`

2. 结果/快照层
   - `article_processing_results`
   - `hot_topic_processing_results`
   - `briefings`
   - `reports`
   - `user_profiles`

3. 支撑链/观测层
   - `briefing_schedules`
   - `briefing_dispatch_logs`
   - `opportunity_execution_results`
   - `ingestion_runs`
   - `ai_processing_runs`
   - `operation_logs`
   - `replay_tasks`
   - `summary_generation_tasks`
   - `user_sessions`
   - `feedback_submissions`

### 2.3 接口标准

1. 前端页面只消费 Workers 投影后的接口对象，不直接假设底表结构
2. 跨内容对象统一引用继续使用 `content_ref = content_type:id`
3. `content_ref` 是接口层稳定引用，不要求每个原始表都物理存这一列
4. 正式接口返回 shape 必须与 `packages/contracts` 或前端显式 API 类型一致

### 2.4 前端标准

正式前端页面分三类：

1. 必须上云并走 D1 的正式页
   - `TodayPage`
   - `HotTopicsPage`
   - `ArticlePage`
   - `ActionsPage`
   - `JournalPage`
   - `GrowthPage`
   - `MyPage`
   - `CollectionsPage`
   - `ProfilePage`
   - `HistoryLogsPage`
   - `HistoryBriefPage`
   - `WeeklyReportPage`
   - `MonthlyReportPage`
   - `AnnualReportPage`
   - `InterestConfigPage`
   - `NotificationSettingsPage`
   - `ChatPage`

2. 可以保留静态/半静态，但不能伪装成正式事实页
   - `AboutPage`
   - `WelcomePage`
   - `LoginPage`

3. 明确保留为 demo，不混入正式主线
   - `/demo/*`

## 3. 云端补齐判断

当前正式前端里最明显还没真正上云的不是主内容页，而是：

1. `HelpFeedbackPage`
   - 当前只有本地 `console.log`
   - 没有云端接口
   - 没有数据库落地

2. `SettingsPage`
   - 推送时间已走云端
   - 声音/震动开关仍停留在本地切换

3. 文案层认知仍有旧口径
   - 仍有“数据存储在本地，后续支持云端同步”这类旧描述

## 4. 三轮思考

### 第一轮：如果不修数据库-接口错位会怎么样？更好的做法是什么？

方案 A：继续允许 Workers 直接写“猜测字段名”的 SQL，靠页面出问题后再修。

方案 B：前端单独做兼容映射，把后端错误 shape 全部吞掉。

方案 C：把 D1 migration 明确为唯一真源，Workers 路由按真实列重写，前端只消费对齐后的正式返回。

结论：选 C。

原因：

1. 方案 A 会持续产生“接口存在但实际查不到数据”的隐性故障
2. 方案 B 会把 schema 漂移从后端转移到前端，长期更难维护
3. 只有方案 C 能让“数据库 -> 接口 -> 页面”重新变成单向链路

### 第二轮：如果重新引入统一 `contents` 大总表会怎么样？更好的做法是什么？

方案 A：重新回到一个大一统 `contents` 表，把热点、文章、机会全塞进去。

方案 B：继续保持 `rss_articles / hot_topics / opportunities` 三张来源事实表，同时在接口层统一 `content_ref`。

方案 C：维持三张来源事实表 + 结果层投影，不在这一轮强行重做总表。

结论：当前选 C。

原因：

1. 方案 A 理论上更统一，但会把当前已落地的 D1 主链整体推翻
2. 当前真正阻塞链路的是“接口与事实层失配”，不是“少一个总表”
3. 先把 `content_ref + projection` 稳住，比现在就重做总表更安全

### 第三轮：如果把所有前端未上云项都一次性云端化会怎么样？更好的做法是什么？

方案 A：保持现状，继续允许正式页面里混本地提交和假反馈。

方案 B：把所有设置项、所有静态页、所有 demo 全部一起云端化。

方案 C：只补齐已经进入正式主线、且已有明确业务意义的云端缺口：
   - 反馈提交入库
   - 声音/震动正式设置入库
   - 过时本地化文案清退

结论：选 C。

原因：

1. 方案 A 会让正式页面继续出现“看起来能用，实际没落库”的假链路
2. 方案 B 范围过大，会把主题/语言等尚无稳定 schema 的能力强行拉进数据库
3. 方案 C 能在控制风险的前提下，先让正式页不再说谎

## 5. 最终决策

本轮正式决策如下：

1. 数据库层
   - 以 `infra/cloudflare/d1/migrations` 为唯一 schema 真源
   - 新增 `feedback_submissions` 作为帮助反馈页的正式落库表

2. 接口层
   - 修复 Workers 对不存在列的查询
   - 修复 Chat 会话接口与前端契约 shape 不一致的问题
   - 修复 Chat 的待确认分流，使“未确认时不写业务事实”重新成立

3. 前端层
   - `HelpFeedbackPage` 改为正式调用云端接口并写入 D1
   - `SettingsPage` 的声音/震动切换改为正式同步到 `user_settings`
   - 清理“数据仍只在本地”的过时文案

## 6. 本轮不做的事

1. 不在这一轮强制重建统一 `contents` 总表
2. 不在这一轮把 demo 页面改成正式业务页
3. 不在这一轮把主题风格、语言切换强行写入数据库
4. 不在这一轮让 Python 重新成为正式在线主后端

## 7. 验收口径

本轮完成后至少应满足：

1. 正式页面不再依赖不存在的数据库列
2. Chat 识别/确认/会话读取的返回 shape 与前端一致
3. 反馈页提交后有真实数据库落点
4. 设置页至少有一组此前本地化的正式设置切到云端
5. 旧“本地存储才是正式事实”的页面文案被清退
