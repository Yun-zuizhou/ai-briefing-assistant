## 1. 这份文档解决什么问题

这份文档不讨论“现在要不要直接物理迁目录”，那一题已经由 [2026-04-16-app迁移安全评估.md](/E:/python/杂谈-想法记录与实践/文档/进行中/待二次判断/2026-04-16-app迁移安全评估.md) 给出了结论：`现在不能直接硬迁，只能先做兼容层、分批切换、最后物理迁移。`

1. 在当前复杂依赖面下，`app/` 现在应该按什么结构继续作为唯一 Python 主包稳定存在。
2. 后续新增、删除、修改接口和功能时，应该往哪一层放，避免继续把过渡逻辑越堆越大。
3. 当前仓库里哪些文档可以参考，哪些文档已经不适合作为正式指导。
4. 如何按“容易 -> 困难”的顺序，把当前项目中的 `过渡 / 兜底 / 虚拟 / 假成功 / 假趋势` 逻辑逐步收敛成正式态。
5. 三轮执行后，最终应当做出什么决策，才能保证目标不是“看起来推进了”，而是真的落成。

## 2. 当前真实复杂度锚点

结合当前代码与已完成的迁移评估，这里只承认以下现实：

1. `app/` 当前仍是 Python 运行、验证、脚本依赖、文档锚点共同引用的主包。
2. 当前直接依赖 `app.*` 的文件共有 `87` 个，其中：
   - `5` 个测试文件
   - `12` 个工具脚本
   - `70` 个 `app/` 内部文件
3. `dev` 入口仍直接绑定 `uvicorn app.main:app`。
4. 根路径推导、`.env` 读取、D1 wrangler 读取、本地 SQLite 位置，都仍以内置“仓库根 + 根目录 app”口径为主。
5. `app/` 目录当前至少包含：
   - `18` 个 `app/api/v1` 文件
   - `28` 个 `app/services` 顶层文件
   - `21` 个 `app/models` 文件
   - `5` 个 `app/crawler` 顶层文件
6. 当前仓库内仍存在较多过渡/兜底逻辑：
   - `virtual_data` 体系
   - `fallback` 响应
   - `transitional` 状态标记
   - `Math.random()` 生成的趋势值
   - 前端本地 demo/bootstrap 状态

一句话判断：

`app/` 现在不是“历史包袱”，而是“仍在承担主责任的正式 Python 主包”；因此当前最重要的不是挪目录，而是先把它整理成一套能长期稳定演进的正式态内部结构。

## 4. 当前 app 的正式职责划分

在不做物理迁移的前提下，当前根目录 `app/` 的正式职责应被明确定义为 5 层：

1. API 协议层：`app/api/v1/*`
   - 负责路由、请求解析、响应 DTO、最薄的编排
   - 不再继续堆放复杂规则和大量 fallback

2. 领域服务层：`app/services/*`
   - 负责领域规则、聚合、读写编排
   - 这是今后最主要的业务承载层

3. 数据访问层：`app/services/d1_*`、`app/database.py`、后续可抽出的 store/repository
   - 负责 D1 / SQLite 数据读写
   - 不直接承担页面逻辑和产品文案

4. 数据模型层：`app/models/*`
   - 负责 SQLAlchemy 模型与数据库对象表达
   - 不混入接口 DTO 和聚合结果逻辑

5. 采集与离线处理层：`app/crawler/*`
   - 负责 RSS / 热点 / 离线采集 / 调度
   - 不直接给页面返回“伪正式内容”

补充说明：

`app/services/data/virtual_data.py` 不属于正式层，而属于待清退兼容层。`

## 6. 可稳定增删改的接口及功能目录指导

后续只要涉及接口或功能改动，都先判断它属于哪一类，再决定写入目录。

### 6.1 接口分层规则

#### A. 页面聚合读侧

适用对象：

1. `dashboard/today`
2. `actions/overview`
3. `content/by-ref`
4. `preferences/growth-overview`
5. `reports/weekly|monthly|annual`

目录指导：

1. 路由保留在 `app/api/v1/*.py`
2. 聚合和组装逻辑下沉到 `app/services/<domain>/`
3. DTO 保留在 `page_schemas.py` 或后续拆分出的 `schemas/`

禁止：

1. 在路由里直接写大段排序、推荐、画像、报告拼装
2. 在路由里直接写大量 fallback 分支

#### B. 对象 CRUD

适用对象：

1. `todos`
2. `notes`
3. `favorites`
4. `history`
5. `user_settings`
6. `user_interests`

目录指导：

1. 路由层只处理参数与响应
2. 真实读写逻辑统一下沉到 store/service
3. 写入时明确副作用：是否写 `history_entries`、是否影响 Today / Growth / Reports

禁止：

1. CRUD 路由顺手返回页面聚合结构
2. CRUD 路由失败时自动改走虚拟数据成功返回

#### C. 命令动作接口

适用对象：

1. `chat/recognize`
2. `chat/execute`
3. `chat/reclassify`
4. `actions/check-in`
5. 后续的调度执行命令

目录指导：

1. 命令识别和命令执行必须分开
2. 命令执行结果必须真实落业务表或显式失败
3. 命令失败不允许再返回“虚拟对象已创建”

#### D. 系统与支撑链接口

适用对象：

1. `system/*`
2. `rss/*`
3. `api_config/*`

目录指导：

1. 系统可观测接口与真实执行接口分开
2. 配置接口不直接承担调度执行
3. 采集接口不再承担页面 mock 生产

### 6.2 新功能放置决策表

| 新功能类型 | 应放位置 | 不应放位置 |
| --- | --- | --- |
| 页面聚合读侧 | `app/services/<domain>/assembler or service` + `app/api/v1` 路由薄编排 | 直接堆在 route |
| 单对象 CRUD | `stores` / `d1_*store` / 对应 service | `dashboard.py` / `reports.py` 等聚合接口 |
| Chat 动作命令 | `app/services/chat/*` | `virtual_data.py` |
| 报告构建 | `app/services/reports/*` | `reports.py` 大函数继续膨胀 |
| 画像构建 | `app/services/profile/*` | `preferences.py` 直接堆所有逻辑 |
| 调度执行 | `app/services/system/*` | `preferences.py` 假装写设置就等于执行 |

## 7. 当前过渡/兜底逻辑的真实分布

当前需要清退到正式态的逻辑，可以分成 6 类：

### 7.1 虚拟数据兜底

位置：

1. `app/services/data/virtual_data.py`
2. `app/api/v1/todos.py`
3. `app/api/v1/notes.py`
4. `app/api/v1/favorites.py`
5. `app/api/v1/history.py`
6. `app/api/v1/chat.py`
7. `app/api/v1/dashboard.py`

风险：

1. 接口失败时仍可能返回“看起来成功”的虚拟结果
2. 用户会误以为数据已正式落库
3. 会干扰 Today / Reports / Growth 的真实性判断

### 7.2 过渡态标记未收敛

位置：

1. `processing_stage = transitional`
2. `detail_state = transitional`
3. Today / content / reports 的过渡态文案

风险：

1. 虽然承认当前仍在过渡，但没有明确退出条件
2. 容易让“过渡态”长期常驻

### 7.3 假趋势 / 合成值

位置：

1. `apps/edge-worker/src/routes/reports.ts` 的 `Math.random()`
2. 前端 HotTopics/报表中的随机涨跌显示
3. `AppContext` 中 mock 时间与本地 bootstrap 数据

风险：

1. 展示看起来丰富，但不是真实数据
2. 会污染对“正式态”的判断

### 7.4 前端 fallback 内容对象

位置：

1. `WeeklyReportPage.tsx`
2. `MonthlyReportPage.tsx`
3. `AnnualReportPage.tsx`
4. 部分详情页和组件内 fallback 文案

风险：

1. 页面即使后端异常，也会显示“完整但虚构”的最低版本
2. 用户难以区分“数据为空”和“接口失败”

### 7.5 采集层 fallback 数据

位置：

1. `app/crawler/platform_crawlers.py` 的 `_get_fallback_data()`

风险：

1. 采集失败会生成假热点
2. 这类假内容一旦进入主链，就会污染 Today 和报告

### 7.6 假成功写接口

当前最敏感的位置：

1. Python `chat` 相关虚拟执行兜底
2. 前端本地聊天缓存与 demo bootstrap
3. system 支撑链接口目前更多是事实表写入，还不是正式执行器

## 8. 目标：把所有过渡/兜底逻辑转换到正式态

唯一例外：

`联网搜索 -> 生成摘要` 这个步骤暂时允许不真正生成，但不允许再返回伪摘要或伪加工结果。`

它的正式态定义应是：

1. 数据接口已确定。
2. 任务状态已确定。
3. 缺 provider 时返回 `pending_provider` / `not_configured` / `failed`。
4. 页面按“未生成”展示空状态或待生成状态。
5. 后续只补 provider 实现，不再改数据契约。

也就是说，这个例外允许“功能未完成”，但不允许“用假数据伪装成功”。

## 9. 按容易与否的正式化方案

### 9.1 第一梯队：最容易，必须先收

这些属于“低风险、高收益、马上就该做”的项。

1. 停止扩张 `virtual_data.py`
2. 把 `todos / notes / favorites / history` 的虚拟成功兜底改成：
   - 正式数据
   - 或显式失败
   - 或空数据
3. 把前端报告页的 fallback 完整对象改成：
   - 正式空状态 DTO
   - 接口失败提示
   - 不再渲染伪报告内容
4. 把 HotTopics / Reports 中的随机涨跌显示改成：
   - 真实值
   - 或未提供状态
5. 明确 `api_config` 是否仍属于正式前台链路；若不是，则降级或隔离

目标：

`先把“假成功、假内容、假趋势”从用户最直接可见的页面层清出去。`

### 9.2 第二梯队：中等难度，收主链真实度

1. 统一 `hot_topics / opportunities` 的正式接口口径，避免前端仍走旧路由名
2. 把 Today、Content、Reports 的 `transitional` 收敛成“有退出条件的过渡态”
3. 把 Chat 的 execute/reclassify 完全收成“真实写表或明确失败”
4. 把前端 `AppContext` 的 demo/bootstrap 逻辑与正式态隔离
5. 把 `Growth / My / HistoryBrief` 统一到正式接口空状态，不再依赖前端本地补齐

目标：

`把“主闭环可跑”提升成“主闭环结果真实可信”。`

### 9.3 第三梯队：最难，但决定是否能彻底进入正式态

1. 采集层 `_get_fallback_data()` 改成：
   - seed 数据只允许出现在工具链 / demo 同步
   - 正式采集失败只写失败状态和日志，不产出假热点
2. system 支撑链从“事实表 + 可观测接口”升级到“真实执行链”
3. 为“联网搜索 -> 生成摘要”定义正式任务接口：
   - 请求对象
   - 结果对象
   - 状态机
   - provider 缺失状态
4. 清掉 `chat.py` / `reports.py` / `dashboard.py` 中仍然过大的聚合逻辑

目标：

`让正式态不再依赖任何“假执行、假采集、假生成”来维持页面完整性。`

## 10. 三轮执行策略

### 第一轮：建立正式态约束，先收用户可见假结果

本轮目标：

1. 确立本文为 `app` 正式态指导基线
2. 禁止新增虚拟数据职责
3. 清理最直接影响用户认知的假内容和假趋势

本轮动作：

1. 后端：
   - `todos / notes / favorites / history` 不再以 `virtual_data` 作为默认成功兜底
   - Chat 保留识别 fallback，但执行写入不再回退虚拟对象
2. 前端：
   - 周报/月报/年报改为空状态渲染，不再用完整 fallback 报告对象伪装成功
   - HotTopics/报告页去掉随机涨跌
3. 文档：
   - 把本文确立为 `app` 正式态基线

本轮验收：

1. 页面失败时显示明确空状态/失败状态，而不是伪内容
2. 不再出现“接口失败但页面看起来像成功”的情况

### 第二轮：主链全面正式化

本轮目标：

1. Today -> Content -> Chat -> Actions -> Growth -> Reports 全链结果真实化
2. 清掉主链上的 route 口径错位和本地 demo 干扰

本轮动作：

1. 统一热点/机会正式接口命名和前端请求口径
2. Today / Content / Reports 的 `transitional` 状态补退出条件
3. AppContext 只保留用户态与最小 UI 状态，不再 bootstrap 正式内容
4. Chat 只允许真实写表或显式失败

本轮验收：

1. 主闭环所有页面都只消费正式接口结果
2. Today、Actions、Growth、Reports 不再依赖前端 demo seed 补洞

### 第三轮：支撑链正式化 + 为后续 LLM 能力留正式接口

本轮目标：

1. 把“采集、调度、摘要生成”从过渡态收成正式任务链
2. 给后续联网搜索-生成摘要补 provider 留稳定接口，不再需要返工契约

本轮动作：

1. 采集失败只写失败状态，不再生成 fallback 内容
2. `system/*` 区分：
   - 可观测接口
   - 执行命令接口
   - 任务状态接口
3. 新增“摘要生成任务”的正式状态模型：
   - `queued`
   - `running`
   - `pending_provider`
   - `succeeded`
   - `failed`

本轮验收：

1. 正式态下不存在任何“假采集、假生成、假成功”
2. 没有 provider 时系统返回“未配置”而不是伪摘要

## 11. 三轮之后的最终决策

### 第一轮决策

如果第一轮做完，仍有大量页面依赖 fallback 完整对象才能维持展示：

`判定为 no-go，禁止进入第二轮扩面，先继续收第一轮。`

### 第二轮决策

如果第二轮做完，主闭环已经全部转成正式接口真实结果，且 demo/bootstrap 不再污染正式链：

`允许进入第三轮。`

如果第二轮后仍然必须依赖本地 demo 或虚拟写入来维持主闭环：

`判定为 no-go，禁止谈物理迁移，先继续清主链。`

### 第三轮决策

第三轮完成后，最终决策应是：

1. `app/` 在当前根目录下已经具备正式态内部结构。
2. 所有过渡/兜底逻辑都已转成：
   - 正式数据
   - 正式状态
   - 或显式失败
3. 唯一未落地的“联网搜索 -> 生成摘要”能力，仅剩 provider 实现未接入。
4. 由于其数据接口、状态模型和任务链已经固定，后续接大模型 API 时不再需要返工主链契约。

因此第三轮后的最终判断应是：

`允许把“app 物理迁移”重新放回候选，但前提已经不再是“为了整理目录而迁”，而是“正式态已经成立，迁目录只是在做物理归位”。`

## 12. 最终结论

这轮结合当前现实后的最终结论是：

1. 你现在最需要的不是“立刻把 app 挪到 apps/api”，而是一份能稳定约束 `app/` 后续演进的正式态架构基线。
2. 当前文件区里有迁移安全评估、阶段总表和技术现实清单，但没有一份专门回答“`app` 应该怎么长期稳定组织”的主线文件，因此本文是必要补充。
3. 后续新增、删除、修改接口时，必须按“页面聚合读侧 / 对象 CRUD / 命令动作 / 系统支撑链”四类来组织，不能继续把 fallback 和业务规则堆进 route。
4. 所有 `过渡 / 兜底 / 虚拟 / 假成功 / 假趋势` 逻辑都应该按“三轮、由易到难”的顺序清退。
5. 三轮完成后，你能得到的目标形态是：
   - `app/` 作为当前唯一 Python 主包，结构稳定
   - 主闭环全部正式化
   - 支撑链状态正式化
   - 联网搜索-生成摘要只差 provider 接入，不差接口设计

一句话总决策：

`当前阶段不迁 app 目录本身；先把 app 内部收成正式态，再在正式态成立后决定是否只做物理归位。`
