# 2026-04-16 app 迁移安全评估

- 记录日期：2026-04-16
- 记录类型：进行中主线评估
- 目标：评估 `app -> apps/api` 是否已经具备安全迁移条件，并给出可执行的分批迁移方案、验收标准与纠正机制。

## 1. 这轮先回答什么问题

这轮不直接做物理迁移，先回答 4 个问题：

1. `app/` 当前到底还承担什么职责。
2. 如果现在直接迁到 `apps/api`，最容易打坏哪些链路。
3. 安全迁移必须先补哪些兼容层。
4. 达到什么条件后，才允许进入真正的物理迁移。

## 2. 三轮“如果不这样做还有什么方法”

### 第一轮：这轮是否直接迁目录

1. 方案 A：现在直接把 `app/` 物理迁到 `apps/api`。
2. 方案 B：继续长期保留根目录 `app/`，不再推进。
3. 方案 C：先单开安全迁移评估轮，先写清风险、兼容层与闸门，再决定是否进入物理迁移。

结论：选 C。当前 `app/` 仍承担离线运行、参考实现、测试承接、脚本依赖和文档锚点职责，直接移动会同时影响运行链、验证链和协作文档链。

### 第二轮：如果后续要迁，采用什么迁法

1. 方案 A：一次性改目录、改 import、改脚本、改测试、改文档。
2. 方案 B：新建 `apps/api`，但长期保留根目录 `app/` 作为第二主包。
3. 方案 C：先补兼容壳和路径抽象，再分批切脚本、切测试、切文档，最后才做物理迁移。

结论：选 C。当前大量引用直接写死 `app.*`，如果没有兼容壳与路径抽象，一次性硬迁很容易把 `dev / check / verify / uvicorn` 一起打坏。

### 第三轮：这轮验收到什么程度

1. 方案 A：只写一个简单建议，不形成主线文档。
2. 方案 B：形成独立主线评估文档，明确 `go / no-go` 条件、分批方案和回退机制。
3. 方案 C：一边评估一边顺手迁一部分目录。

结论：选 B。当前阶段总表已经明确要求“先评估，不贸然移动”，这轮最稳的结果是把迁移条件和禁止动作写清楚，而不是边评估边开迁。

## 3. 当前真实依赖面

### 3.1 代码与脚本依赖规模

本轮清点结果：

1. `app / tests / tools` 内直接依赖 `app.*` 的文件共 `87` 个。
2. 其中 `app/` 自身内部引用文件 `70` 个。
3. `tests/` 直接依赖文件 `5` 个：
   - `tests/test_api_mainline.py`
   - `tests/test_content_module_imports.py`
   - `tests/test_content_projection.py`
   - `tests/test_dashboard_today.py`
   - `tests/test_persona_keywords.py`
4. `tools/` 直接依赖文件 `12` 个：
   - `tools/scripts/bootstrap_local_runtime.py`
   - `tools/scripts/bootstrap_d1_credentials.py`
   - `tools/scripts/apply_d1_remote.py`
   - `tools/scripts/demo_sync.py`
   - `tools/scripts/dev.mjs`
   - `tools/scripts/observe_content_result_usage.py`
   - `tools/scripts/observe_hot_topic_result_usage.py`
   - `tools/scripts/run-backend.mjs`
   - `tools/scripts/test_rss_aggregation.py`
   - `tools/scripts/verify_d1_behavior_mainline.py`
   - `tools/scripts/verify_article_result_hot_path.py`
   - `tools/scripts/verify_hot_topic_result_hot_path.py`

结论：`app/` 不是单纯“放在根目录不够整齐”的问题，而是一个仍被运行链、验证链和离线工具链共同引用的主包。

### 3.2 运行入口仍直接绑定根目录 `app`

当前 Python 运行入口仍直接写死：

1. `tools/scripts/dev.mjs` 使用 `uvicorn app.main:app`
2. `tools/scripts/run-backend.mjs` 使用 `uvicorn app.main:app`

这意味着只要物理移动 `app/`，默认 `npm.cmd run dev` 和单独后端启动链都会立即受影响。

### 3.3 路径敏感代码仍存在根路径假设

以下文件仍依赖 `Path(__file__).resolve().parents[2]` 推导仓库根：

1. `tools/scripts/apply_d1_remote.py`
2. `tools/scripts/bootstrap_d1_credentials.py`
3. `tools/scripts/bootstrap_local_runtime.py`
4. `tools/scripts/demo_sync.py`
5. `tools/scripts/generate_d1_seed.py`
6. `tools/scripts/generate_d1_remote_seed.py`
7. `tools/scripts/split_d1_seed.py`
8. `app/services/d1_client.py`

当前这些推导是以“Python 主包位于根目录 `app/`”为前提设计的；如果直接挪到 `apps/api/app` 或 `apps/api`，根路径计算需要一起抽象，否则容易误读：

1. `.env`
2. `apps/edge-worker/wrangler.toml`
3. `var/local/info_collector.db`
4. `apps/web/demo/mock-data`

### 3.4 配置加载仍以仓库根为默认口径

当前配置写法仍默认“从仓库根运行”：

1. `app/config.py` 使用 `env_file=".env"`
2. `app/config.py` 默认数据库为 `sqlite:///./var/local/info_collector.db`
3. `app/config.py` 的 `MOCK_DATA_PATH` 默认指向 `apps/web/demo/mock-data`
4. `app/database.py` 也默认读取 `sqlite:///./var/local/info_collector.db`

这类配置本身不一定阻止迁移，但它要求迁移前先统一“仓库根解析”与“运行工作目录”口径，不能只改 import。

### 3.5 文档口径仍大量引用 `app/`

当前文档内与 `app/` 相关的文件共有 `53` 个，且当前阶段明示引用至少包括：

1. `文档/进行中/当前阶段总表.md`
2. `文档/进行中/2026-04-16-仓库结构收口与接口契约归位.md`

结论：如果未来真的进入物理迁移，文档不能后补，必须按“主线文档 -> 总表 -> 相关技术文档”同步，否则很容易造成新对话误判。

## 4. 当前阶段结论：现在还不能直接物理迁移

当前 `app -> apps/api` 还不具备“直接一次性硬迁”的条件，核心原因有 5 个：

1. 默认 `dev` 启动链仍直接依赖 `app.main:app`。
2. Python 验证脚本与专项观测脚本仍直接 `import app.*`。
3. 测试入口仍显式依赖 `app.*`，直接迁移会影响当前 `check` 稳定性。
4. 根路径推导、`.env` 解析、D1 wrangler 读取仍存在位置假设。
5. 文档链还没有切换到“兼容迁移态”说明，贸然迁移会让协作口径先乱掉。

因此，这轮的最终判断是：

`允许继续做迁移准备，不允许直接做一次性物理迁移。`

## 5. 允许执行的分批方案

### 第 0 批：评估批（本轮）

目标：

1. 写清当前依赖面、风险、兼容层、go/no-go 闸门。
2. 把“不能直接硬迁”写入当前主线口径。

产出：

1. 本文档
2. 必要时同步 `当前阶段总表`

### 第 1 批：兼容层批

前置目标：

1. 先抽“仓库根解析”与“路径常量”公共口径。
2. 允许新目录逐步落位，但不立即删除根目录 `app`。

建议动作：

1. 抽一个 Python 侧统一根路径工具，替代散落的 `parents[2]`。
2. 为 Python 包引入兼容壳：
   - 可以是根目录 `app` 保留为兼容入口，真实代码逐步落到 `apps/api`
   - 或先让 `apps/api` 仅承接运行入口和包装层，根目录 `app` 暂不删除
3. 先把 `uvicorn` 入口、验证脚本入口改成“可切换路径”，而不是只认一个固定模块路径。

验收：

1. `npm.cmd run dev`
2. `npm.cmd run dev:python`
3. 现有 Python 验证脚本

都仍能工作。

### 第 2 批：测试与脚本切换批

目标：

1. 让测试和工具脚本不再把“根目录 `app`”当成唯一真实位置。
2. 继续保持 `check` 不退绿。

建议动作：

1. 改测试入口的导入策略或兼容解析。
2. 改脚本里的 `app.main:app` 为可配置入口。
3. 验证 D1 脚本、seed 脚本、观测脚本仍能找到仓库根和 wrangler 配置。

验收：

1. `npm.cmd run check`
2. `npm.cmd run check:workers`
3. `npm.cmd run db:d1:prepare`
4. `python tools/scripts/verify_d1_behavior_mainline.py`

### 第 3 批：物理迁移批

进入条件：

1. 前两批全部稳定通过。
2. 根目录 `app` 已不再承担唯一入口职责。
3. 迁移后的兼容壳已被验证可回退。

建议动作：

1. 再执行真实目录迁移。
2. 最后清理根目录 `app` 兼容壳或降级为薄代理层。
3. 同步文档口径。

## 6. go / no-go 闸门

### 6.1 允许进入物理迁移的条件

至少同时满足：

1. `npm.cmd run check` 持续全绿。
2. `npm.cmd run dev` 与 `npm.cmd run dev:python` 都已通过迁移后入口复验。
3. `db:d1:prepare`、D1 apply / verify / observe 脚本都已通过新路径验证。
4. 测试与脚本不再要求“只能从根目录 `app` 导入”。
5. 主线文档与总表已同步迁移口径。

### 6.2 任何一条命中即判定 no-go

1. 迁移后必须改大量测试才能勉强通过。
2. `uvicorn`、验证脚本、D1 脚本出现路径解析不稳定。
3. 需要同时长期维护两个 Python 主包。
4. 文档还没切到迁移口径就先动目录。
5. 会让当前唯一门槛 `check` 退化成“必须手工挑测试跑”。

## 7. 验收标准

这轮评估完成的标准是：

1. 已形成独立进行中文档，而不是把评估混写进旧主线。
2. 已明确写出“当前为什么不能直接迁”。
3. 已明确写出“允许先做哪些准备动作”。
4. 已明确写出“什么条件下才允许进入物理迁移”。
5. 已明确写出“命中哪些情况必须停止迁移”。

## 8. 纠正机制

1. 后续凡是涉及 `app -> apps/api` 的结果级变化，先更新本文档，再判断是否需要更新 `当前阶段总表.md`。
2. 在兼容层没有形成前，禁止直接删除根目录 `app/`。
3. 在 `dev / check / db:d1:prepare / verify_d1_behavior_mainline` 没有全部复验前，禁止进入物理迁移批。
4. 如果后续发现迁移窗口不成立，应把结论停留在评估文档，不强行推进到项目级口径。

## 9. 下一步入口

当前最合理的下一步不是“开迁目录”，而是：

1. 先设计 Python 侧统一根路径解析与兼容入口方案。
2. 先把 `uvicorn` 入口与验证脚本入口改成可切换口径。
3. 先做一轮“兼容层批”最小实现，再决定是否进入测试切换批。

一句话结论：

`app -> apps/api` 值得推进，但当前只能按“兼容层优先、分批切换、最后物理迁移”的顺序做，不能一次性硬迁。`
