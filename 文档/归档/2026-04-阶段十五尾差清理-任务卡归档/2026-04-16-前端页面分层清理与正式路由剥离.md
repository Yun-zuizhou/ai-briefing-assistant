# 2026-04-16-前端页面分层清理与正式路由剥离

- 文档状态：进行中
- 文档定位：梳理当前前端页面分层、数据源归属、与 Workers 后端的对应关系，并作为“demo 页面从正式路由剥离”的执行依据
- 更新时间：2026-04-16

## 1. 当前问题定义

当前 `apps/web` 不是单一前端系统，而是 3 套前端逻辑并存：

1. 正式产品页：已切到 `apiService + Workers`
2. demo / preview 页：仍保留强演示化布局和本地 mock / 本地状态
3. 废弃或兼容别名页：保留历史入口或包装壳，但不应继续作为正式产品认知的一部分

由此产生的直接问题：

1. 路由层无法一眼区分“正式页面”和“演示页面”
2. `AppContext` 里的本地状态与正式 API 读写并存，容易误判业务真源
3. `api.ts` 中仍保留旧时代接口方法，和当前 Workers 路由不完全一致
4. 用户会把预览页 / 演示页的视觉和行为变化误认为“正式前端被改坏”

1. 不以 Workers 为主数据源
2. 更偏视觉展示 / 历史演示验证
3. 当前继续挂在正式路由，会污染“产品真实前端”的理解

## 5. 前端服务层与后端路由对应

### 5.2 当前前端 `api.ts` 中仍残留但后端已不存在的旧接口

以下方法目前在 `apps/web/src/services/api.ts` 中仍保留，但 `apps/edge-worker` 无对应路由：

1. `getLearningResources`
2. `getLearningResource`
3. `generateBriefing`
4. `aiGenerateBriefing`
5. `aiGenerateProfile`
6. `aiSummarizeIdeas`
7. `getStats`

判断：

1. 这些方法属于历史接口残留
2. 当前即使未在正式页面中大量调用，也会干扰后续维护者对“正式 API 面”的判断
3. 后续应单独做一轮 `api.ts` 清理
