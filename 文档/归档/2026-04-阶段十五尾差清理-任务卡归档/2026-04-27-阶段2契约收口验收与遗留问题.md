## 1. 本轮结论

阶段 2 的三个最小契约收口点已经完成，可以进入收尾验收：

1. `Actions`：`GET /api/v1/actions/overview` 已返回 `topPriority / suggestedNextActions`，今日主行动不再长期由前端猜测。
2. `Journal`：新增 `GET /api/v1/journal/overview`，记录页不再只依赖前端五接口拼装。
3. `Today`：`GET /api/v1/dashboard/today` 已扩展 `leadItem / dailyAngle / freshness / extensionSlots`，并优先消费真实 `briefings.payload`。

当前不建议继续横向扩新接口。下一步应以移动端主路径人工验收、真实 payload 质量检查和 P0/P1 小缺口修复为主。

本轮发现并处理 1 个 P1 小缺口：

- `TodayPage` 中 `leadItem.itemType = briefing` 且存在 `contentRef` 时，原逻辑会按文章详情打开；已改为进入 `/history-brief`，避免把简报对象误投到内容详情页。

## 5. 前后端改造判断

当前阶段不需要新增接口，也不需要新增 D1 migration。

如果后续人工验收发现问题，按以下边界处理：

1. 前端小修
   - 路由跳转、文案层级、按钮位置、兜底展示。
   - 不改 contracts，不改 Workers。
2. 后端契约补丁
   - 字段已存在但含义不清，或需要补一两个可选字段。
   - 必须先改 `packages/contracts`，再改 `apps/edge-worker` 和前端 validator。
3. 数据生成质量问题
   - `briefings.payload` 结构不稳定、主判断质量不足、扩展位缺失。
   - 优先修生成链或 seed / fixture，不先改页面。
4. schema 问题
   - 只有在已有表无法表达长期对象时，才新增 `infra/cloudflare/d1/migrations`。

## 6. 人工验收清单

建议使用移动端视口优先验收，桌面只作辅助。

| 路径 | 验收问题 | 通过标准 |
|---|---|---|
| Today -> Article | 今日主判断为热点或机会时，打开内容是否正确 | 能进入内容详情，标题和摘要与主判断一致 |
| Today -> History Brief | 今日主判断为 briefing 时，是否进入历史简报入口 | 不进入文章详情，不出现 unsupported content ref |
| Today -> Chat | 继续追问是否带入标题和内容上下文 | Chat 输入预设可读，来源不暴露内部键 |
| Today -> Todo | 行动建议是否能进入行动页或待办闭环 | 用户能理解下一步，不像内部枚举 |
| Chat -> Journal | 聊天沉淀是否能在记录页被承接 | Journal 能看到“我刚说过的 / 最近沉淀” |
| Actions 主行动 | 主行动是否清楚，按钮是否符合动作类型 | todo 可完成，非 todo 可跳到对应入口 |
| Journal Overview | 新 overview 不可用时页面是否仍可用 | 旧兜底不报错，页面不空白 |
| My 入口 | 退出登录和个人沉淀入口是否分层清楚 | 退出登录不抢首屏主动作 |

## 7. 后续推荐执行顺序

1. 执行移动端人工验收，先记录 P0/P1。
