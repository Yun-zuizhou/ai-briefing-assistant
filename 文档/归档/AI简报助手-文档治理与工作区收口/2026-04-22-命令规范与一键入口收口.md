# 命令规范与一键入口收口

- 建立时间：2026-04-22
- 作用：把当前高频运行事项整理成可直接复制执行的一键命令，减少每次重找脚本与逐步试错

---

## 1. 本轮为什么要做

当前仓库虽然已有较多脚本，但高频任务仍存在三个问题：

1. 名称偏实现层，第一次接手时不容易直接判断该跑哪条
2. 运行、权限、预检、数据准备分散在不同命名空间里
3. 同一类动作缺少统一前缀，容易出现“先猜命令，再试错”的情况

因此本轮目标不是重做脚本，而是：

1. 先整理哪些事项适合包装成一键命令
2. 再统一命名并挂到根级 `package.json`
3. 最后在 `文档/` 下建立稳定的命令入口

---

## 2. 哪些事项适合包装成一键命令

### 2.1 适合直接包装

以下事项可稳定收口为“一行命令即可跑”的任务：

1. 首次环境初始化
2. 启动完整程序
3. 只启动前端
4. 只启动 Workers 后端
5. 启动 Python 回退后端
6. 主门槛校验
7. Workers 权限/secret 预检
8. D1 凭据写入 `.env`
9. 本地 AI provider 同步到 Workers `.dev.vars`
10. D1 seed 生成准备
11. demo 数据同步
12. 浏览器级 E2E 复验

### 2.2 不适合完全伪装成“全自动”的事项

以下事项仍然需要人工决策或外部登录条件，因此只做“标准入口”，不伪装成无前提自动完成：

1. Cloudflare 账号登录本身
2. 新建/申请 Cloudflare API Token
3. 生产 secret 的值来源决策
4. 真正的远端部署发布

这类事项当前的正确做法是：

1. 先把执行入口固定
2. 再明确它需要的外部前提
3. 避免“命令存在，但实际上偷偷依赖未说明的人工步骤”

---

## 3. 本轮采用的统一命名

本轮把高频命令统一收口到 `task:*` 命名空间。

分组规则：

1. `task:init:*`
   - 环境初始化与首次准备
2. `task:start:*`
   - 启动运行类任务
3. `task:access:*`
   - 凭据、secret、权限、配置同步类任务
4. `task:verify:*`
   - 检查、验证、测试类任务
5. `task:data:*`
   - 数据准备、同步、seed 相关任务

## 6. 当前文档功能区分类决定

结合本轮整理与现有文档规范，当前 `文档/` 功能区按下面 5 层理解：

1. **核心口径层**
   - `项目核心总纲.md`
   - `项目核心子纲-阶段演进与归档索引.md`
   - `功能地图.md`
   - `技术现实清单.md`
2. **当前执行层**
   - `进行中/`
   - `UI设计/` 中目录级 `README.md` 明确标出的当前正式 UI 文档
   - `命令规范/`
   - `契约/`
3. **参考专题层**
   - `产品设计/`
   - `技术实现/`
   - `问题清单/`
   - `Agent协同经验/`
   - `文案模板/`
   - `AI技能/`
4. **历史追溯层**
   - `归档/`
   - `过时/`
   - `开发日志/`
   - `学术答辩/`
5. **工具层**
   - `.obsidian/`

这意味着：

1. `文档/` 下的目录不再默认处于同一优先级
2. 顶层专题目录默认应补目录级 `README.md`
3. 顶层 `文档/README.md` 只做总导航，不继续堆每个目录的全部细节解释
