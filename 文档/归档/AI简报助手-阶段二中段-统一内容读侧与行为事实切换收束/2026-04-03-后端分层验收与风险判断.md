# 后端分层验收与风险判断

- 文档状态：进行中
- 建立时间：2026-04-03
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 目的：把“后端通不过”拆成语法层、依赖层、服务层、接口层，避免误判

---

## 1. 本轮实际执行

本轮实际执行了两类后端验收：

1. 全量 `py_compile`
2. `import app.main`

执行结果：

### 1.1 语法层

- 执行结果：`py_compile ok: 52 files`
- 结论：当前 `app/` 下 Python 文件语法通过

### 1.2 运行依赖层

- 执行结果：`ModuleNotFoundError: No module named 'fastapi'`
- 结论：当前环境缺少 `fastapi`，因此无法完成 `app.main` 导入

---

## 2. 这说明问题在什么层

当前问题不在：

1. Python 语法层
2. 文件是否存在
3. 代码目录是否损坏

当前问题在：

`运行依赖层`

更准确地说，是：

1. 当前环境没有安装后端运行所需核心依赖
2. 因此无法验证服务启动层
3. 也无法继续验证真实接口层

---

## 3. 对项目的真实风险

### 3.1 当前已经可以排除的风险

当前可以排除：

1. “后端代码整体语法已坏”
2. “代码文件已经无法继续维护”
3. “当前主线已经被 Python 语法错误阻断”

### 3.2 当前仍然存在的风险

当前仍然存在：

1. 后端服务实际是否能启动，尚未在当前环境确认
2. `/api/v1/dashboard/today`、`/api/v1/content/by-ref` 等主接口的运行态尚未在当前环境确认
3. 若继续做功能联调而不补依赖，会把“环境不可运行”误判成“后端功能坏了”

---

## 4. 当前应如何判断优先级

当前不应把问题表述成：

`后端功能已经坏了，所以先停止结构推进`

更准确的说法应是：

`当前环境未安装后端运行依赖，因此运行态尚未验证；但语法层已通过，所以可以继续做文档、结构和表设计收束，同时把依赖层补齐列为下一轮运行前置动作。`

---

## 5. 当前建议

### 必须先修但不属于“功能大修”的内容

1. 补齐后端运行依赖，至少确保 `fastapi` 可导入
2. 在依赖补齐后，再验证服务启动层与主接口层

### 可以继续推进的内容

1. 数据库和结构建设
2. 文档收束
3. 表设计与当前代码的接轨判断
4. 统一内容层与统一引用规则推进

结论：

`当前不是先修后端功能问题，而是先把后端运行依赖层补齐；在此之前，可以继续推进数据库与结构收束，但不要误把运行环境问题说成后端功能问题。`
