# Chat系统判定消息回收与确认层收束

- 文档状态：进行中
- 建立时间：2026-04-03
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：记录 Chat 在最小确认层之后，继续把“系统怎么理解输入”回收到消息流本身的这轮推进

---

## 1. 本轮目标

本轮不是继续扩新分流类型，而是继续收消息层。

重点只做两件事：

1. 让系统判定消息本身携带更完整的解释信息
2. 让顶部“待确认分流”卡进一步退回到“只负责当前轮操作”

---

## 2. 本轮已完成

### 2.1 系统判定消息已补元数据表达

当前在：

- [useChatLogic.ts](/E:/python/杂谈-想法记录与实践/prototype/src/hooks/useChatLogic.ts)
- [ChatPage.tsx](/E:/python/杂谈-想法记录与实践/prototype/src/pages/ChatPage.tsx)

已补：

1. 候选类型
2. 置信度
3. 来源上下文
4. 识别方式

这意味着当前消息流里的“系统判定”消息已经不再只是：

- “我猜这像某个类型”

而是开始能解释：

- 我怎么理解的
- 还有哪些候选
- 当前输入来自哪里
- 当前识别是按什么方式落下来的

### 2.2 顶部待确认卡进一步收口

当前“待确认分流”卡的口径已调整为：

- 消息流负责记录系统判定
- 顶部卡只保留最后一步确认操作

这标志着当前 Chat 这条线已经继续朝着：

`消息流做解释与回看，顶部卡做当前轮控制`

这个方向推进。

### 2.3 确认消息已补提交语义

当前“已确认”消息不再只是简单一句确认提示，而是明确表达：

1. 已按指定类型继续处理
2. 不再重新询问
3. 会直接按该类型提交

这让“确认 -> 执行”之间的会话衔接更顺，而不再像硬切换。

## 4. 当前正确口径

当前应这样描述：

1. Chat 当前已从“最小确认层 + 顶部待确认卡”继续推进到“系统判定信息开始正式进入消息流”
2. 当前消息流已不仅记录“待确认 / 已确认 / 已执行”，还开始携带候选类型、置信度、来源上下文和识别方式
3. 顶部反馈卡当前进一步收缩为“当前轮操作卡”，而不是继续承担完整解释与日志职责
4. 这仍然不是完整消息系统，但已经从“有状态标签”推进到“有解释性判定消息”的更稳定过渡态

## 5. 下一步建议

顺着当前收口方向，下一步最合理的是：

1. 继续收执行结果消息的表达，减少顶部“当前轮操作”卡的长期驻留感
2. 再补事后纠偏入口，而不是继续堆更多顶部卡片
3. 顺手修正文档和工程口径里“根目录 build / lint”与“prototype 下 build / lint”的混用问题
