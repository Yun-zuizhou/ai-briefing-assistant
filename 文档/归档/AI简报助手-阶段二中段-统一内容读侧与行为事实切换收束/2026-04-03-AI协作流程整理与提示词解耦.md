# AI协作流程整理与提示词解耦

- 文档状态：进行中
- 建立时间：2026-04-03
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：收束当前项目的新对话协作入口、层层汇报机制和提示词拆层方式，降低每轮重复提醒成本

---

## 1. 本轮要解决的问题

当前 AI 协作反复出现三类问题：

1. 新对话开启后，Agent 容易漏掉文档读取顺序或当前阶段判断
2. 文档更新链经常失稳，要么只更过程文档，要么直接碰总纲
3. 当前提示词把“协作规则”和“项目现状”写在一起，导致每轮推进后都要连带重写提示词

这三类问题叠加后，会让项目看起来像“在原地打转”：

1. 执行不是完全没做
2. 但新对话承接成本过高
3. 结果就是协作效率被上下文重建本身吃掉

---

## 2. 本轮整理后的核心结论

后续 AI 协作入口改为三层：

1. 通用协作协议
2. 当前项目状态注入
3. 本轮任务卡

对应文件调整为：

1. `文档/Agent协同经验/项目初始化.md`
   - 改为稳定协作协议版
   - 只保留长期规则
2. `文档/Agent协同经验/项目承接提示词-当前版.md`
   - 改为当前项目状态卡
   - 只维护当前阶段现状
3. `文档/Agent协同经验/AI协作协同经验.md`
   - 负责说明为什么这样拆、如何触发层层汇报、后续怎么复用

因此，后续推进时不再需要：

1. 每轮都重写整段总控提示词
2. 把最新阶段状态继续塞进通用协议
3. 靠口头重复提醒“记得更新子纲，不要直接更新总纲”

---

## 3. 当前固定下来的新对话流程

后续新对话建议固定为：

1. 先加载 `项目初始化.md`
2. 再加载 `项目承接提示词-当前版.md`
3. 最后补本轮任务描述

也就是：

`通用规则 -> 当前状态 -> 本轮任务`

不再使用：

`一段超长提示词把规则、状态、任务、历史备注全部混写`

---

## 4. 层层汇报机制的固定触发条件

本轮固定如下：

1. 只是新增事实、联调、验证：
   - 更新 `文档/进行中/`
2. 当前阶段任务、风险、顺序发生变化：
   - 再更新 `当前阶段总表.md`
3. 形成新的结果级判断、边界口径：
   - 再更新 `项目核心子纲-阶段演进与归档索引.md`
4. 项目级当前判断、阶段切换或对外口径变化：
   - 才继续更新 `项目核心总纲.md`

记忆口令：

`过程先写进行中，阶段判断进总表，结果级结论进子纲，项目级口径才进总纲。`

---

## 5. 当前项目为什么还没有进入数据库同步

当前并不是完全没准备，而是阶段顺序还没走到那一步。

现状已经具备的前置：

1. D1 目标数据库已明确
2. `wrangler.toml` 已有 D1 配置
3. D1 migration 与 remote seed 脚本已预留
4. 当前阶段统一内容读侧与 `content_ref` 边界已基本收口
5. 当前本地联调、默认测试与接口级验证已建立基础锚点

但还没真正进入数据库同步的原因是：

1. Today / 内容层仍是稳定过渡态，不是正式内容层
2. Chat 正式消息持久化和会话上下文还未完成
3. 统一内容层是否实体化为 `contents` 仍未做最终结果级判断
4. 当前后端主读写仍是本地 SQLite，不是 D1
5. D1 同步一旦开始，就需要连带处理迁移、回填、双环境口径和主读切换，不适合在阶段二边界还未收稳时提前抢主线

---

## 6. 当前对“还剩多少工作”的最简判断

如果以“开始真正做数据库同步”为目标，当前至少还差三段工作：

### 第一段：完成阶段二收束前置

1. 继续补 Today / 内容层的边界与测试
2. 继续补 Chat 消息层细化和确认后反馈
3. 对统一内容层是否进入 `contents` 实体化做结果级判断

### 第二段：进入数据库同步设计阶段

1. 基于真实库而不是口头设想整理 D1 正式迁移方案
2. 明确哪些表先迁、哪些继续保留分表事实
3. 明确本地 SQLite 与 D1 的迁移 / 双环境策略

### 第三段：执行数据库正式切换

1. 真正建表到 D1
2. 准备回填或最小种子数据
3. 验证后端主读链路切换
4. 验证回退逻辑是否已下线或缩边

也就是说：

`不是还差十几轮空转，而是至少还差“阶段二收束 -> 迁移设计 -> 正式切换”这三段工作。`

---

## 7. 本轮整理后的正确使用方式

后续如果要继续高频和 AI 协作，最推荐的做法是：

1. 通用规则长期不动，放在 `项目初始化.md`
2. 每轮只维护 `项目承接提示词-当前版.md` 的当前状态
3. 每次具体任务另附一张任务卡
4. 每轮结束都按层层汇报触发条件判断文档更新层级

这样做的直接收益是：

1. 新对话承接更稳
2. 层层汇报更不容易漏
3. 提示词维护成本显著下降
4. 项目推进不再被“重新交代上下文”持续吞噬
