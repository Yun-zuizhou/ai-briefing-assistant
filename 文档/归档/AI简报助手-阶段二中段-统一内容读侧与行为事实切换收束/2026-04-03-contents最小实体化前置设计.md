# contents 最小实体化前置设计

- 文档状态：进行中
- 建立时间：2026-04-03
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：在不立即重表的前提下，明确后续如果要正式建 `contents`，第一批迁移边界、保持不动边界和实施顺序是什么

---

## 1. 本轮目标

本轮不是直接建 `contents`，而是把下面三件事先设计清楚：

1. 如果后续真的建 `contents`，第一批到底迁哪些
2. 哪些内容当前明确不迁
3. 第一批迁移应该怎么落，才不会打断阶段二主线

---

## 2. 当前前提

当前已经成立的事实是：

1. 阶段二当前已形成统一内容读侧稳定过渡态
2. 统一详情入口已稳定为：
   - `content_ref`
   - `/api/v1/content/by-ref`
   - `UnifiedContentDetailResponse`
3. 当前真实内容事实仍然来自：
   - `hot_topics`
   - `rss_articles`
   - `opportunities`

因此如果后续要建 `contents`，它必须解决的不是“让页面第一次能统一读”，而是：

`把已经成立的统一读侧，向统一事实存储层推进`

---

## 3. 第一批明确要迁的范围

如果后续进入 `contents` 实体化，第一批只迁三类内容：

1. `hot_topic`
2. `article`
3. `opportunity`

原因：

1. 这三类已经进入统一读侧
2. 这三类已经有相对清楚的字段映射
3. Today / 详情 / 收藏 / 报告的统一内容问题都首先落在这三类上

---

## 4. 第一批明确不迁的范围

第一批不建议迁以下对象：

1. `notes`
2. `todos`
3. `history_entries`
4. `briefings`
5. `reports`
6. `user_profiles`
7. `favorites`

原因分别如下。

### 4.1 notes / todos / history_entries

它们当前仍应视为：

`用户行为与沉淀事实层`

不是：

`外部内容事实层`

如果把它们一起卷进 `contents`，会立刻把统一内容层问题膨胀成：

`内容事实层 + 用户行为层混合重构`

这不符合当前阶段主线。

### 4.2 briefings / reports

它们当前更像：

`聚合结果层 / 快照层`

而不是：

`基础内容事实层`

因此不应该在第一批跟着一起重构。

### 4.3 favorites

`favorites` 当前虽然和统一内容层关系很近，但更合适的做法不是第一批就重表，而是：

1. 先继续兼容 `content_ref`
2. 等 `contents` 真正确立后，再决定是否把收藏从 `item_type + item_id` 切到 `content_id` 或等价引用

---

## 5. 第一批最小字段边界

第一批 `contents` 如果要建，建议只保留这些最小字段：

1. `id`
2. `content_type`
3. `title`
4. `summary`
5. `content`
6. `source_name`
7. `source_url`
8. `author_name`
9. `category_name`
10. `published_at`
11. `hot_score`
12. `status`

当前不建议第一批就塞进去的字段：

1. 个性化排序分
2. AI 摘要
3. 推荐理由
4. 相关推荐缓存
5. 用户侧交互统计
6. 复杂标签归一化结构

这些字段当前仍属于：

`加工层 / 视图层 / 后续扩展层`

不适合第一批跟表一起落。

---

## 6. 第一批字段映射边界

### 6.1 hot_topics -> contents

建议映射：

1. `content_type = hot_topic`
2. `title <- hot_topics.title`
3. `summary <- hot_topics.summary`
4. `content <- hot_topics.content`
5. `source_name <- hot_topics.source`
6. `source_url <- hot_topics.source_url`
7. `author_name <- hot_topics.author`
8. `category_name <- hot_topics.categories[0]` 的最小等价口径
9. `published_at <- hot_topics.published_at`
10. `hot_score <- hot_topics.hot_value`

### 6.2 rss_articles -> contents

建议映射：

1. `content_type = article`
2. `title <- rss_articles.title`
3. `summary <- rss_articles.summary`
4. `content <- rss_articles.content`
5. `source_name <- rss_articles.source_name`
6. `source_url <- rss_articles.source_url`
7. `author_name <- rss_articles.author`
8. `category_name <- rss_articles.category`
9. `published_at <- rss_articles.publish_time`
10. `hot_score <- 0` 或最小默认值

### 6.3 opportunities -> contents

建议映射：

1. `content_type = opportunity`
2. `title <- opportunities.title`
3. `summary <- opportunities.summary`
4. `content <- opportunities.content`
5. `source_name <- opportunities.source`
6. `source_url <- opportunities.source_url`
7. `author_name <- NULL`
8. `category_name <- opportunities.category`
9. `published_at <- opportunities.published_at`
10. `hot_score <- 0` 或最小默认值

---

## 7. 第一批不应改变的东西

即使后续建 `contents`，第一批也不应同时改变：

1. `content_ref` 格式
2. `/api/v1/content/by-ref` 路由入口
3. `UnifiedContentDetailResponse` 最小契约
4. Today / 详情页当前路由跳转方式

原因：

这些已经是阶段二读侧稳定过渡锚点。

如果建表时把这些一起改掉，就会失去当前阶段已经收住的稳定面。

---

## 8. 第一批实施顺序建议

如果后续进入实体化，建议顺序只能是：

1. 先建 `contents` 最小表
2. 先做只读镜像或最小回填，不切主读
3. 让 `/api/v1/content/by-ref` 在内部增加一层可切换读取策略
4. 验证 Today / ArticlePage / Favorites / Reports / History 是否仍保持现有读契约
5. 最后再决定是否逐步切主读

这意味着当前明确不建议的做法是：

1. 一上来删掉三张原始表角色
2. 一步切所有读写
3. 同时改收藏、报告、历史、推荐、摘要逻辑

---

## 9. 当前一句话结论

`如果后续进入 contents 实体化，第一批只迁 hot_topic / article / opportunity 三类外部内容事实，不迁 notes / todos / history / reports / favorites，并且不改现有 content_ref 与 by-ref 读契约。`
