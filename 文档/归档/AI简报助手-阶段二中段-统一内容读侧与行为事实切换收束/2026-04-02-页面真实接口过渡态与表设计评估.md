## 1. 本轮目标

本轮不直接扩大 Today 页、对话页的困难改造范围，而是先回答 4 个问题：

1. 文件夹中是否已经存在“表设计”与“表/字段清单”。
2. 当前表设计是否足以支撑项目进入“所有页面至少处在真实接口过渡态”。
3. 哪些页面现在仍停留在 `mock / context / 演示态`，必须尽快切到真实接口过渡态。
4. `context 演示态` 本身应该继续保留，还是需要单独设计真实链路。

## 5. 表设计完整性评估

## 5.1 结论先说

当前表设计方向总体是对的，已经能覆盖主闭环的大部分事实层需求。

但是如果目标改成：

`至少保证所有页面都处在真实接口过渡态`

那么当前表设计和真实实现仍然不够，需要继续补强。

## 5.2 已经足够支撑的部分

以下事实层已经足以让页面摆脱纯 mock：

1. `todos`
2. `favorites`
3. `notes`
4. `history_entries`
5. `hot_topics`
6. `opportunities`
7. `rss_articles`
8. `rss_sources`

这些表已经足以支撑：

1. 行动中心
2. 成长中心基础统计
3. 热点列表
4. 机会列表
5. 文章供给与推荐的基础来源

## 5.3 明显不完整或需要增强的部分

如果要让更多页面进入真实接口过渡态，当前至少还缺这些关键事实层：

### A. `user_interests`

当前状态：

- 真实写入还在 `users.interests` JSON/Text 字段
- Today 页推荐又依赖共享虚拟兴趣状态

问题：

- 不利于查询、去重、扩展和后续画像/推荐复用
- 不利于把对话页、兴趣配置页、Today 推荐统一到同一事实层

判断：

`必须补。`

### B. `user_settings`

当前状态：

- 提醒时间、通知设置主要还在前端 context / 本地状态

问题：

- 设置页、通知页还没有统一真实配置事实层

判断：

`必须补。`

### C. `opportunity_follows`

当前状态：

- 机会数据存在
- 但“跟进中的机会”没有独立真实关系层

问题：

- ActionsPage 中的“跟进”难以形成稳定真实接口

判断：

`建议补，且优先级较高。`

### D. `user_profiles`

当前状态：

- 画像页仍基于前端 context 拼装

问题：

- 用户画像无法和真实行为数据脱钩
- Growth / Profile / 报告体系会持续漂浮在演示态

判断：

`必须补，但可以先做过渡版。`

过渡版不要求先做复杂 AI 画像，只要先把“由真实数据汇总后生成的简版画像结果”有地方存。

### E. `reports`

当前状态：

- 周报 / 月报 / 年报主要仍是 mock 数据

问题：

- 成长回顾体系没有真实报告事实层

判断：

`必须补。`

### F. `briefings`

当前状态：

- Today 页有展示结构，但没有真正“某天某个用户的一份简报事实”

问题：

- Today 仍更像即时拼装页，不像真正的简报页

判断：

`建议补。`

Today 的第一步不一定要求完整正式生产链路，但至少要有“简报快照”或“聚合结果缓存”的过渡层。

### G. `contents` 或等价统一内容层

当前状态：

- 热点、RSS、机会分散在多张表
- 文章详情与收藏写入没有统一内容主键

问题：

- Today、热点、文章详情、收藏、报告很难做统一引用

判断：

`需要补统一内容层，或者至少补一个等价的聚合服务层。`

这里不一定必须立即建 `contents` 实体表，但至少要建立：

1. 统一内容 ID 方案
2. 内容来源类型映射
3. 详情页读取与收藏写入的统一引用规则

## 6. 以“所有页面至少进入真实接口过渡态”为目标的最小判断

这里要特别区分：

- `正式态`：真实数据、真实聚合、真实持久化都已成立
- `真实接口过渡态`：页面至少通过真实接口拿结构化数据，底层可以暂时由真实表 + 聚合拼装 + 局部回退组成
- `context 演示态`：页面主要靠前端 context / mockData / 本地派生值自转

当前要求不是让所有页面一步到位进入正式态。

当前合理目标应是：

`所有主要页面至少不再把前端 context / mockData 当主数据源，而是统一转向真实接口过渡态。`

## 8. 对 context 演示态的单独方案

## 8.1 先给结论

`context 演示态不能继续作为主流程的数据主源，但可以继续保留为“过渡缓存层 / 演示兜底层 / 交互即时态”。`

关键是要改定位，而不是一刀删掉。

## 8.2 哪些 context 能继续保留

以下场景中的 context 可以继续保留：

1. 聊天输入中的即时 UI 状态
2. 页面间临时跳转态
3. 未落库前的本地草稿态
4. 后端不可达时的演示回退态
5. 过渡阶段的“写后立即可见”缓存态

这类保留的前提是：

`它们不能再被说成“真实事实来源”。`

## 8.3 哪些 context 不能继续作为主源

以下数据不应继续由 context 充当主数据源：

1. 收藏列表
2. 想法记录列表
3. 历史日志列表
4. 热点列表
5. 画像统计
6. 报告内容
7. 兴趣长期状态
8. 通知设置长期状态

这些都应该切到真实接口过渡态。

## 8.4 对 context 的推荐改造定位

建议把当前 `AppContext` 从“事实源”改成下面三类角色：

### 角色 A：UI 会话态

用于：

1. 聊天消息列表
2. 弹窗开关
3. 页面筛选条件

### 角色 B：过渡缓存态

用于：

1. chat/execute 成功后，前端立即显示执行结果
2. 等待下一次真实接口刷新前的短暂同步

### 角色 C：演示兜底态

用于：

1. 离线演示
2. 后端未就绪时的备用模式

但这类兜底必须在文档和判断中被明确标为：

`演示态 / 回退态`

而不是“已经真实联通”。

## 9. 当前最推荐的落地顺序

如果目标是“至少保证所有页面都进入真实接口过渡态”，当前最合理的顺序不是先全量上正式数据库复杂建模，而是：

1. 先把已经有表支撑的页面切到真实接口
2. 再补最缺的关系表和聚合表
3. 最后再处理 Today / 对话这种高难供给链路

推荐顺序如下：

### 第一批：低成本但收益高，应该先做

1. CollectionsPage -> `/favorites`
2. HistoryLogsPage -> `/history`
3. HotTopicsPage -> `/hot-topics`
4. JournalPage -> `/notes + /history`

这批的特点是：

- 后端表已存在
- 前端改造边界清晰
- 能迅速减少“context 演示态”面积

### 第二批：必须补表后再切

1. InterestConfigPage / GrowthPage 兴趣统计 -> `user_interests`
2. SettingsPage / NotificationSettingsPage -> `user_settings`
3. ActionsPage 跟进机会 -> `opportunity_follows`
4. ProfilePage -> `user_profiles`
5. Reports 系列 -> `reports`

### 第三批：高难内容聚合层

1. TodayPage -> `briefings + 内容聚合层`
2. ChatPage -> 与真实事实层全面统一
3. ArticlePage -> 统一内容层或统一内容引用规则

## 10. 本轮最终判断

### 10.1 用户当前的判断是否成立

如果把问题表述为：

`链路已经基本跑通，问题在于信息获取还不真实`

那么这个判断对 Today / 对话是成立的。

但放到全项目则要补一句：

`除了 Today / 对话之外，很多旁支页面本身也还没有进入真实接口过渡态。`

### 10.2 表设计是否完整

结论：

`方向基本完整，但对“所有页面都进入真实接口过渡态”来说仍不够，需要继续补 user_interests / user_settings / opportunity_follows / user_profiles / reports / briefings，以及统一内容层或等价方案。`

### 10.3 context 演示态应否继续保持

结论：

`可以继续保持，但只能作为过渡缓存层、演示兜底层、即时 UI 会话态，不能继续作为页面主数据源。`

### 10.4 当前是否应该继续真实数据库接入

结论：

`应该继续，但不是先盲目扩大 Today 的复杂真实化，而是先把已有表能支撑的页面先切到真实接口过渡态，再补缺失关系表和聚合表。`

## 11. 本轮建议收口成一句话

当前项目的正确方向不是“立即把所有页面做成正式态”，而是：

`先把所有主要页面从 context / mock 主驱动切到真实接口过渡态，再把 context 收回到缓存与演示兜底角色，最后集中攻克 Today / 对话的真实内容供给链路。`
