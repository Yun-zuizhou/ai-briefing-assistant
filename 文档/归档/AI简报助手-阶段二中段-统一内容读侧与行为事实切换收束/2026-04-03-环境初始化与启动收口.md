## 2. 本轮实际补齐了什么

### 2.1 统一启动入口

根目录已新增：

1. `package.json`
2. `scripts/dev.mjs`
3. `scripts/run-backend.mjs`
4. `scripts/run-frontend.mjs`
5. `scripts/dev.ps1`

当前启动不再依赖直接执行裸 `npm`。

原因是本机当前存在：

`npm` 命令路径异常，但 `npm.cmd` 可用

因此这轮没有强行把“唯一启动命令”压成 `npm run dev`，而是优先把真正稳定可执行的一条命令压成：

`.\scripts\dev.ps1`

这比让后续接手者继续记住“PowerShell 下要用 `npm.cmd`，后端还要额外补 `PYTHONPATH`”更稳定。

### 2.2 环境初始化入口

根目录已新增：

1. `scripts/setup.mjs`
2. `scripts/setup.ps1`
3. `scripts/init-local-db.mjs`

当前 `node scripts/setup.mjs` 会自动完成：

1. 创建根目录 `.env`

3. 检查前端依赖
4. 复用项目自带 `.pydeps_runtime / .pydeps`

### 2.3 前端 API 地址收口

已调整：

`prototype/src/services/api.ts`

当前前端不再把后端地址硬编码死锁在源代码里，而是优先读取：

`VITE_API_ORIGIN`

默认值仍为：

`http://127.0.0.1:5000`

### 2.4 Vite 开发态回归收口

已调整：

`prototype/package.json`

当前前端开发态默认脚本也已补上：

`--configLoader native`

原因是当前本地 Vite / rolldown 组合下，开发态和构建态一样，都不能再继续沿用裸 `vite` 入口。

### 2.5 配置模型补齐

已调整：

`app/config.py`

当前 `.env` 中新增的 D1 相关字段已被 `Settings` 接住，不会再因为配置模型未声明而导致后端启动失败。

## 3. 当前正式态 / 过渡态 / 回退态边界

### 3.1 云端正式目标态

当前项目的云端正式目标数据库仍然是：

`Cloudflare D1 / ai-briefing-assistant-prod`

依据：

1. `wrangler.toml`
2. `cloudflare/d1/migrations/0001_init_schema.sql`
3. `cloudflare/d1/seed.remote.sql`
4. `scripts/generate_d1_remote_seed.py`
5. `scripts/prepare-d1.mjs`

因此，云端正式数据目标并不是新的本地库，也不是新的本地 JSON 持久化。

### 3.2 当前本地运行过渡态

当前项目真正运行时的主读写库仍然是：

`info_collector.db`

也就是：

`DATABASE_URL=sqlite:///./info_collector.db`

这必须明确标注为：

`当前运行过渡态`

而不能误判成：

`当前已经正式上云`

当前代码里还没有“后端直接读写 D1”的实际实现，Cloudflare D1 目前仍停留在：

1. schema 已准备
2. remote seed 已准备
3. 迁移命令入口已预留
4. 但本地运行链路尚未切主读

### 3.3 当前虚拟态

当前仍存在明确虚拟态能力：

1. `app/services/data/virtual_data.py`
2. `dashboard.py` 在用户无真实兴趣时会回退到共享虚拟兴趣
3. `chat.py` 在数据库写入失败时会回退到虚拟待办 / 虚拟记录 / 虚拟关注

这部分必须继续明确标注为：

`虚拟态 / 联调过渡态`

不是正式数据。

### 3.4 当前回退态

以下接口仍带有异常级回退：

1. `favorites`
2. `notes`
3. `todos`
4. `history`
5. `chat.execute`

它们的共同特点是：

`真实数据库链路失败时，返回虚拟数据或虚拟写入响应`

因此当前不能把这些链路误判成“只要页面能出结果，就已经真实稳定”。

### 3.5 当前 mock 态

`mock-data/` 目前仍然存在，而且本轮继续被用于：

1. 初始化本地最小运行数据
2. 生成 D1 seed

它当前的定位是：

`初始化 / 联调 / 过渡辅助数据`

不是当前正式业务数据库本体。

## 5. 当前启动方式是否已经真正收口

### 已收口部分

当前本地开发启动已经基本收口到一条命令：

`.\scripts\dev.ps1`

当前这个命令已经覆盖：

1. 后端依赖拼装
2. FastAPI 启动
3. 前端 Vite 启动
4. 前后端端口对齐
5. 前端 API 地址默认对齐

### 未完全收口部分

当前还没有完全收口到“云端正式态一键启动”，原因不是脚本缺失，而是代码与环境现实还没到那一步：

1. 后端运行主库仍未切到 D1
2. 当前机器没有 `wrangler`
3. 当前会话未持有 Cloudflare 鉴权
4. D1 目前仍是预备态，不是当前本地开发的实际主读链路

因此当前最准确的说法是：

`本地开发启动已收口，云端正式数据库初始化入口已补齐，但云端正式主读链路尚未完成切换。`

## 7. 当前最少命令建议

### 当前直接启动

优先命令：

`.\scripts\dev.ps1`

### 如果后续换机器或清空依赖

先执行一次：

`node scripts/setup.mjs`

再执行：

`.\scripts\dev.ps1`

### 如果要准备 D1 远端种子文件

`node scripts/prepare-d1.mjs`

当前它会完成：

1. 生成 `cloudflare/d1/seed.remote.sql`

## 8. 当前正确口径

本轮之后，当前项目环境与启动的正确口径应更新为：

1. 本地开发已经收口到一条稳定启动命令
2. 当前运行主库仍是本地 SQLite，属于过渡态

不能再继续使用以下错误口径：

1. “项目已经正式切到云端数据库”
2. “所有页面现在都是正式真实态”
3. “本地看到结果就等于已经上云”
