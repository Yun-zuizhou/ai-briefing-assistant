# 统一内容层正式化判断与内容事实层稳定收口

- 文档状态：进行中
- 建立时间：2026-04-03
- 所属阶段：阶段二 - Today / 内容层设计与事实层收束
- 作用：对“统一内容层现在是否应该正式实体化”给出结果级判断，并明确当前内容事实层的稳定收口边界

---

## 1. 本轮要回答的不是泛泛问题，而是三个明确问题

1. 现在要不要正式建 `contents` 实体层
2. 如果现在不建，当前稳定过渡态的边界到底是什么
3. 如果后续要建，第一批最小落地范围应该是什么

---

## 2. 当前代码现实

基于当前真实代码，统一内容层已经部分成立，但成立方式不是“实体表已落地”，而是：

`content_ref -> /api/v1/content/by-ref -> UnifiedContentDetailResponse -> ArticlePage`

当前已成立的事实：

1. `dashboard.py` 已把 Today 主内容区统一输出为 `content_ref`
2. `content.py` 已支持按 `content_ref` 从 `hot_topics / rss_articles / opportunities` 三张事实表统一读取详情
3. `page_schemas.py` 已有统一内容详情契约 `UnifiedContentDetailResponse`
4. `ArticlePage` 已能按统一引用读取正文、标签、元信息、相关推荐
5. `HotTopics / Collections / Today` 已开始共用同一条统一引用链路

当前未成立的事实：

1. 没有真正落地的 `contents` ORM / 表 / 写入链路
2. `favorites` 仍是 `item_type + item_id` 组合引用，不是统一 `content_ref`
3. `reports` 仍主要聚合 `notes / favorites / history / todos`，并未建立统一内容事实层复用
4. `Today` 的内容加工、相关推荐、排序仍是规则型过渡实现
5. 当前不存在“统一内容写入中心”，仍然是三张分表各自写入

---

## 3. 结果级判断

### 3.1 当前不建议立即正式建 `contents` 实体表

这是本轮的明确结论。

原因不是因为统一内容层不重要，而是因为：

1. 当前主链路已经先用 `content_ref + by-ref` 把“统一读取”收住了
2. 当前真正不稳定的，不是“有没有 contents 表”，而是“事实边界和契约是否已经稳定”
3. 如果现在直接建 `contents`，会立刻带来：
   - 三张内容表如何迁移
   - 写入链路改造
   - 收藏与历史如何回填
   - D1 / SQLite 双环境如何同步
   - 现有过渡态测试如何整体重写
4. 这些成本会直接打断阶段二主线，把问题从“内容层收束”扩大成“基础事实层重构”

因此当前最准确的判断是：

`统一内容层应该被正式承认，但暂不应在阶段二当前时点直接实体化为 contents 表。`

### 3.2 当前应该正式化的是“统一内容契约”，不是“统一内容实体表”

当前更应该先正式化的，是以下四件事：

1. `content_ref` 作为唯一统一内容引用键
2. `/api/v1/content/by-ref` 作为统一详情读取入口
3. `UnifiedContentDetailResponse` 作为统一详情契约
4. `Today / HotTopics / Collections / ArticlePage` 全部沿用同一条引用链路

也就是说，当前阶段要先把：

`统一内容的读契约`

收成稳定结果，再决定要不要立刻补：

`统一内容的存储实体层`

---

## 4. 当前稳定过渡态边界

当前内容事实层可以被视为“稳定过渡态”的部分如下。

### 4.1 已稳定的读取边界

1. Today 主内容区使用 `content_ref`
2. 详情页按 `content_ref` 补读
3. 内容类型当前稳定为：
   - `hot_topic`
   - `article`
   - `opportunity`
4. 统一详情最小字段已稳定为：
   - `content_ref`
   - `content_type`
   - `id`
   - `title`
   - `summary`
   - `content`
   - `source_name`
   - `source_url`
   - `author`
   - `category_labels`
   - `tags`
   - `published_at`
   - `quality_score`
   - `detail_state`
   - `related_items`

### 4.2 已稳定的事实来源边界

当前统一内容详情仍然明确来源于三张原始事实表：

1. `hot_topics`
2. `rss_articles`
3. `opportunities`

因此当前统一内容层的真实定位应表述为：

`统一读取层`

而不是：

`统一存储层`

### 4.3 当前仍未稳定、不能误判为正式态的部分

以下内容仍不能误判成“统一内容正式态已完成”：

1. 收藏仍未切成 `content_ref`
2. 报告仍未围绕统一内容事实做复用
3. Today 的排序分仍是规则型
4. 相关推荐仍是规则型近似匹配
5. 内容摘要、正文清洗、来源可信度还没有统一加工中心
6. `contents` 表虽然存在于 SQL 设计锚点，但未进入当前代码现实

---

## 5. 当前阶段正确推进方向

### 第一方向：先把统一内容“读侧”彻底收稳

当前最该继续推进的是：

1. 把更多旧入口统一到 `content_ref`
2. 让收藏、历史、报告读取时也能识别 `content_ref`
3. 为 `content_ref -> by-ref` 补更完整的接口级测试
4. 继续明确哪些字段属于统一内容最小事实字段，哪些仍只是展示辅助字段

### 第二方向：把内容事实边界从“能用”推进到“可解释”

当前最需要补强的不是新页面，而是：

1. 统一排序口径
2. 统一加工字段口径
3. 统一相关推荐的最小规则边界
4. 统一 `detail_state / processing_stage` 的使用边界

也就是让当前内容层从：

`能串起来`

推进到：

`边界清楚、结果可解释`

### 第三方向：为后续实体化预留最小迁移点

虽然当前不建议立刻建 `contents` 表，但应该先把将来会影响实体化的点收住：

1. `content_ref` 必须继续保持稳定格式
2. `content_type` 枚举范围不要继续发散
3. 详情契约字段尽量稳定，不再频繁改名
4. 收藏和报告后续如需进入统一内容层，应优先围绕 `content_ref` 扩展，而不是再造另一套引用

---

## 6. 如果后续决定正式实体化，第一批最小落地范围是什么

如果阶段二后段或阶段三前段决定真的建 `contents`，第一批最小范围应只包含：

1. 三类内容：
   - `hot_topic`
   - `article`
   - `opportunity`
2. 最小字段：
   - `id`
   - `content_type`
   - `title`
   - `summary`
   - `content`
   - `source_name`
   - `source_url`
   - `author_name`
   - `category_name`
   - `published_at`
   - `hot_score`
   - `status`
3. 最小用途：
   - 统一详情读取
   - 收藏统一引用
   - Today / Report 的统一内容复用

第一批不建议一起做的事：

1. 不一起做复杂推荐系统
2. 不一起做 AI 摘要正式落表
3. 不一起做所有历史数据回填
4. 不一起做全量 D1 迁移重构
5. 不一起做内容全量去重与标准化工程

---

## 7. 当前最准确的一句话结论

当前阶段的正确推进不是：

`马上建 contents 表`

而是：

`先把统一内容引用、统一详情契约、统一事实边界收成稳定过渡态，再决定是否把统一内容层正式实体化。`

---

## 8. 本轮输出的最终判断

### 现在建不建

`现在不建。`

### 为什么不建

`因为当前阶段真正缺的不是实体表，而是统一读契约和事实边界的稳定化；现在直接建表会把阶段二主线扩大成事实层重构。`

### 当前过渡态边界

`content_ref + /api/v1/content/by-ref + UnifiedContentDetailResponse + 三张原始事实表承载，是当前统一内容层的稳定过渡边界。`

### 下一步最小落地范围

1. 把更多入口继续统一到 `content_ref`
2. 让收藏 / 报告逐步开始认识 `content_ref`
3. 补统一内容读取与 Today 主链路测试
4. 固定统一内容最小字段与处理状态口径
