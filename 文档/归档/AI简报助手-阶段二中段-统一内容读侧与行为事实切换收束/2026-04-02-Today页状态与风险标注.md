# Today 页状态与风险标注

- 文档状态：进行中
- 建立时间：2026-04-02
- 所属阶段：阶段 4 对话回写与主闭环联调
- 关注对象：Today 页
- 目的：记录 Today 页当前真实状态、成立边界和后续风险

---

## 1. 当前状态结论

Today 页当前可以定义为：

`页面结构已收束、接口联调已基本跑通，但信息获取仍然不真实。`

这里的“联调已基本跑通”指的是：

1. Today 页首屏已经能够通过 `/api/v1/dashboard/today` 获取结构化数据
2. `Chat -> interest -> Today` 链路已经成立
3. 对话更新关注后，Today 推荐内容会变化
4. 同一后端进程内刷新后，推荐结果仍保持一致

这里的“信息获取仍然不真实”指的是：

1. Today 页当前推荐逻辑仍然依赖共享虚拟兴趣状态
2. Today 页当前展示的 `worth_knowing / worth_acting / recommended_for_you` 主要仍是代码内构造内容
3. 当前 Today 页并不是直接从 Cloudflare D1 正式库或统一真实聚合事实层读取

因此，Today 页现在不是“纯 mock 原型”，但也不是“正式真实内容页”。

---

## 2. 当前已成立的部分

### 2.1 页面形态已成立

Today 页当前已经具备：

- 今日总述
- 因你关注而推荐
- 值得知道的
- 值得行动的
- 今日速记入口

这说明 Today 页作为一级主页面的结构已经成立。

### 2.2 对话驱动推荐变化已成立

当前代码中：

1. `chat/execute` 真实写入 `users.interests`
2. 写入成功后同步镜像到共享虚拟兴趣状态
3. `/api/v1/dashboard/today` 读取共享虚拟兴趣状态

因此当前已经成立：

`Chat -> interest 更新 -> Today 推荐变化`

### 2.3 当前不是纯静态展示

Today 页虽然还不是正式数据源驱动，但已经不只是完全静态页面。

它至少具备：

- 接口层
- 推荐变化逻辑
- 与对话页的联动

这说明 Today 页已进入“过渡态可联调页面”，不是单纯原型展示页。

---

## 3. 当前未成立的部分

### 3.1 信息获取不真实

这是当前 Today 页最核心的问题。

当前 Today 页的内容来源还没有做到：

- 真实热点数据聚合
- 真实文章筛选
- 真实机会聚合
- 基于真实兴趣和真实内容的推荐排序

当前代码里：

- `recommended_for_you` 仍由兴趣映射表生成
- `worth_knowing` 为接口内写死的内容
- `worth_acting` 为接口内写死的内容

所以“信息获取还不真实”这个判断是正确的，而且应该明确写出来。

### 3.2 Today 还没有统一真实事实源

当前 Today 页还不是：

- 直接读取 `users.interests`
- 再结合真实热点 / RSS / 机会表进行聚合
- 再统一输出真实 Today 聚合结果

它目前仍然属于：

`真实写入 + 虚拟读取 + 静态内容填充`

### 3.3 云端正式库未进入 Today 主读链路

虽然 Cloudflare D1 远端正式库已经落地，但当前 Today 页并没有直接读取：

- 远端 D1 正式库
- 或其对应的统一后端事实层

所以当前不能把 Today 页误判为“正式上云读取已完成”。

---

## 4. 当前风险

### 风险 1：容易把“链路已通”误判成“信息真实”

当前最容易出现的误判是：

- 看到 Today 页能变化，就以为 Today 已经真实化

但事实是：

- 当前成立的是交互链路
- 不是内容事实层

### 风险 2：Today 的推荐变化不等于 Today 的内容真实

当前 Today 推荐命中虽然能变，但它命中的仍然是虚拟映射和静态内容。

因此如果不单独标注，后续很容易把：

- “推荐逻辑有变化”

误写成：

- “今日内容已真实聚合”

### 风险 3：本地 SQLite 与 Cloudflare D1 容易被混同

当前项目同时存在：

1. 本地 SQLite 运行链路
2. Cloudflare D1 远端正式库
3. 共享虚拟状态

如果不显式标注 Today 页当前读取层，很容易把当前本地读写链路误说成已经正式上云。

---

## 5. 当前最合理的判断

Today 页当前应被定义为：

`主页面结构基本成立，交互联调已基本跑通，但内容事实层仍处于过渡态。`

进一步拆开就是：

1. 页面层：基本成立
2. 对话联动层：基本成立
3. 真实内容获取层：未成立
4. 统一真实事实层：未成立
5. 正式上云读取层：未成立

---

## 6. 对项目阶段判断的影响

这意味着当前阶段的重点不能只理解成“继续接数据库”。

更准确的理解应该是：

当前项目下一步需要把 Today 页从“联调成立”推进到“内容真实成立”。

因此后续工作应至少包含三块：

1. 检查后端已有信息获取能力与项目功能要求的匹配度
2. 检查前端 Today 页字段与后端真实可供给数据的匹配度
3. 再决定哪些数据应该优先接入真实数据库 / 真实聚合接口

---

## 7. 下一步建议

最合理的下一步顺序：

1. 先审视“信息获取 -> 信息加工 -> 信息整合 -> 信息呈现”全链路
2. 明确 Today 页当前哪些字段已有真实来源，哪些仍是虚拟占位
3. 核对后端现有能力与项目功能要求、前端字段要求的匹配度
4. 在匹配度清楚之后，再继续推进真实数据库与真实聚合接入

不建议的顺序：

1. 在当前供给侧未梳理清楚前，直接继续大量数据库接入
2. 在 Today 内容事实层未核清前，误判 Today 已进入正式态
