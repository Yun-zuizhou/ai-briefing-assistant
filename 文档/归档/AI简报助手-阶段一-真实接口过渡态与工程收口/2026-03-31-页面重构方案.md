# 页面重构方案

- 方案名称：AI简报助手（时代与我）页面重构方案
- 归档状态：进行中
- 方案时间：2026-03-31
- 依据文档：
  - `文档/产品设计/需求-功能-呈现映射评估-2026-03-31.md`
  - `文档/UI设计/核心页面呈现评估-2026-03-31.md`
- 方案目标：
  - 区分前端页面架构、后端能力架构、前后端联动边界
  - 将现有页面从“功能堆叠”重构为“围绕主闭环组织”
  - 形成可以直接进入设计和开发排期的执行方案

## 1. 重构目标

本次页面重构不是简单调整视觉层级，而是围绕项目主闭环重组系统：

`今日信息 -> 转化行动 -> 自我记录 -> 周期回顾`

因此，重构目标分为三层：

1. 前端层：重组页面信息架构和导航结构。
2. 后端层：重组页面背后的数据能力和接口边界。
3. 联动层：定义页面与接口之间的职责契约，避免前端继续同时依赖 mock 逻辑和旧接口。

## 2. 重构原则

### 2.1 设计原则

1. 首页只做“今日简报首页”，不再承担混合中枢角色。
2. 对话只做“快捷表达层”，不再承担全部业务浏览。
3. 内容按用户意义组织，不按数据来源组织。
4. 成长相关页面统一为一个连续叙事体系。
5. 所有内容卡片都必须有下一步动作。

### 2.2 技术原则

1. 前端页面结构优先稳定，减少直接依赖零散 mock。
2. 后端接口按页面需求反推，而不是按表结构自然外露。
3. 页面所需数据优先聚合返回，减少前端拼装负担。
4. 对话操作结果必须回写系统，不停留在聊天气泡。

## 3. 前端页面架构重构

## 3.1 新的一级导航

建议将当前结构收敛为 4 个一级主导航：

1. 今日
2. 对话
3. 行动
4. 成长

这 4 个入口比当前“首页 / 热点 / 待办 / 我的”更能反映产品真实价值。

### 3.1.1 今日

定位：

- 今日总览
- 今日筛选结果
- 今日值得看的信息
- 今日值得做的行动

替代关系：

- 取代当前首页的混合结构
- 吸收部分热点页、机会页的首屏价值

### 3.1.2 对话

定位：

- 快捷表达
- 快捷创建
- 快捷调整
- 快捷追问

保留：

- 自然语言入口
- 意图识别反馈

弱化：

- 作为完整浏览容器的角色

### 3.1.3 行动

定位：

- 待办
- 收藏
- 跟进
- 提醒

作用：

- 将“看过的内容”与“准备去做的事”集中管理

### 3.1.4 成长

定位：

- 日志
- 画像
- 历史
- 周报 / 月报 / 年度报告

作用：

- 把“记录自己，看见成长”变成前台主轴

## 3.2 目标页面结构

### 3.2.1 今日页

建议结构：

1. 今日总述
2. 因你关注而推荐
3. 值得知道的
4. 值得行动的
5. 今日速记入口

页面重点：

- 首屏必须在 5 秒内让用户知道今天最重要的 3 到 5 个点。
- 所有区块必须有明确动作。

建议保留组件类型：

- BriefingHero
- RecommendedReasonCard
- KnowledgeCard
- ActionCard
- QuickNoteEntry

建议下线或下沉：

- 重复的热点趋势视觉块
- 与摘要高度重复的卡片流

### 3.2.2 对话页

建议结构：

1. 当前可做的事情提示
2. 快捷示例按钮
3. 对话消息流
4. 识别结果反馈条
5. 操作成功后的跳转入口

页面重点：

- 不只是“聊天”，而是“执行后能落地”。

建议组件类型：

- IntentShortcutPanel
- ChatMessageList
- IntentResultBanner
- FollowUpActionBar

### 3.2.3 行动页

建议将当前 TodoPage、收藏、部分机会跟进能力合并为统一行动中心。

建议结构：

1. 今日待办
2. 来自内容的待办
3. 收藏待处理
4. 跟进中的机会
5. 提醒设置入口

页面重点：

- 把“信息消费后的下一步”集中管理。

建议组件类型：

- TodoSection
- SavedForLaterSection
- OpportunityFollowupSection
- ReminderSummaryCard

页面边界说明：

- `行动页` 是行动体系的主聚合页，负责展示总览，不负责承载全部细节编辑。
- `待办`、`收藏稍后看`、`跟进`、`提醒` 后续可以保留为二级页面，但其存在前提是：
  - 主聚合页已经成立
  - 二级页面承担明确的深度管理职责
- 在重构早期，不应同时把“行动总览页”和多个功能完整独立页并行重做，避免重复实现。

### 3.2.4 成长页

建议将当前 MyPage 改造成成长中心首页。

建议结构：

1. 本周成长摘要
2. 最近记录关键词
3. 画像一句话摘要
4. 历史回顾入口
5. 报告入口矩阵

页面重点：

- 用户应感受到“我这段时间成了什么样的人”。

建议组件类型：

- GrowthSummaryHero
- ThoughtKeywordCloud
- PersonaSnapshotCard
- ReportEntryGrid
- HistoryPreviewList

页面边界说明：

- `成长页` 是成长体系总览页。
- `日志`、`画像`、`历史`、`报告` 可以保留为二级页，但首页必须先能独立表达“成长中心”。
- 设置类页面不再混入成长叙事主结构。

### 3.2.5 设置页

设置页不再挂在成长叙事内部，而作为系统辅助页保留。

建议拆分：

- 通知设置
- 账号设置
- 数据源设置
- AI 提供商设置

## 4. 后端能力架构重构

## 4.1 后端不再按“已有表”思考，而按“页面需要什么”思考

当前后端已有：

- 用户
- 热点
- 机会
- RSS 文章
- 待办
- 意图识别
- AI 服务

但页面重构后，需要后端提供的是“页面聚合能力”，而不仅是资源列表。

## 4.2 页面对应的后端能力划分

### 4.2.1 今日页后端能力

需要新增或补齐聚合接口：

- 今日简报聚合
- 个性化推荐理由
- 值得行动的内容聚合
- 今日记录入口的快捷写入

建议新增接口组：

- `GET /api/v1/dashboard/today`
- `POST /api/v1/dashboard/quick-note`

推荐返回结构：

- today_summary
- recommended_for_you
- worth_knowing
- worth_acting
- quick_note_prompt

### 4.2.2 对话页后端能力

当前意图识别已存在，但还缺“意图执行”层。

建议区分两步：

1. 识别意图
2. 执行动作

建议接口组：

- `POST /api/v1/chat/recognize`
- `POST /api/v1/chat/execute`

执行结果应标准化返回：

- intent_type
- affected_entity
- success_message
- next_action
- deep_link

### 4.2.3 行动页后端能力

建议统一由行动聚合接口提供：

- 待办列表
- 收藏列表
- 跟进列表
- 提醒摘要

建议接口组：

- `GET /api/v1/actions/overview`
- `POST /api/v1/actions/create-from-content`
- `POST /api/v1/actions/favorite`
- `POST /api/v1/actions/follow-opportunity`

### 4.2.4 成长页后端能力

这是当前最弱但最关键的部分。

建议后端聚合：

- 一段时间内的记录摘要
- 兴趣变化
- 行动完成情况
- 画像生成结果
- 周报/月报/年度报告索引

建议接口组：

- `GET /api/v1/growth/overview`
- `GET /api/v1/growth/history`
- `GET /api/v1/growth/reports`
- `POST /api/v1/growth/profile/generate`

## 4.3 建议补充的数据实体

当前重构要成立，后端建议增加或明确以下实体：

### 第一优先级：必须补齐

1. `user_favorite`
2. `user_note`
3. `user_read_history`

理由：

- 这三类数据直接支撑“收藏 / 记录 / 历史回顾”主闭环。

### 第二优先级：可通过现有模型过渡，后续再实体化

4. `user_interest`
5. `user_action_link`
6. `generated_report`

理由：

- 这些能力当前可以先通过现有用户字段、服务层聚合、静态报告索引或轻量表结构过渡。
- 不应在页面重构阶段一次性引入过多新实体，导致后端范围失控。

原则：

- 先保障主闭环成立，再补模型完备性。
- 新实体必须能直接服务页面，不做纯理论扩表。

## 5. 前后端联动方案

## 5.1 联动原则

前端负责：

- 页面结构
- 交互状态
- 临时 UI 状态
- 路由和模块切换

后端负责：

- 数据事实
- 个性化筛选
- 聚合结果
- 意图执行结果
- 长期沉淀数据

前后端共享契约负责：

- 统一数据模型
- 页面所需聚合结构
- 操作结果结构

## 5.2 前后端联动的核心问题

当前最大问题不是没有接口，而是：

1. 前端仍使用本地 mock 作为主驱动。
2. 前端 `api.ts` 中有旧路径和未落地接口。
3. 页面需求没有被拆解成稳定的后端聚合返回结构。

因此，联动重构必须分阶段进行。

## 5.3 联动阶段划分

### 阶段 1：契约统一

目标：

- 清理前端旧接口路径
- 明确每个主页面的接口输入输出

产出：

- 页面接口清单
- 统一 TypeScript 类型
- 后端 Pydantic 响应模型

验收标准：

- 前端不再新增旧路径接口调用
- 4 个主页面都有明确输入输出结构
- 接口命名、响应模型、前端类型定义三者一致

### 阶段 2：聚合接口落地

目标：

- 今日页、行动页、成长页优先接入真实聚合接口
- 对话识别和执行接口完成标准化落地

优先级：

1. 今日页
2. 行动页
3. 成长页
4. 对话执行页

验收标准：

- `dashboard/today`、`actions/overview`、`growth/overview`、`chat/recognize`、`chat/execute` 均有稳定响应
- 前端可在无手工 mock 注入的情况下完成主页面首屏展示
- 对话执行结果可返回标准化 deep link 和实体结果

### 阶段 3：对话结果回写系统

目标：

- 对话创建关注、待办、记录、提醒后，所有页面能实时体现变化

验收标准：

- 通过对话新增待办后，行动页可见
- 通过对话记录想法后，成长页或日志页可见
- 通过对话调整关注后，今日页推荐内容发生变化

### 阶段 4：mock 逐步退场

目标：

- mock 仅保留设计演示和兜底，不再承担核心主流程

验收标准：

- 首页、行动页、成长页、对话执行结果不再依赖 mock 作为主数据源
- mock 数据只在接口失败、演示模式或局部开发场景使用

## 5.4 页面与接口映射

| 页面 | 前端职责 | 后端职责 | 联动关键点 |
|---|---|---|---|
| 今日页 | 呈现聚合结果、组织视觉优先级 | 输出今日简报聚合数据 | 一个接口返回首屏核心内容 |
| 对话页 | 输入、反馈、跳转 | 意图识别 + 执行动作 | 返回标准化结果和 deep link |
| 行动页 | 聚合展示不同行动状态 | 输出行动总览与变更接口 | 统一行动实体 |
| 成长页 | 呈现连续叙事 | 输出成长摘要、历史、报告索引 | 统一成长聚合结构 |
| 设置页 | 表单交互 | 保存偏好、源配置、AI 配置 | 前端按模块分设置项 |

## 6. 执行计划

## 6.1 第一阶段：信息架构重构

目标：

- 先定页面，不急着改所有代码

任务：

1. 确认 4 个一级导航
2. 输出页面树
3. 确认哪些旧页面合并、哪些保留
4. 产出新路由草案

预计耗时：1 到 2 天

完成判定：

- 新页面树明确
- 每个旧页面都有去向
- 主聚合页与二级页边界明确

## 6.2 第二阶段：接口契约重构

目标：

- 让前端页面有稳定依赖

任务：

1. 列出今日页、行动页、成长页、对话页的字段需求
2. 更新前端 `api.ts`
3. 定义后端聚合响应模型

预计耗时：2 到 3 天

完成判定：

- 前端 API 调用清单完成统一
- 每个主页面的字段来源明确
- 已形成前后端共享字段模型

## 6.3 第三阶段：关键页面实现

目标：

- 先做主闭环，不追求一次改完所有页面

优先顺序：

1. 今日页
2. 对话页
3. 行动页
4. 成长页

预计耗时：5 到 8 天

完成判定：

- 4 个主页面均有可运行版本
- 页面主信息不依赖手动拼装 mock
- 能跑通至少一条完整主闭环

## 6.4 第四阶段：旧页面收束

目标：

- 清理重复页面和重复入口

任务：

1. 合并文章页
2. 合并收藏页
3. 将报告页收纳进成长体系
4. 调整设置入口位置

预计耗时：2 到 3 天

完成判定：

- 重复入口移除
- 旧页面仅保留必要兼容壳
- 页面结构与导航结构一致

## 7. 可执行清单

1. 重画路由结构。
2. 将 `HomePage` 改造为 `TodayPage`。
3. 将 `MyPage` 改造为 `GrowthPage`。

5. 给 `ChatPage` 增加快捷示例和执行反馈层。

### 7.2 后端清单

1. 新增 dashboard 聚合接口。
2. 新增 actions 聚合接口。
3. 新增 growth 聚合接口。
4. 将 chat 能力拆为 recognize 和 execute。
5. 补 user_note、favorite、history 等实体或服务层封装。

### 7.3 联动清单

1. 更新前端 API 基础路径和接口定义。
2. 为每个主页面建立 TS 类型。
3. 为每个聚合接口建立 Pydantic 响应模型。
4. 规定对话执行结果标准返回结构。
5. 保留 mock 作为兜底，但从主流程退场。

## 8. 结论

这份方案的核心不是“把页面改得更好看”，而是把产品重新组织成一个真正成立的系统。

页面重构后，前端会更清楚自己在承载什么：

- 今日负责筛选后的信息
- 对话负责低成本表达
- 行动负责转化
- 成长负责沉淀和回顾

后端也会更清楚自己在提供什么：

- 不是零散资源接口
- 而是页面所需的聚合能力和执行结果

前后端联动完成后，项目才会从“原型 + 若干接口”进入“结构清晰、可以持续迭代”的产品状态。
