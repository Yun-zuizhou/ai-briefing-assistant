## 6. 当前主要不一致问题

## 6.1 Base URL 与路由前缀不统一

前端当前基础地址是：

`http://localhost:5000/api`

但后端主资源路由实际是：

- `/api/v1/hot-topics`
- `/api/v1/opportunities`
- `/api/v1/todos`
- `/api/v1/rss`
- `/api/v1/intent`

结果：

- `getHotTopics()` 和 `getOpportunities()` 当前路径不正确
- `recognizeIntent()` 因为额外拼了 `/v1/...`，反而恰好正确

这说明前端 API 层存在“部分旧路径 + 部分新路径”混用。

## 6.2 首页对响应结构的预期与后端返回不完全一致

前端 `HomePage.tsx` 当前写法把 `hotTopicsResponse.data` 和 `opportunitiesResponse.data` 直接传入卡片生成逻辑。

但后端实际返回结构为：

```json
{
  "total": 123,
  "items": [...]
}
```

因此当前前端若直接使用 `data`，会拿到列表对象而不是数组，存在结构不一致问题。

## 6.3 对话页仍是“后端识别 + 前端执行”

当前 `useChatLogic.ts` 的流程是：

1. 调后端做 intent recognize
2. 将结果转成前端本地 intent
3. 本地直接修改：
   - interests
   - todos
   - pushTime
   - thoughtRecords

这意味着：

- 后端只参与识别
- 执行仍然发生在前端本地状态
- 当前对话流程尚未进入真实系统回写阶段

这与后续 `chat/execute` 方向是一致的重构重点。

## 6.4 `api.ts` 中存在明显的历史规划接口残留

以下接口目前更像历史规划残留，而不是现阶段可直接接入项：

- learning-resources
- briefings/generate
- ai/generate-briefing
- ai/generate-profile
- ai/summarize-ideas
- stats

建议：

- 在阶段 2 不把这些接口视为“待修复即可使用”
- 应明确标记为：
  - 旧规划接口
  - 当前未实现
  - 暂不纳入主闭环改造

## 7. 结论与建议

## 7.1 当前真实情况总结

前端当前真正依赖的 API 很少，只有：

1. 热点列表
2. 机会列表
3. 意图识别

这说明当前系统状态是：

- 前端大量功能仍由本地状态和 mock 驱动
- 后端已经具备更多资源能力，但前端尚未接入
- API 层已经积累了旧接口残留，需要清理

## 7.2 对 T2-1 的结论

`T2-1` 可以视为已完成第一轮梳理。

下一步应立即进入：

1. `T2-2` 主页面字段清单最终版
2. `T2-3` TypeScript 类型
3. `T2-4` Pydantic 响应模型
4. `T2-5` 路由和路径统一

## 7.3 当前最优先修复项

按优先级排序：

1. 修正前端基础路径与资源路径不一致问题
2. 修正首页对后端响应结构的错误预期
3. 明确对话页当前是“识别已接后端、执行仍在前端本地”
4. 标记并隔离历史规划接口，避免混入主闭环
