## 2. 新信息架构总览

项目新的信息架构建议固定为四条主线：

1. 今日
2. 对话
3. 行动
4. 成长

这四条主线分别承载：

- 今日：今天最值得知道和最值得行动的内容
- 对话：低成本表达和快捷执行
- 行动：待办、收藏、跟进、提醒
- 成长：记录、历史、画像、报告

核心逻辑保持为：

`今日信息 -> 行动转化 -> 自我记录 -> 周期回顾`

## 4. 页面树与模块清单

## 4.1 今日页 `/today`

### 页面目标

- 让用户在最短时间内理解今天的重点
- 强化“这份内容和我有关”
- 让信息可以快速转化成行动或记录

### 页面模块

1. 今日总述
2. 因你关注而推荐
3. 值得知道的
4. 值得行动的
5. 今日速记入口

### 模块字段清单

#### 模块 A：今日总述

需要字段：

- date_label
- issue_number
- summary_title
- summary_text
- mood_tag

字段来源：

- 前端：`date_label`

- interest_name
- recommendation_reason

字段来源：

#### 模块 D：值得行动的

## 6. 前端状态与后端字段边界

## 6.1 前端本地状态负责

前端应只长期保留以下状态：

1. 登录态缓存
2. 当前页面 UI 状态
3. 输入框临时值
4. 过渡性乐观更新状态
5. 本地兜底展示状态

## 6.2 后端必须负责

后端必须提供以下事实型数据：

1. 今日聚合结果
2. 用户兴趣与推荐关系
3. 待办、收藏、跟进、提醒真实状态
4. 日志、历史、画像、报告聚合结果
5. 对话执行结果

## 6.3 不应继续长期混用的内容

以下内容不应再长期由前端 mock 作为主数据源：

1. 今日简报首屏核心数据
2. 热点和机会列表主数据
3. 用户成长摘要
4. 对话执行后的系统状态

mock 后续仅用于：

- 设计演示
- 接口失败兜底
- 开发阶段局部占位

## 7. 前后端契约草案

## 7.1 今日页聚合接口

建议接口：

`GET /api/v1/dashboard/today`

建议返回结构：

```json
{
  "date_label": "2026年3月31日",
  "issue_number": 128,
  "summary": {},
  "recommended_for_you": [],
  "worth_knowing": [],
  "worth_acting": [],
  "quick_note_prompt": "今天有什么想法想记下来？"
}
```

## 7.2 对话识别与执行接口

建议接口：

- `POST /api/v1/chat/recognize`
- `POST /api/v1/chat/execute`

建议返回结构：

```json
{
  "intent_type": "create_todo",
  "entities": {},
  "success": true,
  "affected_entity": {
    "type": "todo",
    "id": 123
  },
  "success_message": "已加入待办",
  "deep_link": "/actions/todos"
}
```

## 7.3 行动页聚合接口

建议接口：

`GET /api/v1/actions/overview`

## 7.4 成长页聚合接口

建议接口：

`GET /api/v1/growth/overview`
