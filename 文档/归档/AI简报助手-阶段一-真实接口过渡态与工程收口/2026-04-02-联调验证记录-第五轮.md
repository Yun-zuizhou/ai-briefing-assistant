# 联调验证记录 - 第五轮

- 文档状态：进行中
- 建立时间：2026-04-02
- 所属阶段：阶段 4 对话回写与主闭环联调
- 对应任务：`T4-3 对话调整关注回写`
- 本轮目标：验证 `Chat -> interest -> Today` 链路是否成立，并明确当前成立边界

---

## 1. 本轮结论

本轮可以明确下结论：

`Chat -> interest -> Today` 链路在“本地 SQLite 可写 + Today 读取共享虚拟兴趣状态”的当前阶段约束下，已经成立。

但需要明确：

这条链路当前还不是“真实数据库统一事实源”的最终态，而是：

1. `chat/execute` 真实写入 `users.interests`
2. 写入成功后同步镜像到共享虚拟兴趣状态
3. `/api/v1/dashboard/today` 继续读取共享虚拟兴趣状态

因此，本轮成立的是：

`真实写入成功 -> 镜像到虚拟兴趣状态 -> Today 推荐变化`

而不是：

`Today 直接读取真实数据库兴趣字段`

---

## 2. 首次验收失败原因

首次执行接口级验收时，`chat/execute` 没有进入真实写入成功分支，而是直接回退到虚拟写入。

排查结果确认：

不是意图识别失败，也不是用户不存在，而是本地 SQLite 缺少 `history_entries` 表。

具体表现为：

1. `users` 表存在
2. `chat/execute` 在更新兴趣后会追加 `history_entries`
3. 提交事务时因为 `history_entries` 不存在而报错
4. 后端异常被当前代码捕获后，整条写入链路回退为虚拟写入

这解释了为什么第一次看到的是：

- `successMessage = 已更新关注内容（虚拟写入）`
- `resultSummary = 新增关注：AI。当前数据库不可写，结果来自虚拟回退。`

---

## 3. 本轮修正动作

为了让本地接口级验收具备真实写入前置条件，本轮执行了：

1. 在项目本地目录安装运行依赖到 `.pydeps_runtime`
2. 执行 `app/init_db.py`
3. 补齐当前本地 SQLite 缺失表结构

本轮补齐后，本地数据库至少已经具备：

- `users`
- `todos`
- `history_entries`
- `notes`
- `favorites`

从而允许 `chat/execute` 完成兴趣更新与历史记录写入。

## 4. 最终验收过程

本轮验收方式：

1. 将 `users.id=1` 的兴趣重置为空数组
2. 重置共享虚拟状态
3. 先请求一次 `/api/v1/dashboard/today`
4. 再调用 `/api/v1/chat/execute?user_id=1`
5. 再次请求 `/api/v1/dashboard/today`
6. 再做一次同进程内刷新读取验证

本轮输入：

- 对话输入：`我想关注AI`

## 6. 当前成立边界

本轮成立的是：

1. `Chat -> add_interest -> users.interests` 真实写入成立
2. 真实写入成功后镜像到共享虚拟兴趣状态成立
3. `Today` 基于共享虚拟兴趣状态的推荐变化成立
4. 同一后端进程内刷新后保持一致成立

本轮尚未成立的是：

1. `Today` 直接读取真实数据库兴趣字段
2. 应用重启后仍然只依赖真实数据库单一事实源
3. 基于 Cloudflare D1 的远端正式读写联调

---

## 7. 当前风险与下一步

当前风险：

1. `Today` 仍然不是直接读真实数据库，而是读共享虚拟兴趣状态
2. 本地 SQLite 表结构曾经落后于模型定义，说明当前本地运行环境存在模式漂移风险
3. Cloudflare D1 已落地，但本地 FastAPI 默认数据库仍是 SQLite，尚未切到统一正式事实源

最合理的下一步：

1. 把 Today 页兴趣来源从共享虚拟状态收束到真实事实层
2. 明确本地 SQLite 与 Cloudflare D1 的阶段分工
3. 再进入“真实数据库统一事实源”的第二轮联调验证
