# 项目级下一阶段收益排序与后端核验收口

- 文档状态：进行中
- 建立时间：2026-04-05
- 所属阶段：阶段十二 - 项目级下一阶段收益排序
- 文档职责：在阶段十与阶段十一连续收口后，对当前主线数据库/后端能力做项目级核验，并收出下一条真正值得进入执行面的业务主线

---

## 1. 当前要回答的问题

阶段十二当前不再继续补局部页面尾差，而是先回答 3 个问题：

1. 当前数据库是否已经达到“主线级基本落成”
2. 当前关键后端接口是否已经达到“主线级基本建立完成”
3. 在这个前提下，下一条真正值得进入执行面的业务主线到底是谁

---

## 2. 当前主线数据库与后端能力核验

### 2.1 已稳定成立

当前可以明确收口为“已稳定成立”的部分：

1. 主闭环正式读侧：
   - `dashboard/today`
   - `actions/overview`
   - `preferences/growth-overview`
   - `content/by-ref`
2. Chat 正式执行与会话链：
   - `chat/execute`
   - `chat/reclassify`
   - `chat/sessions`
   - `chat/sessions/{session_id}/messages`
3. 周期报告主链：
   - `GET /api/v1/reports`
   - `GET /api/v1/reports/weekly`
   - `GET /api/v1/reports/monthly`
   - `GET /api/v1/reports/annual`
4. 当前主线所需正式承接位：
   - `chat_session / chat_message`
   - `reports`
   - `briefings`
   - `user_profiles`
   - `opportunity_follows`

当前判断：

- 这些已经足够支撑“主线级基本稳定”
- 不应再把它们误写成“仍未建立”

### 2.2 已后置，不是当前缺口

当前明确属于“已后置”的部分：

1. `reports / briefings / user_profiles / opportunity_follows` 的进一步覆盖面扩展
2. Chat 完整多会话管理、会话归档、会话搜索
3. 统一内容层重新拉起
4. `HistoryLogsPage` 更高层统一读侧扩展
5. 年报分享图片生成等增强型能力

当前判断：

- 这些不是当前主线缺口
- 不应借阶段十二核验之名重新拉回执行面

### 2.3 真实缺口

当前真正还能构成“下一条业务主线”的缺口，不在数据库和主接口是否存在，而在：

1. `HistoryBriefPage` 当前仍是整页 mock
2. 但前端已经存在：
   - `getReports()`
   - `getWeeklyReport()`
   - `getMonthlyReport()`
   - `getAnnualReport()`
3. 后端也已经存在：
   - `GET /api/v1/reports`
   - `GET /api/v1/reports/weekly`
   - `GET /api/v1/reports/monthly`
   - `GET /api/v1/reports/annual`

这意味着：

- 当前真正的缺口不是“报告接口没建”
- 而是“历史简报 / 周期回顾回看页还没有真正消费已成立的报告读侧”

---

## 3. 当前阶段性结论

阶段十二当前应正式收口为：

1. 数据库当前可以判断为：
   - `主线级基本落成`
   - 但不能写成“产品级全量落成”
2. 后端关键接口当前可以判断为：
   - `主线级基本建立完成`
   - 但不能写成“未来所有扩展接口都已完成”
3. 当前项目级真实缺口，不是数据库/接口再补全设计，而是：
   - `已成立报告读侧还没有真正收回到历史简报 / 周期回顾回看产品面`

因此当前下一条真正值得进入执行面的业务主线，应正式收口为：

- `历史简报与周期回顾回看正式化`

---

## 4. 为什么是这条主线

当前它比其他候选更高收益，原因是：

1. 它直接连接“周期回顾 -> 历史回看”这条业务链
2. 它不是重新扩对象层，而是把已经成立的报告读侧继续收回到用户可感知页面
3. `HistoryBriefPage` 当前仍是明显 mock，这比继续打磨 `Chat / Journal / HistoryLogs` 的局部细节更像新的业务缺口
4. 当前依赖已经存在：
   - 数据库承接位已成立
   - 报告接口已成立
   - 报告详情页已成立
5. 因而这条线具备：
   - 收益高
   - 依赖清楚
   - 不需要重新设计对象层

---

## 5. 当前下一步入口

如果阶段十二继续往下走，下一步最合适的入口应是：

1. 新开一份并列主线文档，专门承接：
   - `HistoryBriefPage` 是否切到 `getReports()`
   - 周报 / 月报 / 年报入口如何在历史回看页正式消费
2. 第一优先先判断：
   - `HistoryBriefPage` 当前最小正式读侧应该长什么样
3. 第二优先再判断：
   - 历史简报页是否需要直接跳到对应周报 / 月报 / 年报详情

---

## 6. 一句话结论

`阶段十二当前已经可以明确判断：数据库与关键后端接口都已达到主线级基本稳定，当前真正值得进入执行面的下一条业务主线不是继续补后端设计，而是“历史简报与周期回顾回看正式化”。`
