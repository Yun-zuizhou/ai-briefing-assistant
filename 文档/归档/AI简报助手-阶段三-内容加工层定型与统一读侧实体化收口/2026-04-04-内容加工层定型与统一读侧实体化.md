## 9. 正式加工结果层最小结果对象定义

基于当前阶段性收口判断，现阶段先收敛一个**最小结果对象**，用于回答：

“如果后续真的进入更正式的加工结果层，第一批最小结果到底承接什么？”

### 9.1 当前建议对象名

建议名称：

- `content_processing_result`

命名理由：

1. 当前要表达的是“加工结果”，不是原始内容事实
2. 当前还不想提前把它绑定到 `contents`
3. 当前也不想误导成最终正式表名，因此先用结果对象名，而不是实体表名

### 9.2 当前最小结果对象职责

这个对象当前只承接 3 类信息：

1. **来源锚点**
   - 它是由哪个原始内容事实加工而来
2. **稳定加工结果**
   - 哪些字段已经足够稳定，值得被视为“结果”
3. **结果状态**
   - 这个结果是在什么规则版本、什么时间点生成出来的

这个对象当前**不承接**：

1. 原始全文事实
2. 视图级瞬时字段
3. 页面展示临时文案
4. 用户行为事实
5. Chat 执行副作用

### 9.3 当前建议字段

当前推荐的最小字段集合如下：

#### A. 结果锚点字段

1. `source_content_ref`
   - 原始来源统一引用，例如 `article:1`
2. `source_content_type`
   - 原始来源类型，例如 `article`
3. `source_item_id`
   - 原始来源主键，便于内部联查

#### B. 稳定结果字段

1. `normalized_title`
   - 标准化标题
2. `normalized_summary`
   - 标准化摘要
3. `normalized_category_labels`
   - 标准化分类集合
4. `normalized_tags`
   - 标准化标签集合
5. `quality_score`
   - 当前可复用的质量分结果
6. `published_at`
   - 标准化发布时间

#### C. 结果状态字段

1. `processing_version`
   - 当前加工规则版本
2. `processed_at`
   - 本次结果生成时间
3. `is_stale`
   - 当前结果是否已过期

### 9.4 当前明确不进入最小结果对象的字段

以下字段当前**不建议**写进最小结果对象：

1. `ranking_score`
   - 因为 Today / Reports / Related 的排序语境不同，当前仍是视图级计算值
2. `match_score`
   - 因为它依赖兴趣、主题、上下文，是临时匹配值
3. `relevance_reason`
   - 因为它是视图文案，不是稳定结果事实
4. `insights`
   - 因为它仍是规则型解释文案
5. `related_items`
   - 因为它是读侧关系结果，不是单对象稳定结果
6. `hot_score`
   - 因为它更接近原始热点域或特定视图域，不适合作为通用结果字段

### 9.5 当前生命周期建议

当前建议把它视为**可重算的轻结果对象**，生命周期先按下面定义：

1. **生成时机**
   - 读取 `article` 详情时发现没有结果缓存
   - 或原始事实有更新时触发重算
2. **刷新时机**
   - 原始标题、摘要、分类、标签、质量分发生变化
   - 加工规则版本变化
3. **失效时机**
   - `processing_version` 落后于当前规则版本
   - 来源内容被删除或引用失效
4. **覆盖策略**
   - 当前建议允许整对象重算覆盖，不做增量 patch

### 9.6 当前存储判断

当前结论是：

1. 这个最小结果对象**已经值得被定义**
2. 但当前仍**不建议立即独立持久化**
3. 第一阶段更建议把它当成：
   - 明确的结果对象定义
   - 代码层结果结构
   - 是否落库前的判断基线

也就是说，当前推荐路径是：

先把“最小结果对象定义”收清楚，再决定它是否要从代码层进入持久化结果层。

### 9.7 为什么这样定义

理由：

1. 当前这组字段已经跨 `Today / content / reports` 三侧稳定出现
2. 它们大多是“标准化后还能复用”的结果，不是瞬时视图值
3. 这组字段足够小，不会把阶段三直接拖进表设计和 migration
4. 它能把“正式加工结果层”与“读侧投影层”第一次真正区分开

### 9.8 当前方案的缺陷

缺陷也要明确保留：

1. **结果对象仍偏 article-first**
   - 当前虽然名字通用，但现实上是围绕 `article` 最先收口的
2. **还没覆盖关系型结果**
   - 相关推荐、主题趋势、推荐解释仍在对象外
3. **还没回答是否落库**
   - 这个定义只解决“结果是什么”，没解决“结果存哪里”
4. **还没回答批量生成**
   - 当前只定义了按需生成 / 重算思路，没有定义批处理链路
5. **还没回答多规则并存**
   - 若未来存在多套排序或推荐版本，当前对象仍显得偏薄

### 9.9 当前收口结论

当前可以把“正式加工结果层”的第一步收口为：

1. 先承认 `content_processing_result` 作为最小结果对象已经成立
2. 先不把它落成表
3. 先不把它并入 `contents`
4. 先把它作为下一轮判断和实现的统一基线

## 10. `content_processing_result` 的适用范围判断

基于上一节已经定义的最小结果对象，当前继续判断：

`content_processing_result` 是不是已经足够扩到更多内容类型，还是只适合 `article` 首轮落地。

### 10.1 当前结论

当前结论是：

1. **它已经不是只能服务 `article` 的私有对象**
2. **但当前也还不适合一次性扩到全部内容类型**
3. **最合理的落地范围是：**
   - 第一批正式适用：`article`
   - 第二批可候选扩展：`hot_topic`
   - 当前不建议纳入：`opportunity`

也就是说，当前更准确的判断不是：

- “只适合 `article`”

而是：

- “以 `article` 为首轮正式落地对象，并为 `hot_topic` 预留兼容空间，但暂不把 `opportunity` 拉进来”

### 10.2 为什么 `article` 最适合首轮落地

`article` 最适合作为第一批正式对象，理由有 4 个：

1. **字段稳定度最高**
   - 标题、摘要、分类、标签、质量分、发布时间这组字段在 `article` 上最完整
2. **加工语义最清晰**
   - 当前“标准化标题 / 摘要 / 标签 / 分类”这套定义，本质上就是围绕 `article` 收出来的
3. **来源锚点最明确**
   - `rss_articles` 当前已经是 D1 中较稳定的原始事实层承接位
4. **读侧复用价值最高**
   - `content/by-ref` 的 `article` 详情、本轮主线本来就围绕 `article` 内容流水线推进

所以：

如果下一轮真的要从“对象定义”走到“最小实现”，`article` 仍然应该是唯一正确的首轮对象。

### 10.3 为什么 `hot_topic` 可以作为第二批候选

`hot_topic` 当前已经具备一部分扩展条件，但还不建议和 `article` 同时并入第一轮正式落地。

可扩展的理由：

1. 它同样具备 `title / summary / category / tags / quality_score / published_at`
2. 当前读侧也已经接入统一内容引用与统一详情链路
3. 读侧投影和轻加工逻辑与 `article` 已存在较高重合

但它还不适合立刻并入第一轮，原因是：

1. **热点域自带额外语义**
   - `hot_score`、热点波动、热点解释都更像热点域专属事实
2. **加工结果与事实边界没完全分开**
   - 当前热点聚合和热点解释仍更接近视图层、聚合层而不是稳定结果层
3. **若现在一起推进，会抬高对象边界复杂度**
   - 会让第一轮结果对象同时背负“内容标准化”与“热点域语义兼容”

所以：

当前对 `hot_topic` 的正确态度是：

- 先确保 `content_processing_result` 的字段设计对它不冲突
- 但不把它纳入第一轮正式落地范围

### 10.4 为什么 `opportunity` 当前不建议纳入

`opportunity` 当前最不适合并入第一轮甚至第二轮结果对象扩展，原因是它天然更偏“行动域”，而不是“内容标准化结果域”。

主要问题有 4 个：

1. **业务语义不同**
   - `opportunity` 的核心价值不只是内容展示，而是转行动、转待办、跟进执行
2. **关键字段异质性更强**
   - `deadline`、`reward`、`is_remote`、状态类字段，并不是通用内容结果字段
3. **排序逻辑更偏场景型**
   - 当前 worth acting 的排序更依赖行动语境，而不是单纯内容标准化结果
4. **如果强行并入，会把最小对象拉胖**
   - 会让“加工结果层”过早混入行动层语义

所以当前应明确：

- `opportunity` 仍然继续走统一读侧投影
- 但不进入 `content_processing_result` 的第一轮或第二轮扩展范围

### 10.5 当前最推荐的适用范围定义

当前建议把适用范围收口为：

1. **首轮正式适用类型**
   - `article`
2. **兼容但暂不落地类型**
   - `hot_topic`
3. **当前排除类型**
   - `opportunity`

这条边界的价值在于：

1. 保持第一轮对象足够干净
2. 给后续 `hot_topic` 扩展留下空间
3. 避免把行动域提前混入结果对象

### 10.6 为什么这样选

理由总结：

1. 当前主线本来就围绕 `article` 内容流水线推进
2. `article` 是最像“标准化内容结果”的对象
3. `hot_topic` 与它接近，但还带热点域附加语义
4. `opportunity` 则明显是另一类对象，不该为了统一而强统一

### 10.7 当前方案的缺陷

缺陷也必须明确：

1. **对象命名仍然偏通用**
   - 但实际首轮适用范围却是 `article-first`
2. **会产生阶段性“名大于实”**
   - 名字叫 `content_processing_result`，但首轮只正式服务 `article`
3. **`hot_topic` 的进入时机仍未完全定清**
   - 当前只是判断它“可扩展”，不是已经成立
4. **`opportunity` 的独立结果层问题被延后了**
   - 未来若行动域也要正式化，可能需要另一类结果对象，而不是继续扩当前对象

### 10.8 当前一句话结论

`content_processing_result` 当前已经不再是只能服务 `article` 的私有概念，但它的第一轮正式落地范围仍应严格收口为 `article`；`hot_topic` 只保留兼容扩展位，`opportunity` 当前不纳入。

## 11. `content_processing_result` 是否进入独立持久化结果层

基于前两步已经收口出的结论：

1. `content_processing_result` 已成立
2. 它当前应采用 `article-first`

这一节继续判断：

在 `article-first` 的前提下，它现在是否还应停留在代码层，还是已经值得进入独立持久化结果层。

### 11.1 当前结论

当前结论是：

1. **现在还应继续停留在代码层**
2. **当前还不值得进入独立持久化结果层**
3. **当前更不应借这个对象提前推动 `contents` 实体化**

换句话说：

当前已经到了“可以认真讨论落库”的程度，但还没到“现在就该落库”的程度。

### 11.2 为什么当前仍应停留在代码层

当前更适合继续停留在代码层，理由主要有 5 个：

#### 1. 当前结果对象还没有形成足够强的复算成本

现在这批字段主要是：

1. 标题标准化
2. 摘要标准化
3. 标签 / 分类标准化
4. 质量分读取与标准化
5. 发布时间标准化

这些结果虽然已经值得定义为“结果对象”，但它们的计算成本仍然偏轻。

这意味着：

当前还没有出现“每次读取都重算代价过高，必须持久化缓存”的硬压力。

#### 2. 当前结果字段仍偏“轻结果”，不是真正的重加工产物

真正值得独立持久化的结果层，通常至少满足以下之一：

1. 生成成本高
2. 依赖多阶段流水线
3. 会被多读侧频繁重用
4. 需要稳定审计、版本追踪、回滚

而当前的 `content_processing_result` 还只满足：

1. 有一定跨读侧复用价值

但它还没有强到需要：

1. 持久化版本审计
2. 复杂重算调度
3. 离线批处理产出

#### 3. 当前生命周期虽已能描述，但还没有真实运行压力验证

我们已经能定义：

1. 何时生成
2. 何时刷新
3. 何时失效
4. 允许整对象重算覆盖

但这些目前仍是**逻辑上可定义**，不是**运行上已被证明必须独立持久化**。

也就是说：

现在把它落库，会先获得复杂度；但能不能换来足够收益，当前还没有被真实证明。

#### 4. 当前首轮适用范围过窄

现在首轮正式对象仍严格收口为：

1. `article`

这意味着：

当前若直接建设独立持久化结果层，本质上是在为一个 `article-first` 的轻结果对象建立存储基础设施。

这个时点的风险是：

1. 结构过早固定
2. 后续 `hot_topic` 纳入时重新改边界
3. 结果层与原始事实层关系被过早写死

#### 5. 当前真正缺的不是“存储位置”，而是“结果层价值证明”

现在最需要被证明的，不是：

- “这个对象能不能存”

而是：

- “把它存下来，到底能带来什么明显收益”

例如：

1. 是否显著减少多接口重复加工
2. 是否能稳定支持多读侧一致性
3. 是否能显著减少未来重逻辑分散
4. 是否能支撑更重的内容解释链路

这些收益当前都还只是趋势，不是已被验证的硬需求。

### 11.3 当前不进入独立持久化结果层的推荐策略

当前最推荐的策略是：

1. **继续保留结果对象定义**
2. **继续停留在代码层**
3. **把它当成“准结果层对象”而不是“已持久化结果层对象”**

具体可理解为：

1. 对象边界先定清
2. 字段语义先定清
3. 生命周期先定清
4. 只有当下一步出现明确收益，再决定是否落库

### 11.4 什么时候才值得进入独立持久化结果层

当前建议设 4 个触发条件，只要出现其中 2 个以上，再重新判断“是否落库”：

1. **同一结果对象被多个接口频繁重复计算**
   - 且重复逻辑已明显影响维护成本
2. **结果字段明显变重**
   - 例如接入更复杂摘要、主题归并、解释链
3. **需要稳定版本审计**
   - 例如要比较不同加工版本的输出差异
4. **需要批处理 / 预生成**
   - 例如不希望读时计算，而要在写时或离线任务中提前生成

在这些条件没成立前，继续停留在代码层更稳。

### 11.5 为什么这样判断

理由总结：

1. 当前对象已经值得被定义，但还不够重
2. 当前收益主要在“边界清晰”，不在“存储提效”
3. 现在落库，拿到的是复杂度；现在不落库，失去的收益还不明显
4. 先以代码层结果对象稳定一轮，更利于后续判断是否真的值得持久化

### 11.6 当前方案的缺陷

这个判断也有明显缺陷，需要明确告诉你：

1. **会延后部分重复计算**
   - 因为不落库，就意味着读时仍可能重复做轻加工
2. **会延后结果版本治理**
   - 现在只有对象定义，没有真正的持久化版本审计
3. **后续若需求突然变重，可能还要补一次迁移判断**
   - 也就是今天省下来的复杂度，未来可能还要补回来
4. **容易让人误以为“定义了对象就已经正式化”**
   - 实际上当前仍然只是“准结果层”，不是正式结果层

### 11.7 当前收口结论

在 `article-first` 的前提下，`content_processing_result` 当前最合理的位置仍然是：

1. **定义成立**
2. **代码层承接**
3. **暂不独立持久化**
4. **暂不并入 `contents`**

### 11.8 当前一句话结论

`content_processing_result` 现在已经值得被当作“准结果层对象”稳定下来，但当前还不值得进入独立持久化结果层；先把它继续留在代码层，是更稳也更省复杂度的选择。

## 12. 从代码层升级到独立持久化结果层的触发条件

基于上一节判断，当前继续收口一个真正可执行的问题：

**什么样的触发条件出现时，我们才把 `content_processing_result` 从代码层升级到独立持久化结果层。**

### 12.1 当前总原则

当前不采用“感觉复杂了就落库”的模糊标准，而采用：

1. **核心触发条件**
2. **辅助触发条件**
3. **明确排除条件**

只有当：

1. **至少命中 1 个核心触发条件**
2. **并且至少再命中 1 个辅助触发条件**

才建议正式启动“独立持久化结果层”。

### 12.2 核心触发条件

以下任意一条成立，都可视为核心触发条件命中：

#### 核心触发 1：重复计算已经形成真实维护压力

满足以下任意 2 条即可判定命中：

1. `article` 相关结果已被 3 个及以上读侧反复计算
2. 同一批结果字段在 2 个及以上模块中出现独立重算逻辑
3. 因结果重算逻辑漂移，已出现一次真实口径不一致修复

#### 核心触发 2：结果字段明显变重

满足以下任意 1 条即可判定命中：

1. 结果对象开始纳入更复杂摘要或解释链
2. 结果对象开始纳入主题归并、聚类、去重归并之类重加工逻辑
3. 单次结果生成已经不再是“轻标准化 + 轻规则计算”

#### 核心触发 3：出现明确的预生成需求

满足以下任意 1 条即可判定命中：

1. 读时生成已经影响接口稳定响应
2. 产品侧明确要求预生成 / 预热 / 批处理
3. 需要把结果层从“随读现算”切到“写时生成 / 定时生成”

#### 核心触发 4：需要稳定版本审计

满足以下任意 1 条即可判定命中：

1. 需要比较不同规则版本下的输出差异
2. 需要对结果版本进行回滚或冻结
3. 答辩、演示、验收要求结果可追溯，而不仅是代码可解释

### 12.3 辅助触发条件

以下条件用于辅助判断“现在升级是否划算”：

#### 辅助触发 1：适用范围已超出 `article-first`

满足以下任意 1 条即可判定命中：

1. `hot_topic` 已明确进入第二批正式适用范围
2. 同一结果对象已被不止一种内容类型稳定复用

#### 辅助触发 2：生命周期已经被真实约束

满足以下任意 1 条即可判定命中：

1. 已明确存在生成、刷新、失效、重算中的至少 2 种真实运行场景
2. 已出现结果过期或规则升级后的真实刷新需求

#### 辅助触发 3：结果对象已成为跨接口稳定契约

满足以下任意 1 条即可判定命中：

1. 多个接口已经稳定依赖同一结果对象字段集
2. 若不持久化，跨接口一致性维护成本明显变高

#### 辅助触发 4：已有清晰收益证明

满足以下任意 1 条即可判定命中：

1. 能明确说明持久化后会减少哪几处重复加工
2. 能明确说明持久化后会提升哪一段链路的稳定性
3. 能明确说明持久化后会降低哪一类修改风险

### 12.4 当前明确不构成触发的情况

以下情况当前**不应**被当成升级触发：

1. 只是代码看起来更整齐了
2. 只是已经存在结果对象命名
3. 只是觉得“以后可能会有用”
4. 只是为了顺手建表
5. 只是为了把阶段三做得更像正式架构

### 12.5 当前推荐的执行判定法

当前建议用下面这条简单规则：

1. 命中 `1 个核心触发 + 1 个辅助触发`
   - 可以进入“持久化结果层设计”
2. 命中 `2 个及以上核心触发`
   - 可以进入“持久化结果层实现准备”
3. 只命中辅助触发，不命中核心触发
   - 继续停留在代码层
4. 一个都没命中
   - 继续保持现状，不升级

### 12.6 为什么这样定义

理由：

1. 先要求出现真实压力，再允许引入存储复杂度
2. 防止因为阶段推进惯性，把“对象定义”误推成“必须落库”
3. 让升级条件从“主观感觉”变成“可核查事实”
4. 给后续阶段切换提供明确门槛，而不是继续反复讨论

### 12.7 当前方案的缺陷

这个触发机制也有缺陷：

1. **偏保守**
   - 可能会比“先建再说”更慢进入结果层持久化
2. **部分条件仍需人工判断**
   - 比如“维护压力是否明显”“收益证明是否清晰”
3. **还没有量化阈值**
   - 当前是规则门槛，不是精确指标体系
4. **容易受阶段目标影响**
   - 如果后续答辩或演示突然要求结果可追溯，触发节奏会提前

### 12.8 当前一句话结论

只有当 `content_processing_result` 同时出现**真实重复压力或重加工压力**，并且已经具备**跨类型复用、生命周期约束或清晰收益证明**时，才应从代码层升级到独立持久化结果层；在这之前，继续停留在代码层更稳。

### 16.1 小设计目标

当前小设计**不解决**：

#### 失效

1. 规则版本落后
2. 来源 article 已删除
3. 来源 article 字段已更新但结果未刷新

#### 覆盖策略

3. 避免一开始把所有 article 读链一起切过去

理由总结：

1. 当前已经证明 article 结果对象存在跨接口复用价值
2. 但这种价值目前仍集中在 article 域
3. 因此最稳的设计不是“全域统一结果层”，而是“article-first 独立结果层”
4. 独立结果层比直接进 `contents` 更容易控制边界
5. 这也给后续判断 `hot_topic` 是否纳入留下缓冲层

### 16.9 当前小设计的缺陷

缺陷必须明确保留：

1. **会先引入一张只服务 article 的结果表**
   - 结构上不够优雅，但边界更稳
2. **可能产生两层事实观感**
   - `rss_articles` 与 `article_processing_results`
3. **仍未解决跨类型统一**
   - 只是把问题缩小，不是彻底解决
4. **初期收益可能不够大**
   - 因为当前结果字段仍偏轻
5. **如果后续不真正进入实现，这个小设计本身不会直接带来收益**

### 16.10 当前收口结论

当前 `article-first content_processing_result` 的独立持久化结果层小设计，应收口为：

1. 若未来要落库，先独立建 `article_processing_results`
2. 只承接 article 的稳定加工结果
3. 不承接视图排序值、解释文案、关系型读侧结果
4. 不并入 `contents`
5. 当前完成小设计即可，仍不直接实现

### 20.8 当前与 Today / Reports 的边界

   - 不应反向定义 `hot_topic_processing_result`

一句话说：

`hot_topic_processing_result` 应被 Today / Content 消费，但不应由 Reports 趋势语义来定义。

### 20.9 当前一句话小设计结论

当前 `hot_topic_processing_result` 的最小小设计已经可以收口为：

1. 若后续要推进，先作为 `hot_topic-first` 的代码层结果对象
2. 只承接标准化标题 / 摘要 / 分类 / 标签、质量分、发布时间与结果状态
3. 不承接 `hot_value`、热点解释、热点波动与趋势比较

## 22. `hot_topic-first` 独立持久化结果层小设计

基于上一节结论，当前进入的不是实现，而是严格收口的 `hot_topic-first` 独立持久化结果层小设计。

### 22.1 当前小设计目标

这一轮小设计只解决 4 件事：

1. 若 `hot_topic` 继续推进，最小独立结果层长什么样
2. 它与 `hot_topics` 原表如何分层
3. 首批读链应按什么顺序接入
4. 哪些热点域语义明确不进入结果层

这一轮不解决：

1. 热点趋势快照表
2. 热点解释持久化
3. `article` 与 `hot_topic` 共表
4. `contents` 实体化

### 22.2 当前推荐形态

当前推荐形态是：

1. 若未来进入实现，先独立建一张结果表
2. 表名建议先用：
   - `hot_topic_processing_results`
3. 它与 `hot_topics` 保持：
   - `1 -> 0/1`

也就是说：

1. `hot_topics` 继续承接热点原始事实与域语义
2. `hot_topic_processing_results` 只承接稳定标准化结果
3. Trends / Reports 继续停留在趋势层，不反向挤进结果层

### 22.3 当前为什么不直接并入 `contents`

当前不建议直接并入 `contents`，理由和 `article` 一致，但对 `hot_topic` 更强：

1. 当前这轮要解决的是热点结果层，不是统一内容实体
2. `hot_topic` 自带热度、互动、趋势语义，比 `article` 更容易把边界拉混
3. 如果现在并入 `contents`，会把：
   - 结果层成立
   - 趋势层成立
   - 统一实体成立
   三个问题绑死

所以当前更稳的路径是：

先独立结果层，再决定未来是否进入更大的统一实体。

### 22.4 当前建议字段

当前建议字段严格只围绕“稳定标准化结果”展开：

#### 主键与锚点

1. `id`
2. `source_hot_topic_id`
   - 对应 `hot_topics.id`
3. `source_content_ref`
   - 例如 `hot_topic:136`

#### 稳定结果字段

1. `normalized_title`
2. `normalized_summary`
3. `normalized_category_labels_json`
4. `normalized_tags_json`
5. `quality_score`
6. `published_at`

#### 结果状态字段

1. `processing_version`
2. `processed_at`
3. `is_stale`

#### 当前约束建议

1. `source_hot_topic_id` 唯一
2. `source_content_ref` 唯一
3. `processing_version` 非空
4. `processed_at` 非空

### 22.5 当前明确不进表的字段

以下字段当前明确不进 `hot_topic_processing_results`：

1. `hot_value`
2. `view_count`
3. `like_count`
4. `comment_count`
5. `relevance_score`
6. `hot_comments`
7. `guide_questions`
8. `match_score`
9. `ranking_score`
10. `relevance_reason`
11. Reports 的 `current_heat / previous_heat / change / trend`

原因很明确：

1. 前 7 项属于热点原始事实或热点域语义
2. 中间 3 项属于视图级排序或解释
3. 最后一组属于跨时间趋势层语义

### 22.6 当前生成、刷新与失效策略

若未来进入实现，当前推荐策略是：

#### 生成

1. 首先只在 `content/by-ref` 的 `hot_topic` 详情链路上按需生成
2. 结果不存在时，读时生成并返回

#### 刷新

满足以下任一条件时整行覆盖：

1. `title / summary / categories / tags / quality_score / published_at` 变化
2. `processing_version` 变化

#### 失效

满足以下任一条件时标记 `is_stale = true`：

1. 规则版本落后
2. 来源热点被删除
3. 标准化字段已更新但结果未刷新

#### 当前不作为失效触发

1. `hot_value` 变化
2. 互动计数变化
3. 趋势方向变化

因为这些变化说明的是“热点在波动”，不是“标准化结果失真”。

### 22.7 当前首批接入顺序

如果未来进入实现，当前推荐顺序是：

1. 先接 `content/by-ref` 的 `hot_topic` 详情链路
2. 再接 `content.related` 中的 `hot_topic` 候选链路
3. 再接 Today 的 `worth_knowing`
4. 最后再评估 Today 的 `recommended_for_you`

这样安排的原因是：

1. 详情链路最容易验证结果层读写是否正确
2. 当前主热路径是 `content.related.d1`，第二步接它最能验证收益
3. `worth_knowing` 的消费强度已经明显高于详情链
4. `recommended_for_you` 仍然更依赖兴趣命中与视图排序，放最后更稳

### 22.8 当前为什么这样设计

这轮设计这样收，是因为：

1. `hot_topic` 已有热路径，但收益仍主要集中在标准化结果复用
2. 真正复杂的热点域语义还不适合现在塞进结果层
3. 若先独立结果表，后续可更清楚判断：
   - 标准化结果值不值得持久化
   - 热点语义是否需要单独层
   - 是否真的有必要进一步走向统一实体

### 22.9 当前小设计的缺陷

缺陷也必须明确保留：

1. 会先引入一张只服务 `hot_topic` 的结果表
2. `hot_topics` 与 `hot_topic_processing_results` 会形成双层观感
3. `today.recommended` 的收益不如 `content.related / worth_knowing` 明确
4. 如果后续把热点解释也想落库，还需要再拆一层
5. 当前小设计仍然不回答 `contents` 是否成立

### 22.10 当前收口结论

当前 `hot_topic-first` 独立持久化结果层小设计，应收口为：

1. 若未来进入实现，先独立建 `hot_topic_processing_results`
2. 只承接标准化标题 / 摘要 / 分类 / 标签、质量分、发布时间与结果状态
3. 不承接 `hot_value`、互动计数、热点解释、热点波动与趋势字段
4. 先按 `content.detail -> content.related -> today.worth_knowing -> today.recommended` 顺序接入
5. 不并入 `contents`
6. 当前完成小设计即可，仍不直接实现

4. 如果把它接到结果层，真正复用到的仍只是轻标准化字段，收益很薄，反而会把“热点列表就是热点原表视图”这条边界打糊

所以当前更稳的做法是：

让 `hot_topics` 列表继续停留在热点域原表直出，不接结果层。

### 24.3 为什么 Reports 当前不值得继续接

当前判断：

- **不值得继续接**

原因：

1. Reports 现在消费的核心不是单条热点内容的标准化结果
2. 它主要消费的是：
   - `content_ref`
   - 真实历史 / 收藏 / 记录痕迹
   - 趋势计算结果
   - 洞察文案
3. Reports 的热点引用位虽然可能指向 `hot_topic:1`
4. 但这并不意味着它需要 `hot_topic_processing_results`

一句话说：

Reports 当前更像“趋势层 / 解释层”，不是“标准化结果层”的下一条收益链。

### 24.4 为什么聚合 / 同步 / 爬虫链不该接结果层

当前判断：

- **明确不该接**

原因：

1. 聚合器的职责是保存或更新热点原始事实
2. `/sync` 与爬虫链负责的是：
   - 抓取
   - 存储
   - 同步原始热点数据
3. 它们不应该承担标准化结果层写入职责
4. 否则会把：
   - 原始事实采集
   - 结果层生成
   - 趋势语义
   混到同一条链里

所以这部分当前不仅“收益不高”，而且“职责上就是错层”。

### 24.5 当前最终判断

基于这轮覆盖面盘点，当前结论应收口为：

1. 现有 4 条链路已经覆盖了 `hot_topic_processing_results` 当前最有收益的消费面
2. 额外候选面里：
   - `hot_topics` 列表接口收益偏薄
   - Reports 属于趋势层，不适合接结果层
   - 聚合 / 同步 / 爬虫链职责上不应接结果层
3. 因此当前**没有新的明确收益**值得继续扩大覆盖面

### 24.6 当前执行结论

当前执行结论正式收口为：

1. `hot_topic_processing_results` 当前覆盖面已经足够
2. 当前不再继续接：
   - `hot_topics` 列表接口
   - Reports
   - 聚合 / 同步 / 爬虫链
3. 当前阶段应停在：
   - 结果层最小实现已成立
   - migration 治理已收口
   - 覆盖面暂不继续扩大
4. 后续只有在出现新的明确收益时，才重新打开覆盖面扩张判断

一句话结论：

`hot_topic_processing_results` 当前最有收益的 4 条链路已经接完；现阶段没有新的明确收益值得继续扩大覆盖面，因此这条线可以阶段性停在这里。`
