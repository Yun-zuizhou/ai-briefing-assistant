# 正式页面职责审查与整改顺序

- 文档状态：现行文档
- 更新日期：2026-04-22
- 适用范围：`apps/web/src/pages`
- 文档职责：对照“正式应用功能排布方案”，逐页审查当前正式页面是否职责清楚、是否越界，并给出整改优先级

---

## 1. 文件定位

本文件不是视觉评审文档，而是页面职责评审文档。

它主要回答：

1. 当前哪些正式页面已经接近“正式应用页面”
2. 哪些页面还像“展示板 / 原型页 / 控制台页”
3. 后续页面整改应该先改谁

本文件与以下文档配套使用：

- `2026-04-22-正式前端UI当前方案总表.md`
- `2026-04-22-正式应用功能排布方案.md`
- `2026-04-21-正式前端UI改造执行清单.md`

---

## 2. 审查标准

每个页面统一按 5 个问题判断：

1. 这页是否只有一个主任务
2. 首屏是否只有一个主焦点
3. 页面是否承担了不该由它承担的任务
4. 页面是否混入研发态 / 中间态表达
5. 页面是否像正式应用页面，而不是功能展示板

审查结论分为 4 档：

- `A`：结构基本清楚，可继续细化
- `B`：结构可用，但主次仍不稳
- `C`：结构明显混杂，需要优先整改
- `D`：页面职责严重越界，已明显像原型 / 控制台 / 展示板

---

## 3. 一级页面审查

### 3.1 `TodayPage`

当前定位：

- 简报首页
- 今日信息入口

当前问题：

- 同时承担“今日总述 / 推荐 / 值得知道 / 值得行动 / 今日速记”5 个舞台
- “因你关注而推荐”和“值得知道”在当前首屏权重接近，主次冲突
- 首屏 CTA 同时指向 `Chat` 和 `HotTopics`，主动作不够单一
- 页面整体更像“首页聚合大看板”，而不是“今天先看什么”的应用首页

审查结论：

- 等级：`C`

整改方向：

- 保留“今日一句判断”作为唯一首屏锚点
- “推荐给你”下沉，不再占据和核心内容同级的舞台
- 明确一条主动作：
  - 优先建议保留“去行动”或“去记录”，不要同时等重

### 3.2 `ActionsPage`

当前定位：

- 行动转化和待办推进页

当前问题：

- “行动节律”大卡、统计卡、筛选、待办、收藏、跟进同时并列，重心太散
- 页面在“执行面板”和“成果展示”之间摇摆
- 顶部打卡模块视觉权重过高，会压过真正需要处理的待办清单
- 更像“功能很多的行动展示页”，不够像“先处理什么”的工作台

审查结论：

- 等级：`C`

整改方向：

- 把“今日最该推进的事项”提到最前
- 打卡、统计、收藏待处理降级为辅助区
- 页面默认先服务“处理待办”，不是先展示“行动感”

### 3.3 `ChatPage`

当前定位：

- 低成本表达入口

当前问题：

- 首屏出现大量解释性内容
- 会话状态、模式状态、确认状态、读取状态被前置
- 页面同时承担：
  - 输入入口
  - 系统说明
  - 调试说明
  - 会话状态展示
  - 结果摘要展示
- 更像“系统控制台 + 执行面板”，不像正式应用中的表达入口

审查结论：

- 等级：`D`

整改方向：

- 默认只保留：
  - 输入区
  - 示例区
  - 最近消息
- 会话读取状态、候选意图、模式说明全部后置或折叠
- “页面职责”这类解释块从首屏移除

### 3.4 `JournalPage`

当前定位：

- 日志记录页

当前问题：

- 当前页同时承担：
  - 今日叙事
  - 成长轨迹
  - 想法记录列表
  - 对话入口
- “日志页”里混入了较强的成长总结和叙事包装
- 真正的“快速记录 / 回看记录”主任务还不够明确

审查结论：

- 等级：`B`

整改方向：

- 强化“记录”本身
- 把成长摘要降级为辅助模块
- 让“新增记录 / 最近记录 / 进入成长回看”成为更清楚的主线

### 3.5 `MyPage`

当前定位：

- 个人中心和分发页

当前问题：

- 当前页定位最接近正确方向，但仍有“成长摘要过重”的倾向
- 容易滑向“小型成长页”，而不是入口页

审查结论：

- 等级：`A`

整改方向：

- 继续保持“入口页”角色
- 控制摘要长度
- 让跳转入口比展示信息更重要

## 4. 二级页面审查

### 4.1 `HotTopicsPage`

当前定位：

- 热点总览与阅读入口

当前问题：

- 同时存在“趋势图 + 详情列表 + 弹层详情”
- 一页承担了：
  - 排行榜
  - 可视化趋势
  - 列表详情
  - 弹层详情
- 内容重复度较高，主任务不够纯

审查结论：

- 等级：`C`

整改方向：

- 二选一：
  - 趋势总览 + 列表跳详情
  - 列表优先 + 趋势作为辅助
- 不建议继续保留“页内弹层详情”作为主承接

### 4.2 `ArticlePage`

当前定位：

- 内容详情阅读页

当前问题：

- 同时承担：
  - 阅读
  - 收藏
  - 分享
  - 原文跳转
  - AI 摘要
  - 相关推荐
  - 机会转待办
- 这些功能都合理，但当前视觉权重过于平均
- 页面会给人“功能全，但阅读主轴不够强”的感觉

审查结论：

- 等级：`B`

整改方向：

- 重新确立“阅读正文”是主任务
- 收藏 / 分享 / 原文作为辅助操作
- AI 摘要和相关推荐降级，不和正文争主舞台

### 4.3 `CollectionsPage`

当前定位：

- 收藏内容的后续处理页

当前判断：

- 角色方向是对的，但应继续避免变成“内容混合页”

审查结论：

- 等级：`B`

整改方向：

- 更强调“稍后看 / 下一步处理”
- 不做新的内容总览舞台

### 4.4 `GrowthPage`

当前定位：

- 成长回看页

当前问题：

- 页面本身质量不差，但它不应再被误当成一级导航主入口
- 目前结构中已经有摘要、关键词、画像、历史、报告入口，属于典型二级整合页

审查结论：

- 等级：`B`

整改方向：

- 保持二级页定位
- 继续围绕“回看”而不是“入口分发”

---

## 5. 当前优先级排序

### 第一优先

1. `ChatPage`
2. `TodayPage`
3. `ActionsPage`

原因：

- 这 3 页决定用户会不会把它当成正式应用
- 它们也是当前一级导航里最容易显出“原型感 / 展示感 / 控制台感”的页面

### 第二优先

1. `HotTopicsPage`
2. `JournalPage`
3. `ArticlePage`

原因：

- 这 3 页是主链路的重要承接页
- 如果它们结构不稳，会拖累整个应用的“产品感”

### 第三优先

1. `MyPage`
2. `GrowthPage`
3. `CollectionsPage`
4. `Settings` 系列

原因：

- 它们的结构方向已经基本正确
- 更多是收细节，而不是重构角色

---

## 6. 页面整改顺序建议

后续建议按下面顺序改：

1. `ChatPage`
  - 先从“控制台”改成“表达入口”
2. `TodayPage`
  - 先从“聚合大首页”改成“今日判断页”
3. `ActionsPage`
  - 先从“展示墙”改成“执行面板”
4. `HotTopicsPage`
  - 先从“多层重复详情页”改成“热点总览页”
5. `JournalPage`
  - 先从“叙事包装页”改成“记录页”
6. `ArticlePage`
  - 强化阅读主轴

---

## 7. 当前一句话结论

当前正式前端“最像正式应用”的一级页是 `MyPage`，  
“最不像正式应用”的一级页是 `ChatPage`，  
而真正决定整个产品会不会站住的，是 `TodayPage + ActionsPage + ChatPage` 这组三页能不能先收成清楚的任务入口。
