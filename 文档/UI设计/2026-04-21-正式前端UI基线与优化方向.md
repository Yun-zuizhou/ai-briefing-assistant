# 正式前端 UI 基线与优化方向

> 文档状态：现行文档
> 当前总表：`文档/UI设计/2026-04-22-正式前端UI当前方案总表.md`
> 说明：本文件负责提供 UI 判断基线；如果与旧原型文档、旧品牌文档冲突，以当前总表和本文件为准。

- 文档日期：2026-04-21
- 适用范围：`apps/web` 正式前端
- 目标：为后续 UI 优化提供稳定、统一、可执行的判断基线
- 当前判断：正式前端已经具备产品骨架，但尚未完成“产品化收口”，因此会呈现出较强的原型感和中低保真状态

---

## 1. 先说结论

当前前端的问题，不是“没有设计”，而是“设计概念已存在，但系统没有闭环”。

现状更准确的描述是：

- 已经有正式页面、正式路由、正式数据接入。
- 已经有明确视觉方向：纸张感、报头感、编辑感、个人简报气质。
- 但正式页面仍保留明显的联调态、原型态、局部试制态痕迹。

因此后续 UI 工作的重点不是“重新换一套风格”，而是：

- 把当前的报刊/纸张主题从“视觉表皮”升级为“产品级界面系统”。

---

## 2. 当前 UI 为什么有强烈“原型感”

### 2.1 展示壳感过强

当前正式前端被固定在手机预览尺寸中，整体观感更像“设计稿展示窗口”而不是“可正式使用的 Web 应用”。

这种感觉会直接带来三个问题：

- 云端访问时像嵌在背景上的样机，不像正式产品。
- 桌面端缺少空间组织，无法建立产品级留白和层级。
- 用户会默认把它理解成“移动端原型”。

### 2.2 主题很强，但信息层级不够强

当前报头、纸张、描边、边框、金色装饰等元素都很明确，但内容区块之间的层级差不够大。

具体表现为：

- 很多卡片都采用相近的背景、描边、字号和密度。
- 重要信息和辅助信息的视觉权重差太小。
- 页面更像“多个样式统一的内容块并列”，而不是“有明确主次关系的一屏产品”。

### 2.3 正式前台混入了联调和内部说明

当前正式页面中还存在较多研发态、验证态、测试态表达，例如：

- 测试账号快速切换
- 当前为调试页
- 会话来源、正式会话、缓存会话、候选意图等说明
- 对用户解释系统如何识别、如何分流、如何写回

这些内容对研发很有价值，但会削弱用户对“正式产品”的感知。

### 2.4 正式页并没有真正运行在一套稳定组件系统上

当前很多正式页面主要靠局部内联样式直接堆叠完成。

这会导致：

- 同类元素跨页不一致
- 按钮、标签、输入框、卡片难以统一迭代
- 页面每次微调都需要手改多处
- 视觉稳定性依赖页面作者，而不是依赖系统

### 2.5 排版精度停留在“可读”，还没进入“产品级阅读节奏”

当前多数页面虽然能读，但还没有建立高分辨率的信息节奏：

- 标题层级不够拉开
- 元信息、描述、正文、操作说明之间的区分度不足
- 正文和控件几乎使用同一语言气质
- 一屏内“先看哪里、后看哪里”不够明确

### 2.6 状态设计不够完整

一个成熟产品界面，除了静态样式，还依赖明确的状态体系：

- 默认态
- 悬停态
- 按下态
- 选中态
- 禁用态
- 加载态
- 成功态
- 错误态

当前页面里这些状态大多是局部处理，还没有形成统一规则，所以整体完成度会偏低。

---

## 3. 当前 UI 为什么看起来“偏低保真”

低保真不是因为颜色淡，也不是因为风格克制，而是因为界面还停留在“块级表达”而不是“产品表达”。

当前很多页面更像是在表达：

- 这里有一个区块
- 这里有一段信息
- 这里有一个按钮

但高保真产品界面需要表达的是：

- 这屏的主目标是什么
- 用户当前最应该读哪一句
- 读完之后最自然的下一步是什么
- 哪些信息是主要叙事，哪些只是辅助上下文
- 哪个动作是主动作，哪个动作只是补充

换句话说：

- 你现在已经有“页面”。
- 接下来要做的是“界面决策精度”。

---

## 4. 正式前端的稳定产品定义

后续所有 UI 决策，都以这句话为最高判断基线：

> 这不是新闻站，不是后台系统，也不是聊天机器人壳层。  
> 它是一个有编辑感的个人情报工作台。

这个定义意味着：

- 内容不是为了浏览而浏览，而是为了帮助用户形成判断。
- 对话不是产品本体，而是低成本表达入口。
- 页面不能只展示信息，还必须帮助用户转成行动、记录和长期沉淀。
- 风格上可以保留编辑感，但必须具备工具级清晰度。

---

## 5. 未来 UI 的五条总原则

### 5.1 内容优先，装饰服务于内容

纸张感、报头感、边框、纹理都可以保留，但不能抢走内容层级的工作。

判断标准：

- 如果一个装饰不帮助理解结构，就应弱化或删除。

### 5.2 标题可以有编辑感，正文和控件必须理性

后续不建议全站所有文本都继续使用同一种 serif 表达。

更稳的方向是：

- 标题、报头、栏目名：保留 serif/editorial 气质
- 正文、控件、标签、表单、元信息：使用更理性的 sans 体系

### 5.3 一屏一个主任务

每个页面都必须能回答：

- 用户进来是干什么的
- 这屏最重要的一件事是什么
- 看完后应该点哪里

如果一屏同时承担解释系统、展示状态、教育用户、提供多个操作入口，那就会重新滑回原型态。

### 5.4 正式前台和调试前台必须分离

以下内容不应出现在正式主链路页面中：

- 测试账号快捷入口
- 调试目标说明
- 联调术语
- 内部状态判定细节
- 面向研发的解释性语言

### 5.5 同类元素只允许一种正式写法

只要某种元素在两个页面以上重复出现，就不应该继续以页面内联样式存在。

优先组件化的对象：

- Button
- Card
- Input / Textarea
- Tag / Status
- SectionHeader
- Empty / Loading / Error State

## 6. 视觉系统基线

## 6.1 视觉方向

正式前端的视觉方向定为：

- 温暖编辑感
- 纸张与档案感
- 克制而非花哨
- 信息秩序强于装饰冲击

不建议改成：

- 通用 SaaS 蓝白后台
- 纯复古海报
- 过度拟物化的报纸仿真
- 依赖大面积纹理和装饰边框的视觉方案

正确方向应该是：

- “编辑感的数字产品”，不是“复古报纸主题秀场”。

## 6.2 字体策略

建议形成双字体体系：

- 标题字体：`Noto Serif SC`
- 正文字体：`Noto Sans SC`

用途划分：

- 报头、页面标题、栏目标题：`Noto Serif SC`
- 正文、按钮、表单、说明、元信息：`Noto Sans SC`

这样做的原因：

- 当前全站统一 serif 会强化“概念稿感”和“装饰感”
- 混合字体体系更容易拉开层级
- 中文环境下可读性更稳定

## 6.3 排版层级

建议后续固定 6 档文字层级：

- L1：页面主标题
- L2：区块标题
- L3：卡片标题
- B1：正文
- B2：辅助描述
- M1：元信息/标签/状态

要求：

- 同一层级跨页一致
- 同一页面中尽量不要临时造字号
- 元信息必须明显轻于正文
- 操作按钮文字不能和正文同权重

## 6.4 色彩策略

当前暖纸张体系可以保留，不建议大换血。

建议色彩角色固定为：

- `paper`：页面背景
- `paper-warm`：轻卡片背景
- `ink`：主标题与主操作
- `ink-light`：正文
- `ink-muted`：辅助信息
- `accent`：强调、提醒、主 CTA
- `gold`：少量编辑感点缀
- `success / warning / error`：语义状态

要求：

- `accent` 只用于真正重要的强调和主动作
- `gold` 只作少量点缀，不参与主层级竞争
- 成功、警告、错误必须可独立识别，不应完全复用品牌色

## 6.5 组件形态

后续正式组件建议统一形态：

- 卡片：克制描边 + 轻微层次，不要每张都像装裱板
- 按钮：主次分明，状态完整
- 输入框：必须显得可编辑、可聚焦、可提交
- 标签：不再承担“视觉主角”，只承担标记作用
- 空状态：不再只有提示句，要明确下一步动作

## 7. 页面级基线

## 7.1 Welcome 页

定位：

- 说明产品是什么
- 让用户愿意继续进入

不再承担：

- 多轮 onboarding 教学
- 复杂滑屏演示
- 过多品牌概念解释

未来结构建议：

- 一句产品定义
- 一句用户收益
- 一个主 CTA
- 一个次级入口

## 7.2 Login 页

定位：

- 最快完成进入

不再承担：

- 测试身份切换展示
- 研发验证入口
- 过多引导说明

未来原则：

- 登录和注册流程更简
- 测试入口隐藏到开发态
- 表单控件统一组件化

## 7.3 Today 页

定位：

- 每日进入后的第一屏

页面任务必须稳定为：

- 先看今日判断
- 再决定看内容、转行动、做记录

未来结构建议：

- 今日一句总述
- 值得知道的
- 值得行动的
- 今日记录入口

不要继续并列堆太多“都很重要”的区块。

## 7.4 Chat 页

定位：

- 最低成本的系统入口

不再定位为：

- 系统原理展示页
- 状态说明页
- 联调结果解释页

未来原则：

- 默认看到输入和示例
- 高级状态折叠收纳
- 执行结果清楚回落到其他页面
- 页面更像控制台，不像后台调试面板

## 7.5 二级内容页

定位：

- 承接内容理解、行动转化和回看沉淀

未来原则：

- 信息分组清晰
- 主内容区稳定
- 侧向动作集中
- 页面标题、来源、时间、摘要、后续动作关系明确

## 8. 哪些旧设计思路继续保留，哪些停止使用

## 8.1 保留

- “每天一份简报，帮你收集世界，记录自己，看见成长”这一产品气质
- 报头式标题语言
- 纸张/档案感底色
- “值得知道 / 值得行动 / 记录自己”这一主结构
- 内容与个人沉淀并置的产品特征

## 8.2 停止继续扩散

- 纯蓝白现代 SaaS 色板
- 全站统一 serif 的正文策略
- 大量内联样式继续增长
- 正式页中出现调试说明
- 正式页中暴露测试账号切换
- 页面继续沿用“固定手机壳展示”

说明：

- 旧文档中的早期蓝色品牌稿、手写感方案、通用现代风方案，不再作为正式前端的默认方向。
- 后续以当前暖纸张编辑感路线为正式主线。

---

## 9. 正式前端的“禁止项”

以下现象一旦继续出现，就会持续强化原型感：

- 页面固定在手机尺寸展示
- 一屏出现多个同级主动作
- 卡片全部长得一样重
- 同类控件每页单独写一套样式
- 正式页面继续出现研发说明文字
- 标签样式过重，抢正文注意力
- 所有文字都用同一种标题气质
- 空状态只有一句提示，没有下一步
- 加载态、错误态、成功态没有统一语言

---

## 10. 接下来 UI 优化的正确顺序

### 第一阶段：建立基线

- 去掉固定手机壳思路，改为响应式正式外壳
- 建立标题 serif + 正文 sans 的双字体系统
- 收敛 color tokens、spacing tokens、text scale

### 第二阶段：建立组件系统

- Button
- Card
- Input / Textarea
- Tag / Badge / Status
- SectionHeader
- Empty / Loading / Error State

### 第三阶段：重做四个样板页

- Welcome
- Login
- Today
- Chat

要求：

- 这四页先成为全站风格样板
- 其余页面以后按这四页的系统迁移

### 第四阶段：收口其余正式页面

- Hot Topics
- Article
- Collections
- Reports
- Settings
- Me / Growth

### 第五阶段：分离研发态入口

- Lab 页面保留，但明确归到内部验证链路
- 测试入口从正式用户链路剥离

---

## 11. 一句话作为所有后续设计评审标准

每次准备改 UI 前，先问一句：

> 这个改动，是在增强“个人情报工作台”的正式感，还是在继续强化“主题原型”的展示感？

如果答案偏向后者，就不应该继续做。

---

## 12. 本文档的使用方式

后续每次做 UI 设计或页面优化，都按下面顺序判断：

1. 这页的主任务是什么？
2. 这一屏最重要的一句话是什么？
3. 这一屏只有一个主动作吗？
4. 装饰有没有压过信息层级？
5. 正式用户是否会看到研发/联调语言？
6. 这里是不是应该沉淀成组件，而不是继续内联？

当以上 6 个问题都能回答清楚时，页面才算进入正式产品化状态。
