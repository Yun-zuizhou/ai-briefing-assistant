# 核心页面具体设计方案

- 文档状态：现行文档
- 更新日期：2026-04-22
- 适用范围：`apps/web` 正式前端核心页面
- 文档职责：给出可以直接照着做的具体风格、具体布局、具体展示和具体设计，不再停留在抽象原则层

---

## 1. 文件定位

本文件是“具体方案文档”，解决的是：

1. 具体风格长什么样
2. 具体页面首屏长什么样
3. 具体区块顺序怎么排
4. 具体组件用什么展示方式

本文件默认服务以下 3 个最高优先页面：

1. `ChatPage`
2. `TodayPage`
3. `ActionsPage`

---

## 2. 视觉方向定稿

### 2.1 风格名称

当前正式风格统一命名为：

> 温暖编辑室

它不是复古报纸秀场，而是：

- 有纸张感
- 有编辑感
- 但本质上是数字工具

### 2.2 用户看到的气质

用户第一眼应该感受到：

- 像一款被认真编辑过的移动应用
- 不冰冷
- 不花哨
- 不像后台
- 不像原型展示页

### 2.3 风格边界

保留：

- 暖纸张背景
- 深墨色标题
- 少量金色或红褐色点缀
- 有编辑标题感的页头

禁止：

- 整页深色背景包一个手机壳
- 满屏装饰边角
- 所有区块都厚边框 + 厚阴影
- 所有文字都像标题
- 按钮全部黑块化

---

## 3. 全局视觉规则

### 3.1 页面外壳

#### 正式目标视口

- 正式设计只认移动端竖屏
- 目标宽度区间：`360px - 430px`
- 保持单列沉浸感
- 内容边距略紧，但不做“430px 样机框”

#### 桌面预览口径

- 桌面浏览器只做调试与演示预览
- 页面居中即可，不展开为桌面专用布局
- 不做后台式左右边栏

### 3.2 字体

#### 标题字体

- `Noto Serif SC`

使用位置：

- 页头标题
- 区块标题
- 重点句

#### 正文字体

- `Noto Sans SC`

使用位置：

- 正文
- 说明
- 表单
- 按钮
- 标签
- 状态信息

### 3.3 颜色

#### 主背景

- `paper`: 偏浅暖纸色
- `paper-warm`: 区块底色

#### 主文字

- `ink`: 深墨色
- `ink-light`: 正文色
- `ink-muted`: 辅助信息色

#### 强调色

- `accent`: 暗砖红
- `gold`: 少量提示和编辑感点缀

#### 语义色

- `success`: 深绿
- `warning`: 土金
- `error`: 深红

### 3.4 卡片

统一规则：

- 背景偏浅，不抢正文
- 边框 1px 或 2px，不要每张卡都厚重
- 阴影很轻，只做层次，不做戏剧效果
- 圆角建议统一为小圆角或直角，不混用

### 3.5 按钮

统一 4 类：

- `Primary`
  - 暗砖红底，白字
- `Secondary`
  - 浅纸底，深字，细边框
- `Ghost`
  - 无底，仅文字与 hover 背景
- `TextLink`
  - 像编辑部里的栏目跳转

## 4. `ChatPage` 具体方案

### 4.1 页面目标

让用户一进来就知道：

- 这里是说一句话的地方
- 不需要先懂系统
- 说完就能触发动作

### 4.2 页面结构

```text
┌─────────────────────────────────────┐
│ 页头：对话 / 低成本表达入口         │
├─────────────────────────────────────┤
│ 首屏主卡：一句提示 + 输入框          │
│ 示例动作 Chips                       │
├─────────────────────────────────────┤
│ 最近会话 / 最近结果                  │
├─────────────────────────────────────┤
│ 消息区                               │
├─────────────────────────────────────┤
│ 底部固定输入栏                        │
└─────────────────────────────────────┘
```

### 4.3 首屏具体展示

首屏只保留一个主卡：

- 标题：
  - `今天想让我帮你处理什么？`
- 副文案：
  - `可以直接说关注、待办、想法，或一句调整。`
- 一个大输入框
- 一排快捷示例 chips：
  - `我想关注 AI`
  - `明天提醒我投简历`
  - `今天记一条想法`
  - `把这条机会转成待办`

### 4.4 最近结果区

紧接主卡下方，不要先放系统说明。

展示方式：

- 标题：`最近一条结果`
- 内容：
  - 上次做成了什么
  - 一个跳转按钮

例如：

```text
已帮你记成待办：完善简历
[去待办查看]
```

### 4.5 消息区样式

用户消息：

- 偏浅米白底
- 深字
- 更像便签

系统消息：

- 浅暖底
- 左侧细色条
- 不用过重气泡

状态标签：

- 小而轻
- 放在消息角落
- 不要大面积占位

### 4.6 需要隐藏到折叠区的内容

收进“高级状态”折叠区：

- 正式会话 / 缓存会话
- 候选意图数量
- 模式切换说明
- 读取状态
- sourceContext / matchedBy 这类中间信息

折叠标题建议：

- `查看处理细节`

### 4.7 页面效果关键词

- 像对话台，不像调试台
- 像表达入口，不像控制台

## 5. `TodayPage` 具体方案

### 5.1 页面目标

用户进入后 5 秒内知道：

- 今天最重要的判断是什么
- 今天最该推进什么
- 如果只做一件事，应该做哪件

### 5.2 页面结构

```text
┌─────────────────────────────────────┐
│ 页头：今日 / 日期 / 副标题           │
├─────────────────────────────────────┤
│ Hero：今日一句判断                   │
│ 主 CTA：去行动                       │
│ 次 CTA：记一句                       │
├─────────────────────────────────────┤
│ 值得知道的 3 条                      │
├─────────────────────────────────────┤
│ 值得行动的 2 条                      │
├─────────────────────────────────────┤
│ 因你关注而推荐（下沉）               │
├─────────────────────────────────────┤
│ 今日速记                             │
└─────────────────────────────────────┘
```

### 5.3 首屏 Hero 具体设计

Hero 不是大装饰区，而是大判断区。

内容包括：

- 上标：
  - `今日判断`
- 主标题：
  - 一句 18 到 24 字的总述
- 副说明：
  - 今天已筛出多少条值得知道 / 值得行动

按钮：

- 主按钮：
  - `先去处理`
  - 跳到 `/todo`
- 次按钮：
  - `记一句想法`
  - 跳到 `/chat`

### 5.4 值得知道的

数量控制：

- 首屏只展示 3 条

每条结构：

- 来源
- 标题
- 一句摘要
- 轻量热度 / 标签

卡片点击进入 `Article`

### 5.5 值得行动的

数量控制：

- 首屏只展示 2 条

每条结构：

- 动作类型
- 标题
- 为什么值得做
- 一个直接动作按钮

按钮文案建议统一：

- `转成待办`

### 5.6 推荐区

“因你关注而推荐”不再占第二主舞台。

应改成：

- 放在“值得行动的”之后
- 更像个性化补充
- 默认只展示 2 到 3 条

### 5.7 今日速记

保留，但变成极轻的末尾收束区。

样式：

- 一句提示
- 一个按钮

不要再做成独立强视觉大块。

### 5.8 页面效果关键词

- 像日报首页
- 像判断台
- 不像内容堆场

## 6. `ActionsPage` 具体方案

### 6.1 页面目标

用户进入后马上知道：

- 现在先做哪一项
- 今天还剩什么
- 哪些内容只是辅助信息

### 6.2 页面结构

```text
┌─────────────────────────────────────┐
│ 页头：行动 / 今日日期                │
├─────────────────────────────────────┤
│ 今日最该推进                         │
├─────────────────────────────────────┤
│ 待办筛选：今日 / 未来 / 已完成       │
├─────────────────────────────────────┤
│ 待办列表                             │
├─────────────────────────────────────┤
│ 跟进中                               │
├─────────────────────────────────────┤
│ 收藏待处理                           │
├─────────────────────────────────────┤
│ 打卡与统计（下沉）                   │
└─────────────────────────────────────┘
```

### 6.3 首屏主区

把现在的“行动节律大卡”改成更轻的“今日最该推进”。

结构：

- 标题：
  - `今天先推进这 1 件`
- 展示 1 条最高优先待办
- 右侧或下方 2 个动作：
  - `标记完成`
  - `去详情`

如果没有高优先项：

- 显示空状态
  - `今天还没有明确的推进项`
  - 按钮：`去对话里新增`

### 6.4 待办筛选区

保留 `今日 / 未来 / 已完成`

但视觉要更轻：

- 小标签式 segmented control
- 不做厚重黑底大按钮

### 6.5 待办列表

每一条只展示：

- 勾选框
- 标题
- 截止时间
- 优先级
- 来源

不再额外塞装饰信息。

### 6.6 跟进中

展示“正在推进的机会 / 项目”

每条结构：

- 标题
- 当前状态
- 下一步

这一区要比“收藏待处理”更靠前。

### 6.7 收藏待处理

角色是“稍后处理池”，不应压过待办主链路。

展示方式：

- 小卡片列表
- 可快速进入内容详情

### 6.8 打卡和统计

这部分不删除，但下沉到底部。

展示方式：

- 一个轻量摘要行

例如：

```text
已连续打卡 7 天 · 今日待办 3 项 · 已完成 1 项
```

不再占据首屏主视觉。

### 6.9 页面效果关键词

- 像执行面板
- 像今天要打开的一款移动应用
- 不像成果海报

## 7. 通用组件展示规则

### 7.1 `SectionHeader`

结构：

- 左：区块标题
- 右：轻量查看更多

不允许：

- 左右两边都过重

### 7.2 `ContentCard`

统一字段顺序：

1. eyebrow
2. title
3. summary
4. meta
5. action

### 7.3 `ActionCard`

比 `ContentCard` 更强调按钮。

统一字段顺序：

1. action type
2. title
3. why this matters
4. CTA

### 7.4 `EmptyState`

统一结构：

1. 标题
2. 一句说明
3. 一个下一步按钮

### 7.5 `StatusPill`

统一要求：

- 小尺寸
- 低饱和
- 只表达状态，不当视觉主角

## 8. 三页首屏对比摘要

### `Chat`

主标题：

- `今天想让我帮你处理什么？`

主动作：

- 直接输入一句话

### `Today`

主标题：

- `今日一句判断`

主动作：

- `先去处理`

### `Actions`

主标题：

- `今天先推进这 1 件`

主动作：

- 勾掉 / 推进当前最高优先任务

## 9. 当前可直接进入实现的结论

如果后续要开始真正改页面，最推荐的顺序是：

1. 先按本文件改 `ChatPage`
2. 再按本文件改 `TodayPage`
3. 再按本文件改 `ActionsPage`

因为这 3 页已经具备清晰的具体方案，不再停留在抽象层。
