## 1. 总体策略

这轮 UI 改造不采用“全站重画”方案，而采用“样板页先行 + 组件先收口 + 逐页迁移”的方式。

执行原则：

- 先建立稳定视觉系统，再动页面细节。
- 先改高频主链路页面，再改低频次级页面。
- 先清理正式前台中的研发态痕迹，再谈视觉精修。
- 先把重复元素组件化，再继续做局部美化。

一句话目标：

> 把当前正式前端从“有主题的原型态页面”推进到“可持续扩展的产品级界面”。

---

## 2. 改造阶段总览

## 阶段 A：建立全局基线

目标：

- 去除固定手机壳展示思路
- 建立正式排版与字体基线
- 统一色彩、间距、圆角、阴影、边框规则

产出：

- 全局响应式外壳
- 字体策略定稿
- 全局 tokens 整理
- 基础状态样式定稿

## 阶段 B：收口基础组件

目标：

- 所有正式页面共用一套正式组件

产出：

- Button
- Card
- Input / Textarea
- Tag / Badge / Status
- SectionHeader
- Empty / Loading / Error State

## 阶段 C：重做四个样板页

目标：

- 用四个主页面验证视觉方向和产品表达是否成立

页面：

- Welcome
- Login
- Today
- Chat

## 阶段 D：迁移其余正式页面

页面：

- Hot Topics
- Article
- Collections
- Me / Growth
- Reports
- Settings

## 阶段 E：研发态剥离与视觉收尾

目标：

- 正式主链路只保留正式产品表达
- 调试页和研发入口不再污染品牌感知

---

## 3. 第一轮必须优先处理的文件

## 3.1 全局与布局层

第一轮优先文件：

- `apps/web/src/index.css`
- `apps/web/src/components/layout/PageLayout.tsx`
- `apps/web/src/components/layout/TabBar.tsx`
- `apps/web/src/components/layout/Header.tsx`

需要完成的事情：

- 去掉固定 `430 x 932` 画布思路
- 改为正式响应式容器策略
- 移动端保留单栏沉浸感
- 桌面端建立合理留白和容器宽度
- 保留当前暖纸张气质，但不再做“样机展示”

完成标准：

- 桌面端访问时不再像手机截图贴在深色背景上
- 移动端仍保留当前单栏体验
- Header / TabBar / PageContent 在结构上成为稳定骨架

---

## 3.2 Token 与全局视觉语言

第一轮优先文件：

- `apps/web/src/styles/design-tokens.css`
- `apps/web/src/styles/design-tokens.ts`
- `apps/web/src/index.css`
- `apps/web/tailwind.config.js`

需要完成的事情：

- 整理当前已存在但未完全收口的颜色变量
- 补齐正式组件需要的变量缺口
- 确定标题字体与正文字体
- 固定字号层级、间距层级、圆角层级、阴影层级

当前必须修复的问题：

- `Button` 等组件使用了当前未完整定义的变量体系
- `design-tokens` 与正式页面没有形成唯一事实源

完成标准：

- 组件使用的变量必须都能在正式主题中找到定义
- 不再新增“页面私有 token”
- 视觉变量成为真正单点事实源

---

## 3.3 基础组件层

第一轮优先文件：

- `apps/web/src/components/ui/Button.tsx`
- `apps/web/src/components/ui/Input.tsx`
- `apps/web/src/components/ui/Card.tsx`
- `apps/web/src/components/ui/Tag.tsx`
- `apps/web/src/components/ui/EmptyState.tsx`
- `apps/web/src/components/ui/LoadingSpinner.tsx`
- `apps/web/src/components/ui/ErrorToast.tsx`
- `apps/web/src/components/ui/ConfirmModal.tsx`

需要完成的事情：

- 让这些组件真正适配当前正式主题
- 明确每个组件的 `primary / secondary / subtle / danger / ghost` 语义
- 补齐 hover / focus / active / disabled / loading / selected 状态
- 避免继续使用 demo 时代遗留的视觉变量命名

完成标准：

- 正式页面里的按钮、输入框、卡片、标签不再继续页面内联手写
- 状态反馈统一
- 组件可覆盖当前四个样板页的大部分重复 UI

## 4. 第一轮样板页改造顺序

## 4.1 Welcome 页

目标：

- 用最短时间说清产品是什么
- 减少 onboarding 原型感

当前问题：

- 滑屏 onboarding 感较强
- 信息像演示页，不像正式入口页
- 页面更强调概念展示，不够强调立即进入

改造方向：

- 改成单屏价值说明
- 保留品牌气质，但减少“演示流程”
- 一个主 CTA，一个次 CTA
- 页面更像产品首页，而不是引导动画页

验收标准：

- 3 秒内能回答“这是什么产品”
- 页面不依赖滑动才能理解核心价值
- 首屏主 CTA 明确

---

## 4.2 Login 页

目标：

- 降低登录页联调感
- 提升正式产品可信度

当前问题：

- 测试账号快速切换直接暴露在正式页面中
- 表单区块仍偏试制态
- 登录与注册切换仍有原型卡片感

改造方向：

- 正式用户视图中移除测试入口
- 统一表单控件和按钮
- 强化标题、说明、表单、主动作之间的阅读节奏
- 减少“演示账户”气息

验收标准：

- 用户不会把它当作内部测试面板
- 登录和注册都能一眼看懂
- 表单控件风格与全局一致

---

## 4.3 Today 页

目标：

- 成为正式主首页
- 建立最强的产品感

当前问题：

- 区块很多，但主次不够明确
- “今日总述 / 推荐 / 值得知道 / 值得行动 / 今日速记”之间还可以继续压缩层级冲突
- 卡片表达偏均匀，导致焦点分散

改造方向：

- 强化“今日一句判断”作为首屏锚点
- 明确“知道什么 / 做什么 / 记什么”三段式结构
- 卡片层级拉开，不再全都一样重
- 主 CTA 集中，不让每个区块都像主角

验收标准：

- 用户进入页面后能立即知道今天先看什么
- 首屏不拥挤
- 主次信息明显分离

---

## 4.4 Chat 页

目标：

- 把聊天页做成真正的低成本系统入口

当前问题：

- 页面解释性文本较多
- 会话状态、模式状态、判定状态等信息前置过多
- 更像研发控制台，不像正式产品入口

改造方向：

- 默认只保留必要引导和输入入口
- 高级状态折叠或后置
- 系统反馈更清楚，但不占主界面注意力
- 保留“快捷控制台”定位，不让它变成系统说明书

验收标准：

- 初次进入用户知道这里能做什么
- 输入区和示例区是主焦点
- 调试术语不成为主界面叙事

---

## 5. 第一轮不做的事情

为了防止改造失焦，第一轮明确不做：

- 不全站同步重绘所有页面
- 不先动 Reports 系列页面
- 不先做复杂动画系统
- 不重新定义品牌名称与品牌故事
- 不先做多套视觉主题探索
- 不因为局部页面不好看就推翻当前暖纸张路线

原因：

- 当前最大问题不是风格选择错误，而是系统没有收口

---

## 6. 正式页面中的研发态清理清单

以下内容需要在正式主链路中逐步清理：

- 测试账号快捷切换
- 调试目标说明
- “当前是阶段十六调试页”这类研发文案
- 会话缓存、正式会话、候选意图、匹配来源等内部术语
- 引导用户理解系统机制的长说明

这些内容的处理方式：

- 移到内部页面
- 移到开发态开关
- 移到折叠面板
- 或完全从正式前台剥离

---

## 7. 视觉验收标准

每次完成一个页面改造后，都按以下标准检查：

### 7.1 结构

- 这页是否只有一个主任务
- 首屏是否只有一个主焦点
- 主动作是否明确

### 7.2 排版

- 标题、正文、元信息是否拉开层级
- 辅助信息是否足够轻
- 同层级文本是否保持一致

### 7.3 组件

- 是否复用了正式组件
- 是否还有明显页面内联重复块
- 状态是否完整

### 7.4 视觉

- 是否仍然像“手机原型截图”
- 装饰是否压过内容
- 标签和边框是否过重
- 页面是否具备产品级呼吸感

### 7.5 产品表达

- 用户是否能迅速知道这页在干什么
- 是否混入研发态语言
- 是否强化了“个人情报工作台”的定位

## 8. 第二轮迁移建议

当四个样板页稳定后，按下面顺序迁移其余页面：

1. `HotTopicsPage`
2. `ArticlePage`
3. `CollectionsPage`
4. `MyPage`
5. `GrowthPage`
6. `WeeklyReportPage`
7. `MonthlyReportPage`
8. `AnnualReportPage`
9. `SettingsPage`
10. `NotificationSettingsPage`
11. `HelpFeedbackPage`
12. `AboutPage`

迁移原则：

- 每迁移一个页面，都优先替换为正式组件
- 不再允许继续新增大量页面内联样式
- 每次迁移必须同时清理研发态文案

---

## 9. 实施方式建议

推荐实施节奏：

### 第 1 次改造

- 全局壳层
- token 收口
- Button / Card / Input / Tag

### 第 2 次改造

- Welcome
- Login

### 第 3 次改造

- Today
- Chat

### 第 4 次改造

- 其他正式页面逐步迁移

这样做的好处：

- 每一轮都能看到明确质量提升
- 不会和后端联调形成大面积冲突
- 风格可以先在样板页上验证，再扩散

---

## 10. 一句话执行标准

后续每次准备动手改 UI 时，先检查一句：

> 这次改动是在建立正式界面系统，还是在继续给原型补皮肤？

只有前者，才值得做。
