## 2. 当前判断

### 2.1 当前问题不是视觉，而是应用结构

当前正式前端已经具备统一风格方向，但还没有完全呈现出正式应用的结构特征。

现在更像：

- 一组风格接近的页面
- 一组已经接入真实接口的读侧页面
- 一组还在承接研发说明、调试口径和中间态表达的产品壳

而不是：

- 一个拥有稳定一级导航
- 稳定任务分区
- 稳定跳转逻辑
- 稳定主次页面关系

的正式应用。

### 2.2 当前应用的一句话结构定义

> 一个围绕“今天先看什么 -> 现在做什么 -> 想到什么就说 -> 之后如何回看”展开的移动端个人情报助手应用。

## 3. 正式应用一级导航

当前正式应用统一收口为 5 个一级入口：

1. `Today / 简报`
2. `Actions / 待办`
3. `Chat / 对话`
4. `Journal / 日志`
5. `My / 我的`

这 5 个入口与当前代码主线一致：

- 当前底部导航已是 `对话 / 简报 / 待办 / 日志 / 我的`
- 但目前更像技术上已经存在，还没有完全产品化说明

### 3.1 一级导航定义

#### `Today / 简报`

定位：

- 每日进入后的第一屏
- 负责“先知道今天最重要的事”

唯一主任务：

- 帮用户完成今天的信息判断

不应承担：

- 过多系统解释
- 太多并列操作入口
- 收藏、设置、成长回顾等次级任务

#### `Actions / 待办`

定位：

- 把信息和机会转成推进中的动作

唯一主任务：

- 帮用户决定“现在要推进什么”

不应承担：

- 过多内容浏览
- 画像展示
- 报告阅读

#### `Chat / 对话`

定位：

- 整个系统的低成本表达入口

唯一主任务：

- 让用户用一句话发出意图

不应承担：

- 复杂状态教育
- 系统原理说明
- 正式会话机制展示

#### `Journal / 日志`

定位：

- 记录当天的想法、片段和笔记

唯一主任务：

- 快速留下值得以后回看的内容

不应承担：

- 热点阅读主链路
- 待办推进主链路

#### `My / 我的`

定位：

- 个人中心和分发页

唯一主任务：

- 把用户带去“成长、收藏、设置、历史”

不应承担：

- 成为成长页本体
- 直接展示过重的信息墙

## 4. 二级页面分组

二级页面不应再和一级入口竞争注意力，统一按“阅读承接 / 回顾承接 / 设置承接 / 内部入口”分组。

### 4.1 阅读承接页

这些页面用于承接从主页面点进去的内容：

- `HotTopicsPage`
- `ArticlePage`
- `CollectionsPage`

它们的角色是：

- 展开看
- 深入看
- 保存后再看

它们不是一级导航。

### 4.2 成长与回顾承接页

这些页面用于承接“日志之后的长期沉淀”和“我的页之后的回顾”：

- `GrowthPage`
- `ProfilePage`
- `HistoryLogsPage`
- `HistoryBriefPage`
- `WeeklyReportPage`
- `MonthlyReportPage`
- `AnnualReportPage`

它们的角色是：

- 回看
- 归纳
- 总结

它们不应再被误当成一级区本身。

### 4.3 设置与支持承接页

- `SettingsPage`
- `NotificationSettingsPage`
- `AiProviderSettingsPage`
- `HelpFeedbackPage`
- `AboutPage`

它们统一归入 `My` 的下级。

### 4.4 认证与启动页

- `WelcomePage`
- `LoginPage`
- `InterestConfigPage`

它们属于启动链路，不属于登录后的正式一级导航。

### 4.5 内部 / 验证入口

- `PreviewPage`
- `AiDigestLabPage`

这些页面不属于正式主用户链路。

它们应被明确视为：

- 内部预览
- 实验入口
- 验证态页面

## 5. 正式应用主链路

### 5.1 用户主闭环

当前正式应用的主闭环统一定义为：

1. 在 `Today` 看见今天最重要的内容
2. 在 `Actions` 把值得推进的内容转成动作
3. 在 `Chat` 用一句话补充关注、待办、记录或调整
4. 在 `Journal` 留下当天值得保存的想法
5. 在 `Growth / Reports` 回看自己的变化

### 5.2 页面跳转原则

页面跳转统一按下面的逻辑组织：

- `Today -> Article`
  - 继续阅读
- `Today -> Chat`
  - 记录、转待办、表达关注
- `Today -> HotTopics`
  - 查看更完整内容池
- `Actions -> Chat`
  - 追加动作表达
- `Article -> Chat`
  - 从内容转待办 / 转记录
- `Journal -> Growth`
  - 从记录进入长期沉淀
- `My -> Growth / Collections / Settings / History`
  - 作为个人入口分发

### 5.3 页面之间不应再出现的混乱

后续应避免：

- `Chat` 承担内容浏览
- `My` 承担成长页主体
- `Today` 同时承担推荐总页、热点总页、待办总页、记录总页
- `HotTopics` 同时是趋势总览、详情容器和弹层入口

## 6. 每个一级页面的推荐结构

### 6.1 `Today`

推荐结构：

1. 今日一句判断
2. 值得知道的
3. 值得行动的
4. 今日记录入口

删减原则：

- 不把所有内容都堆成同权重卡片
- 不把“推荐给你”做成另一套主舞台

### 6.2 `Actions`

推荐结构：

1. 今日最该推进的事项
2. 未来待处理
3. 已完成
4. 跟进中 / 收藏待处理

删减原则：

- 少做装饰大卡
- 更像执行面板，不像展示墙

### 6.3 `Chat`

推荐结构：

1. 输入区
2. 示例区
3. 最近结果 / 最近会话
4. 高级状态折叠区

删减原则：

- 先让用户开口，再解释系统

### 6.4 `Journal`

推荐结构：

1. 今日记录入口
2. 最近记录
3. 可回看入口

删减原则：

- 让“记录”本身成为主任务

### 6.5 `My`

推荐结构：

1. 用户摘要
2. 成长入口
3. 收藏入口
4. 历史入口
5. 设置入口

删减原则：

- 它是入口页，不是“大而全页面”

## 7. 当前应移走或降级的内容

### 7.1 从正式主链路移走

以下内容不应继续在正式一级页面里前置出现：

- 测试账号快捷入口
- 会话缓存 / 正式会话 / 候选意图等内部术语
- 调试说明
- Lab / Preview 性质入口
- 对系统机制的长解释

### 7.2 收进折叠区或内部态

以下内容允许保留，但应后置：

- Chat 的识别过程信息
- 调试/验证用状态说明
- 开发时的辅助入口

## 9. 后续改造优先级

为了让前端更像“正式应用程序”，后续应先处理结构，再处理视觉尾差。

推荐顺序：

1. 统一一级导航口径
2. 明确每页唯一主任务
3. 把二级页和一级页彻底分层
4. 剥离研发态入口
5. 再继续组件化和页面细化

## 11. 一句话执行标准

> 后续每改一个页面，都先问一句：这个页面是在完成一个清楚的用户任务，还是还在继续当“功能展示板”？

如果更像后者，就还没有达到正式应用结构。
