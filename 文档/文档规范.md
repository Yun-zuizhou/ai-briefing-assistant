# 文档规范

## 一、目录结构

文档中心统一收束到 `文档` 目录内部，采用以下层级结构：

- **顶层：主控文档层** - 少量总控文档，如项目核心总纲、功能地图、技术现实清单
- **顶层补充：主控子纲层** - 用于承接主控文档不宜持续膨胀的阶段过程摘要、归档索引与阶段汇报
- **子目录：专题文档层** - 按主题分类的文档，如产品设计、技术实现、UI设计等
- **子目录：协同经验层** - 用于沉淀 Agent 协同、阶段切换、文档治理等可复用经验
- **进行中/：当前阶段执行层** - 正在推进的任务和记录
- **过时/：历史沉淀层** - 已降级、已失效或不再作为当前依据的历史文档
- **归档/：阶段归档层** - 某一阶段全部任务完成后，从 `进行中/` 成批转移出的阶段性资料，按项目单独归档

## 二、命名规则

### 1. 顶层主控文档命名
放在 `文档` 顶层，只保留少量固定文件名：
- `项目核心总纲.md`
- `功能地图.md`
- `技术现实清单.md`

### 1.1 顶层主控子纲命名
当主控文档需要瘦身，但又不能直接删除阶段过程时，可新增子纲文档：

- `项目核心子纲-阶段演进与归档索引.md`
- `主题子纲-用途.md`

### 2. 专题文档命名
放在各专题目录内，命名遵循：
- `主题.md`
- `主题-版本.md`
- `主题-日期.md`

### 2.1 协同经验文档命名
放在如 `Agent协同经验/` 这类经验目录内，命名遵循：

- `主题经验.md`
- `主题整理经验.md`
- `主题协同经验.md`

### 3. 进行中文档命名
放在 `文档\进行中`，统一强约束：
- `YYYY-MM-DD-主题.md` - 适合联调记录、阶段推进、当日决策、局部重构
- `当前-主题.md` - 适合始终滚动更新的当前状态文档

补充规则：

1. 同一阶段下，如果某条主线继续产生“小设计 / 最小实现 / migration治理 / 验证结论”等不同承载层内容，不得持续堆进同一个主线文件
2. 应在 `进行中/` 下继续拆分并列子文件，例如：
   - `YYYY-MM-DD-主线主题.md`
   - `YYYY-MM-DD-主线主题-最小小设计.md`
   - `YYYY-MM-DD-主线主题-最小实现.md`
   - `YYYY-MM-DD-主线主题-D1治理.md`
3. 主线文件只保留：
   - 为什么当前切到这条主线
   - 当前真实现状
   - 当前边界
   - 当前阶段性结论
   - 当前下一步入口
4. 小设计、最小实现、治理、验证，应拆到并列子文件承接，并由主线文件做入口引用

### 4. 过时文档命名
放在 `文档\过时`：
- 保持原名直接收纳
- 或加前缀 `过时-`

### 5. 阶段归档文档命名
放在 `文档\归档\某项目`：
- 保持原名直接归档
- 如有必要，可按阶段补充前缀，例如 `阶段1-主题.md`

### 6. 版本命名规则
持续迭代但有明显版本切换的长期文档：
- `xxx-v2.md`
- `xxx-v3.md`

### 7. 禁止使用的命名
避免使用以下后缀：
- `最终版`
- `最新版`
- `最新`
- `new`
- `完整版`
- `修正版`
- `可执行版`

## 三、文档状态分类

### 主文档
少数几个，但层级不同：
- `项目核心总纲.md`：只接收子纲汇报后的结果级更新
- `项目核心子纲-阶段演进与归档索引.md`：默认承接主线进行中文档的阶段汇报，并向总纲汇报
- `进行中\当前阶段总表.md`：当前阶段执行面主控文档
- `进行中\当前测试与验收总表.md`：当前测试数字唯一事实源
- `功能地图.md`
- `技术现实清单.md`

### 现行文档
当前仍有效，但不是全局锚点：
- 当前技术方案
- 当前 UI 规范
- 当前产品设计文档

### 参考文档
仍有价值，但不再作为第一事实源：
- 某次阶段评估
- 某轮方案讨论
- 某阶段字段清单

### 过时文档
只保留历史上下文，且不再作为当前执行依据：
- 旧版本产品定位
- 根目录旧说明文件
- 早期立项材料副本

### 阶段归档文档
某阶段完成后，从 `进行中` 中成批转出的过程资料：
- 某项目某阶段的推进记录
- 某项目某阶段的联调文档
- 某项目某阶段的阶段总结

## 五、文档降级策略

对准备降级的文件，在开头添加统一头部：

```md
> 文档状态：参考文档
> 当前优先阅读：项目核心总纲 / 当前阶段总表
> 说明：本文件保留历史上下文，不再作为当前阶段的第一事实源。
```

## 六、阶段完成后的归档规则

当某一阶段的全部任务完成后，必须执行一次归档动作：

1. 将该阶段在 `文档\进行中` 下的相关文件统一整理出来。
2. 在 `文档\归档` 下建立对应项目目录，格式为 `文档\归档\某项目`。
3. 将该阶段已完成的过程文档、记录文档、阶段性方案文档移动到对应项目归档目录。
4. `文档\进行中` 只保留当前阶段仍在推进、仍需频繁更新的文件。

这是强约束，不应让已完成阶段的文件长期滞留在 `进行中`。

补充规则：

1. `文档\进行中` 是当前执行层独立工作区，可以承接更细的迭代记录。
2. 当前不要求极限压缩到极少文件。
3. 当前推荐规模：
   - 约 20 份上下
4. 超过该量级时，必须继续执行归档或职责瘦身。

## 七、阅读优先级

1. **第一优先**：项目核心总纲
2. **第二优先**：项目核心子纲
3. **第三优先**：进行中/当前阶段总表
4. **第四优先**：进行中/当前测试与验收总表
5. **第五优先**：功能地图 / 技术现实清单
6. **第六优先**：专题文档 / 协同经验文档
7. **第七优先**：归档文档
8. **第八优先**：过时文档

补充说明：

1. 上述顺序用于“建立全局上下文”，不等于“当前执行判断优先级”
2. 若要判断“这一轮现在该做什么”，默认以：
   - `当前阶段总表.md`
   - `当前测试与验收总表.md`
   - 对应 `进行中/` 主线文档
   为准
3. 若要判断“项目级当前口径是什么”，默认以：
   - `项目核心总纲.md`
   为准
4. 如果“总纲 / 子纲 / 总表 / 主线文档”之间出现冲突，必须先回到对应 `进行中/` 主线文档核实事实，再判断是否需要向上层文档汇报修正

## 八、测试口径更新规则

为避免项目状态在多份文档中漂移，测试结果采用“单一事实源 + 结果级同步”规则：

1. 测试数字只允许写在：
   - `文档/进行中/当前测试与验收总表.md`
2. 文档中的测试数字，只有在真实执行对应测试入口后才能更新。
3. 其他核心文档只允许写：
   - 已复验通过 / 未通过
   - 以《当前测试与验收总表》为准
4. 默认需要同步结果级状态的文档包括：
   - `文档/进行中/当前阶段总表.md`
   - `文档/项目核心总纲.md`
   - `文档/Agent协同经验/项目承接提示词-当前版.md`
   - `文档/README.md`
   - `文档/技术现实清单.md`
5. 过程文档如需记录测试，只描述“已验证 / 已补强 / 已恢复”等过程事实；默认不再维护具体数字。

## 八点一、结果级变化同步硬规则

为避免跳过主线事实层直接改上层文档，新增以下硬规则：

1. 只要形成结果级变化，默认先检查对应 `文档/进行中/` 主线文档是否已经更新
2. 如果主线文档还没更新，不得直接继续同步 `当前阶段总表.md`、`项目核心子纲-阶段演进与归档索引.md`、`项目核心总纲.md`
3. 默认同步顺序必须是：
   - `主线进行中文档 -> 当前阶段总表 -> 项目核心子纲 -> 项目核心总纲`
4. 上述同步链是“允许继续上推时的顺序约束”，不是“默认每次都必须全链更新”
5. 是否继续从主线文档上推到总表、子纲、总纲，仍应按触发条件判断：
   - 只新增执行事实：停在主线文档
   - 当前阶段任务 / 风险 / 顺序变化：再更新总表
   - 形成阶段性结果级判断：再更新子纲
   - 形成项目级当前口径变化：再更新总纲
6. `项目承接提示词-当前版.md` 属于新对话状态注入，不参与替代这条汇报链
7. 如果某轮只同步了总表或状态卡，而没有先同步主线文档，应视为同步链路未完成，必须补齐

这条规则是强约束，不依赖当轮对话是否额外提醒。

## 八点二、突破升级触发器

为防止小任务演进成大突破时遗漏上层同步，新增以下硬规则：

1. 每次任务完成后，AI 必须额外检查以下 5 个问题：
   - 用户可见主路径变了吗？
   - 数据源或 schema 真源变了吗？
   - API 路由面或接口契约变了吗？
   - 测试门槛或验收结果变了吗？
   - 当前阶段优先级、风险或排序变了吗？
2. 只要 5 个问题中任意一项回答为“是”，就不能停在局部主线文档。
3. 必须继续按汇报链上推：
   - 主线文档
   - 当前阶段总表
   - 项目核心子纲
   - 项目核心总纲
4. `项目承接提示词-当前版.md` 仍只负责状态注入，不替代突破升级触发器。

## 九、主控文档职责边界

为避免多个主控文件重复承载同一层信息，当前项目的核心文件采用“专人专事”边界：

### 1. 项目核心总纲

职责：

1. 只保留项目级结果判断、核心原则、当前阶段判断和下一步主线
2. 回答“项目现在到底在做什么”
3. 作为项目级统一口径出口

不再承担：

1. 大量阶段过程记录
2. 每轮推进明细
3. 详细当前文档清单
4. 协作方法论说明

### 2. 项目核心子纲

职责：

1. 承接阶段划分、阶段结果级结论和归档入口
2. 负责把主线进行中文档的阶段结论整理后再向总纲汇报
3. 回答“这个阶段是怎么演进到现在的”

不再承担：

1. 每轮新增执行事实流水账
2. 过多局部结论逐条累加
3. 当前阶段执行顺序管理
4. 项目级最终口径输出

### 3. 当前阶段总表

职责：

1. 承担当前阶段执行面管理
2. 只保留当前阶段目标、主线、风险、顺序、相关文档和退出动作
3. 回答“当前阶段现在怎么推进”

不再承担：

1. 项目长期历史说明
2. 过长的已完成细项流水账
3. 阶段过程级全量摘要
4. 项目级原则和长期定位

### 3.1 当前测试与验收总表

职责：

1. 承接当前测试数字唯一事实源
2. 维护主门槛、辅助门槛与最近一次复验结果
3. 回答“现在测试到底跑到了什么状态”

不再承担：

1. 当前阶段主线排序
2. 项目级方向判断
3. 历史阶段汇报
4. 开发过程流水

### 4. 项目功能文档

职责：

1. 只负责产品功能、页面全貌和能力概览
2. 作为产品视角参考文档
3. 回答“这个产品有什么、用户怎么用”

不再承担：

1. 当前阶段状态
2. 当前执行主线细节
3. 云端切换进度
4. 测试口径主维护职责

### 5. README

职责：

1. 只做导航入口
2. 帮助快速定位文档目录和专题区
3. 不与总纲、总表争夺当前状态解释权

不再承担：

1. 当前重点判断
2. 当前阶段解释
3. 项目级结果判断
4. 主线执行顺序说明

### 6. 项目承接提示词-当前版

职责：

1. 只承接当前项目状态注入
2. 只保留当前日期、当前阶段名称、当前执行主线、正式态/过渡态/虚拟态/回退态、当前测试口径和当前最优先动作
3. 服务新对话快速接手

不再承担：

1. 长篇阶段成果枚举
2. 项目长期历史说明
3. 稳定协作规则
4. 详细过程记录

### 7. 项目初始化

职责：

1. 只承接稳定协作协议
2. 只定义阅读顺序、汇报链、输出规则等长期方法论

不再承担：

1. 当前项目状态
2. 当前测试数字
3. 当前数据库状态
4. 本轮任务细节

### 8. 进行中主线文档

职责：

1. 只承接当前主线事实、边界和推进判断
2. 作为当前阶段最细的一层真实工作面
3. 作为当前主线的导航入口，而不是无限膨胀的总承载文件

不再承担：

1. 项目级判断
2. 阶段归档索引
3. 长期导航职责
4. 同一主线下所有小设计、最小实现、治理、验证的持续堆叠

### 9. 归档目录与阶段说明

职责：

1. 承接已稳定、已完成、不再高频修改的过程明细
2. 提供追溯入口
3. 回答“之前这一阶段具体怎么做过”

不再承担：

1. 当前阶段主线职责
2. 当前执行顺序职责
3. 当前项目状态出口

## 十、项目管理团队分工

当前项目管理团队采用“文件即岗位”的轻量分工：

1. `项目核心总纲.md`：项目负责人
2. `项目核心子纲-阶段演进与归档索引.md`：阶段总结与归档负责人
3. `进行中/当前阶段总表.md`：当前阶段执行负责人
4. `项目功能文档.md`：产品功能负责人
5. `README.md`：导航负责人
6. `Agent协同经验/项目承接提示词-当前版.md`：新对话状态交接负责人
7. `Agent协同经验/项目初始化.md`：协作协议负责人
8. `进行中/当前主线文档`：当前工作面事实负责人
9. `归档/阶段目录/阶段说明.md`：历史追溯负责人

执行原则：

1. 每个岗位只做自己的事，不跨层抢职责
2. 当某个文件开始持续吸收不属于本岗位的信息时，应立即下放到 `进行中/` 或 `归档/`
3. 阶段切换时，优先检查 `项目核心子纲`、`当前阶段总表`、`项目承接提示词-当前版` 是否已经出现膨胀迹象
