## 1. 本轮目标

当前不是回头重做上一阶段页面切换，也不是把 Today 误写成正式完成，而是先把以下问题收束清楚：

1. Today 当前哪些部分已经是事实层过渡态。
2. Today 当前哪些部分仍停留在结构化拼装态。
3. 内容详情、收藏、推荐、报告后续要依赖什么统一引用规则。
4. 当前阶段代码推进时，最小优先补什么，不该先补什么。

## 2. 当前代码事实判断

基于 `app/api/v1/dashboard.py`、`apps/web/src/pages/TodayPage.tsx`、`apps/web/src/pages/ArticlePage.tsx` 的当前实现，可以把 Today 拆成四层：

### 2.1 信息获取

当前已成立：

1. `worth_knowing` 已从真实 `hot_topics` 表读取。
2. `worth_acting` 已从真实 `opportunities` 表读取。
3. `recommended_for_you` 已按真实 `hot_topics / opportunities` 与兴趣关键词做最小匹配。

当前未成立：

1. RSS 文章尚未正式进入 Today 主供给位。
2. 真实正文、真实摘要、真实相关推荐尚未形成统一内容读取链路。
3. 用户兴趣事实层仍是 `users.interests` 与共享虚拟兴趣状态并存。

### 2.2 信息加工

当前已成立：

1. Today 已按“值得知道 / 值得行动 / 因你关注而推荐”完成后端结构化输出。
2. 基础相关性说明已经有最小生成逻辑。

当前未成立：

1. 没有统一排序分。
2. 没有统一内容质量分与推荐分合并规则。
3. 没有正文、摘要、标签、来源可信度的统一加工层。

### 2.3 信息整合

当前已成立：

1. Today 已经是聚合接口，不再只是前端 mockData 自转。

当前未成立：

1. `hot_topics / rss_articles / opportunities` 仍是分表分类型输出。
2. Today 到 Article 详情当前仍主要依赖路由 state 传值。
3. 收藏表 `favorites` 仍使用 `item_type + item_id` 组合引用，尚未有统一内容引用键。

### 2.4 信息呈现

当前已成立：

1. Today 页结构已经稳定。
2. Today -> ArticlePage 跳转已可用。
3. ArticlePage 收藏已接到 `/api/v1/favorites`。

当前未成立：

1. 详情页不是“可独立读取”的正式详情页。
2. 无正文层、无 AI 摘要层、无相关推荐层。
3. Today 的推荐区还没有把“推荐内容卡片”真正展开成统一内容卡片系统。

## 3. 当前阶段的正式态 / 过渡态 / 虚拟态 / 回退态

### 3.1 正式态

当前阶段内，Today 相关还没有完整正式态。

当前仅可视为局部正式的部分：

1. 热点表、机会表本身作为内容供给表已真实存在。
2. 收藏接口 `/api/v1/favorites` 已是真实接口。

### 3.2 过渡态

当前 Today 主体属于：

`真实供给表 + 后端聚合接口 + 前端结构呈现 + 详情页字段传递`

这就是当前主线的真实接口过渡态。

### 3.3 虚拟态

当前仍存在的虚拟态主要有：

1. 兴趣回退到共享虚拟兴趣状态。
2. 详情页正文与相关推荐仍是占位表达。
3. 推荐排序与内容加工仍属规则占位态。

### 3.4 回退态

当前回退态主要有：

1. Today 请求失败时，前端文案提示“可能来自后端回退数据”。
2. 详情页若没有路由 state，则只能返回 Today，无法独立拉取详情。

结论：

`Today 当前已经不是 mock 原型态，但也绝不是正式内容层；它是“真实供给已接入、统一内容层未收口”的过渡态。`

---

## 4. 当前最核心缺口

### 4.1 缺统一内容引用规则

这是当前最值得先补的一层。

如果没有统一引用规则，就会持续出现：

1. Today 能看，但详情页无法独立读取。
2. 收藏已真实化，但只能收藏“页面传来的对象”，很难沉淀为统一内容事实。
3. 报告、历史、推荐无法稳定复用同一内容标识。

### 4.2 缺内容加工层

当前热点和机会只是“被选出来”，还没有被真正加工成：

1. 可复用摘要
2. 可解释推荐理由
3. 可比较排序分
4. 可进入详情页的统一内容对象

### 4.3 缺详情正文层

当前 ArticlePage 只能展示：

1. 标题
2. 来源
3. 摘要
4. 收藏操作

正文、AI 摘要、相关推荐都还没有进入当前主链路。

## 5. 当前最合理的结构判断

### 5.1 统一内容层应当存在

当前最合理的判断不是继续长期维持“多张表直接给页面各自输出”，而是至少建立：

1. 统一内容引用键
2. 统一内容类型
3. 统一来源字段
4. 统一摘要字段
5. 统一详情读取规则

### 5.2 但当前不必立即重表

本阶段不必先上完整 `contents` 正式实体表，也不必先做重型迁移。

当前更合适的是先做：

1. 统一内容引用键方案
2. Today 输出统一内容引用字段
3. 详情页按统一引用规则收口，统一详情入口固定为 `/api/v1/content/by-ref`
4. 收藏、报告、推荐后续都沿用这一规则

当前已经受这条规则影响的页面至少包括：

1. `HotTopicsPage`
2. `CollectionsPage`

这属于：

`先统一引用，再决定是否实体化成 contents 表`

## 6. 当前建议的数据最小方案

当前阶段建议采用：

### 6.1 统一内容引用键

格式建议：

`{content_type}:{id}`

例如：

1. `hot_topic:12`
2. `opportunity:7`
3. `article:35`

### 6.2 Today 输出最小统一内容元数据

Today 中所有可跳详情、可收藏、可进入报告引用的内容，应至少带：

1. `content_ref`
2. `content_type`
3. `id`
4. `title`
5. `summary`
6. `source_name`
7. `source_url`

### 6.3 详情页当前过渡态

在正式详情接口未补齐前，详情页应至少承认：

1. 当前是“引用已统一、详情仍为过渡态”
2. 若已有路由 state，则优先使用
3. 后续应允许按 `content_ref` 独立读取

## 7. 当前阶段推进顺序

### 第一优先级

1. 建立并固定 Today 的统一内容引用规则。
2. 让 Today -> ArticlePage 的跳转带上 `content_ref`。
3. 补默认 `pytest` 入口下的 Today 聚合 smoke / 规则测试。

### 第二优先级

1. 梳理 `hot_topics / rss_articles / opportunities` 的字段映射。
2. 决定详情页读取优先顺序。
3. 规划 AI 摘要与相关推荐的最小接入点。

补充前置项：

4. `article` 详情链路若要进入正式数据库切换，必须先完成 `rss_articles` 重包的分片规则设计。
5. 在 `rss_articles` 分片规则未明确前，不应直接推进 `article` 类型的云端正式详情切换。

## 9. 本轮结论

当前项目进入阶段二后的正确判断应为：

1. Today 已经拥有真实供给表与真实聚合接口。
2. Today 当前主要问题不再是“有没有接口”，而是“有没有统一内容层与统一事实引用规则”。
3. 当前最小正确推进，不是立刻重表，而是先把统一内容引用键和详情过渡态收束清楚。
4. 默认测试入口下一步应优先补 Today 聚合规则和内容引用 smoke，而不是停留在纯环境级 smoke。
5. `Today` 当前与 D1 的前置收束不应被误判为“所有内容类型都要一起切”；`article` 详情链路当前仍受 `rss_articles` 重包约束，应先完成分片规则设计，再决定何时执行。
