- 文档状态：进行中
- 建立日期：2026-04-27
- 文档定位：承接本轮“正式页面功能与问题梳理 + 阶段性解决方案”
- 适用范围：`apps/web` 正式前端、`apps/edge-worker` Workers API、`packages/contracts` 页面契约、`infra/cloudflare/d1/migrations` D1 事实表

## 1. 本轮结论摘要

1. 当前五个页面都位于正式前端主路由下，均受 `ProtectedRoute` 保护：
   - `/me` -> `apps/web/src/pages/MyPage.tsx`
   - `/chat` -> `apps/web/src/pages/ChatPage.tsx`
   - `/log` -> `apps/web/src/pages/JournalPage.tsx`
   - `/today` -> `apps/web/src/pages/TodayPage.tsx`
   - `/todo` 与 `/actions` -> `apps/web/src/pages/ActionsPage.tsx`
2. 当前页面数据主链路基本来自 `apiService` 调 Workers `/api/v1/*`，不是 demo mock 主驱动。
3. 当前最需要先修的是页面职责和信息层级，不是新增大功能。
4. Chat 页问题最明确：正式会话读写链已存在，但前端渲染只默认显示最近 4 条，并且消息区嵌在内部滚动容器中，直接导致可浏览范围太小。
5. Actions 页数据契约相对完整，但首屏仍被打卡和统计占据，导致“待办页好像只有打卡有效”。
6. Journal 页真实读取 `notes + growth-overview`，但“今日叙事 / 成长轨迹”权重大于“记录本身”，没有充分承接 Chat 沉淀。
7. Today / 简报页已完成阶段 1 前端重排与阶段 2 每日结果对象契约收口：`dashboard/today` 当前可优先消费 `briefings.payload`，并保留热点、机会与兴趣即时聚合兜底。
8. My 页职责方向最接近正确：个人中心和入口分发；主要问题是退出登录与成长入口同级前置，位置和语义不合理。

## 2. 页面职责总表

| 页面 | 路由 | 当前实际职责 | 理论职责 | 差距 |
|---|---|---|---|---|
| 我的 | `/me` | 个人摘要 + 成长入口 + 沉淀入口 + 退出登录 | 个人中心和二级入口分发页 | 退出登录过于前置，成长摘要略重 |
| 对话 | `/chat` | 输入、模式切换、确认、结果摘要、会话读取、处理细节混合页 | 最低成本表达入口 | 真实对话内容被 Hero、卡片、折叠状态压缩 |
| 记录 | `/log` | 今日叙事 + 成长轨迹 + 最近 notes | 快速记录和回看记录 | 记录主任务不够强，Chat 沉淀关系不够清楚 |

## 4. 逐页详细分析

### 4.1 我的页

#### 当前页面职责判断

- 现在实际在做：用 `getGrowthOverview()` 读取成长摘要，展示用户名、画像摘要、关键词、真实记录数、近期回看数，并分发到收藏、画像、历史日志、历史简报、设置、帮助反馈。
- 理论上应该做：个人中心和二级入口分发页，帮助用户进入成长、收藏、设置、历史，不应成为成长页本体。
- 差距：方向基本正确，但“进入成长页”和“退出登录”并排同级，退出登录作为危险/账号动作被放得太靠前。

#### 当前主要问题

- 功能问题：退出登录功能本身可用，调用 `logout()` 后跳 `/welcome`。
- 信息架构问题：账号操作没有单独分组，和成长入口混在首屏主卡。
- UI/交互问题：退出登录不是日常主动作，应放到设置入口、账号区末尾或二次确认。
- 数据源/接口问题：只读 `preferences/growth-overview`，没有专门的 `my/overview` 或账号摘要接口。
- 页面协作问题：`My` 与 `Growth` 的边界仍依赖前端约束，后端仍给的是成长概览数据。

#### 前端代码定位

- 页面文件：`apps/web/src/pages/MyPage.tsx`
- 关键组件：`Masthead`、`MetricMiniCard`、`NavigationEntryCard`、`PageNoticeCard`、`Button`、`Tag`
- 相关样式：`apps/web/src/styles/my-pages.css`
- 内联样式：当前页面基本无明显内联样式。
- 硬编码文案：`ENTRY_ITEMS`、`进入成长页`、`退出登录`、`个人沉淀入口`。
- 局部假状态：无明显假业务状态；加载失败时入口仍可用。

#### 后端/数据依赖

- 当前 API：`GET /api/v1/preferences/growth-overview`，`POST /api/v1/auth/logout`
- 正式链路：Workers + D1，`AppContext` 通过 `auth/me` 还原用户。
- mock/localStorage：无正式业务 mock；认证由 cookie session 驱动。
- 接口调整判断：阶段 1 不需要新增接口；如果后续要做账号页，可考虑新增 `GET /auth/me` 扩展字段或 `GET /account/overview`，但不是当前最小需求。

#### 改进建议

- 低风险前端调整：把退出登录移到页面底部“账号与安全”区，或收进“设置”页；首屏只保留成长、收藏、历史等入口。
- 需要接口配合：暂无。
- 需要 schema 支撑：暂无。
- 暂不建议：不要把 My 页改成成长详情页，不要新增大面积账号管理系统。

#### 优先级

- P1：退出登录位置影响正式产品感。
- P2：入口分组和摘要层级优化。

### 4.2 对话页

#### 当前页面职责判断

- 现在实际在做：低成本输入、模式切换、示例、动作上下文、待确认、结果摘要、最近消息、会话读取状态和高级细节。
- 理论上应该做：最低成本表达入口，用户一进来先能说一句话，最近消息和最近结果应围绕输入服务。
- 差距：页面承担了完整控制台和会话状态展示，真实消息浏览被多个卡片挤压。

#### 当前主要问题

- 功能问题：消息列表默认只显示最近 4 条，更多历史被折叠；用户反馈“可浏览区域太小”与此直接相关。
- 信息架构问题：Hero、模式 chip、空状态卡、结果摘要、处理细节与消息列表竞争首屏。
- UI/交互问题：`chat-message-list-scroll` 内部滚动叠加 `PageContent` 外层滚动，移动端体验容易只剩很短滚动区域。
- 数据源/接口问题：正式会话链存在，但前端仍保留 `localStorage` 作为缓存兜底。
- 页面协作问题：Article/Today/Collections 能跳 Chat 并带上下文，但 ActionContext 里仍显示 `内容引用`，偏内部。

#### 前端代码定位

- 页面文件：`apps/web/src/pages/ChatPage.tsx`
- 关键组件/函数：`HeroCard`、`MessageList`、`MessageItem`、`PendingConfirmationCard`、`ResultSummaryCard`、`ActionContextCard`、`ConversationEmptyState`
- 关键 hook：`apps/web/src/hooks/useChatLogic.ts`
- 相关样式：`apps/web/src/styles/chat-page.css`
- 内联样式：页面组件内无明显 style 属性，但局部结构较重。
- 硬编码文案：模式、示例、处理细节、读取方式、会话编号等均在页面内。
- 局部假状态：`setTimeout` 模拟 typing；`localStorage` 缓存作为会话兜底。

#### 后端/数据依赖

- 当前 API：
  - `GET /api/v1/preferences/interests`
  - `POST /api/v1/chat/recognize`
  - `POST /api/v1/chat/execute`
  - `POST /api/v1/chat/reclassify`
  - `GET /api/v1/chat/sessions?limit=20`
  - `GET /api/v1/chat/sessions/:session_id/messages`
- 正式链路：Workers + D1，`chat_sessions`、`chat_messages` 已在 D1 migration 中成立。
- mock/localStorage：`jianbao_chat_state_v1` 仍用于缓存兜底；正式写失败不再虚拟成功。
- 接口调整判断：阶段 1 不需要；阶段 2 可考虑增加“最近结果摘要 / 最近会话轻量读侧”，或给 session 增加更适合 UI 的标题与 lastResult，但不能先做。

#### 改进建议

- 低风险前端调整：取消“只显示最近 4 条”的默认限制；让消息列表参与页面主滚动，或给消息区明确更大 `min-height` / `max-height`；把模式和高级状态降级。
- 需要接口配合：如要完整会话历史页，再新增 session 列表分页 / session detail 页面。
- 需要 schema 支撑：完整历史页可能需要 session title、pinned、archived 等字段；当前不建议先做。
- 暂不建议：不要把 Chat 直接扩成完整 IM 历史系统；先保证表达入口和最近消息可读。

#### 优先级

- P0：消息区可浏览范围过小，阻碍当前页面人工验收。
- P1：控制台感强，正式产品感不足。

### 4.3 记录页

#### 当前页面职责判断

- 现在实际在做：读取 notes 与 growth-overview，生成“今日叙事”、成长轨迹和最近想法记录。
- 理论上应该做：快速留下和回看值得沉淀的内容，承接 Chat 生成的 note / fragment。
- 差距：真正的记录列表在后半页，前面叙事和成长包装更重。

#### 当前主要问题

- 功能问题：没有页面内新增记录，只能跳 Chat；`AI生成` 只是前端 `setTimeout`，没有真实生成接口。
- 信息架构问题：“今日叙事 / 成长轨迹 / 想法记录”顺序让记录主任务后置。
- UI/交互问题：删除记录可用，但缺少编辑、筛选、来源说明的正式策略。
- 数据源/接口问题：`notes` 与 `growth-overview` 分开读，未形成“记录页 overview”。
- 页面协作问题：Chat 可写 note，但 Journal 未明确按“聊天沉淀 / 手动记录 / 历史行为 / 收藏”分层。

#### 前端代码定位

- 页面文件：`apps/web/src/pages/JournalPage.tsx`
- 关键组件：`Masthead`、`Button`、`Tag`
- 相关样式：`apps/web/src/styles/journal-feedback.css`
- 内联样式：无明显 style 属性。
- 硬编码文案：`今日叙事`、`成长轨迹`、`想法记录`、`AI生成`、`记录进行中`。
- 局部假状态：`AI生成` 只是 1.5 秒前端状态。

#### 后端/数据依赖

- 当前 API：
  - `GET /api/v1/notes`
  - `DELETE /api/v1/notes/:id`
  - `GET /api/v1/preferences/growth-overview`
- 正式链路：Workers + D1，`notes`、`history_entries`、`briefings`、`opportunity_follows` 参与成长概览。
- mock/localStorage：无正式 mock；生成按钮是前端假状态。
- 接口调整判断：阶段 1 不需要；阶段 2 如要承接更多沉淀，建议新增 `GET /journal/overview` 或扩展 contracts，聚合 notes、chat-created notes、history、favorites、period review。

#### 改进建议

- 低风险前端调整：把“想法记录”上移为主区；将“成长轨迹”降为回看入口；移除或降级 `AI生成` 假动作。
- 需要接口配合：记录来源、记录类型、关联 chat message / content ref 的展示需要后端字段更清楚。
- 需要 schema 支撑：如要正式区分用户笔记、聊天沉淀、周期回顾摘录，可能需要 note_type / source_context / chat_message_id。
- 暂不建议：不要先做复杂日记编辑器。

#### 优先级

- P1：记录页薄弱，无法承担沉淀入口。
- P2：移除假生成状态和强化来源说明。

### 4.4 简报页

#### 当前页面职责判断

- 现在实际在做：基于 `dashboard/today` 展示总述、值得知道、值得行动、因关注推荐、今日速记。
- 理论上应该做：今日信息输入主舞台，帮助用户快速判断今天先看什么、先做什么。
- 差距：阶段 1 已先解决首屏主判断不够突出，阶段 2 已补每日结果对象契约；后续差距主要转为 `briefings.payload` 的生产生成质量和人工验收。

#### 当前主要问题

- 功能问题：内容点击、转待办、去记录都已连通。
- 信息架构问题：章节导航 + 五段内容使页面像聚合目录，首屏主判断不够锋利。
- UI/交互问题：信息密度和卡片权重偏平均，缺少“今天最重要的一条”。
- 数据源/接口问题：`dashboard/today` 当前可优先消费 `briefings.payload` 作为每日结果对象，缺失时继续读热点、机会、兴趣做即时聚合；若后续改用 `result_payload` 命名，需先新增 migration。
- 页面协作问题：Today -> Chat / Todo / Article 链路成立，但“今日速记”和“值得行动”没有形成更清楚的沉淀闭环。

#### 前端代码定位

- 页面文件：`apps/web/src/pages/TodayPage.tsx`
- 关键组件：`TodaySummaryCard`、`TodayContentCard`、`TodaySection`、`TodayInfoBox`
- 相关样式：`apps/web/src/styles/today-page.css`
- 内联样式：无明显 style 属性。
- 硬编码文案：`值得知道的`、`值得行动的`、`因你关注而推荐`、`今日速记`、空状态文案。
- 局部假状态：无前端 mock，但后端 `issueNumber: 128` 是固定值，summary 文案偏过程口径。

#### 后端/数据依赖

- 当前 API：`GET /api/v1/dashboard/today`
- 正式链路：Workers + D1，读取 `hot_topics`、`opportunities`、`user_interests`。
- mock/localStorage：无正式 mock。
- 接口调整判断：阶段 2 已完成最小结果对象契约；后续若继续增强，应先验证 `briefings.payload` 的生产质量，再评估是否接入 `summary_generation_results` 或新增 snapshot。

#### 改进建议

- 低风险前端调整：减少同权重区块，突出今日判断；默认展示 3 条知道、2 条行动；推荐和速记下沉。
- 需要接口配合：后端摘要文案应去掉过渡表达；如果要“新鲜、有趣、可拓展”，需要后端给 daily angle / lead item / extension slots。
- 需要 schema 支撑：当前不需要新增 schema；若要改成 `briefings.result_payload` 或专门 `today_snapshot`，需新增 migration 并同步 contracts。
- 暂不建议：不要先大改内容采集和 AI 生成链。

#### 优先级

- P1：简报作为今日输入主舞台，产品感不足会影响主链路。
- P2：卡片结构和文案新鲜度优化。
- P3：每日结果对象和扩展位。

### 4.5 待办页

#### 当前页面职责判断

- 现在实际在做：读取 `actions/overview`，展示打卡、统计、待办筛选、收藏待处理、后续跟进。
- 理论上应该做：行动转化与推进页，让用户马上知道现在先做什么。
- 差距：打卡和统计抢占首屏，待办和跟进被后置。

#### 当前主要问题

- 功能问题：待办完成、删除、打卡都可调用正式接口。
- 信息架构问题：打卡、统计、待办、收藏、跟进并列；“今日最该推进”缺席。
- UI/交互问题：来源直接展示 `chat/content/manual`，仍偏内部枚举。
- 数据源/接口问题：`actions/overview` 已有 todos、savedForLater、followingItems、reminderSummary，但没有明确的 top priority / suggestedNextAction。
- 页面协作问题：Chat 可生成 todo，Today 可引导转待办，机会跟进也进 overview；但页面没有把这些来源组织成闭环。

#### 前端代码定位

- 页面文件：`apps/web/src/pages/ActionsPage.tsx`
- 关键组件：`ContentListCard`、`Button`
- 相关样式：`apps/web/src/styles/actions-page.css`
- 内联样式：无明显 style 属性。
- 硬编码文案：`行动节律`、`今日打卡`、`待办事项`、`收藏待处理`、`后续跟进`。
- 局部假状态：无明显前端假数据；streak 来自后端 `getActivityStreak`。

#### 后端/数据依赖

- 当前 API：
  - `GET /api/v1/actions/overview`
  - `POST /api/v1/actions/check-in`
  - `PUT /api/v1/todos/:id`
  - `DELETE /api/v1/todos/:id`
- 正式链路：Workers + D1，`todos`、`favorites`、`opportunity_follows`、`briefing_schedules`、`history_entries`。
- mock/localStorage：无正式 mock。
- 接口调整判断：阶段 1 不需要；阶段 2 可扩展 `ActionsOverviewData`，增加 `primaryAction`、`sourceLabel`、`actionReason`、`originPage`。

#### 改进建议

- 低风险前端调整：把“今日最该推进的 1 件”提到顶部；打卡和统计下沉为摘要行；跟进中排到收藏前；sourceType 显示为中文自然语言。
- 需要接口配合：后端给主推荐待办、行动来源文案和下一步理由。
- 需要 schema 支撑：如要完整闭环，需要 `opportunity_execution_results` 与 todo/follow 的状态在 UI 中可读。
- 暂不建议：不要先做复杂项目管理看板。

#### 优先级

- P1：待办主任务不明确，直接影响正式产品感。
- P2：来源文案和分组强化。
- P3：机会执行结果闭环。

## 6. 分阶段解决方案

### 阶段 0：事实核查

目标：

- 确认页面职责、路由、真实数据源、后端接口、当前问题清单。

交付物：

- 页面职责矩阵：见第 2 节。
- 页面问题矩阵：见第 3 节。
- 前后端依赖矩阵：见第 5 节。

验收方式：

- 文档核查完成。
- 不更新测试数字。
- 如继续进入实现前，先确认本文件与 `当前阶段总表`、UI 当前方案总表无冲突。

### 阶段 1：低风险前端收口

目标：

- 不改接口、不改 schema，先修明显体验问题。

建议任务：

1. Chat：取消默认只看最近 4 条或改成更自然的历史折叠；扩大消息区；降低高级状态权重。
2. Actions：把“今日最该推进”前置；打卡和统计下沉；跟进中排到收藏前；sourceType 中文化。
3. Journal：把记录列表上移；成长轨迹降级；移除或弱化 `AI生成` 假动作。
4. Today：突出今日判断；缩短总述；推荐和速记下沉；卡片数量和权重收束。
5. My：退出登录移到底部账号区或设置页；首屏只保留个人入口分发。

验收方式：

- 主门槛：`npm.cmd run check`
- UI 阶段收口后：`npm.cmd run task:verify:proof`
- 涉及浏览器级主路径时：`npx.cmd playwright test e2e/app-mainline.spec.ts`
- 测试数字只更新 `文档/进行中/当前测试与验收总表.md`

### 阶段 2：前后端契约收口

目标：

- 如果页面职责需要正式数据支撑，最小化调整接口和数据结构。

候选调整：

1. `apps/edge-worker`
   - `dashboard/today`：已去掉用户可见过渡文案，并已增加 `leadItem / dailyAngle / freshness / extensionSlots` 的可选契约。
   - `actions/overview`：增加 `primaryAction`、自然语言 `sourceLabel`、`actionReason`。
   - `journal/overview`：如确定新增，聚合 notes、history、favorites、chat-created notes。
   - `chat/sessions`：如做完整会话历史页，增加分页与更好 session 标题。
2. `packages/contracts`
   - 所有新增页面字段必须先落到 `packages/contracts/src/page-data.ts`。
   - 不允许前端私造字段长期漂移。
3. `infra/cloudflare/d1/migrations`
   - 阶段 2 默认先不改 schema。
   - 只有当需要 `note_type`、`chat_message_id`、`today_snapshot`、`session title/pinned/archived` 等持久字段时，再新增 migration。

风险与测试：

- Workers 改动必须跑：`npm.cmd run check:workers`
- 契约和前端联动必须跑：`npm.cmd run check`
- D1 migration 必须补 Workers 路由测试和 D1 主链验证；测试数字仍只写测试总表。

### 阶段 3：逐页实施顺序

推荐顺序：

1. 对话页
2. 待办页
3. 记录页
4. 简报页
5. 我的页

理由：

- Chat 是当前职责混杂和可用性问题最明确的页面，且阶段 1 可纯前端处理。
- Actions 已有较完整数据契约，低风险重排收益高。
- Journal 是 Chat 沉淀的承接页，应在 Chat 修完后接上。
- Today 涉及今日信息输入主舞台，后续可能牵动 `dashboard/today` 文案和契约，应放在低风险页之后。
- My 方向最正确，且退出登录位置是局部调整，可最后收口。
