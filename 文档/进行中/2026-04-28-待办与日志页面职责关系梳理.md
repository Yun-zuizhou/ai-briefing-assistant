## 2. 当前重叠的本质

两个页面现在不是接口层没有分工，而是页面呈现层仍把同一批对象用了两次。

### 2.1 直接重叠对象

| 对象 | 待办页用法 | 日志页用法 | 当前问题 |
|---|---|---|---|
| 当前待办 | 主任务、列表、完成、删除 | 作为“我推进过的”摘要 | 日志页会显示仍待处理的事项，和待办页抢“下一步行动”的心智 |
| 机会跟进 | 后续跟进列表 | 作为“我推进过的”摘要 | 两页都像跟进列表，只是日志页不能操作 |
| 收藏待处理 | 收藏待处理队列 | 作为“我留下的”摘要 | 可接受，但日志页需要明确只是留存入口 |
| 打卡 / 已完成 | 行动节律摘要 | 沉淀计数的一部分 | 可接受，但不应成为日志页主模块 |

### 2.2 用户心智上的重叠

用户进入“待办”时，核心问题是：

> 我现在先做什么？

用户进入“日志”时，核心问题应该是：

> 我最近留下了什么？哪些东西值得回看或继续进入？

当前日志页的“我推进过的”容易把第二个问题拉回第一个问题，尤其当它展示的是 active todos，而不是完成痕迹、创建痕迹或行动摘要时。

---

## 3. 推荐结论

短期不建议合二为一。建议保留两个一级页面，但必须把边界从“对象类型”改成“用户状态”：

| 页面 | 应定义为 | 核心动词 | 时间方向 | 交互强度 |
|---|---|---|---|---|
| 待办 | 行动推进页 | 做、完成、删除、跟进、稍后处理 | 现在 -> 下一步 | 强操作 |
| 日志 | 个人沉淀页 | 看见、回看、整理、继续进入 | 刚刚 -> 过去 -> 周期回看 | 轻操作 |

一句话分工：

- 待办页负责“把未完成的事往前推”。
- 日志页负责“把已经发生或已经留下的东西组织起来”。

---

## 4. 页面协作关系

推荐链路：

```txt
Chat / Today
  -> 创建记录、待办、收藏、跟进
  -> Actions 处理未完成事项
  -> Journal 回看说过、做过、留下过的痕迹
  -> Growth / HistoryBrief 做周期理解
```

页面间配合规则：

1. `Actions` 可以显示 active todos、future todos、followingItems、savedForLater，并提供完成 / 删除 / 跟进操作。
2. `Journal` 可以引用 todos 和 followingItems，但只应显示“行动痕迹”或“行动池摘要”，不应承担待办列表管理。
3. `Journal` 中的行动入口只跳到 `/actions`，不在本页完成、删除或重排待办。
4. `Journal` 的主内容应是 notes 和留存对象，行动只是沉淀的一类证据。
5. `Growth`、`HistoryLogs`、`HistoryBrief` 继续作为二级回看和证据层，不回流抢日志页首屏。

## 5. 推荐布局调整

### 5.1 待办页布局

待办页应继续强化“执行优先”：

1. 主行动卡
   - 标题：今天先推进这 1 件
   - 来源：后端 `topPriority`
   - 主按钮：完成 / 去跟进 / 继续查看
2. 行动队列
   - 分段：今日 / 未来 / 已完成
   - 只放 todo，不混收藏和历史
3. 跟进项目
   - 放机会跟进、申请、等待反馈
   - 作为行动对象，不作为历史对象
4. 稍后处理
   - 放收藏待处理
   - 保持弱于待办和跟进
5. 行动节律
   - 打卡、连续天数、提醒时间
   - 放底部或摘要条，不抢首屏

### 5.2 日志页布局

日志页应继续弱化“当前待做”，强化“个人材料”：

1. 最近沉淀
   - 说过 / 做过 / 留下
   - 文案避免像任务看板
2. 我刚说过的
   - notes 为主
   - 支持删除，暂不做复杂编辑器
3. 行动痕迹
   - 不再叫“我推进过的”或减少 active todo 暗示
   - 优先展示已完成、已创建、已跟进过的摘要
   - 如果没有完成痕迹，只展示“行动池有 N 项，去待办处理”
4. 我留下的
   - favorites + history 摘要
   - 跳转 HistoryLogs / Collections
5. 长期回看
   - Growth / HistoryBrief 入口
   - 不搬成长页主体内容

## 6. 需要调整的具体点

### 6.1 命名统一

当前底部导航叫“日志”，页面 Masthead 叫“记录”。建议二选一：

1. 保守方案：底部导航改为“记录”，页面继续叫“记录”。
2. 更准确方案：底部导航和页面都叫“沉淀”。

如果继续叫“日志”，用户会期待日记编辑器或时间流水；当前产品实际更像“个人沉淀入口”。

### 6.2 JournalOverview 的数据语义

`buildJournalOverview()` 当前把 active todos 直接塞进 `progressItems`。建议后续调整为：

1. `progressItems` 优先用 completed todos、recently created todos、recent follow history。
2. active todos 不作为列表展示，只作为摘要 CTA：
   - `行动池还有 N 件待推进`
   - `去待办处理`
3. 如果要长期保留 active todo 摘要，契约层应拆成：
   - `actionPoolSummary`
   - `progressEvidenceItems`

### 6.3 ActionsOverview 与 JournalOverview 的边界

`ActionsOverviewData` 是操作页契约，应该回答：

- 主行动是什么
- 待办队列是什么
- 跟进队列是什么
- 稍后处理是什么

`JournalOverviewData` 是沉淀页契约，应该回答：

- 最近说过什么
- 最近留下了哪些材料
- 最近有哪些行动痕迹
- 有哪些长期回看入口

不建议继续让 `JournalPage` 自己兜底拼 `getActionsOverview()` 作为长期方案；当前兜底可以保留，但正式语义应收在 `journal/overview`。

## 7. 是否合二为一

### 7.1 不建议短期合并

不建议现在直接合并，原因：

1. 两页服务的是不同用户状态：一个是执行，一个是回看。
2. 当前底部主导航已经承认两页是不同入口，强行合并会牵动导航、路由、E2E 和页面契约。
3. `ActionsOverviewData` 和 `JournalOverviewData` 已经分开，说明契约层正在走向分治。
4. 合并容易变成“一个更大的杂糅页”，不能从根上解决重叠。

### 7.2 可选合并形态

如果后续必须减少一级入口，建议不是简单删除一页，而是合并为一个一级页面：

```txt
我的事
  - 待处理
  - 已留下
```

或：

```txt
行动与记录
  - 行动
  - 沉淀
```

但这属于导航信息架构调整，应在 Today、Chat、My 的入口关系一起审查后再做。

## 8. 推荐实施顺序

阶段 1：纯前端呈现收口

1. 将底部“日志”和 Masthead“记录”统一命名。
2. 调整日志页“我推进过的”为“行动痕迹”或“行动池摘要”。
3. 日志页不再逐条展示 active todos；active todos 只作为跳转待办页的摘要。
4. 待办页继续保留主行动、待办队列、跟进队列、稍后处理和节律摘要。

阶段 2：契约语义收口

1. 调整 `buildJournalOverview()` 的 `progressItems` 语义。
2. 必要时扩展 `JournalOverviewData`，拆出 `actionPoolSummary`。
3. 补 Workers route test，避免 Journal overview 回到 active todo 列表。

阶段 3：导航层评估

1. 如果用户仍觉得两个入口多，再评估合并为“行动与记录”。
2. 合并前先更新 TabBar、路由、E2E 和页面文档。

---

## 9. 当前决策建议

本轮建议采用：

> 保留两个页面；待办页做强操作，日志页做轻回看；日志页只承接行动痕迹，不承接当前待办管理。

下一步最小改动范围：

1. `apps/web/src/pages/JournalPage.tsx`
2. `apps/web/src/styles/journal-feedback.css`
3. 如进入契约阶段，再改：
   - `apps/edge-worker/src/services/behavior/builder.ts`
   - `packages/contracts/src/page-data.ts`
   - `apps/edge-worker/tests/journal.route.test.ts`
