# 2026-04-22 system 执行链最小接入方案

- 文档状态：进行中
- 记录日期：2026-04-22
- 记录类型：当前阶段执行设计
- 目标：把当前 `/api/v1/system/*` 从“可观测接口”推进到“至少一条具备真实状态流转的执行链”

---

## 1. 当前代码现实

当前 `apps/edge-worker/src/routes/system.ts` 已具备以下能力：

1. `GET /api/v1/system/chain-health`
   - 面向当前登录用户返回支撑链事实层计数。
2. `POST /api/v1/system/summary-tasks`
   - 登录用户可创建 `summary_generation_tasks` 事实。
3. `GET /api/v1/system/summary-tasks`
   - 登录用户可查看自己的摘要任务。
4. `GET /api/v1/system/summary-tasks/:id`
   - 登录用户可查看自己的摘要任务详情。
5. `GET /api/v1/system/summary-tasks/:id/result`
   - 登录用户可查看自己的摘要结果。
6. `POST /api/v1/system/ingestion-runs`
   - internal token 保护下可写入采集运行事实。
7. `POST /api/v1/system/ai-processing-runs`
   - internal token 保护下可写入 AI 加工运行事实。
8. `POST /api/v1/system/operation-logs`
   - 登录用户可写入可回放操作日志。
9. `POST /api/v1/system/replay-tasks`
   - 登录用户可为自己的操作日志创建回放任务。
10. `GET /api/v1/system/replay-tasks`
   - 登录用户可查看自己的回放任务。

一句话判断：

`当前 system 已经不是“没有支撑链”，而是“有事实层、有查看入口，但还缺执行器与状态流转”。`

---

## 2. 当前最小缺口

从“具备事实表”到“具备真实执行链”，当前最小缺的不是更多表，而是以下 3 类能力：

1. **状态推进**
   - 当前任务大多只会“创建一条记录”，缺少 `queued -> running -> succeeded / failed` 的正式推进入口。
2. **结果回写**
   - 当前 `summary_generation_results` 有读口，但没有明确的正式写回链路约定。
3. **执行者身份**
   - 当前 internal token 已能保护内部接口，但还没有一条稳定的“谁来拿任务、谁来回写结果”的最小 contract。

---

## 3. 当前不建议同时做的事

本轮不建议同时推进：

1. Cron 调度器
2. 队列系统
3. Replay 控制台
4. 多类任务统一编排器
5. 所有支撑链一起执行化

原因很简单：

`这些能力会把范围迅速扩到“平台建设”，而当前阶段更需要的是先拿一条真实执行链形成证明。`

---

## 4. 最推荐先打通的垂直链

当前最推荐先打通的是：

`summary_generation_tasks -> summary_generation_results`

原因：

1. 用户侧已经有正式创建与查看入口。
2. 事实表已经存在，不需要再做表设计。
3. 结果查看路径已经存在，只差执行回写。
4. 相比 ingestion / ai-processing / replay，这条链最接近用户可见收益。
5. 当前阶段十六的 Python 摘要执行链可以先作为“最小执行者”，不必额外新建平台。

---

## 5. 最小接入方案

### 5.1 第一步：定义 internal 执行回写 contract

为摘要任务增加 2 类 internal 执行入口：

1. `POST /api/v1/system/summary-tasks/:id/start`
   - 作用：
     - 把 `summary_generation_tasks.status` 从 `queued / pending_provider` 推到 `running`
     - 写入 `started_at`
2. `POST /api/v1/system/summary-tasks/:id/complete`
   - 作用：
     - 写入 `summary_generation_results`
     - 更新 `summary_generation_tasks.status = succeeded`
     - 写入 `finished_at / updated_at`
3. `POST /api/v1/system/summary-tasks/:id/fail`
   - 作用：
     - 更新 `summary_generation_tasks.status = failed`
     - 写入 `error_message / finished_at / updated_at`

这 3 个入口都走：

- `Authorization: Internal <token>`

这样可以先把“执行者是谁”固定为：

`受 internal token 保护的执行器`

不先纠结是 Python 脚本、Cron、Queue consumer 还是别的执行器。

### 5.2 第二步：把 Python 摘要脚本接成最小执行者

当前建议先由已有 Python 摘要链承担执行者角色：

1. 任务创建后，Python 执行器拿到任务 ID
2. 调用 `/start`
3. 产出结果后，调用 `/complete`
4. 异常时，调用 `/fail`

这样可以用最小成本拿到：

`queued -> running -> succeeded/failed`

这一条正式状态机证明。

### 5.3 第三步：补最小自动化证明

最小自动化应至少补 3 类测试：

1. internal token 缺失时，`/start /complete /fail` 返回 `401/403`
2. 合法 internal token 下，摘要任务状态可推进到 `running`
3. `/complete` 后，用户能从 `GET /summary-tasks/:id/result` 读到结果

---

## 6. 为什么不先做 replay 或 ingestion

### replay

当前 replay 更适合作为“故障恢复工具”，不是最短用户价值链。

### ingestion / ai-processing

这两条更偏内部观测与流水线事实，先做执行化也可以，但短期内较难直接形成用户可见收益。

所以当前更合理的排序是：

1. 先打通 `summary task`
2. 再扩 `ingestion / ai-processing`
3. 最后再补 `replay`

---

## 7. 当前建议的下一步实现顺序

1. 在 `apps/edge-worker/src/routes/system.ts` 新增 `summary-tasks/:id/start|complete|fail`
2. 在 `apps/edge-worker/src/services/system/store.ts` 新增摘要任务状态更新与结果写入函数
3. 为 3 个 internal 执行入口补 Vitest
4. 让现有 Python 摘要执行链调用这 3 个 internal 接口
5. 再补一轮链路健康与用户读侧验证

---

## 8. 一句话结论

当前 `/api/v1/system/*` 的下一步，不是“大而全的任务平台”，而是：

`先把 summary task 这条最短垂直链做成真正会流转状态、会回写结果、用户能看到结果的正式执行链。`
