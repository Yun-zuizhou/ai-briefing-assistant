## 6. 本轮与自动化的关系

当前自动化已覆盖：

1. 受保护路由鉴权
2. welcome -> register -> interest-config -> today
3. today -> article -> collections
4. actions / growth / settings / help-feedback
5. chat / hot-topics / journal / profile / history
6. weekly / monthly / annual / history-brief

因此人工验收当前不再负责“证明功能存在”，而是负责：

1. 证明正式页面体验和文案已收口。
2. 补截图级证据。
3. 补自动化不适合承接的演示口径。

## 6.1 当前建议自动化补跑顺序

当本轮修改主要集中在页面文案、共享契约、Chat 展示层、内容详情或报告页时，建议优先按下面顺序补跑：

1. `apps/edge-worker` 定向 Vitest
   - 适用：鉴权、内容详情、反馈、报告等 Workers 路由返回形态变化
2. `apps/web` build
   - 适用：页面文案、共享契约、路由页展示、类型入口变化
3. `python -m pytest tests/test_api_mainline.py tests/test_dashboard_today.py -q`
   - 适用：Python 影子契约、页面静态契约、共享字段命名变化
4. `npx.cmd playwright test e2e/app-mainline.spec.ts`
   - 适用：需要再确认正式 UI、欢迎页拦截、登录后主链跳转是否仍一致
5. `npm.cmd run check`
   - 适用：准备把这轮结果并入口径前，回到唯一主门槛做最终复验

补充说明：

1. 上述顺序用于“先快后全”，不替代唯一主门槛。
2. 只有 `check` 通过后，才允许把结果级状态同步为“已复验通过”。
