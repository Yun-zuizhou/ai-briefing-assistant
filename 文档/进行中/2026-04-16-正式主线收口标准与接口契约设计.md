# 2026-04-16 正式主线收口标准与接口契约设计

- 记录日期：2026-04-16
- 记录类型：进行中主线设计文档
- 目标：为“正式主线已收口”建立统一的标准定义、问题定义、测试口径、接口契约和扩展留位，避免后续实现再次回到口头判断和临时兜底。

## 1. 文件定位

这份文档不是实现记录，也不是阶段总结。

这份文档只回答 5 类问题：

1. 现在什么问题必须被承认为“正式态阻塞项”。
2. 什么标准一旦不满足，就不能宣称“主线正式收口”。
3. 测试应该按什么口径证明“达标”，而不是只证明“代码能跑”。
4. 鉴权、Chat、报告、支撑链等接口边界应该怎么定义。
5. 哪些能力现在不用一次做完，但必须提前留出稳定扩展位。

## 2. 这份文档要解决的误判

当前项目已经证明了：

1. `workers + D1 + pages` 这一条链可以跑通。
2. `check:workers` 与前端构建可以通过。
3. 主链若干页面已经能读取真实接口。

但当前仍不能把“可跑”误判为“正式收口”，因为还存在以下口径漂移：

1. Workers 主线里的旧 query/header 身份透传虽已切掉，但资源级授权与生产验收还没完全收口。
2. Chat 在后端失败时仍可能回退到前端本地写状态。
3. 前端本地状态仍承担了部分业务事实和身份来源职责。
4. 报告仍存在合成结果和未严格定义的指标。
5. 支撑链目前主要是事实层与观测层，不等于执行链已成立。
6. 当前“部分字段未补齐”的正式表达仍需继续收口，但不再适合继续沿用 `transitional` 这一含混命名。
7. 自动化测试虽已切到 cookie session 认证上下文，但生产人工验收与资源级权限验证还没补齐。

一句话判断：

`当前真正要收口的，不是“能不能继续开发”，而是“以后什么才算正式链，什么再也不能混入正式链”。`

问题本质：

`过渡态不是不能存在，而是当前没有 owner、退出条件和替代字段。`

## 4. 正式主线的严格定义

| 术语 | 严格定义 | 不算达标的情况 |
| --- | --- | --- |
| 正式主线 | 用户可见的正式功能，其读写结果以 Workers + D1 的真实接口和真实表为单一依据 | 前端本地改状态并向用户显示成功 |
| 真鉴权 | 受保护接口只接受服务端可验证的认证凭据，并从认证上下文读取 `userId` | `user_id`、`x-user-id`、`x-user-email`、任意邮箱自动建用户 |
| 单主线写入 | 所有正式业务写操作都只在后端成功提交后才被前端承认为成功 | 后端失败，前端本地兜底写待办/兴趣/设置 |
| 草稿态 | 用户输入已保留，但尚未写入正式业务事实的临时状态 | 把草稿态伪装成“已创建 / 已更新” |
| UI Cache | 仅用于界面回显、草稿、滚动位置、会话临时缓存的本地状态 | 把身份、待办、兴趣、报告等正式事实长期放在本地作为真源 |
| 报告可信结果 | 每个数值都能追溯到明确表和明确统计规则；数据不足时显式返回不足 | 用固定数字或假趋势填满页面 |
| 支撑链执行化 | 支撑任务有明确定义的状态机、执行入口、失败重试和观测结果 | 只有写表和查表，没有真正执行者 |
| 过渡态 | 暂时存在的兼容字段或加工阶段标记，必须有 owner、退出条件和替代方案 | 长期出现在正式返回里，但没有退场计划 |

## 5. 正式收口硬标准

下面这些要求是“是否达标”的硬标准，采用 EARS 风格记录。

### 5.1 身份与权限

1. `WHEN` 任一受保护路由收到请求 `THEN` 系统只从服务端验证通过的认证上下文读取用户身份。
2. 系统 `SHALL NOT` 再接受 `user_id` query、`x-user-id`、`x-user-email`、`x-user-name` 作为正式身份输入。
3. `WHEN` 认证失败或凭据缺失 `THEN` 接口必须返回 `401`，且不得自动建用户。
4. `IF` 认证通过但资源不属于当前用户 `THEN` 接口必须返回 `403`。

### 5.2 Chat 与正式写链

1. `WHEN` `chat/execute` 或 `chat/reclassify` 执行失败 `THEN` 前端不得改动正式业务状态。
2. 系统 `SHALL` 把“正式成功”和“草稿保留/失败待重试”区分为两类不同状态。
3. `WHEN` Chat 成功写入正式对象 `THEN` 前端以服务端返回和正式刷新结果为准，而不是以本地推测为准。

### 5.3 本地状态边界

1. 系统 `SHALL` 将本地状态拆分为 `auth/session`、`formal data`、`draft/cache` 三类，并明确只有第三类允许长期留在本地。
2. 系统 `SHALL NOT` 让 `localStorage` 再承担正式身份来源。
3. `WHERE` 需要离线连续体验时，允许保留草稿与界面缓存，但必须和正式事实隔离。

### 5.4 报告可信度

1. `IF` 数据不足以生成真实趋势 `THEN` 报告返回 `insufficient_data`，而不是填固定数字。
2. 系统 `SHALL` 为所有核心统计字段定义数据来源表和统计规则。
3. `IF` 某个指标当前无法被真实计算 `THEN` 该指标返回 `null` 或被移除，并通过 `dataQuality` 标注原因。

### 5.5 支撑链与过渡态

1. `WHEN` 支撑链任务被创建 `THEN` 它必须进入明确定义的状态机，而不是只写一条观测日志。
2. 系统 `SHALL` 为 `partial` 一类字段定义明确语义、退出条件与替代字段。
3. 系统 `SHALL NOT` 把没有退出计划的过渡标记长期保留在正式响应里。

## 6. 问题分类：现在必须解决 vs 先留接口

## 6.1 开发阶段必须立刻做完的

1. 正式鉴权链与认证中间件。
2. 测试口径切换到正式认证上下文。
3. Chat 失败时禁止本地正式写回。
4. 本地状态边界重划，至少先切断“本地身份来源”和“本地正式业务事实”。
5. 报告的伪趋势与错误指标修正。

## 6.2 现在必须严格定义，但可以分批接入实现的

1. 支撑链状态机与执行接口。
2. `partial` 的明确语义、替代字段与退出条件。
3. 报告的 `dataQuality` 与可解释性字段。
4. 前端草稿态与失败重试态。

## 6.3 当前明确不作为本轮硬门槛的

1. OAuth / 第三方登录。
2. 统一 `contents` 事实总表。
3. 智能推荐模型、复杂画像流水线。
4. 完整消息中心或正式通知投递系统。

## 7. 接口契约定义

## 7.1 认证上下文契约

目标：把“拿到谁的身份”从业务路由里抽离出来，形成统一 `AuthContext`。

```ts
type AuthContext = {
  userId: number
  sessionId: string
  email: string
  username: string
  authMethod: 'password'
  issuedAt: string
  expiresAt: string
}
```

后端约束：

1. 现有 `resolveUserId()` 目标上应收敛为 `requireAuthContext()`。
2. 所有受保护读接口和所有写接口都只从 `AuthContext.userId` 取用户。
3. `AuthContext` 是正式业务读取用户的唯一来源。

## 7.2 认证传输契约

当前已落地的正式线定义为：

1. 浏览器端受保护 API 通过服务端可验证的 session cookie 进入 `AuthContext`。
2. 前端不得再传 `X-User-Email` / `X-User-Name` / `X-User-Id`，也不得再拼 `user_id` query 作为正式身份。
3. `AuthContext` 的身份来源当前以 `user_sessions + cookie` 为准，而不是前端本地用户对象。

扩展留位：

1. 当前已先把“服务端可验证 session -> AuthContext”固定为正式契约，不再把 `Bearer token` 误写成已落地事实。
2. 浏览器中的传输载体允许后续在 `cookie / bearer` 之间继续收敛，但本轮禁止再把 `localStorage user` 当作身份真源。
3. 无论后续是否切换为 `Authorization: Bearer`，都不得改变受保护路由对 `AuthContext` 的下游消费方式。

## 7.3 认证接口契约

### `POST /api/v1/auth/register`

请求：

```json
{
  "email": "user@example.com",
  "password": "string",
  "username": "string"
}
```

成功响应：

```json
{
  "user": {
    "id": 123,
    "email": "user@example.com",
    "username": "user"
  }
}
```

补充说明：

1. 当前会话由服务端在 `Set-Cookie` 中写入，不在 JSON 中回传 `accessToken`。

### `POST /api/v1/auth/login`

请求：

```json
{
  "email": "user@example.com",
  "password": "string"
}
```

成功响应与 `register` 一致，服务端同时写入正式 session cookie。

### `GET /api/v1/auth/me`

用途：

1. 校验当前 session 是否有效。
2. 给前端恢复正式登录态。

成功响应：

```json
{
  "user": {
    "id": 123,
    "email": "user@example.com",
    "username": "user"
  }
}
```

### `POST /api/v1/auth/logout`

成功响应：

```json
{
  "success": true
}
```

## 7.4 Chat 执行接口契约

### 成功写入

`POST /api/v1/chat/execute` 与 `POST /api/v1/chat/reclassify` 保持当前统一结果形态，但额外补齐以下约束：

1. `success=true` 才代表正式业务事实已提交。
2. `affectedEntity` 必须只引用后端实际写入的对象。
3. 前端收到 `success=true` 后，再触发正式会话刷新与相关页面刷新。

### 失败写入

失败响应统一至少包含：

```json
{
  "success": false,
  "errorCode": "EXECUTION_FAILED",
  "message": "执行失败，未写入数据库",
  "retryable": true,
  "draftSaved": false
}
```

前端约束：

1. 不得在失败分支调用本地 `processIntent` 去改正式业务状态。
2. 如需连续体验，只允许进入 `draft/error` UI 状态。
3. `draft/error` 状态不得在文案上伪装成“已创建 / 已更新”。

## 7.5 本地状态契约

前端状态分 3 类：

| 类别 | 可否进入本地持久化 | 说明 |
| --- | --- | --- |
| Session 展示态 | 可短期持久化，但不得作为服务端身份真源 | 例如 token 过期前的显示态、登录页恢复态 |
| Formal business state | 不可作为主真源 | 例如 todos、interests、notes、reports、push settings |
| Draft / UI cache | 可以 | 例如聊天输入草稿、会话滚动位置、最近打开页签 |

硬规则：

1. `todos/interests/notes/reports/settings` 不再以 `localStorage` 持久化结果作为正式展示真源。
2. 若页面首屏需要缓存回显，必须标识为 cache，并在正式接口返回后被覆盖。

## 7.6 报告接口契约

报告响应需要新增统一数据质量层：

```ts
type DataQuality = {
  status: 'ready' | 'partial' | 'insufficient_data'
  reasons?: string[]
}
```

规则：

1. 报告顶层必须返回 `dataQuality`。
2. 主题趋势不足时，`topicTrends` 可以为空数组，但不得再返回硬编码趋势。
3. `daysActive` 的正式定义改为：
   - `COUNT(DISTINCT date(created_at)) FROM history_entries WHERE user_id = ?`
   - 若当前阶段无法保证 `history_entries` 对目标报告周期的覆盖完整性，则返回 `null`，并在 `dataQuality.reasons` 中说明。

## 7.7 支撑链接口契约

### 正式状态机

以下状态作为当前统一最小状态机：

```ts
type SupportTaskStatus = 'queued' | 'running' | 'done' | 'failed' | 'cancelled'
```

适用对象：

1. `briefing_dispatch_logs`
2. `ingestion_runs`
3. `ai_processing_runs`
4. `replay_tasks`

规则：

1. 观测写表不再等于执行完成。
2. `system/*` 中对外展示时，应以状态机状态说明链路位置。
3. 内部写入口与真正执行器可以分步接入，但状态含义必须现在就固定。

### 权限边界

1. `GET /api/v1/system/chain-health`：允许登录用户查看与自己相关的用户级支撑链健康摘要。
2. `POST /api/v1/system/ingestion-runs` 与 `POST /api/v1/system/ai-processing-runs`：定义为内部执行器接口，不直接暴露给普通用户。
3. 当前最小实现口径：
   - 通过 `Authorization: Internal <token>` 进入
   - token 来源于服务端 `INTERNAL_API_TOKEN`
   - 未配置 token 时返回“内部执行器鉴权未配置”，而不是继续裸露开放
4. 当前生产配置口径：
   - 真实线上对象是默认 Worker `ai-briefing-assistant`
   - 首次 secret 已完成写入
   - 后续轮换命令应使用 `wrangler secret put INTERNAL_API_TOKEN --env ""`
5. `GET /api/v1/system/replay-tasks`：必须经过正式认证，并只返回当前用户请求的回放任务。

## 7.8 过渡态替代契约

从现在开始：

1. `transitional` 不再被视为正式语义状态。
2. 若页面需要表达“结果已接入真实链，但加工能力还不完整”，统一改用 `dataQuality` 或解释字段表达。
3. 任何仍保留的临时字段，都必须在对应文档或代码注释里写明：
   - owner
   - exit condition
   - replacement field

## 8. 测试口径定义

## 8.1 三层门槛

| 层级 | 目标 | 当前口径 |
| --- | --- | --- |
| 仓库级合并门槛 | 防止通用回归 | `npm.cmd run check` |
| 阶段级收口门槛 | 证明正式主线达标 | `npm.cmd run check:stage` + 新增收口专项验证 |
| 发布级验收门槛 | 证明用户链路与权限链真实可用 | 自动化通过 + 人工截图级证据 |

补充说明：

1. `npm.cmd run check` 继续保留为项目级唯一准入门槛。
2. 但“正式主线已收口”的结论不能只看 `check`，还必须看专项口径。

## 8.2 自动化专项口径

以下专项用例必须成立，才能宣称本轮收口达标：

1. 鉴权契约测试：
   - 所有受保护接口在无 session 时返回 `401`
   - Workers 主线不再接受 `?user_id=1`
   - Workers 主线不再接受 `x-user-id`
2. Chat 正式写链测试：
   - `chat/execute` 失败时，前端不会显示正式成功
   - 后端失败时，D1 中不会出现误写对象
3. 本地状态边界测试：
   - 刷新页面后，正式数据来自接口恢复，而不是 `localStorage` 真源
4. 报告可信度测试：
   - 数据不足返回 `insufficient_data`
   - `daysActive` 按 distinct date 统计或返回 `null`
5. 支撑链状态机测试：
   - 至少一条任务具备 `queued -> running -> done/failed` 的可验证流转

## 8.3 手工验收口径

至少补齐以下截图级证据：

1. 登录成功后访问受保护页面。
2. 未登录时访问受保护页面被拦截。
3. Chat 创建待办成功，刷新后待办仍存在。
4. 模拟后端失败时，前端只显示失败/草稿，不显示“已创建待办”。
5. 数据不足用户打开报告时看到明确的空态或不足说明。

## 9. 扩展留位设计

## 9.1 认证扩展留位

当前必须做完的是“正式认证上下文”。

当前不用做完的是：

1. OAuth
2. 魔法链接
3. 多设备 session 管理
4. `HttpOnly Cookie` 迁移

留位原则：

1. 下游统一只消费 `AuthContext`。
2. 上游认证方式可以扩展，但不改受保护路由签名。

## 9.2 前端连续体验留位

当前必须切掉的是“本地正式写回”。

当前允许保留的是：

1. 输入草稿
2. 失败重试按钮
3. 会话缓存视图

留位原则：

1. 所有连续体验都挂在 `draft/cache`，不挂在正式对象 store。
2. 允许后续把 Chat 失败恢复做得更平滑，但不允许再回到假成功。

## 9.3 支撑链执行器留位

当前必须定义的是状态机、内部接口和权限边界。

当前可以后补的是：

1. Cron
2. Queue consumer
3. Retry policy
4. DLQ / 回放控制台

留位原则：

1. 先统一状态和事实表。
2. 再逐步接入执行器，不返工用户侧 contract。

## 9.4 报告解释力留位

当前必须做的是去伪趋势和修指标。

当前可以后补的是：

1. 趋势解释树
2. 原始内容证据链
3. 可追溯引用卡片

留位原则：

1. 先保证“值是真的”。
2. 再补“为什么是真的”。

## 10. 推荐执行顺序

1. 第一优先：继续补正式鉴权剩余收口，扩大资源级 `403` 覆盖并完成生产人工验收、internal token 轮换流程与传输契约说明。
2. 第二优先：切断 Chat 本地正式写回，把失败分支收敛到 `draft/error`。
3. 第三优先：收紧前端本地状态边界，清掉“localStorage 身份真源”。
4. 第四优先：修报告口径，补 `dataQuality` 与真实统计规则。
5. 第五优先：为 `system/*` 和调度事实层接入正式状态机。
6. 第六优先：清退 `transitional`，统一替换为数据质量或解释字段。

## 11. 退出本轮主线的判定条件

只有同时满足以下条件，才允许宣称“正式主线已收口”：

1. 生产受保护路由不再接受 `user_id` / `x-user-id` / `x-user-email`。
2. 前端登录态来自正式 session/token，而不是本地 `isLoggedIn` 开关。
3. Chat 后端失败时不再本地改正式业务状态。
4. 本地状态已明确分为 `formal` 与 `draft/cache` 两类边界。
5. 报告不再返回硬编码伪趋势，关键指标都有真实定义。
6. 自动化测试已切到正式认证上下文，不再给旧身份口径背书。
7. 至少一条支撑链具备真实状态流转证明，而不只是静态写表。

## 12. 当前结论

这份文档形成后的项目级口径应当是：

1. 当前最该优先收口的不是“支撑链执行器”，而是“身份链 + Chat 正式写链 + 本地状态边界 + 报告可信度”。
2. 支撑链与过渡态不要求这一轮全部做完，但必须先把接口和状态机定义锁定。
3. 从现在开始，任何“主线正式收口”的判断都必须引用本文标准，而不能只引用“测试通过”或“页面看起来能用”。
4. `INTERNAL_API_TOKEN` 当前已具备“本地示例文件 + setup 自动补齐 + 已完成首次生产写入 + 后续轮换命令”的正式配置路径，不再只是代码里有校验、仓库里没有入口。
