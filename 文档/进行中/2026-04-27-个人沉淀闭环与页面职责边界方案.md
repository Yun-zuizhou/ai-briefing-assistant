# 个人沉淀闭环与页面职责边界方案

- 文档状态：进行中
- 建立日期：2026-04-27
- 文档定位：承接“正式页面功能职责与问题梳理”之后的方向级判断，先明确个人沉淀闭环，再决定 Journal 等页面怎么改
- 适用范围：`apps/web` 正式前端、`apps/edge-worker` Workers API、`packages/contracts` 页面契约、`infra/cloudflare/d1/migrations` D1 事实表
- 测试口径：本文只做方案判断，不记录测试数字；测试数字只允许更新到 `文档/进行中/当前测试与验收总表.md`

---

## 1. 本轮结论

1. 下一步不应直接把 `JournalPage` 当作“记录页 UI 排序问题”去修。
2. 更好的做法是先把“个人沉淀”定义成一个跨页面产品闭环，再用这个闭环反推 Journal、HistoryLogs、Growth、Reports、My 的职责边界。
3. 当前项目已经具备正式事实表和正式接口：`notes`、`chat_messages`、`todos`、`favorites`、`history_entries`、`opportunity_follows`、`reports`、`growth-overview` 等都存在，不需要先扩 mock 或新 UI 框架。
4. 阶段 1 应只用现有 API 做前端组织收口，把 Journal 从“薄记录列表 + 成长装饰”调整为“个人沉淀入口”。
5. 阶段 2 再判断是否需要 `journal/overview`、`packages/contracts` 新契约或 D1 schema 字段，不在阶段 1 先扩大范围。

一句话判断：

> 当前正确方向不是“继续修一个页面”，而是把产品骨架从“页面集合”升级为“信息输入 -> 表达 -> 行动 -> 沉淀 -> 回看”的闭环。

---

## 2. 为什么不能直接改 Journal

如果直接改 Journal 页面，会遇到三个问题：

1. `JournalPage` 当前读取的是 `notes + growth-overview`，但用户理解的“记录”不只包括 notes。
2. Chat 已能产生记录、待办、兴趣、修正等对象，但 Journal 没有解释这些对象如何沉淀。
3. Growth、HistoryLogs、HistoryBrief、My 都已经在展示沉淀结果，如果 Journal 只做列表重排，会继续和这些页面抢职责。

因此，Journal 的问题不是“列表放前面还是后面”，而是它到底承接哪类沉淀对象。

## 3. 当前真实事实链

| 沉淀对象 | 当前表 / 数据 | 当前 API | 当前前端入口 | 判断 |
|---|---|---|---|---|
| 我说过的 | `notes`、`chat_messages` | `/api/v1/notes`、`/api/v1/chat/sessions/:id/messages` | `JournalPage`、`ChatPage` | notes 可先作为 Journal 主体；chat_messages 暂不直接做全量历史 |

| 我留下的 | `favorites`、`history_entries` | `/api/v1/favorites`、`/api/v1/history` | `CollectionsPage`、`HistoryLogsPage` | Journal 可展示近期摘要，证据层在 HistoryLogs |

| 页面 | 应承担职责 | 不应承担职责 | 阶段 1 判断 |

| HistoryLogs `/history-logs` | 事实证据层，按时间归档真实历史、收藏、记录 | 首页沉淀入口、成长解释主舞台 | 保持为二级回看页，Journal 只跳转过去 |

| Today `/today` | 今日信息输入主舞台 | 历史沉淀页、周期总结页 | 后续再做信息密度和今日主判断收口 |

## 5. Journal 的新方向

Journal 不应再被定义成“个人记录列表页”，而应定义成：

> 个人沉淀入口：把用户最近说过、推进过、留下过的对象组织成可继续进入的个人材料。

阶段 1 推荐结构：

1. 顶部：最近沉淀摘要
   - 展示今天 / 近期留下了多少条记录、行动、收藏或历史痕迹。
   - 数据优先来自现有 `notes`、`history`、`favorites`、`actions/overview`。
2. 第一段：我刚说过的
   - 主体展示 `notes`。
   - 如果 `source_type` 可读，显示“来自对话 / 手动记录 / 内容记录”等自然表达。
   - 保留跳转 Chat 的入口。
3. 第二段：我推进过的
   - 轻量展示今日待办、后续跟进或最近已完成动作。
   - 不在 Journal 内完成复杂操作，主操作跳到 Actions。
4. 第三段：我留下的
   - 展示收藏和历史痕迹摘要。
   - 详情交给 Collections / HistoryLogs。
5. 底部：长期回看入口
   - Growth 和 HistoryBrief 作为入口存在，不再压在首屏主内容里。

阶段 1 不建议做：

1. 不新增复杂日记编辑器。
2. 不直接读取全量 `chat_messages` 做聊天历史页。
3. 不新增 schema。
4. 不把 Growth 的长期解释搬到 Journal 首屏。
5. 不用 mock 拼“沉淀对象”。

## 6. 阶段 1 最小实现边界

目标：

- 不改接口。
- 不改 D1 schema。
- 不改 `packages/contracts`。
- 用现有 API 让 Journal 的产品职责从“薄记录列表”变成“沉淀入口”。

允许读取的现有 API：

1. `GET /api/v1/notes`
2. `GET /api/v1/history`
3. `GET /api/v1/favorites`
4. `GET /api/v1/actions/overview`
5. `GET /api/v1/preferences/growth-overview`

实现原则：

1. `notes` 是主内容，不再后置。
2. `history / favorites / actions` 只做摘要和入口，不做重复管理。
3. 现有 `AI生成` 前端假状态应移除或降级成“整理入口占位”，不能伪装成真实生成。
4. 所有来源文案必须面向用户，不暴露 `source_type`、`content_ref` 等内部字段。
5. 移动端首屏必须让用户看到“我留下的东西”，而不是先看到成长装饰。

## 7. 阶段 2 契约收口判断

只有当阶段 1 证明 Journal 需要稳定聚合读侧时，才进入阶段 2。

候选接口：

```txt
GET /api/v1/journal/overview
```

候选契约位置：

```txt
packages/contracts/src/page-data.ts
```

候选数据结构方向：

1. `summary`
   - `notesCount`
   - `actionsCount`
   - `favoritesCount`
   - `historyCount`
2. `recentExpressions`
   - 近期 notes
   - 来源标签
   - 可选关联对象
3. `recentProgress`
   - 待办、跟进、完成动作摘要
4. `recentKeeps`
   - 收藏、历史痕迹摘要
5. `reviewEntrypoints`
   - Growth、HistoryLogs、HistoryBrief 入口元数据

默认不新增 D1 migration。

只有出现以下需求时，才考虑 `infra/cloudflare/d1/migrations`：

1. notes 需要正式区分 `note_type`，例如 quick_note / chat_extract / review_excerpt。
2. notes 需要关联 `chat_message_id` 或统一对象引用。
3. Journal 需要持久化用户手动置顶、归档或编辑状态。
4. reports / growth 需要写入 Journal 专用摘要快照。

## 8. 推荐实施顺序调整

原先“Chat -> Actions -> Journal -> Today -> My”的顺序仍基本成立，但 Journal 的执行方式需要调整：

1. Chat：已完成首轮低风险收口。
2. Actions：已完成首轮低风险收口。
3. Journal：下一步执行，但必须按本文件的沉淀闭环做，不按普通 UI 排序做。
4. Today：等 Journal 明确沉淀入口后，再优化今日输入如何流向 Chat / Actions / Journal。
5. My：最后处理退出登录位置和入口分组。

调整理由：

- Journal 是“表达和行动之后”的承接页。
- 如果 Journal 边界不清，Today 的速记、Chat 的沉淀、Actions 的推进都会继续显得割裂。
- My 页问题相对局部，不应抢先决定沉淀体系。

## 10. 当前风险

1. 最大风险不是实现复杂度，而是把 Journal 继续误判成“记录列表页”。
2. 若阶段 1 同时读取 `notes / history / favorites / actions`，前端加载态和失败态需要清楚分区，避免一个接口失败导致整个页面不可用。
3. 如果不新增 `journal/overview`，阶段 1 会有一定前端聚合逻辑；这可以接受，但不能长期扩成复杂业务编排。
4. 如果过早新增 schema，会把产品语义还没稳定的对象固化到数据库，风险高于收益。
5. 如果把 Chat 全量消息直接塞进 Journal，会把最低成本表达入口和沉淀入口重新混在一起。

---
