## 1. 这份文档现在解决什么问题

这次改写只解决一件事：

`把 UI 美化目标重新锚定到 WDB 移动应用，而不是桌面工作台。`

当前问题不是“设计方向太弱”，而是方向被我外推偏了：

1. 把 Web 技术形态误读成了桌面设计目标
2. 把“桌面浏览器可打开”误写成了“桌面端应该被正式设计”
3. 把“个人情报应用”说成了“个人情报工作台”

这次改写后的唯一前提是：

> 正式设计目标是移动端 WDB 应用；桌面浏览器只承担开发调试和演示预览，不反向决定正式设计。

---

## 2. 当前美化的正确目标

本轮 UI 美化不追求：

1. 桌面化
2. 工作台化
3. 响应式双端并行设计
4. 大屏独立布局

本轮正确目标是：

> 把当前“温暖编辑感”收成一款正式、克制、好用的移动应用界面。

用户最终应感受到：

1. 这是一个移动端个人情报助手
2. 有编辑感和纸面感，但首先是手机里要每天打开的工具
3. 信息层级清楚，单手使用顺手
4. 不像原型展示稿，也不像套了手机壳的概念页

一句话标准：

`让页面像一款认真排版的移动应用，而不是一张摆在桌面上的展示卡。`

---

## 3. 本轮不改什么

为了防止“UI 美化”跑偏，本轮明确不改：

1. 不改路由结构
2. 不改核心信息架构
3. 不改后端 API、共享契约和业务逻辑
4. 不新增设计库
5. 不做桌面专用布局
6. 不做整站大动画

本轮只做：

1. 移动端外壳收口
2. 字体与字号收口
3. 组件视觉系统收口
4. 核心页面首屏与信息块美化

---

## 4. 视觉方向定稿

### 4.1 风格名称

当前正式风格仍叫：

> 温暖编辑室

但它现在的解释必须改成：

- 不是桌面编辑台
- 不是复古报纸秀场
- 而是带编辑感的移动端信息助手

### 4.2 用户第一眼该感受到什么

用户第一眼应该感受到：

1. 这是手机里的一款正式应用
2. 不是一张设计稿
3. 不是一个控制台
4. 不是一个后台

### 4.3 风格边界

保留：

1. 暖纸色背景
2. 深墨色标题
3. 少量砖红和金色点缀
4. 编辑标题感

禁止：

1. 深色舞台包裹一个内容画布
2. 手机壳样机感
3. 满屏装饰角和假状态栏
4. 每个区块都厚边框厚阴影
5. 所有文字都像标题

## 5. 设计系统提案

### 5.1 正式目标视口

正式设计目标只认移动端：

1. 宽度区间：
   - `360px - 430px`
2. 方向：
   - 竖屏优先
3. 高度：
   - 使用真实视口高度
   - 去掉 `932px` 固定高度

### 5.2 桌面浏览器的正确角色

桌面浏览器只做两件事：

1. 开发调试预览
2. 演示与截图辅助

因此桌面预览只需要：

1. 居中显示移动端页面
2. 保留适度留白
3. 不再加深色舞台和样机框

但绝不需要：

1. 单独的桌面信息架构
2. 大屏双栏布局
3. 桌面专用交互逻辑

### 5.3 字体系统

建议固定为：

1. 标题：
   - `Noto Serif SC`
2. 正文：
   - `Noto Sans SC`

使用规则：

1. `Serif`
   - 页头标题
   - 区块标题
   - 关键句
2. `Sans`
   - 正文
   - 按钮
   - 表单
   - 标签
   - 状态

当前 `body` 全站 serif 应退出。

### 5.4 字号与留白

当前主要问题不是“字不美”，而是“字太紧、太小、太像展示稿”。

本轮目标：

1. 正文基线从 `12px` 级别提升到真正可读
2. 关键卡片上下留白增加
3. 消息卡、列表卡、表单控件不再显得拥挤
4. 手机端一屏只保留一个首焦点

### 5.5 颜色

当前主色方向保留，但要正式化：

1. `paper / paper-warm / paper-dark`
2. `ink / ink-light / ink-muted`
3. `accent`
4. `gold`

需要补齐的正式 token：

1. `--accent-dark`
2. `--surface-raised`
3. `--surface-soft`
4. `--shadow-soft`

禁止继续使用未定义变量：

1. `--card`
2. `--fg`
3. `--bg-paper`

### 5.6 卡片

卡片应该像“移动应用里的纸卡”，而不是“报纸主题海报块”。

原则：

1. 1px 边框为主
2. 轻阴影为主
3. 小圆角统一
4. 主次层级靠留白和色面，不靠重装饰

### 5.7 动效

本轮只保留 3 类动效：

1. 页面淡入
2. 卡片轻上浮
3. 按钮 hover / press

动效目标是：

`让移动端更顺滑，不让页面更像展示页。`

## 6. 核心页面美化策略

### 6.1 `TodayPage`

目标：

1. 像移动端首页
2. 不是聚合大看板

美化重点：

1. `今日判断` 做成唯一 Hero
2. `值得知道 / 值得行动` 卡片做成移动端信息卡
3. `因你关注而推荐` 下沉
4. `今日速记` 收成轻输入卡

### 6.2 `ChatPage`

目标：

1. 像移动应用里的表达入口
2. 不是调试控制台

美化重点：

1. 首屏主卡保持一句提示 + 输入入口
2. 最近结果卡更像“移动便签”
3. 确认卡更像自然提问，不像系统面板
4. 消息层级更清楚
5. 高级状态永远折叠在下方
6. 结构定义以 `低成本表达为主，结果反馈为辅，会话承接与状态细节后置` 为准，详见 `文档/进行中/2026-04-22-ChatPage成品级主辅结构方案.md`

### 6.3 `ActionsPage`

目标：

1. 像移动待办页
2. 不是行动展示台

美化重点：

1. 今日最该做的项提到最前
2. 打卡与统计降级
3. 列表卡统一为轻任务卡

### 6.4 `ArticlePage`

目标：

1. 强化阅读感
2. 让正文是主舞台

美化重点：

1. 标题区轻量化
2. 正文区舒适化
3. AI 摘要与相关推荐退到辅助层

### 6.5 `Growth / My / Reports`

目标：

1. 像移动应用里的回看页
2. 不是展示页

美化重点：

1. 报告页继续保留可信度提示
2. 去掉伪图形
3. `My` 继续保持入口页角色

如果现在开始实施，第一批最该直接动的是：

1. `apps/web/src/index.css`
   - 去固定 `932px`
   - 去深色舞台
   - 去假状态栏
   - 去装饰角
2. `apps/web/src/styles/design-tokens.ts`
   - 重做字号、留白、阴影、颜色 token
3. `apps/web/src/styles/design-tokens.css`

4. `apps/web/src/components/layout/PageLayout.tsx`
   - 收成移动应用外壳
5. `apps/web/src/components/ui/Button.tsx`
   - 去未定义变量依赖
   - 正式化按钮变体

## 8. 实施顺序建议

### 阶段 1：移动外壳收口

目标：

1. 页面摆脱“样机展示稿”
2. 恢复成真实移动应用容器

### 阶段 2：排版收口

目标：

1. 标题 serif、正文 sans 真正落地
2. 字号和留白回到移动端可读范围

### 阶段 3：组件收口

目标：

1. `Button / Card / Status / Input` 正式化
2. 停止页面继续各写各的视觉系统

### 阶段 4：核心页美化

顺序：

1. `Chat`
2. `Today`
3. `Actions`
4. `Article`
5. `Growth / My / Reports`

---

## 9. 验收标准

这轮 UI 美化完成后，至少要满足：

1. 正式页面看起来像移动应用，不像样机展示稿
2. `body` 不再全站 serif
3. `Masthead` 不再靠 ornaments 和装饰角撑气氛
4. 按钮和卡片不再引用未定义 token
5. `Chat / Today / Actions` 首屏都只有一个主焦点
6. 桌面浏览器打开时只是“移动预览”，而不是“桌面端设计”

---

## 10. 一句话结论

当前 UI 美化的正确方向不是“把 Web 页面做得更像桌面工作台”，而是：

`把已经存在的温暖编辑风，收成一款正式、稳定、移动优先的 WDB 应用界面。`
