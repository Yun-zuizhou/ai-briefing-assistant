# 可与UI改进批次同步修复的问题

## 分析说明
UI改进批次（批次1-7）正在由另一个任务对话执行中。本文件梳理了代码审查发现的问题中，**可以与UI改进批次同步进行、互不阻塞**的问题。这些问题要么与UI改进批次修改的文件有重叠（可一并修改），要么是独立的后端/工程化问题（不受UI改进影响）。

---

## 一、与UI改进批次文件重叠，可一并修改的问题

### 1. 前端组件职责过重 → 与批次3/4/5同步

**对应问题清单：** `02-前端组件与状态管理问题.md` 第1条

**重叠分析：**
- UI改进批次3（主页面）正在修改 `TodayPage.tsx`、`ActionsPage.tsx`、`GrowthPage.tsx`、`MyPage.tsx`
- UI改进批次4（次页面）正在修改 `ChatPage.tsx`、`ArticlePage.tsx`、`HotTopicsPage.tsx`
- UI改进批次5（报告页）正在修改 `WeeklyReportPage.tsx`、`MonthlyReportPage.tsx`、`AnnualReportPage.tsx`

**同步建议：**
在批次3/4/5进行内联样式清理和组件替换时，**同步进行页面组件拆分**：
- `TodayPage.tsx` 拆分为 `TodayHeader` + `ArticleList` + `ActionSummary`
- `ChatPage.tsx` 拆分为 `ChatMessageList` + `ChatInput` + `ChatSidebar`
- 报告页拆分为 `ReportHeader` + `ReportStats` + `ReportContent`

**价值：** 避免同一文件被多次重构，减少回归测试成本。

## 三、需要等待UI改进完成后才能进行的问题

以下问题**不应同步进行**，因为会与UI改进批次产生冲突或依赖：

| 问题 | 原因 |
|------|------|
| 建立共享类型包 (`packages/shared-types`) | 需要前后端同时改造，会干扰UI改进的前端文件 |
| 引入全局状态管理 (Zustand) | 需要修改前端状态管理逻辑，与批次3/4的页面重构冲突 |
| 前端路由守卫 | 依赖批次7的ErrorBoundary和AppContext改造完成 |
| 数据库访问层抽象 (Repository层) | 纯后端大重构，与UI改进无关，但应独立排期 |
| 缓存策略实现 | 架构层改进，应在UI改进稳定后评估 |

## 五、风险提示

1. **文件冲突风险**：同步修改页面组件时，需与UI改进对话协调，避免同时修改同一文件的同一区域。
2. **测试回归风险**：后端API错误格式变更后，前端错误处理逻辑需要同步适配，建议分阶段发布。
3. **范围蔓延风险**：同步修复的问题应严格控制范围，避免将"同步修复"演变为"大规模重构"。
