# 测试覆盖与边界处理问题

## 问题概述
在审查测试文件及核心服务逻辑时，发现测试覆盖存在盲区，且部分业务逻辑在边界条件下表现不够健壮。此类问题可能导致线上故障难以被及时发现。

---

## 1. 测试覆盖不全面，核心业务流缺乏集成测试

**问题描述：**
- 现有测试主要集中在 `auth-guard.route.test.ts` 等路由守卫层面，验证了 401 响应。
- 但以下核心业务流缺乏自动化测试：
  - 用户完整的登录 -> 访问 Dashboard -> 执行 Action -> 生成 Report 的端到端流程。
  - Chat 模块的消息发送、意图解析、会话存储流程。
  - 数据同步与离线状态处理（如有）。

**影响范围：**
- 重构或新增功能时，难以确保未破坏现有业务流。
- 依赖人工测试，效率低且容易遗漏。

**建议：**
- 补充核心业务流的集成测试（Integration Test），覆盖关键用户旅程。
- 对 `reports/builder.ts`、`behavior/builder.ts` 等核心服务函数补充单元测试，验证边界输入。

---

## 2. 边界条件与异常输入处理不足

**问题描述：**
- `apps/edge-worker/src/services/behavior/builder.ts` 中的 `buildGrowthKeywords` 函数：
  - 当 `interests` 为空数组时，会填充默认关键词。但未处理 `interests` 为 `null` 或 `undefined` 的情况。
- `buildPersonaSummary` 函数：
  - 当 `counts` 对象中的字段缺失时，可能导致模板字符串中出现 `undefined`。
- `apps/edge-worker/src/services/reports/builder.ts` 中的报告生成逻辑：
  - 当 `notes`, `favorites`, `todos` 等数组均为空时，生成的报告内容可能过于空洞，但未触发警告或降级逻辑。

**影响范围：**
- 极端情况下可能返回包含 `undefined` 的文案，影响用户体验。
- 数据异常时，系统静默处理，不利于问题排查。

**建议：**
- 对所有公共服务函数的输入参数增加防御性编程（如使用 Zod 校验或默认值）。
- 在关键数据处理节点增加日志记录，便于追踪数据质量问题。

---

## 3. 认证与授权测试场景单一

**问题描述：**
- `auth-guard.route.test.ts` 仅测试了“无 session 时返回 401”的场景。
- 缺少以下场景的测试：
  - session 过期或无效时的处理。
  - 已登录用户访问需要特定权限（如管理员）的路由时的 403 处理。
  - 并发请求下 session 的一致性。

**影响范围：**
- 安全逻辑变更时缺乏回归测试保障。
- 潜在的权限提升漏洞难以被及时发现。

**建议：**
- 扩展认证测试套件，覆盖 session 过期、权限不足、并发访问等场景。
- 如有角色权限体系，补充基于角色的访问控制（RBAC）测试。

---

## 4. 前端测试缺失

**问题描述：**
- `apps/web` 目录下未发现明显的单元测试或组件测试文件（如 `.test.tsx` 或 `.spec.ts`）。
- 前端核心组件（如 `TodayPage`, `ChatPage`）的交互逻辑缺乏自动化验证。

**影响范围：**
- UI 逻辑变更时，依赖人工点击验证，效率低。
- 复杂交互（如 Chat 消息流、表单校验）容易引入回归问题。

**建议：**
- 引入 Vitest + React Testing Library 作为前端测试框架。
- 优先为核心业务组件和自定义 Hook 补充单元测试。
- 对关键用户旅程补充 E2E 测试（如使用 Playwright）。

---

## 5. 数据库迁移与 Schema 变更缺乏测试

**问题描述：**
- 项目中使用 Cloudflare D1 作为数据库，但未看到数据库迁移脚本或 Schema 版本管理工具（如 Drizzle ORM 的迁移）。
- 缺乏对数据库 Schema 变更的自动化测试，无法验证迁移脚本是否正确。

**影响范围：**
- Schema 变更时，可能因迁移脚本错误导致数据丢失或服务不可用。
- 新环境部署时，难以确保数据库状态正确。

**建议：**
- 引入 Drizzle ORM 或类似工具的迁移管理机制。
- 在 CI 流程中增加数据库迁移的自动化验证步骤。
