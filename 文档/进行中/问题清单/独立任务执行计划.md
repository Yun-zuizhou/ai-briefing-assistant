## 任务1：统一API错误响应格式（高优先级）

### 现状
- `index.ts` 中已有 `app.onError` 全局错误处理器，但格式简单：`{ error: 'Internal Server Error' }`
- `app.notFound` 返回：`{ error: 'Not Found' }`
- 各路由中可能使用不同的错误格式
- `auth.ts` 使用 `HTTPException` 抛出错误，但响应格式不一致

### 目标
统一所有API错误响应格式为：
```json
{
  "error": "人类可读的错误描述",
  "code": "ERROR_CODE",
  "status": 401
}
```

### 执行步骤
1. 在 `apps/edge-worker/src/utils/` 下创建 `error-handler.ts`
2. 定义标准错误响应类型和错误码枚举
3. 修改 `index.ts` 的 `app.onError` 使用新的错误处理器
4. 修改 `app.notFound` 使用统一格式
5. 确保 `HTTPException` 被正确转换为统一格式
6. 更新测试文件中的错误断言

### 验证方式
- `npm run test:workers` 通过
- 所有错误响应包含 `error`, `code`, `status` 三个字段

## 任务2：加固边界条件处理（高优先级）

### 现状
- `buildGrowthKeywords(interests: string[])`：未处理 `null`/`undefined`
- `buildPersonaSummary(interests: string[], counts: ProfileCounts)`：未处理 `counts` 字段缺失
- `buildPeriodicReportPayload`：未处理空数据降级

### 目标
所有公共服务函数对异常输入具有防御性

### 执行步骤
1. 修改 `services/behavior/builder.ts`：
   - `buildGrowthKeywords`：处理 `null`/`undefined`，返回默认关键词
   - `buildPersonaSummary`：处理 `counts` 字段缺失，使用默认值
2. 修改 `services/reports/builder.ts`：
   - `buildPeriodicReportPayload`：处理空数据，返回降级报告
   - `calcStreak`：处理空数组
3. 为上述函数补充单元测试

### 验证方式
- 新增单元测试覆盖边界条件
- `npm run test:workers` 通过

## 执行顺序

```
任务1（错误格式统一） → 任务2（边界条件加固） → 任务3（认证测试扩展） → 任务4（JSDoc补充）
```

**理由**：
- 任务1是基础设施，影响所有错误响应，优先完成
- 任务2依赖任务1的错误格式（边界条件测试需要断言统一格式）
- 任务3和任务4相互独立，可在任务2之后并行

**与线A（UI改进）的并行关系**：
- 线A批次1（基础层）可与线B任务1（错误格式统一）同时启动
- 线A批次2（组件层）可与线B任务2（边界条件加固）同时启动
- 线A批次3-5（页面层）可与线B任务3-4（认证测试/JSDoc）同时启动

## 风险与回退

| 风险 | 缓解措施 |
|------|----------|
| 错误格式变更影响前端错误处理 | 保持`error`字段人类可读，前端可先适配，后续再切换`code`字段 |
| 边界条件变更影响现有行为 | 所有变更均有单元测试覆盖，确保向后兼容 |
| 测试扩展导致测试时间增加 | 仅增加少量测试用例，不影响整体执行时间 |
| 与线A任务冲突 | 线B为纯后端修改，与线A前端文件无重叠，不会冲突 |
