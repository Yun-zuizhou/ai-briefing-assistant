# 架构设计与扩展性问题

## 问题概述
在审查项目整体架构时，发现部分设计决策在当前阶段虽能运行，但在未来扩展（如新功能模块、多平台支持、性能优化）时可能成为瓶颈。此类问题需要提前规划，避免后期大规模重构。

---

## 1. 前后端职责边界模糊

**问题描述：**
- 意图解析（Intent Parsing）逻辑同时存在于前端 (`apps/web/src/utils/intentParser.ts`) 和后端 (`apps/edge-worker/src/services/chat/intent.ts`)，职责未清晰划分。
- 部分数据格式化逻辑（如报告文案生成）由后端承担，导致 API 返回体包含大量展示层信息。

**影响范围：**
- 前后端耦合度高，独立迭代困难。
- 若未来需要支持小程序、移动端等新客户端，后端文案逻辑难以复用。

**建议：**
- 明确前后端职责边界：
  - 后端：负责数据存储、业务逻辑、权限控制、意图解析（核心算法）。
  - 前端：负责 UI 渲染、用户交互、展示层文案、轻量级状态管理。
- 将意图解析等核心算法统一收敛到后端，前端通过 API 获取解析结果。

---

## 2. 缺乏模块化与插件化设计

**问题描述：**
- 后端服务层代码按功能平铺（如 `services/chat/`, `services/reports/`, `services/behavior/`），但各模块之间缺乏清晰的接口契约。
- 新功能（如未来可能新增的“热点追踪”模块）难以以“插件”形式接入，可能需要修改核心代码。

**影响范围：**
- 功能迭代时，容易牵一发而动全身。
- 团队并行开发效率低，容易产生代码冲突。

**建议：**
- 定义模块接口规范，每个业务模块暴露统一的入口（如 `Module.init()`, `Module.routes()`, `Module.handlers()`）。
- 考虑引入依赖注入（DI）或简单的模块注册机制，支持新功能模块热插拔。

---

## 3. 数据库访问层未抽象

**问题描述：**
- 后端代码中，SQL 查询直接散落在路由处理函数或服务函数中，未形成独立的 Repository / DAO 层。
- `apps/edge-worker/src/utils/db.ts` 仅为工具函数，未对具体业务实体（如 `User`, `Todo`, `Note`）进行封装。

**影响范围：**
- 数据库 Schema 变更时，需要全局搜索并修改所有相关 SQL。
- 难以实现单元测试中的数据库 Mock，测试依赖真实 D1 环境。

**建议：**
- 为每个核心实体建立 Repository 层（如 `repositories/user.repository.ts`），封装所有数据库操作。
- Repository 层定义接口，支持未来切换数据库（如从 D1 迁移到 PostgreSQL）时最小化改动。

---

## 4. 缓存策略缺失

**问题描述：**
- 项目中未看到任何缓存层实现（如 Redis、Cloudflare Cache API、或内存缓存）。
- 对于读多写少的数据（如用户配置、热点内容、报告模板），每次请求都直接查询数据库。

**影响范围：**
- 高并发场景下，数据库压力较大。
- 部分计算密集型操作（如报告生成）重复执行，浪费资源。

**建议：**
- 评估引入 Cloudflare Cache API 或 Workers KV 作为缓存层。
- 对以下数据优先实施缓存：
  - 用户配置和权限信息（短时效缓存）。
  - 生成的报告内容（长时效缓存，或直到数据变更时失效）。
  - 热点内容和元数据（中等时效缓存）。

---

## 5. 未考虑多租户与数据隔离

**问题描述：**
- 当前数据模型中，未看到显式的多租户支持（如 `tenant_id` 字段）。
- 若未来需要支持团队版、企业版，或 SaaS 化部署，数据隔离将成为重大改造点。

**影响范围：**
- 未来扩展多租户时，需要大规模修改数据库 Schema 和查询逻辑。
- 数据安全隔离缺乏底层保障。

**建议：**
- 在核心数据表中预留 `tenant_id` 或 `workspace_id` 字段（即使当前单租户场景下使用默认值）。
- 在数据库访问层统一注入租户隔离条件，为未来多租户切换做准备。

---

## 6. 外部服务依赖缺乏降级策略

**问题描述：**
- 项目中可能依赖外部服务（如 AI 模型 API、第三方登录、推送服务），但未看到明确的降级（Fallback）或熔断（Circuit Breaker）策略。
- 若外部服务不可用，可能导致核心功能（如 Chat、报告生成）完全不可用。

**影响范围：**
- 外部服务故障时，系统可用性受严重影响。
- 用户体验差，缺乏优雅的错误提示和替代方案。

**建议：**
- 为关键外部服务调用增加超时控制和重试机制。
- 实现降级策略：如 AI 服务不可用时，返回预设的兜底文案或提示用户稍后重试。
- 考虑引入简单的熔断器模式，避免故障扩散。
