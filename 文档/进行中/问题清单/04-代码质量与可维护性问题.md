# 代码质量与可维护性问题

## 问题概述
在审查项目整体代码结构、构建流程及工程化实践时，发现部分代码存在重复、命名不一致及工程化配置不足等问题。此类问题长期积累将显著增加技术债务。

---

## 1. 代码重复与逻辑耦合

**问题描述：**
- 前后端存在功能重复的逻辑：
  - `apps/web/src/utils/intentParser.ts` 与 `apps/edge-worker/src/services/chat/intent.ts` 均实现了意图解析逻辑。
  - 前后端的数据格式化函数（如日期格式化、数字格式化）各自独立实现，未共享。
- 后端服务层部分函数（如 `buildPeriodicReportPayload`）内部逻辑过于冗长，包含大量临时变量和嵌套计算。

**影响范围：**
- 同一逻辑变更需修改多处，容易遗漏。
- 服务层函数难以单元测试，可读性差。

**建议：**
- 将意图解析等核心逻辑统一收敛到后端，前端仅做展示层适配。
- 提取共享工具函数到公共包（`packages/utils`），供前后端共同使用。
- 对冗长函数进行拆分，遵循“单一职责原则”。

---

## 2. 命名规范与代码风格不一致

**问题描述：**
- 文件命名风格不统一：部分使用 PascalCase（如 `TodayPage.tsx`），部分使用 camelCase（如 `useChatLogic.ts`），后端路由文件使用 kebab-case（如 `auth-guard.route.test.ts`）。
- 变量命名存在混用：`snake_case` 与 `camelCase` 在前后端代码中均有出现，尤其在数据库模型和 API 交互层。
- 部分函数命名不够语义化，如 `builder.ts` 中的 `build` 系列函数，未能清晰表达其“格式化/渲染”而非“构建对象”的职责。

**影响范围：**
- 代码可读性降低，增加代码审查成本。
- 新成员难以快速理解代码规范。

**建议：**
- 制定并文档化项目命名规范：
  - 前端组件/页面：PascalCase
  - 前端 Hook/工具：camelCase
  - 后端路由/服务：kebab-case 或 camelCase（统一即可）
  - 数据库字段：snake_case
  - TypeScript 变量/函数：camelCase
- 配置 ESLint + Prettier 规则，强制统一代码风格。

---

## 3. 缺乏统一的日志与监控机制

**问题描述：**
- 后端代码中未看到结构化的日志记录（如使用 `console.log` 零散打印，或完全无日志）。
- 缺乏请求链路追踪（Trace ID）和性能监控（如请求耗时、数据库查询耗时）。
- 前端未集成错误监控（如 Sentry），无法捕获线上运行时错误。

**影响范围：**
- 线上问题排查困难，缺乏上下文信息。
- 性能瓶颈难以定位。

**建议：**
- 后端引入结构化日志库（如 `pino`），统一日志格式，包含 Trace ID、用户 ID、请求路径等上下文。
- 在关键路径（API 入口、数据库查询、外部调用）增加性能打点。
- 前端集成错误监控 SDK，捕获并上报未处理异常。

---

## 4. 构建与部署流程缺乏自动化校验

**问题描述：**
- `tools/scripts/check.mjs` 脚本存在，但未明确其在 CI/CD 流程中的触发时机。
- 缺乏对 TypeScript 类型、Lint 规则、测试覆盖率的强制性门禁（Gate）。
- 未看到预提交（pre-commit）钩子配置（如 Husky + lint-staged）。

**影响范围：**
- 不符合规范的代码容易进入主分支。
- 类型错误或测试失败可能在部署后才被发现。

**建议：**
- 在 CI 流程中增加以下强制步骤：
  1. TypeScript 类型检查（`tsc --noEmit`）
  2. ESLint 检查
  3. 单元测试与集成测试
  4. 构建验证
- 配置 Husky + lint-staged，在提交前自动运行类型检查和代码格式化。

---

## 5. 文档与代码注释不足

**问题描述：**
- 核心服务函数（如 `buildPeriodicReportPayload`, `calcStreak`）缺乏 JSDoc 注释，难以快速理解参数和返回值含义。
- 项目根目录及子项目缺乏 README 文件，新成员难以了解项目结构、启动方式及开发规范。
- 复杂的业务逻辑（如报告生成算法、意图解析规则）缺少设计文档说明。

**影响范围：**
- 知识传递依赖口头沟通，容易丢失。
- 代码维护成本高，重构风险大。

**建议：**
- 为核心公共函数补充 JSDoc 注释。
- 编写项目 README，包含：项目简介、技术栈、目录结构、启动命令、测试命令、部署流程。
- 对复杂业务逻辑编写设计文档（如 `docs/design/reports.md`）。

---

## 6. 硬编码与魔法值

**问题描述：**
- `apps/edge-worker/src/services/behavior/builder.ts` 中硬编码了默认关键词和文案：
  ```typescript
  keywords.push('持续探索', '记录成长', '行动实践')
  ```
- `apps/edge-worker/src/services/reports/builder.ts` 中硬编码了报告类型与文案的映射关系。
- 部分数值常量（如分页大小、最大关键词数量）直接以魔法值形式出现。

**影响范围：**
- 业务文案调整需修改代码并重新部署。
- 魔法值分散在代码各处，修改时容易遗漏。

**建议：**
- 将业务文案抽取到配置文件或数据库中，支持动态调整。
- 将数值常量抽取为命名常量（如 `const MAX_KEYWORDS = 6`），集中管理。
