# 前端组件与状态管理问题

## 问题概述
在审查 `apps/web` 前端代码过程中，发现组件设计、状态管理及页面逻辑存在多处可优化点。部分问题可能影响性能、可维护性及用户体验。

---

## 1. 页面组件体积过大，职责不单一

**问题描述：**
- `TodayPage.tsx`, `ActionsPage.tsx`, `ChatPage.tsx` 等核心页面组件代码行数过多（部分超过 300 行），同时承担了数据获取、状态管理、UI 渲染、事件处理等多重职责。
- 例如，`ChatPage.tsx` 中既包含消息列表渲染，又包含输入框逻辑，还包含会话切换逻辑。

**影响范围：**
- 代码可读性差，新成员上手成本高。
- 单元测试困难，难以独立测试某个功能单元。
- 复用性低，相似逻辑在不同页面重复实现。

**建议：**
- 按照“容器组件 + 展示组件”模式重构，将数据获取和状态管理抽离到自定义 Hook 或状态管理库中。
- 将大页面拆分为多个子组件，如 `ChatMessageList`, `ChatInput`, `ChatSidebar` 等。

---

## 2. 状态管理缺乏统一方案

**问题描述：**
- 目前前端状态管理混合使用了多种方式：
  - React `useState` / `useReducer` 管理局部状态。
  - `useChatLogic.ts` 等自定义 Hook 封装了部分业务状态。
  - 部分全局状态（如用户信息）通过 props drilling 传递。
- 缺乏统一的全局状态管理方案（如 Zustand、Jotai 或 Context + Reducer）。

**影响范围：**
- 跨组件状态共享困难，容易出现状态不同步。
- 状态逻辑分散，难以追踪数据流。

**建议：**
- 评估引入轻量级状态管理库（如 Zustand），统一管理用户会话、全局配置等跨页面状态。
- 对现有自定义 Hook 进行梳理，明确哪些是“状态逻辑”，哪些是“副作用逻辑”。

---

## 3. 前端路由与权限控制耦合不足

**问题描述：**
- 前端路由配置中，未看到基于角色的路由守卫实现。权限控制主要依赖后端返回 401，前端被动处理。
- `LoginPage.tsx` 登录成功后，跳转逻辑硬编码，未统一管理未授权访问的拦截和重定向。

**影响范围：**
- 用户体验不佳，访问无权限页面时先看到闪烁的页面内容，然后才跳转登录。
- 安全依赖后端，前端缺乏第一道防线。

**建议：**
- 在前端路由层增加路由守卫（如 React Router 的 `loader` 或 `element` 包裹鉴权组件）。
- 统一处理 401 响应，自动清除本地会话并跳转登录页。

---

## 4. 部分组件存在重复 UI 逻辑

**问题描述：**
- 多个页面（如 `GrowthPage.tsx`, `MyPage.tsx`, `HistoryBriefPage.tsx`）中，存在相似的卡片布局、列表渲染、空状态展示等 UI 逻辑，但未抽象为通用组件。
- 例如，“加载中”、“空数据”、“错误重试”等状态展示逻辑在各页面重复实现。

**影响范围：**
- UI 风格难以统一，改动成本高。
- 重复代码增加维护负担。

**建议：**
- 建立基础 UI 组件库，如 `<DataState status="loading|empty|error" />`、`<InfoCard />`、`<ActionList />` 等。
- 推动设计系统落地，确保各页面使用统一的组件和样式变量。

---

## 5. Chat 模块前端逻辑复杂度高

**问题描述：**
- `useChatLogic.ts` 和 `ChatPage.tsx` 中，消息流处理、意图解析、会话管理、错误重试等逻辑交织在一起。
- `intentParser.ts` 中的意图解析逻辑与后端 `apps/edge-worker/src/services/chat/intent.ts` 存在功能重叠，前后端各自维护一份解析规则。

**影响范围：**
- 意图解析规则变更时，需要前后端同步修改，容易遗漏。
- 前端逻辑过重，影响页面性能（如消息列表频繁重渲染）。

**建议：**
- 将意图解析逻辑统一收敛到后端，前端仅负责展示解析结果。
- 对 Chat 模块进行更细粒度的组件拆分，并考虑使用 `useMemo` / `useCallback` 优化渲染性能。
