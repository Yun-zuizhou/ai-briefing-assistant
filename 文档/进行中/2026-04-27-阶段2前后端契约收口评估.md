## 1. 本轮结论

五个重点页阶段 1 已完成后，下一步不应继续做页面级小修。阶段 2 的主线应转为“页面职责对应的正式后端契约”，重点不是新增大功能，而是把当前前端临时聚合逻辑收回到 Workers + contracts。

本轮开工前核查结论如下；Actions、Journal、Today 的执行结果见第 10-12 节：

1. `Today / 简报页` 当前仍由 `GET /api/v1/dashboard/today` 即时聚合 `hot_topics`、`opportunities`、`user_interests`，没有直接消费每日简报结果对象。
2. 真实 D1 schema 中 `briefings` 表已有 `payload` 字段，但当前迁移文件没有 `result_payload` 字段；后续文档若写 `briefings.result_payload`，实现时必须先校正为真实字段或明确新增迁移。
3. `Journal / 记录页` 当前一次拉取 `notes`、`growth-overview`、`history`、`favorites`、`actions/overview` 五个接口，在前端拼成沉淀入口；这适合阶段 1，不适合作为长期契约。
4. `Actions / 待办页` 已有 `GET /api/v1/actions/overview`，但契约缺少 `topPriority` / `suggestedNextAction` 这类“今天先推进什么”的正式字段，当前由前端推导。
5. `My / 我的页` 阶段 2 暂不需要新增接口；它继续作为入口分发页即可。

阶段 2 推荐顺序：

1. `ActionsOverviewData` 补主行动字段。
2. 新增 `JournalOverviewData` 与 `GET /api/v1/journal/overview`。
3. 扩展 `TodayPageData` 的每日简报结果对象能力。

## 3. Actions 契约收口

### 当前问题

`actions/overview` 已经返回今日待办、未来待办、完成事项、收藏待处理、机会跟进、提醒摘要和打卡状态。但“今天先推进这 1 件”目前由前端从多个列表里自行推导。

这会造成：

1. 前端长期承担业务排序规则。
2. 多端或后续页面复用时容易出现排序不一致。
3. 无法解释为什么这件事被推荐为主行动。

### 最小契约方案

在 `packages/contracts/src/page-data.ts` 中对 `ActionsOverviewData` 增加可选字段，保持向后兼容：

```ts
export interface SuggestedActionItem {
  source: 'todo' | 'opportunity_follow' | 'saved_item'
  id: number | string
  title: string
  reason: string
  priorityLabel?: string
  dueLabel?: string
  primaryActionLabel: string
  deepLink?: string
}

export interface ActionsOverviewData {
  topPriority?: SuggestedActionItem | null
  suggestedNextActions?: SuggestedActionItem[]
  // existing fields...
}
```

Workers 最小实现：

1. 在 `buildActionsOverview` 中按 `urgent/high + deadline + follow.nextStep + savedForLater` 生成 `topPriority`。
2. `suggestedNextActions` 最多返回 3 条，来源可来自 todo、follow、saved item。
3. 不改 D1 schema，不新增 endpoint。

测试方式：

1. 更新 `apps/edge-worker/tests/actions.route.test.ts`，断言 `topPriority`。
2. 更新 `apps/web/src/services/api.ts` validator。
3. 跑 `npm.cmd run check`。
4. 涉及页面主路径后跑 `npx.cmd playwright test e2e/app-mainline.spec.ts`。

风险：低。只扩展现有响应，字段可选，前端可渐进切换。

## 4. Journal 契约收口

### 当前问题

`JournalPage` 当前为了显示“最近沉淀 / 我刚说过的 / 我推进过的 / 我留下的 / 长期回看”，一次读取五个接口：

1. `notes`
2. `preferences/growth-overview`
3. `history`
4. `favorites`
5. `actions/overview`

这说明页面职责已经明确，但数据契约仍停留在前端拼装阶段。

### 最小契约方案

新增 `JournalOverviewData`，新增 Workers 路由：

`GET /api/v1/journal/overview`

建议 contracts：

```ts
export interface JournalOverviewData {
  summary: {
    expressionCount: number
    progressCount: number
    keptCount: number
    reviewCount: number
    summaryText: string
  }
  recentNotes: NoteApiItem[]
  progressItems: Array<{
    id: string
    title: string
    meta: string
    detail: string
    deepLink?: string
  }>
  keptItems: Array<{
    id: string
    title: string
    sourceLabel: string
    detail: string
    createdAt?: string
    deepLink?: string
  }>
  review: {
    availableCount: number
    keywords: string[]
  }
}
```

Workers 最小实现：

1. 新建 `apps/edge-worker/src/routes/journal.ts`。
2. 复用现有 behavior store 查询：notes、todos、favorites、history、opportunity follows、reports/growth summary。
3. 在 `packages/contracts/src/page-data.ts` 增加 `JournalOverviewData`。
4. 在 `apps/web/src/services/api.ts` 增加 validator 与 `getJournalOverview()`。
5. 前端 `JournalPage` 从多接口并发切为优先读 `journal/overview`，失败时可短期保留现有多接口兜底。

是否需要 D1 migration：否。

测试方式：

1. 新增 `apps/edge-worker/tests/journal.route.test.ts`。
2. 更新前端 validator 静态契约测试。
3. 跑 `npm.cmd run check`。
4. 跑 `npx.cmd playwright test e2e/app-mainline.spec.ts`。

风险：中。新增 endpoint，但不改表；主要风险是聚合口径与现有页面显示不一致。

## 5. Today 契约收口

### 当前问题

`TodayPageData` 当前已经有 `summary`、`recommendedForYou`、`worthKnowing`、`worthActing`、`quickNoteEntry`，但缺少正式“今日主判断”字段。阶段 1 只能在前端推导 lead item，并在前端过滤后端过渡文案。

更关键的是，文档曾多次提到 `briefings.result_payload`，但真实 D1 迁移里 `briefings` 表字段是 `payload`。因此 Today 阶段 2 不能直接按 `result_payload` 开工。

### 最小契约方案

先不新增表。扩展 `TodayPageData` 可选字段：

```ts
export interface TodayLeadItem {
  contentRef?: string
  itemType: 'hot_topic' | 'opportunity' | 'briefing'
  title: string
  summary: string
  sourceLabel?: string
  relevanceLabel?: string
  primaryActionLabel: string
  secondaryActionLabel?: string
}

export interface TodayExtensionSlot {
  slotType: 'ask' | 'todo' | 'save' | 'review'
  title: string
  description: string
  actionLabel: string
  deepLink?: string
  sourceContentRef?: string
}

export interface TodayPageData {
  leadItem?: TodayLeadItem | null
  dailyAngle?: string
  freshness?: {
    latestPublishedAt?: string
    sourceCount: number
    generatedAt?: string
  }
  extensionSlots?: TodayExtensionSlot[]
  // existing fields...
}
```

Workers 最小实现路径：

1. 保留 `GET /api/v1/dashboard/today`，不新增 `/briefing/today`。
2. 在 `dashboard/today` 内优先查询用户当天最新 ready briefing：
   - `briefings.user_id`
   - `briefings.briefing_date`
   - `briefings.briefing_type`
   - `briefings.payload`
   - `briefings.generated_at`
3. 如果 `briefings.payload` 可解析且满足 `TodayPageData` 或兼容字段，则优先使用其中的主判断字段。
4. 如果没有 briefing payload，继续使用当前 hot topics + opportunities 的即时聚合兜底。
5. 去掉用户可见的“真实聚合阶段 / 继续收口”文案，过程状态如需保留只能进内部字段或测试文档。

是否需要 D1 migration：

阶段 2.1 不需要。先消费 `briefings.payload`。

只有当确认要把字段从 `payload` 改成 `result_payload` 或 `briefing_payload_json` 时，才新增 migration。当前不建议先做字段重命名。

测试方式：

1. 扩展 `apps/edge-worker/tests/dashboard.route.test.ts`：
   - 有 briefing payload 时优先使用 payload lead。
   - 无 briefing payload 时走现有即时聚合兜底。
   - 响应不包含用户可见过渡文案。
2. 更新 `apps/web/src/services/api.ts` validator。
3. 更新 `TodayPage` 优先使用 `leadItem`。
4. 跑 `npm.cmd run check`。
5. 跑 `npx.cmd playwright test e2e/app-mainline.spec.ts`。

风险：中高。Today 是主入口，且涉及 briefing payload 兼容解析；应放在 Actions 和 Journal 之后。

## 6. 不建议做的方向

1. 不建议先新增 D1 表 `today_snapshot`。
   - 现有 `briefings.payload` 已可承接每日结果对象，先用真实已有字段验证契约。
2. 不建议直接把 Journal 继续做成前端五接口拼装。
   - 这会让页面职责越来越清楚，但数据契约越来越散。
3. 不建议让 Actions 的 `topPriority` 长期留在前端。
   - 今日主行动是业务判断，不应由 UI 层长期推导。
4. 不建议马上重命名 `briefings.payload`。
   - 迁移成本不高，但会牵动历史数据、测试和报告口径；先以现字段消费。
5. 不建议把 My 页纳入阶段 2。
   - My 页现在是入口分发页，`growth-overview` 足够支撑当前职责。

## 7. 推荐实施顺序

### 2.1 Actions 主行动契约

原因：

1. 不新增 endpoint。
2. 不改 schema。
3. 前端已经有对应 UI，只是把推导逻辑后移到 Workers。
4. 风险最低，最适合阶段 2 开局。

交付：

1. `packages/contracts` 扩展 `ActionsOverviewData`。
2. `apps/edge-worker` 扩展 `buildActionsOverview`。
3. `apps/web` 优先使用 `topPriority`。
4. 测试全绿。

### 2.2 Journal Overview 聚合契约

原因：

1. 解决前端五接口拼装问题。
2. 不需要 D1 migration。
3. 能把“个人沉淀入口”正式固化为后端契约。

交付：

1. 新增 `JournalOverviewData`。
2. 新增 `GET /api/v1/journal/overview`。
3. `JournalPage` 切到新接口，短期保留兜底。
4. 测试全绿。

### 2.3 Today 每日结果对象契约

原因：

1. Today 是主入口，风险最大。
2. 需要先明确 `briefings.payload` 与文档中 `result_payload` 的口径差异。
3. 需要 payload 兼容解析和兜底路径。

交付：

1. 扩展 `TodayPageData`。
2. `dashboard/today` 优先消费 `briefings.payload`，无 payload 时保持现有聚合。
3. 前端优先使用 `leadItem`、`extensionSlots`。
4. 测试全绿。
