# 前端设计资产盘点与落点清单

- 文档日期：2026-04-22
- 适用范围：`apps/web`
- 目标：把“已经存在的设计”从零散内联中盘出来，形成可统一、可替换、可讨论的样式资产目录

---

## 1. 当前判断

当前前端并不是“没有设计”，而是：

1. 已经形成了一套稳定的复古纸张风格语言
2. 这套语言在很多页面里被反复手写
3. 同一套语言没有被明确命名，因此“哪里该用、哪里不该用”变得不稳定

所以当前最该做的，不是继续凭感觉改页面，而是先把现有设计资产命名、归档、记录落点。

---

## 2. 已抽出的样式资产入口

代码侧目录：

- `apps/web/src/styles/design-tokens.ts`
- `apps/web/src/styles/design-tokens.css`
- `apps/web/src/styles/style-recipes.ts`

其中：

- `design-tokens.*`：颜色、字体、间距、阴影等基础 token
- `style-recipes.ts`：对当前“已经存在的设计语言”做出的命名抽象与使用落点记录

---

## 3. 当前已识别的设计语言

### 3.1 报头编辑样式

特点：

- 暖纸底色
- 衬线主标题
- ornament / divider / meta 三段式
- 强栏目感，不偏工具感

当前落点：

- `Masthead`
- `SecondaryHeader`
- `TodayPage`
- `GrowthPage`
- `MyPage`
- `ChatPage`

风险：

- 报头是有设计的，但当前已经被当成“默认容器”，容易导致所有页面头部都过重

### 3.2 区块标题样式

特点：

- 左侧红色短竖条
- 大写感栏目标题
- 右侧弱操作入口

当前落点：

- `TodayPage`
- `GrowthPage`
- `MyPage`
- 设置/通知类页面

风险：

- 已形成统一识别度，但当前标题尺寸、右侧操作位置、点击区域还不完全统一

### 3.3 暖纸面板

特点：

- `paper-warm`
- `1px solid border`
- `14px` 左右内边距
- 承载说明、摘要、表单区块

当前落点：

- `TodayPage` 推荐区、速记区
- `ChatPage` 顶部反馈卡
- `GrowthPage` 摘要区
- `AboutPage`
- `AiProviderSettingsPage`
- `PreviewPage`

风险：

- 这是当前最泛滥的容器，已被同时用于“说明块 / 空状态 / 正式内容块 / 表单块”

### 3.4 带内框摘要面板

特点：

- 外层暖纸盒
- 内层虚线 inset
- 顶部眉标 + 右侧计数 + 两个操作按钮

当前落点：

- `TodayPage` 今日总述

风险：

- 已有明确设计感，但目前只有一个页面在用，没有被定义成“摘要型卡片”

### 3.5 内容列表卡

特点：

- `paper` 或 `paper-warm`
- 顶部眉标
- 中间标题
- 底部摘要/辅助信息

当前落点：

- `TodayPage`
- `ActionsPage`
- `AiDigestLabPage`
- `ArticlePage` 相关推荐

风险：

- 同一种内容卡在多个页面里反复出现，但每页都自己写 padding / gap / meta 排列

### 3.6 行动型内容卡

特点：

- 在内容卡基础上增加回报、截止、操作按钮
- 偏“机会转行动”

当前落点：

- `TodayPage` 值得行动的
- `ActionsPage` 部分待办块

风险：

- 这是目前最适合单独抽成组件的一类，因为它同时有内容、状态和动作

### 3.7 空状态卡

特点：

- 仍使用正式卡片容器
- 居中排版
- 常用 `24px` padding

当前落点：

- `TodayPage`
- `ActionsPage`
- `GrowthPage`
- `Report pages`

风险：

- 当前空状态和内容状态差异不够大，属于“风格有了，但语义没分层”

### 3.8 主操作按钮 / 次操作按钮

特点：

- 主按钮：`ink` 或 `accent`
- 次按钮：浅底 + 描边

当前落点：

- `TodayPage`
- `MyPage`
- `ArticlePage`
- `Report pages`
- `WelcomePage`

风险：

- 已形成明显视觉语言，但当前还存在“页面内自己手搓按钮长相”的问题

### 3.9 标签胶囊

特点：

- 深底浅字
- 阴影偏移
- 存在感强

当前落点：

- `TodayPage`
- `GrowthPage`
- `MyPage`
- `ProfilePage`
- `AiDigestLabPage`

风险：

- 这个设计已经很强，但当前被用于“兴趣词 / 关键词 / 风险项 / 摘要点”多种语义，容易失去区分

### 3.10 Chat 气泡系统

特点：

- 助手气泡：暖纸底 + 装饰角 + 状态标签
- 用户气泡：深色底 + 白字

当前落点：

- `ChatPage`

风险：

- 已形成一套独立视觉语法，但当前所有构成元素几乎都在页面文件内联，不利于统一

---

## 4. 当前用法中“行业上不行”的部分

以下不是“审美意见”，而是正式项目里不应继续放任的做法：

1. 同一种卡片结构在 2 个页面以上重复出现，还继续在页面里手写内联
2. 同一种按钮在不同页面里由页面自己定义 padding / border / fontSize
3. 空状态继续使用与正式内容卡几乎相同的视觉重量
4. `span/div + onClick` 与 `button` 混用承担同级操作
5. 同一层级文本跨页反复出现 `11px / 12px / 13px / 14px` 临时定义，却没有命名语义
6. 已有稳定设计语言，但没有“样式名 -> 页面落点 -> 语义场景”的登记
7. 组件已经存在，页面仍绕开组件继续直接手写视觉结构

---

## 5. 建议的统一顺序

不是先改视觉，而是先改使用边界：

1. 先统一“哪种设计语言对应哪种语义”
2. 再统一“哪些页面必须复用同一种样式配方”
3. 最后才决定某个设计本身要不要微调

建议先统一的 5 类：

1. 内容列表卡
2. 行动型内容卡
3. 空状态卡
4. 顶部反馈卡 / 说明卡
5. 主按钮 / 次按钮

---

## 6. 下一步最适合做什么

在不重做视觉的前提下，最值得先推进的是：

1. 让 `TodayPage` / `ChatPage` / `ActionsPage` 先统一使用同一套卡片语法
2. 把 `Tag` 的使用语义拆成至少两类：兴趣词 / 状态词
3. 把“空状态”从“内容卡”里分出来
4. 把 `Masthead` 保留为页面身份层，不继续拿它承担页面主要信息层级
