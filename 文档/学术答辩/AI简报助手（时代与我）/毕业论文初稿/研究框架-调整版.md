### 外文摘要（Abstract）

（250—300个词）

This paper designs and implements a personal information management application called "Briefing Assistant App" with "Daily Briefing" as its core carrier. Aiming at the pain points of current information overload, difficulty in information organization, and insufficient information awareness, the system proactively helps users obtain valuable content from the ocean of information by intelligently refining scattered information into highly condensed briefings. The frontend is implemented using React + TypeScript + Tailwind CSS. The backend evolved from Python + FastAPI to TypeScript + Hono + Cloudflare Workers. The database uses Cloudflare D1 (SQLite-compatible), and the system supports integration with 7 major AI providers including DeepSeek, OpenAI, Anthropic, and Gemini. Core features include rule-based intent recognition with keyword matching, interest-driven content recommendation, session persistence with multi-turn dialogue, and periodic report generation. This paper focuses on the specific application of AI tools in technology selection, database design, API development, and large model integration, exploring a "briefing-driven" information interaction model.

**Key words:** Personal Information Management; Intelligent Briefing; Intent Recognition; Large Language Model; AI-Assisted Development; Cloudflare Workers; React

### 第1章 绪论

#### 1.1 研究背景与意义

- 1.1.1 信息过载时代的个人信息管理困境
  - 信息分散在多个平台，用户在各平台间来回跳转
  - 信息获取后缺乏有效记录和沉淀机制
  - 用户缺乏对主题发展趋势的感知能力
- 1.1.2 人工智能技术在个人信息管理领域的应用前景
  - 大语言模型在文本处理、意图理解方面的能力
  - AI辅助开发工具降低全栈开发门槛的趋势
- 1.1.3 研究目的与意义（事实边界）
  - 目标定位：在毕设周期内完成可运行、可验证的简报助手原型，不把当前结果外推为通用产品结论
  - 实践意义：围绕“简报浏览-记录-待办-回顾”构建最小闭环，验证可用性
  - 工程意义：形成可复验的实现流程（统一脚本、统一测试门槛、权限收口）
- 1.1.4 设计可行性背景
  - 当前主链：apps/web（React + TypeScript）→ apps/edge-worker（Hono + Workers）→ Cloudflare D1；Python链路保留为离线/参考
  - 当前可用能力：今日简报、内容详情、收藏/待办/笔记、会话管理、报告查看、基础意图执行
  - 当前验证口径：以 `npm run check` 与《当前测试与验收总表》为测试数字事实源
  - 可行性判断：技术栈成熟、实现路径清晰、验证机制已建立、范围可控（信息过载与混合架构研究支撑见[10][11][12]）

#### 1.2 国内外研究现状

- 1.2.1 信息聚合类应用发展现状
  - 语鲸、即刻、小红书等产品的功能特点与不足
  - Notion AI、Readwise等工具的学习成本与本地化问题
- 1.2.2 AI辅助开发工具发展现状
  - AI辅助代码生成、设计、测试的工具演进
  - Trae IDE等AI辅助开发平台的出现
- 1.2.3 用户画像与智能摘要技术研究现状
  - 静态画像与动态演化画像的研究进展
  - 多源信息整合与智能摘要技术
- 1.2.4 现有研究的不足与本课题的切入点

#### 1.3 研究内容、设计原理与方法

- 1.3.1 设计目的
  - 降低多平台信息整理成本，提供“每日简报 + 行动管理”的轻量入口
  - 验证“简报驱动交互”在个人信息管理场景中的可用性
  - 在资源受限条件下验证“规则优先、生成补充”的工程路径
  - 本课题面向毕业设计场景，目标是完成一个可运行、可验证的个人信息管理原型，而不是直接给出可规模化复制的产品结论。基于当前项目实现，系统已经形成“前端（React + TypeScript）—在线后端（Hono + Cloudflare Workers）—数据层（Cloudflare D1）”的主链，Python链路作为离线参考与回退验证保留。当前能力覆盖简报浏览、内容详情、收藏/待办/笔记、会话管理与报告查看，且已有统一测试门槛与记录口径。结合信息过载问题研究与LLM混合架构研究，本课题采用“先可用、后增强”的实施策略具有现实可行性[10][11][12]。
  - 本课题的设计目的主要有三点：第一，在信息分散和筛选成本较高的场景下，提供“每日简报 + 行动管理”的轻量入口，降低用户在多平台间反复切换与整理的成本；第二，验证“简报驱动交互”是否能够支持用户完成浏览、记录、待办和回顾这一最小闭环；第三，在毕设资源约束下，验证“规则优先、生成补充”的工程路径，并沉淀可复验的开发与验收流程。
- 1.3.2 设计原理
  - 最小闭环原理：优先保障主流程可用，再扩展功能
  - 真实链路原理：正式页面以真实后端与正式数据链路为主，demo数据不作为业务真源
  - 分层解耦原理：前端、路由、服务、存储分层，降低迭代耦合
  - 稳定优先原理：意图识别当前以规则/关键词为主，语义增强作为后续方向
  - 安全前置原理：受保护资源走会话鉴权，内部执行接口走token控制
  - 本系统遵循最小闭环、真实链路、分层解耦、稳定优先和安全前置五项原则。最小闭环强调先保障主流程可用，避免功能扩面导致主线失稳；真实链路强调正式页面以真实后端接口与正式数据链路为准，演示数据不作为业务真源；分层解耦强调前端、路由、服务与存储分层实现，便于迭代和排错；稳定优先强调当前意图识别以规则/关键词为主，语义增强作为后续演进方向；安全前置强调受保护资源与内部执行接口采用不同鉴权策略，控制越权和误调用风险。
- 1.3.3 设计理论基础（借鉴边界）
  - 信息过载研究：支撑简报化与结构化呈现的必要性[10]
  - 检索/规则/生成协同研究：支撑混合架构的工程可行性[1][3][11][12]
  - 意图识别研究：支撑“规则主线 + 语义增强”演进路径[4][14][15]
  - 摘要评测与AI开发实证研究：用于约束质量验证口径，避免过度外推[16][17][18][19][20]
  - 本研究对理论的使用方式是“借鉴并约束到当前工程边界”。在问题层面，信息过载研究为“为何需要简报化与结构化呈现”提供了依据[10]；在方法层面，检索/规则/生成协同研究为混合架构的可行性提供了支持[1][3][11][12]；在交互层面，意图识别相关研究支持“规则主线 + 语义增强”的分阶段演进路径[4][14][15]；在验证层面，摘要评测与AI开发实证研究提示应以覆盖率、可追溯性和可复验结果约束结论表达，避免仅凭主观体验外推系统效果[16][17][18][19][20]。
- 1.3.4 研究内容概述
  - 设计以"每日简报"为核心的个人信息管理应用
  - 实现信息聚合、兴趣匹配、意图识别、互动闭环、周期回顾五大功能
- 1.3.5 研究方法与技术路线
  - 文献研究法、竞品分析法、原型迭代法、AI辅助开发法
  - 方法边界：采用“先可用再增强”的迭代策略，不以一次性端到端重构为目标

## 论文格式说明与注意事项

### 图表编号规则

- 图表不多的话：图1—图20, 表1—表20（一贯到底）
- 图表多的话：分章编号，如：图2-1（图2.1），表2-1（表2.1）
- 图题放在图的正下方，表题放在表的正上方
- 图表及图题、表题的文字比正文减小一个字号
- 建议用三线表，表格转页应重复表头并在右上角标"（续表）"

### 公式编号规则

- 一律采用阿拉伯数字分章依序编排
- 如："式（2-13）" "式（4-5）"
- 标注应于该公式所在行的最右侧
- 公式书写方式应在文中相应位置另起一行居中横排
- 较长的公式只可在符号处（+、-、\*、/、≤≥等）转行

### 章节编号格式

- 章："1" "2" "3" ...
- 节："1.1" "1.2" ...
- 小节："1.1.1" "1.1.2" ...
- 或采用："一、" "（一）" "1." "（1）" 等格式

### 其他注意事项

- 正文用中文撰写，字数要求一般为8000字
- 摘要采用第三人称撰写，用语规范，明确简练
- 关键词间用分号隔开，列出3～6个
- 中外文标点不混用
- 必要的地方应随页加脚注
- 正文中不应加入程序的源代码（除非必需）
- 采用国家正式公布实施的简化汉字和法定的计量单位
- 术语、符号、代号必须统一，符合规范化要求
- 新的专业术语、缩略语应加以注释
- 国外新的专业术语、缩略语，必须在译文后用小括号注明原文
