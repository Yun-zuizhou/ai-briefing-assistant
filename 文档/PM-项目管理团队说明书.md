# PM-项目管理团队说明书

- 文档状态：现行参考文档
- 适用对象：项目 PM / 新对话交接 / 文档职责巡检
- 建立时间：2026-04-04
- 使用方式：用于 3 分钟内判断当前项目管理文档体系是否清晰、是否有人越权抢职责

---

## 1. 这份说明书解决什么问题

这份说明书不是新的主控文档。

它只做一件事：

把当前项目的核心文档体系压缩成一页 PM 视角的“管理团队结构图”，帮助你快速判断：

1. 每个文件现在负责什么
2. 每个文件不负责什么
3. 信息应该怎么流动
4. 哪些文件开始变胖时，应把内容下放到哪里

---

## 2. 团队结构图

当前项目管理团队分为三层：

### 上层：定方向、定口径、定制度

1. `文档/项目核心总纲.md`
   - 岗位：项目负责人
   - 负责：项目级判断、核心原则、当前阶段判断、下一步主线

2. `文档/README.md`
   - 岗位：导航负责人
   - 负责：文档入口、目录导航、阅读顺序

3. `文档/项目功能文档.md`
   - 岗位：产品功能负责人
   - 负责：产品能力、页面全景、用户使用方式

4. `文档/文档规范.md`
   - 岗位：制度负责人
   - 负责：文档结构、职责边界、归档制度、测试口径规则

### 中层：承接阶段、组织执行、对外交接

1. `文档/进行中/当前阶段总表.md`
   - 岗位：当前阶段执行负责人
   - 负责：当前阶段目标、主线、风险、顺序、相关文档、退出动作

2. `文档/项目核心子纲-阶段演进与归档索引.md`
   - 岗位：阶段总结与归档负责人
   - 负责：阶段划分、阶段结论、归档入口、向总纲汇报

3. `文档/Agent协同经验/项目承接提示词-当前版.md`
   - 岗位：新对话状态交接负责人
   - 负责：当前日期、当前阶段名称、当前执行主线、当前测试口径、当前最优先动作

4. `文档/Agent协同经验/项目初始化.md`
   - 岗位：协作协议负责人
   - 负责：稳定协作规则、阅读顺序、汇报链、输出要求

### 底层：承接当前事实、保存历史过程

1. `文档/进行中/当前主线文档`
   - 岗位：当前工作面事实负责人
   - 负责：当前主线事实、边界、缺口、推进判断

2. `文档/归档/阶段目录/阶段说明.md`
   - 岗位：历史追溯负责人
   - 负责：阶段过程入口、历史明细定位、归档边界说明

## 3. 每个岗位不负责什么

### 项目负责人不负责

1. 阶段过程流水账
2. 当前执行面细节
3. 新对话状态注入

### 导航负责人不负责

1. 当前阶段解释
2. 当前执行主线判断
3. 项目级结果口径

### 产品功能负责人不负责

1. 当前阶段管理
2. 当前执行主线
3. 测试口径主维护

### 制度负责人不负责

1. 当前项目状态判断
2. 某轮具体推进说明

### 当前阶段执行负责人不负责

1. 项目长期口径
2. 阶段过程长摘要
3. 新对话状态注入

### 阶段总结与归档负责人不负责

1. 每轮推进流水
2. 当前阶段顺序管理
3. 项目级最终口径

### 新对话状态交接负责人不负责

1. 长篇成果汇总
2. 项目长期历史
3. 协作方法论

### 当前工作面事实负责人不负责

1. 项目级判断
2. 阶段归档索引
3. 长期导航职责

### 历史追溯负责人不负责

1. 当前执行顺序
2. 当前主线判断
3. 当前项目状态出口

## 4. 信息流怎么走

日常推进时，信息流固定为：

`底层事实 -> 当前阶段总表 -> 项目核心子纲 -> 项目核心总纲`

补充硬规则：

1. 只要形成结果级变化，必须先检查对应 `文档/进行中/` 主线文档是否已经更新
2. 如果主线文档还没更新，不得直接修改 `当前阶段总表`、`项目核心子纲`、`项目核心总纲`
3. `项目承接提示词-当前版` 只负责状态注入，不能替代主线文档的先行职责

新对话承接时，阅读顺序固定为：

`文档规范 -> 项目核心总纲 -> 项目核心子纲 -> 当前阶段总表 -> 当前主线文档 -> 项目功能文档 -> 代码`

这两条顺序不要混。

---

## 5. 阶段切换时谁先动

阶段切换或阶段完成时，优先按下面顺序调整：

1. 先更新 `文档/进行中/` 主线文档
2. 再更新 `文档/进行中/当前阶段总表.md`
3. 形成阶段结论后，再更新 `文档/项目核心子纲-阶段演进与归档索引.md`
4. 只有项目级口径变化时，最后更新 `文档/项目核心总纲.md`
5. 完成后执行归档，并补 `归档/阶段目录/阶段说明.md`

## 6. 文件变胖时怎么判断

### `项目核心总纲.md` 变胖信号

1. 开始写大量阶段过程
2. 开始维护详细当前文档清单
3. 开始解释协作规则

处理方式：

- 下放到 `当前阶段总表`、`项目核心子纲` 或 `项目初始化`

### `项目核心子纲-阶段演进与归档索引.md` 变胖信号

1. 开始逐条累积每轮推进判断
2. 开始像流水账一样记录推进
3. 开始承担当前执行顺序

处理方式：

- 下放到 `进行中/` 主线文档或 `归档/`

### `当前阶段总表.md` 变胖信号

1. 已完成项越来越细碎
2. 开始变成阶段摘要仓
3. 开始复制状态卡内容

处理方式：

- 下放到 `进行中/` 主线文档、`项目核心子纲` 或 `项目承接提示词-当前版`

### `项目承接提示词-当前版.md` 变胖信号

1. 开始写成长篇成果总结
2. 开始复制总表和子纲
3. 开始承载历史演进

处理方式：

- 下放到 `当前阶段总表` 或 `进行中/`

## 7. PM 的最小巡检动作

每次推进一轮后，你只需要快速检查这 4 个问题：

1. `当前阶段总表` 还是不是“执行负责人”
2. `项目核心子纲` 还是不是“阶段总结负责人”
3. `项目承接提示词-当前版` 还是不是“最小状态交接负责人”
4. `项目核心总纲` 还是不是“项目负责人”

如果这一轮形成了结果级变化，再额外检查第 5 个问题：

5. 对应 `文档/进行中/` 主线文档有没有先更新；如果没有，说明同步顺序违规，必须先补主线文档再看上层文件是否需要修正

如果这 4 个没乱，整个文档管理团队一般就还稳。

## 8. 一句话版本

上层定方向，中层承接阶段，底层承接事实，历史单独归档，新对话单独交接，制度由 `文档规范.md` 统一管理。
