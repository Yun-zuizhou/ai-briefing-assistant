# AI简报助手功能设计和实现表

- 生成日期：2026-04-16
- 文档目的：把当前项目主线功能拆成可执行的“功能设计 -> 模块落位 -> 接口边界 -> 数据链路 -> 实现难点 -> 替代方案”表，直接指导后续功能需求细化和代码实现
- 适用范围：
  - 正式前端：`prototype/src`
  - 正式后端：`workers/src`
  - 正式数据层：`cloudflare/d1/migrations`
  - Python `app/`：保留为业务语义参考实现、测试参考和回退知识源，不再作为后续主扩展承载位
- 设计原则：
  1. 只以当前正式技术主线为准，不再基于旧版 React Native / FastAPI 架构做未来实现设计
  2. 功能必须优先挂在“今日信息 -> 行动转化 -> 自我记录 -> 周期回顾”主闭环上
  3. 接口按“页面聚合读侧”和“结构化对象写侧”分离
  4. 数据链路必须能追溯到真实表，禁止重新回到前端假状态拼装

## 3. 推荐的代码组织方案

当前 `apps/edge-worker/src/routes/chat.ts`、`apps/edge-worker/src/routes/reports.ts` 已经明显承担过多职责。建议按“路由编排 / 领域服务 / SQL 仓储 / DTO 类型”四层继续整理：

| 层级 | 推荐目录 | 职责 |
| --- | --- | --- |
| 路由层 | `apps/edge-worker/src/routes/*.ts` | 解析请求、鉴权预留、调用 service、返回 DTO |

## 4. 接口清晰性设计准则

## 4.1 读写分离

| 类型 | 规则 |
| --- | --- |
| 页面聚合接口 | 只返回页面所需 DTO，例如 `TodayPageData`、`ActionsOverviewData`、`GrowthOverviewData` |
| 对象接口 | 只处理单一对象，例如 `todos`、`favorites`、`notes`、`user_settings` |
| 动作接口 | 只处理具备副作用的命令，例如 `chat/execute`、`actions/check-in` |

不要出现的情况：

1. `overview` 接口顺手创建待办
2. `settings` 接口顺手生成简报
3. `content/by-ref` 顺手写收藏

## 4.2 `content_ref` 统一引用

- 所有内容型跳转统一用 `content_ref`
- 推荐、收藏、报告热点、历史内容都不要再单独发明一套跳转参数
- `content_ref` 格式固定为 `content_type:id`

## 4.3 Chat 契约

- `recognize` 只认意图，不写表
- `execute` 只做执行，不承担页面渲染格式
- `reclassify` 只做纠偏，不再让前端手动二次改表
- `chat_messages` 是会话沉淀层，不是业务真源；业务真源仍是 `todos/notes/user_interests/user_settings`

## 4.4 报告契约

- 报告对象只通过 `report_id` 被历史回看引用
- 报告 payload 是结果快照，不替代行为事实表
- 若页面需要“为什么得出这个结论”，应在 payload 中补来源摘要，而不是直接从页面反查散表

## 5. 关键数据链路总表

| 主链路 | 上游输入 | 中间处理 | 下游结果 |
| --- | --- | --- | --- |
| Today 推荐链 | `user_interests` + `hot_topics` + `opportunities` | 匹配、打分、聚合 | `TodayPageData` |

## 6. 三轮核验思考

这里不是重复列功能，而是针对“这个功能是不是必要”“是否必须按当前方式实现”“是否还能更稳地落地”做三轮反证。

## 6.1 第一轮：如果不要这个功能，目标还能不能达成

| 功能 | 如果不要该功能 | 是否还能实现目标 | 结论 |
| --- | --- | --- | --- |
| Today 个性化推荐 | 只保留热点列表，不做兴趣推荐 | 可以做“信息浏览”，但不能实现“时代与我”的个体相关性 | 必须保留 |
| 统一内容详情 | 各页面各自读各自详情接口 | 可以短期跑通，但会导致收藏、报告、相关推荐无法共用同一引用体系 | 必须保留 |
| Chat 新增兴趣 | 只允许手工进入设置页加兴趣 | 可以实现配置目标，但会损失“自然语言即配置”的主产品特色 | 应保留 |
| Chat 创建待办 | 只允许手工在待办页建任务 | 仍能实现行动管理，但“信息到行动”的转化效率会明显下降 | 应保留 |
| Chat 记录想法 | 只保留日志页手写记录 | 仍能记录，但无法把随手输入转化为沉淀，使用摩擦更大 | 应保留 |
| 行动页打卡与节律 | 去掉打卡，仅保留待办列表 | 可以实现任务管理，但成长感和连续反馈会变弱 | 可降级但不建议删除 |
| 周报/月报/年报 | 只看原始历史数据，不生成报告 | 可以勉强回顾，但无法形成周期总结，主闭环会断在“沉淀后无回看” | 必须保留 |
| 推送偏好设置 | 不做推送时间配置 | 当前主链仍能跑，但“主动提醒/晨晚简报”目标无法成立 | 可以暂缓执行链，但设置入口要保留 |

第一轮结论：

1. 真正不能删的是：`Today 推荐`、`统一内容详情`、`Chat 写入`、`周期报告`
2. 可以降级但不宜删除的是：`打卡节律`、`推送执行链`
3. 因此后续开发优先级仍应围绕主闭环，而不是外围扩面

## 6.2 第二轮：如果不按当前方式实现，还能怎么实现

| 功能 | 当前方式 | 可替代方式 | 为什么当前方式仍更适合 |
| --- | --- | --- | --- |
| Today 推荐 | 后端聚合页面 DTO | 前端拉多个接口自行拼装 | 当前项目推荐规则和兴趣匹配都在后端更稳定，前端拼装会导致规则散落 |
| 内容详情 | `content_ref + by-ref` | 每种内容单独详情 API | 当前项目内容来源多且仍在过渡期，统一引用能降低页面复杂度 |
| Chat 识别与执行 | `recognize/execute` 双接口 | 一个接口同时识别并写入 | 双接口更适合确认、纠偏和测试，尤其当前规则识别并不完美 |
| 兴趣配置 | `user_interests` 独立事实表 | 继续只写 `users.interests` JSON | 独立表更适合查询、状态控制、后续统计和推荐学习 |
| 报告生成 | 行为事实聚合后缓存到 `reports` | 每次页面实时重算 | 缓存结果更利于历史回看、固定 `report_id` 和一致性 |
| 行动页 | `overview` 聚合读侧 + `todos` 对象写侧 | 页面直接分别拉待办/收藏/跟进多个接口 | 当前页面强依赖“统一体感”，overview 更适合首屏；同时保留对象接口保证边界清晰 |
| 机会转待办 | 借 Chat 动作上下文转写 | 在详情页直接写 `todos` | 通过 Chat 保留“动作解释层”，也更符合当前产品主交互；后续可补快捷 command，但不该替换 Chat 主链 |

第二轮结论：

1. 当前方式并不是唯一方式，但大部分核心功能已经选择了“聚合读侧 + 对象写侧 + Chat 命令入口”的架构
2. 这套方式适合当前项目，因为它兼顾了产品体验和代码边界
3. 真正需要调整的不是方向，而是把过大的 route/hook 继续拆小

## 6.3 第三轮：如果要用最小复杂度完成目标，哪些实现必须收束，哪些可以延期

| 功能 | 当前应坚持的最小实现 | 可以延期的部分 | 最终建议 |
| --- | --- | --- | --- |
| Today 推荐 | 兴趣命中 + 质量分 + 热度规则排序 | 机器学习推荐、复杂权重学习 | 先稳住规则排序和解释字段 |
| 内容详情 | 统一 `content_ref`、正文兜底、相关推荐最小版 | 统一内容事实表、复杂相关推荐引擎 | 先保证统一打开与最小可读性 |
| Chat | 规则识别 + 候选确认 + 写入结构化对象 + reclassify | LLM 意图识别、复杂多轮计划代理 | 先把命令处理拆模块、提高可维护性 |
| 推送 | 保留 `user_settings` 与设置页 | 正式调度实例表、投递日志、消息中心 | 标注为下一阶段，不应伪装成已完成 |
| 画像 | 规则型 persona/keywords | 实时画像流水线、推荐反馈学习 | 保持展示型画像即可 |
| 机会闭环 | 机会详情 -> Chat -> 待办 | 机会结果回收、申请状态自动同步 | 下一步优先补 `todo` 与 `follow` 的强绑定 |
| 报告 | 周/月/年报按真实行为聚合并落库 | 报告解释树、内容引用级证据链 | 保持 `report_id` 和历史回看稳定最重要 |

第三轮结论：

1. 当前项目最应该做的是“把已成立的主链做稳”，不是再开新抽象层
2. 可以立即指导开发的优先顺序应是：
   1. 拆 `chat.ts` 和 `useChatLogic.ts`
   2. 清理 Today/Article/Actions 的页面级尾差
   3. 补“机会 -> todo -> follow”强绑定
   4. 决定是否正式建设统一 `contents` 表
   5. 最后再补推送调度执行链

## 7. 可直接执行的开发约束

后续写功能需求或开始改代码时，建议直接遵守下面这组约束：

1. 新功能先判断属于哪一类：
   - 页面聚合读侧
   - 对象 CRUD
   - Chat 命令动作
   - 周期聚合结果
2. 新功能先落到正式主线目录：
   - 前端：`prototype/src`
   - 后端：`workers/src`
   - 数据：`cloudflare/d1/migrations`
3. 不再把新增正式功能主要写到 `app/`
4. 每个新接口都要先定义响应 DTO，再写 route
5. 每个新写入动作都要明确：
   - 写哪张表
   - 是否写 `history_entries`
   - 是否影响 Today / Growth / Reports / Actions 中任一读侧
6. 每个新页面动作都要明确最终 deepLink 或回流目标，避免写完后“对象落库了，但用户无感”

## 8. 最终结论

这份表给出的核心判断是：

1. 当前项目已经具备可运行的主闭环，但代码组织上最需要继续收束的是 `Chat`、`Reports`、`Dashboard` 三块聚合逻辑
2. 后续实现必须锚定 `workers + prototype + D1` 这条正式链，不再用历史 Python 架构指导未来扩展
3. 功能设计要继续坚持三件事：
   - 统一引用：`content_ref`
   - 读写分离：页面聚合接口 vs 对象接口 vs 命令接口
   - 结果回流：每次写入都要能反馈到 Today / Actions / Growth / Reports 至少一处

如果按这份表推进，后续的功能需求拆解、接口设计和代码落位都可以直接有据可依，不需要再从“这个功能该写在哪一层、哪张表、哪个模块”重新讨论一轮。
