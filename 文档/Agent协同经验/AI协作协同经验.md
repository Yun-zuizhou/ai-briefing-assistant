# AI协作协同经验

- 文档状态：现行文档
- 适用对象：新对话承接、Agent 协作、提示词治理、层层汇报失稳场景
- 建立时间：2026-04-03

---

## 1. 这次要解决的核心问题

当前协作反复失稳，不是因为规则不够多，而是因为三类信息长期写在同一段提示词里：

1. 稳定不该频繁变化的协作协议
2. 每轮都会变化的项目现状
3. 只属于某一类任务的场景要求

直接后果是：

1. 新对话一开，Agent 很容易漏掉真正该先做的上下文建立动作
2. 层层汇报链经常被跳过，变成“只改过程文档”或“直接改总纲”
3. 每轮推进后都要顺手重写一遍总控提示词，提示词本身开始成为维护负担

这次整理后的核心结论是：

`把 AI 协作入口拆成“通用协作协议 + 当前项目状态注入 + 本轮任务卡”三层。`

---

## 2. 现在采用的三层结构

### 2.1 第一层：通用协作协议

作用：

1. 只定义稳定执行规则
2. 只定义阅读顺序、正式态判断、文档更新链、输出格式
3. 不写死当前日期、当前阶段、当前测试数、当前数据库状态

当前承载文件：

- `文档/Agent协同经验/项目初始化.md`

### 2.2 第二层：当前项目状态注入

作用：

1. 只描述“这个项目此刻到了哪里”
2. 只更新当前阶段、已完成、未完成、正式态/过渡态/虚拟态/回退态
3. 允许高频更新，但不重写通用协作规则

当前承载文件：

- `文档/Agent协同经验/项目承接提示词-当前版.md`

### 2.3 第三层：本轮任务卡

作用：

1. 只描述这轮具体要做什么
2. 可以是“环境初始化”“数据库同步”“页面联调”“答辩排查”等场景化任务
3. 用完即走，不承担项目长期状态维护职责

当前典型参考：

1. `文档/Agent协同经验/环境初始化.md`
2. `文档/Agent协同经验/老师-环境初始化.md`
3. `文档/Agent协同经验/老师-数据库同步提示词.md`

## 3. 新对话的正确使用顺序

后续新对话建议固定为：

1. 先加载 `项目初始化.md` 的通用协作协议
2. 再加载 `项目承接提示词-当前版.md` 的当前项目状态卡
3. 最后再补本轮任务说明

不要再把三层内容重新揉成一段超长提示词。

正确顺序是：

`通用规则 -> 当前状态 -> 本轮任务`

而不是：

`当前状态 + 通用规则 + 本轮任务 + 历史遗留说明` 全部混写。

---

## 4. 层层汇报机制怎么判断

这是最容易失控的地方，必须固定成触发规则。

| 发生了什么 | 至少更新到哪一层 |
| --- | --- |
| 只是新增执行事实、联调记录、验证结果 | `文档/进行中/` 对应主线文档 |
| 当前阶段任务、风险、顺序发生变化 | `文档/进行中/` + `当前阶段总表.md` |
| 形成新的结果级判断、边界口径、阶段性结论 | `文档/进行中/` + `当前阶段总表.md` + `项目核心子纲-阶段演进与归档索引.md` |
| 项目当前阶段、对外口径、核心判断发生变化 | 在前三级基础上，再更新 `项目核心总纲.md` |

记忆口令：

`过程先写进行中，阶段判断进总表，结果级结论进子纲，项目级口径才进总纲。`

---

## 5. 为什么老师的提示词更容易复用

老师提示词的优点不在于“字更多”，而在于结构更清楚：

1. 先给稳定的执行方法论
2. 再给具体场景任务
3. 不把某一天的项目现状写死进长期总控里

这次重构沿用了这个思路，但做了适配：

1. 保留“黑盒执行、少让用户记命令、少讲原理”的风格
2. 加入当前项目强依赖的文档工作流
3. 单独固定“主线进行中文档 -> 当前阶段总表 -> 子纲 -> 总纲”的汇报链

因此，后续更新时：

1. 项目推进了，只更新状态卡
2. 换了任务场景，只换任务卡
3. 只有协作方法真的变了，才改通用协作协议

---

## 6. 每轮结束前必须自检的 6 件事

1. 有没有先按 `文档规范 -> 总纲 -> 子纲 -> 当前阶段总表 -> 当前阶段文档 -> 项目功能文档 -> 代码` 建立上下文
2. 有没有先判断这轮任务是不是当前阶段主线
3. 有没有明确区分正式态、过渡态、虚拟态、回退态
4. 有没有明确说明这轮应更新到哪一层文档
5. 有没有在最终输出里写清“当前阶段记录已写入哪里”
6. 如果没有更新总纲，是否明确说明“为什么这轮还不到总纲级别”

---

## 7. 当前项目的落地方式

这次整理后，当前项目建议这样使用：

1. `项目初始化.md` 负责稳定协作协议
2. `项目承接提示词-当前版.md` 负责当前阶段状态注入
3. `环境初始化.md`、数据库同步提示词等负责场景任务
4. 每轮重要推进完成后，仍按“主线进行中文档 -> 当前阶段总表 -> 子纲 -> 总纲”的链路判断是否继续上报

---

## 8. 本次经验结论

本次整理证明：

1. 提示词失控，本质上常常不是模型不听话，而是输入结构本身混层
2. 想让新对话稳定，重点不是再加更多提醒，而是把“规则、状态、任务”拆开
3. 层层汇报机制必须写成触发条件，不能只靠临场提醒
4. 项目现状应该写进状态卡，而不是反复污染总控提示词
