# 文档整理经验

- 文档状态：现行文档
- 适用对象：Agent 协同推进、阶段切换、文档归档与总纲瘦身场景
- 建立时间：2026-04-03

---

## 1. 本次经验要解决什么问题

当项目从一个阶段切换到下一个阶段时，如果：

1. `进行中/` 堆积了大量上一阶段执行记录
2. `项目核心总纲.md` 同时承担“当前判断”和“历史流水账”两种职责
3. Agent 每次进入上下文都要重新穿过上一阶段细节

那么即使代码没乱，文档也会开始拖慢推进。

本次整理的核心经验是：

`阶段切换时，不要只补新文档，要同时做一次“总纲瘦身 + 子纲承接 + 归档收口”；并且后续每轮推进默认同步“主线进行中文档 + 子纲”，由子纲向总纲汇报。`

---

## 2. 这次采用的整理方法

### 2.1 三层结构

本次采用三层结构：

1. 总纲层
   - 只保留子纲汇报后的当前有效判断、关键结果、下一步主线
2. 子纲层
   - 默认承接主线进行中文档的阶段过程摘要、归档索引、阶段汇报
3. 归档层
   - 承接完整过程文档和历史执行记录

### 2.2 进行中目录的保留原则

`进行中/` 只保留：

1. 当前阶段还会频繁更新的文档
2. 当前阶段仍需反复引用的边界判断文档
3. 当前阶段总表

不再保留：

1. 已完成的联调轮次记录
2. 已完成的“第一步收口”过程文档
3. 仅用于上一阶段的工程修复记录

### 2.3 总纲瘦身原则

`项目核心总纲.md` 不再承担：

1. 大段阶段流水账
2. 每次联调的细节说明
3. 大量归档级过程展开

而改为：

1. 接收子纲汇报后的阶段结论
2. 给出当前状态边界
3. 给出子纲和归档入口

---

## 3. 本次整理的可复用步骤

后续如再做阶段切换，可直接复用：

1. 先确认新阶段主线是否已经和旧阶段不同
2. 列出 `进行中/` 中文档的“保留 / 归档 / 升级为专题”清单
3. 新建“当前阶段总表”
4. 新建“项目核心子纲”承接总纲不宜继续展开的阶段过程
5. 建立阶段归档目录与阶段说明
6. 将上一阶段过程文档整体迁入归档
7. 先更新子纲，再由子纲回写总纲、文档规范和经验文档

---

## 4. 协同执行时的注意事项

1. 不要把“整理文档”做成全仓库命名大修
2. 优先整理高频入口文档，而不是全量历史文档
3. 归档前要先写阶段说明，否则归档目录会变成新垃圾堆
4. 总纲必须保留结果级信息，不能瘦成空壳
5. 子纲与总纲不能并列接收同层级更新，必须保持“主线文档 -> 子纲 -> 总纲”的汇报链
5. 经验文档应沉淀“为什么这样做”和“以后怎么复用”

---

## 5. 本次整理的结论

本次整理证明：

1. 在进入新阶段前做一次轻量归档，是值得的
2. `总纲 + 子纲 + 归档` 比“总纲持续变胖”更适合长期协同
3. Agent 协同不是只依赖代码上下文，也强依赖文档入口是否干净

---

## 6. 后续采纳建议

建议后续每次阶段切换都做一次最小检查：

1. `进行中/` 是否仍然代表当前工作面
2. `项目核心总纲.md` 是否仍然可以 5 分钟内读懂当前状态
3. 是否需要让子纲继续承接阶段过程
4. 是否需要新增一份经验文档沉淀协同方式
