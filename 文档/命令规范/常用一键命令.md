# 常用一键命令

- 更新日期：2026-04-24
- 适用环境：Windows PowerShell
- 默认执行位置：仓库根目录 `E:\python\杂谈-想法记录与实践`

## 1. 先整理：哪些事项适合包装

当前最适合做成“一键命令”的高频事项有 5 组：

1. 环境初始化
2. 本地启动
3. 权限 / 凭据 / secret 收口
4. 检查与验收
5. 数据准备与同步

## 2. 推荐直接复制的命令

| 事项 | 直接运行的命令 | 作用 | 补充说明 |
|---|---|---|---|
| 首次初始化 | `npm.cmd run task:init` | 安装依赖、补 `.env`、补 `.dev.vars`、初始化本地数据 | 新机器第一次进入仓库先跑这一条 |
| 启动完整程序 | `npm.cmd run task:start` | 同时启动前端 + Workers 后端 | 日常联调默认用这条 |
| 启动完整程序（轻量） | `npm.cmd run task:start:fast` | 同时启动前端 + Workers 后端，但跳过本轮本地 D1 准备 | 适合本机已初始化、只想快速进入界面时使用 |
| 只跑前端 | `npm.cmd run task:start:web` | 只启动 `apps/web` | 适合纯 UI 调整 |
| 只跑 Workers 后端 | `npm.cmd run task:start:workers` | 只启动 `apps/edge-worker` | 适合接口联调 |
| 跑 Python 回退链 | `npm.cmd run task:start:python` | 启动 Python 离线/参考后端 | 仅在需要回退验证时使用 |
| 跑 Python 回退链（轻量） | `npm.cmd run task:start:python:lite` | 启动 Python 后端但关闭 RSS/热点后台调度 | 适合接口验证，不跑后台采集 |
| 主门槛检查 | `npm.cmd run task:verify` | 执行统一 `check` | 当前唯一准入门槛 |
| 主门槛检查（带摘要） | `npm.cmd run task:verify:proof` | 执行 `check` 并自动生成门禁摘要 | 摘要落地到 `var/logs/gate-check/latest-summary.md` |
| 主门槛检查（严格双命令证明） | `npm.cmd run check:proof:strict` | 执行 `check + check:workers` 并生成双命令摘要 | 建议用于阶段收口或对外同步前留痕 |
| Workers 检查 | `npm.cmd run task:verify:workers` | 执行 Workers 类型检查 + 测试 | 用于专项排查 |
| 浏览器级 E2E | `npm.cmd run task:verify:e2e` | 执行 Playwright E2E | 适合页面级复验 |
| 写入 D1 凭据 | `npm.cmd run task:access:d1` | 把稳定 D1 配置写入根目录 `.env` | 用于 D1 凭据收口 |
| Workers 权限预检 | `npm.cmd run task:access:workers` | 检查本地 `.dev.vars` 与远端 `INTERNAL_API_TOKEN` secret | 需要已具备 Cloudflare 登录/权限 |
| 推送 Workers 内部 token | `npm.cmd run task:access:workers:push-token` | 把本地 `.dev.vars` 里的 `INTERNAL_API_TOKEN` 推到默认 Worker secret | 有远端副作用，会改线上 secret |
| 同步咨询 provider | `npm.cmd run task:access:consult` | 把根目录 `.env` 的 AI provider 配置同步到 Workers `.dev.vars` | 适合本地 consult 联调前执行 |
| 生成 D1 seeds | `npm.cmd run task:data:prepare` | 生成远端 seed 文件与分片 | 不等于已经 apply 到远端 |
| 同步 demo 数据 | `npm.cmd run task:data:sync-demo` | 执行 demo 数据同步 | 用于演示数据更新 |

## 3. 三套推荐顺序

### 3.1 新机器首次进入

```powershell
npm.cmd run task:init
npm.cmd run task:access:d1
npm.cmd run task:access:workers
npm.cmd run task:start
```

### 3.2 日常开发

```powershell
npm.cmd run task:start
```

本机已经初始化且只需要快速打开完整界面时：

```powershell
npm.cmd run task:start:fast
```

需要只改页面时：

```powershell
npm.cmd run task:start:web
```

### 3.3 发布前或交付前复验

```powershell
npm.cmd run task:verify
npm.cmd run task:verify:e2e
```

更新测试结论文档时，建议先跑：

```powershell
npm.cmd run task:verify:proof
```

阶段收口或对外汇报前，建议跑：

```powershell
npm.cmd run check:proof:strict
```

## 4. 权限类命令的边界

权限 / 凭据类命令当前能标准化，但不能假装“零前提自动完成”：

1. `task:access:d1`
   - 能自动把本机已可发现的 D1 配置写入 `.env`
2. `task:access:workers`
   - 能自动检查本地 `.dev.vars` 与远端 Worker secret
3. `task:access:workers:push-token`
   - 能自动把本地 token 推到远端 secret

但以下前提仍需你本机已经满足：

1. 已有可用的 Cloudflare 登录态或 API 权限
2. `.dev.vars` 中已有正确的 `INTERNAL_API_TOKEN`
3. 远端默认 Worker 就是当前正式对象

## 5. 当前不推荐再直接记底层命令的事项

以下事项后续默认优先记 `task:*`，不再优先记底层脚本名：

1. 环境初始化
2. 启动前端
3. 启动完整程序
4. Workers 权限预检
5. D1 凭据收口
6. AI provider 同步
7. E2E 复验
