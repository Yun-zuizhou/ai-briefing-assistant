# Navigation Hierarchy 标准

## 这轮解决的问题

当前正式 UI 副本里，报头和底部导航虽然都存在，但职责没有完全分开：

- 报头有时承载快捷跳转
- 底部导航又在很多非一级页面持续出现
- 用户会感到“头部和底部都像在导航，但都不像唯一主导航”

这会带来两个问题：

1. 页面上下都在抢导航注意力
2. 二级页和详情页也保留底部导航，层级感不清晰

## 当前判断

这不是单纯组件皮肤问题，而是页面层 + 组件层的共同问题。

涉及组件：

- `PageLayout`
- `Masthead`
- `TabBar`

## 本轮形成的正式标准

### 1. 底部导航只服务一级主页面

底部导航的职责固定为全局一级主路由切换：

- 对话
- 简报
- 待办
- 日志
- 我的

它不应该继续承担二级页和域内子页的存在感表达。

### 2. 报头承担当前页面上下文与域内快捷入口

报头负责：

- 当前页面身份
- 当前页面时间/副标题
- 当前领域内的快捷入口

例如：

- `我的` 页中的 `成长 / 设置`
- `成长` 页中的 `日志 / 画像 / 历史 / 简报`

这类入口不是全局一级导航，而是上下文导航。

### 3. 二级页默认不显示底部导航

从本轮开始：

- `PageLayout` 的 `secondary` 变体不再默认显示底部导航

这意味着：

- 设置页
- 关于页
- 收藏页
- 详情页
- 个人域子页

都不再默认同时带着底部导航。

## 为什么这比“让报头彻底替代底部导航”更稳

因为两者不是互相替代关系，而是层级分工关系：

- 底部导航负责全局主路径
- 报头负责当前页上下文

如果直接让报头取代底部导航，会让一级主路由失去固定承载位置；  
如果两者都做导航，又会继续重叠。

真正稳定的方式，是明确分层，而不是二选一。

## 本轮实际落地

1. `PageLayout` 的 `secondary` 变体默认隐藏底部导航
2. `GrowthPage` 作为“我的域内子页”，改为使用 `secondary` 导航层级

## 仍不进入本轮标准的内容

以下内容本轮不处理：

1. 一级主路由是否要重命名
2. 底部导航图标是否要换
3. 报头 metaLinks 是否要升级成更强的子导航组件

这些适合在后续“导航组件深化迭代”里单独处理。

## 迁回正式前端时的优先顺序

建议按这个顺序迁回 `apps/web/src`：

1. `PageLayout` 中 `secondary` 的底部导航策略
2. `GrowthPage` 的页面层级归类
3. 逐步检查其他二级页是否仍有保留底部导航的必要
