# Action Summary Panel 标准

## 这轮解决的问题

当前待办页里的摘要区如果只摆三组数字，会出现这些问题：

- 用户知道有多少，但不知道现在该先做什么
- 数字会抢走注意力，行动语义反而变弱
- 页面一开始就像仪表盘，不像行动面

所以“行动摘要”更适合先给一句话判断，再把数字降成辅助信息。

## 当前判断

这块不只是页面局部排版问题，而是一个正在成型的业务组件。

当前先在待办页验证，组件名：

- `ActionSummaryPanel`

## 本轮形成的标准

### 1. 行动摘要必须先用文字回答“现在该做什么”

当前正式方向：

- 先给一句总结性的行动判断
- 再补充提醒信息
- 最后再放辅助统计

也就是说，数字不能再做主标题。

### 2. 统计信息降级为辅助芯片

统计仍然可以保留，但只作为摘要下方的辅助信息：

- 已完成
- 收藏待处理
- 今日待办
- 后续跟进

它们的作用是补充判断，不是取代判断。

### 3. 打卡区如果存在，就必须有轨迹展示

当前正式方向：

- 打卡区不能只显示“已打卡 / 今日打卡”
- 至少要补一个最近 7 天或本周的打卡轨迹

注意：

- 当前副本接口还没有真实逐日打卡历史
- 现在的轨迹展示基于 `streakDays + checkedInToday` 推断最近连续打卡区间

所以它目前是“连续打卡轨迹”，不是精确历史明细。

### 4. 页面顺序要改成“摘要 -> 筛选 -> 列表”

待办页前半屏推荐顺序：

1. 行动摘要
2. 当前待办筛选
3. 待办列表
4. 收藏待处理
5. 后续跟进

## 本轮实际落地

- 新增业务组件：`src/components/business/actions.tsx`
- 待办页按新顺序接入：`src/pages/ActionsPage.tsx`
- 视觉实现：`src/index.css`

## 是否进入正式标准

这轮先进入“可迁移的半标准”：

- 结构已经值得复用
- 但目前仍以待办页业务语义为主

如果后续成长页、我的页、报告页也出现类似“先一句话总结，再补少量数字”的结构，再考虑上升成更通用的摘要面板标准。

## 仍然不进入标准的内容

以下仍然属于待办页局部：

1. “收藏待处理 / 今日待办 / 后续跟进”这些指标命名
2. 当前摘要文案生成逻辑
3. 打卡状态文案与表现

这些都带有明显的行动页业务语义。

## 迁回正式前端时的优先顺序

建议按这个顺序：

1. `ActionSummaryPanel` 组件
2. 摘要区和筛选区相关样式
3. 再评估是否抽象成更通用的摘要面板组件
