# Navigation Typography 标准

## 这轮解决的问题

当前正式 UI 副本中，报头与底部导航需要明确分工：

- 报头主标题使用 `var(--font-serif-cn)`
- 底部导航文字使用 sans 字体

这会导致：

1. 如果没有统一规则，导航文字会在不同页面被随意覆写
2. 导航作为高频操作区，需要优先保证扫描效率
3. 后续新增页签容易再次回到组件内硬编码

## 当前判断

这不是页面局部例外，而是正式组件标准缺失。

目标对象：

- `src/components/layout/TabBar.tsx`
- `src/index.css`

## 本轮形成的正式标准

### 1. 导航文字保持 sans 体系（与报头分工）

底部导航文字统一使用 sans 家族，不跟随报头主标题：

- `var(--font-nav-label)`（默认映射到 `var(--font-sans-cn)`）

不允许导航组件继续直接硬编码字体栈。

### 2. 报头与导航维持统一风格，不做同字体强绑定

底部导航和报头属于同一视觉系统，但承担不同层级：

- 报头是页面身份主标题
- 导航是高频、压缩型入口标签

因此底部导航：

- 使用 sans 保证可读
- 通过颜色、间距、装饰线条保持与报头同一气质

### 3. Token 化的导航文字规格

本轮新增：

- `--font-nav-label`
- `--text-nav-label`
- `--tracking-nav-label`

用于约束导航文字的尺寸和字距，避免再次回到组件内硬编码。

## 这项标准为什么值得迁回正式前端

这项标准非常适合优先迁回 `apps/web/src`，因为：

1. 影响范围小
2. 统一性收益高
3. 不改变导航结构和路由逻辑
4. 几乎没有业务风险

## 仍不进入标准的部分

以下仍然不是本轮要上升成标准的内容：

1. 导航图标本身的线宽和形状
2. 激活态下划线长度
3. 导航栏整体高度

这些属于后续导航体系更大一轮的组件优化，不应和这轮字体统一混在一起。
