# 正式 UI 副本工作区

本目录是从正式前端源码 [`apps/web/src`](E:/python/杂谈-想法记录与实践/apps/web/src) 复制出来的 UI 迭代副本。

## 目的

后续 UI 调整不再基于独立的静态效果图区，而是直接基于正式前端结构的副本进行。

这样做的目标是：

1. 避免效果图和正式工程分叉过远
2. 让组件、页面、装饰标准都建立在真实正式 UI 结构上
3. 后续更容易把改动迁回正式前端

## 当前约定

1. 本目录只作为“正式 UI 的可改副本”
2. 后续每轮 UI 迭代，优先改本目录中的 `src`
3. 每次迭代都要同时判断：
   - 这是 token、组件、页面还是局部例外
   - 是否应沉淀成正式标准
4. 当副本中的方案稳定后，再计划迁回真正的 [`apps/web/src`](E:/python/杂谈-想法记录与实践/apps/web/src)

## 当前新增中的正文布局标准

除 `Viewport Shell / App Shell` 外，副本中已开始收口第二层正式标准：

1. `PageBody`
   - 负责正文整体节奏、底部安全区、页面纵向留白
2. `PageSection`
   - 负责区块切分、统一上下间距和分隔线
3. `PageSectionHeader`
   - 负责区块标题、区块副说明、弱操作入口
4. `PagePanel`
   - 负责正文中的基础信息承载盒，替代页面里反复散写的暖纸说明盒

同时，副本中的底部导航 `TabBar` 文字排版也进入统一标准：

5. `TabBar` typography
   - 导航文字必须与报头共用同一衬线字体家族
   - 不允许底部导航单独切回 sans 字体体系
   - 导航可以保留自己的紧凑字号与激活态，但排版语言必须服从正式标题体系

以及，副本中的导航层级也开始收口：

6. Global nav vs contextual nav
   - 底部导航只服务一级主页面
   - 报头承担当前页面身份与域内快捷入口
   - 二级页、详情页、设置页、个人域子页不再默认显示底部导航

7. Card surface hierarchy
   - 卡片统一允许小圆角，但必须小于 8px
   - 卡片层级优先使用轻阴影和纸面偏移感，不走厚重浮层风格
   - 需要强调层级时，可叠加细虚线内框，但不应把所有卡片都做成重装饰

8. Action summary panel
   - 待办页的“行动摘要”应以文字描述为主，而不是裸排统计卡
   - 统计信息只能作为摘要的辅助信息，不应反客为主
   - 页面顺序应先回答“现在该做什么”，再进入筛选和具体待办
   - 只要保留打卡区，就必须同时给出打卡轨迹或日历，不应只给一个孤立按钮状态
   - 打卡节律应弱于摘要正文和当前待办，不应再次变成第二个主角

这套标准目前先在副本中的主链页面验证：

- `TodayPage`
- `GrowthPage`
- `MyPage`
- `PreviewPage`
- `TabBar`
- common card surfaces
- action summary strip

验证稳定后，优先迁回真正的正式前端 `apps/web/src`。

## 壳层基线

当前副本不再沿用“固定 430 x 932 手机样机锁死”策略。

当前统一基线是：

1. 移动端优先
   - 在手机宽度下自然铺满可用视口
2. 桌面端允许居中预览
   - 但只是预览兼容，不是正式桌面布局
3. 根节点和应用壳层不再硬锁高度为 `932px`
4. 外边框只在桌面预览时保留，用于辅助观察，不作为移动端正式边界
5. 一级页面正文统一在 `main` 内滚动，底部导航不应跟随正文滚到视口外
